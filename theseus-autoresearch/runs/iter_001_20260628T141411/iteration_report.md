## Rejected (feedback for next iteration)

- **ZKP-Enforced Agent Key Rotation Without Human Approver**: AXIS 2 FATAL: Standard EdDSA rotation (old_private_key signs new_public_key → on-chain state update) never exposes either private key and prevents double-rotation via on-chain state invalidation. The proposal's core claim — 'ZKP is the only mechanism that proves simultaneous possession of both keys without exposure' — is false. Standard ECDSA/EdDSA key delegation achieves identical guarantees. The ZKP adds proof generation latency (3–10s), circuit complexity, and a trusted setup ceremony for zero marginal security or privacy benefit.; AXIS 1 CONCERN: The simpler path for Theseus is trivial — a Solidity (or custom VM) registry function that takes (new_pubkey, sig_by_old_key) and atomically invalidates the old key. A senior EVM dev completes this in 2–3 days. The ZKP circuit path (AgentRotation.circom + double-EdDSA witness + PLONK setup) is 2–4 weeks minimum, and the marginal security gain over the standard path is undemonstrated.; AXIS 3 CONCERN: The proposal conflates two distinct failure modes. For COMPROMISED keys (the headline use case), ZKP rotation is self-defeating: generating the rotation proof requires the old private key — an agent whose key is stolen cannot safely invoke this path without the attacker being able to generate a competing rotation proof first. The proposal solves scheduled/epoch-based rotation only, which standard crypto handles trivially.
- **Sybil-resistant agent reputation via ZKP nullifiers for agent marketplaces**: AXIS 2 FAIL: The ZKP unlinkability property directly conflicts with the reputation use case. Reputation systems require persistent, linkable identity within a marketplace context. The card's own design (same nullifier per agent per marketplace) means the agent IS identifiable within any given marketplace — ZKP only hides cross-marketplace correlation. But Theseus agent genesis transactions are already public on-chain, so 'which agent you are' is not hidden data. A standard on-chain nullifier — keccak256(genesisTxHash, marketplaceId, salt) stored in a mapping — provides identical Sybil resistance with zero ZKP overhead, no trusted setup, and trivial verification. The kill signal fires: the ZKP privacy property conceals data that has no economic or security value because it is already public.; AXIS 1 CONCERN: The proposal describes the required circuit modification as 'minor' — replacing the human secret with a genesis tx hash. This self-assessment undermines the moat argument. Semaphore v4 is open source, PSE provides direct developer support, and a single ZKP-curious engineer can adapt the protocol in 4-6 weeks. Theseus is Orange DAO-backed ($100K), typical of YC-adjacent founding teams with high technical depth. The kill signal for 2-week replication in pure Solidity does not fire, but the Semaphore fork path makes 6-week replication realistic — well within the 3-month threshold.; AXIS 3 CONCERN: Theseus is pre-mainnet, pre-marketplace, and funded at $100K. Agent population at launch will be in the tens to low hundreds, where reputation is handled manually or by operator whitelist. Sybil resistance at cryptographic level only becomes necessary when anonymous participation is possible and reputation can be gamed at scale — typically >1K agents with open admission. The named use case (agent marketplace reputation) is valid but premature; Theseus needs a functioning L1 and marketplace first, making this a 12-18 month need, not 3-month.
- **ZKP Agent Birth Certificate via PLONK (No Per-Agent Trusted Setup)**: AXIS 1 (Theseus Builds It) — CONCERN: The core 'proof of legitimate spawn' can be replicated without ZKP expertise in ~2 weeks. Theseus L1 can deterministically derive an agent keypair from the genesis transaction, register the public key in a Solidity registry, and use the spawning transaction hash as the birth certificate. That's standard OpenZeppelin + Hardhat work. The proposal conflates 'no human CA' with 'needs ZKP' — these are not the same property. Bolyra's actual moat is selective disclosure of the 8-bit permission bitmask, not the birth certificate itself.; AXIS 2 (Standard Crypto Suffices) — FAIL: This is the kill signal. The specific ZKP value-add named in the proposal is (a) no per-agent trusted setup and (b) portable credential commitment. But the 'no ceremony' argument only matters if you're using ZKP at all — it's circular. Standard crypto provides proof of legitimate spawn via the L1's own signature on the genesis transaction. For cross-chain portability, Theseus can publish a Merkle root of its agent registry, and any chain can verify inclusion proofs without ZKP. The only genuine ZKP advantage is hiding the full permission bitmask, but at birth-certificate time there is no adversary who benefits from not knowing an agent's permission scope — permissions are typically set by the operator and are non-sensitive at spawn.; AXIS 3 (No Agent Actually Needs This) — CONCERN: No concrete Theseus agent use case is named that fails without ZKP birth certs. The proposal explicitly admits 'evidence rests on structural inference.' The need for permission-private birth certificates only becomes real when (a) an agent has sensitive permission scope it wants to selectively disclose to counterparties, and (b) those counterparties exist and are querying the agent. At Theseus's current scale ($100K funding, pre-mainnet), neither condition holds. The earliest plausible concrete need is agent-to-agent delegation with permission narrowing — but that's Axis 4 work, not birth certificate work.
- **Nullifier-Bound Session Nonces to Block Agent Transaction Replay Attacks**: AXIS 1 FAIL — Replay prevention via nonce registries is trivially buildable without ZKP expertise. A usedNonces[bytes32] => bool mapping is 10 lines of Solidity and a well-understood OpenZeppelin pattern. EIP-4337 account abstraction handles sequential nonces natively. Theseus's 1-2 engineers can ship this in days, not months. ZKP expertise is not required for replay prevention — it is only required if the goal is *unlinkable* nullifiers, which is a privacy claim, not a security claim.; AXIS 2 FAIL — The card's own rationale self-destructs this proposal: 'nonce counters at the wallet layer or simple on-chain nonce mappings achieve replay prevention without ZKP — the ZKP advantage here is unlinkable nullifiers for privacy, not replay prevention per se.' That is an admission that standard crypto suffices for the stated goal. The ZKP overhead (Groth16 proof generation latency per session, ~200K-300K gas for on-chain verification vs ~20K gas for a boolean write) is not justified when the marginal benefit is privacy for session nonces — a property Theseus agents may not need on a transparent L1 where all transactions are visible anyway.; AXIS 3 CONCERN — The replay surface is real in high-frequency agent systems, but the ZKP-specific solution is not distinguishable from simpler alternatives at Theseus's current scale ($100K funded, pre-mainnet, likely single-digit active agents). The need for unlinkable nullifiers specifically requires an agent that (a) executes high-frequency transactions AND (b) needs to hide which sessions it participated in. No named Theseus agent use case satisfies both conditions. At launch scale, a simple nonce counter eliminates the replay surface entirely.
- **ZKP delegation chain proof appended to x402 payment receipts for audit and dispute**: AXIS 2 FAIL — Theseus's 'no human key path' architecture is a structural kill shot for the primary ZKP value proposition. The entire privacy argument rests on 'hiding root identity X and intermediate delegators from the payment receipt.' But in an agent-native chain where there IS no human root and agents ARE the first-class principals, there is nothing sensitive to hide. An EdDSA delegation chain posted on-chain (agent A signs over to agent B signs over to agent C, all public) provides the same audit trail with zero ZKP overhead. The Bolyra rationale assumes human-rooted privacy is the motivator — Theseus's architecture explicitly eliminates that assumption.; AXIS 4 FAIL — Integration complexity is disqualifying at three independent layers: (1) Theseus is an L1 with unknown VM compatibility — if non-EVM, Bolyra's Solidity verifier is useless and requires a custom Groth16 verifier implementation from scratch in Theseus's native runtime, a multi-month effort for Theseus alone; (2) the Delegation circuit requires 'major' modification to add payment-context binding inputs (amount, recipient, receipt hash), triggering a new trusted setup ceremony and regenerated Solidity verifier — Bolyra estimates 10+ days just for the circuit work; (3) there is no finalized x402 receipt format for ZKP attachment, making this a standards problem in addition to a build problem. Three compounding blockers against a $100K-funded pre-mainnet partner is an unacceptable risk ratio.; AXIS 1 CONCERN — A Theseus engineer with basic crypto literacy can build an on-chain delegation registry (agent → agent signed scoped grants, stored as Merkle leaves, referenced in payment receipts) using standard Solidity + OpenZeppelin in 4-6 weeks. This satisfies 80% of the audit use case without ZKP. The 20% gap is the privacy property — but as noted in Axis 2, that property may have no value in Theseus's architecture. The 'ZKP expertise barrier' argument is only valid if privacy is the requirement, which is unproven.
- **Agent-rooted Delegation circuit: L1 genesis as root of trust**: AXIS 2 FAIL — Standard crypto suffices for agent delegation: a Solidity ACL contract enforcing delegateePermissions & delegatorPermissions == delegateePermissions (bitwise AND check) enforces the same cumulative-bit scope narrowing the Delegation circuit provides, with nullifier replay protection via a simple spent-nullifier mapping. The privacy benefit of hiding delegation chains is unclear for an agent-native L1 where chain transparency is a feature, not a bug. The proposal never addresses what concrete harm occurs if delegation chains are visible on-chain, which is the only question that would justify ZKP overhead.; AXIS 4 FAIL — Integration is structurally blocked on unknowns that Bolyra cannot resolve before the Sunday demo: Theseus's VM architecture is undocumented ($100K funded, likely pre-mainnet), the genesis block commitment format is unknown, and the EVM-compatibility status is unconfirmed. The proposal itself admits the entire 2-week sprint is gated on Theseus sharing their genesis commitment format first. If Theseus runs a custom non-EVM VM, Solidity verifiers cannot be deployed and Bolyra would need to implement a custom Groth16 verifier in Theseus's native language — work that dwarfs the circuit modification itself. The claimed timeline is fictional until these prerequisites are satisfied.; AXIS 1 CONCERN — Theseus can defer this: for their current scale ($100K, pre-launch), a hardcoded permission set per agent enforced in a Solidity registry is a 1-week implementation. ZKP circuit design requires specialized Circom expertise, but Theseus's identity problem at launch is not complex enough to demand it. The 'they can't build this themselves' claim is unsubstantiated — the proposal acknowledges it doesn't know Theseus's crypto team composition.
- **EIP-1186 storage proof relay to sync Bolyra IdentityRegistry root onto Theseus L1**: AXIS 1 — CONCERN: The storage proof relay requires zero ZKP expertise. EIP-1186 is a mature standard with production Rust implementations (Helios, Nimbus). The Bolyra-specific work is identifying the IdentityRegistry storage slot — that's one day of effort. Theseus's engineers can build a functional MPT-based root relay in 6–8 weeks using open-source tooling. The proposal itself concedes 'the storage proof relay is EIP-1186 mechanics rather than ZKP itself,' which means Bolyra's moat is absent from the primary deliverable.; AXIS 2 — CONCERN: The trustless property of EIP-1186 is Merkle Patricia Trie verification, not ZKP. Standard crypto (MPT proofs + EdDSA oracle signatures) fully serves the root-pinning function. ZKP only enters at the credential presentation layer, which is a separate concern. At Theseus's current validator count ($100K funding, pre-mainnet), a 2-of-3 multisig oracle updating the pinned root is operationally equivalent and carries near-zero integration overhead. The ZKP overhead is not justified by the marginal trustlessness benefit at this stage.; AXIS 3 — FAIL: No concrete agent use case requiring cross-chain root sync exists at Theseus's current scale. Theseus has essentially zero live agents today. At this stage, agents can directly query Base Sepolia RPC for credential verification with sub-second latency — cross-chain root pinning is a throughput optimization relevant at thousands of simultaneous agent verifications per second. This proposal solves a scaling problem Theseus will not face for at minimum 18–24 months. Kill signal triggered: the need only arises at scale that far exceeds any plausible Theseus 12-month roadmap.
- **Nullifier-Based Cross-Chain Agent Identity for x402 Micropayments**: AXIS 1 (CONCERN): ZKP circuit expertise is a real barrier — AgentNullifier.circom cannot be built in 2 weeks — but BIP32/BIP44-style HD wallet key derivation lets Theseus agents derive chain-specific keypairs from a master key in a single afternoon. This is already standard practice in multi-chain wallets (MetaMask, Coinbase Wallet) and achieves the pseudonymity goal with zero ZKP expertise required. The proposal conflates 'cross-chain pseudonymity' with 'ZKP nullifiers' without ruling out the obvious alternative.; AXIS 2 (FAIL): The stated use case is x402 micropayments on Base. For a payment to succeed, the agent needs funds and a valid signature — not a cross-chain identity proof. HD key derivation already gives each agent an unlinkable Base key. The nullifier only adds value if a Base contract needs to verify Theseus-level permissions before accepting payment, which is not described anywhere in the integration card. The proposal claims 'EdDSA+Merkle leaks linkage' but Tier 2 review correctly flagged this: Merkle proofs can also hide leaf identity, and separate derived keys have no linkage at all. The core ZKP value proposition for this specific use case evaporates under scrutiny.; AXIS 3 (FAIL): No specific Theseus agent use case is named where the agent operationally FAILS without ZKP nullifiers. The deanonymization risk is theoretical — agents can simply use chain-specific keys. For the first 12 months, Theseus at $100K funding will have dozens of agents at most, all likely controlled by the same small team, where cross-chain behavioral fingerprinting is not a meaningful attack surface. The x402 micropayment use case is purely about payment authorization, not identity continuity. The need for ZKP-based cross-chain pseudonymity only materializes at adversarial scale (thousands of agents with sophisticated chain-analysis opponents) that Theseus cannot reach in 12 months.
