# Tier 1 — Discovery Rubric

Scoring guide for raw proposals generated by personas.

## Dimension Scoring Guide

### Agent Need (0-25)

| Range | Description | Example |
|---|---|---|
| 20-25 | Day-1 blocker — agents cannot operate on Theseus without this | Agent identity bootstrapping (how does a new agent prove it exists?) |
| 15-19 | Critical for first 100 agents — operational necessity | Agent-to-agent delegation (agent A authorizes agent B to act on its behalf) |
| 10-14 | Important for ecosystem growth — needed at scale | Reputation scoring for agent marketplace |
| 5-9 | Nice-to-have — some agents need it, not all | Cross-chain credential portability |
| 0-4 | Theoretical — no agent needs this in the first 12 months | Formal verification of agent behavior proofs |

### ZKP Edge (0-25)

| Range | Description | Example |
|---|---|---|
| 20-25 | ZKP is the only viable approach — no alternative achieves the same properties | Proving authorization without revealing the full delegation chain |
| 15-19 | ZKP provides significant privacy/efficiency advantage over alternatives | Selective disclosure of agent capabilities to counterparties |
| 10-14 | ZKP is better but standard crypto is workable | Permission verification (ACLs work, ZKP adds privacy) |
| 5-9 | Marginal advantage — complexity may not be justified | Simple agent authentication (EdDSA signature suffices) |
| 0-4 | No advantage — standard crypto is strictly better | Public key registration (just put it on-chain) |

### Primitive Readiness (0-25)

| Range | Description | Example |
|---|---|---|
| 20-25 | Existing circuit/contract/SDK handles this — ship in < 3 days | AgentPolicy circuit for permission verification |
| 15-19 | Minor extension to existing primitive — ship in < 1 week | Add agent-root variant to Delegation circuit |
| 10-14 | New circuit or contract needed, but pattern is established — 1-2 weeks | Agent-to-agent delegation circuit (similar to existing Delegation) |
| 5-9 | Significant new R&D — new cryptographic patterns — 3-4 weeks | Cross-chain proof relay with chain-specific verification |
| 0-4 | Research project — unknown feasibility — 1+ months | Novel consensus-integrated identity scheme |

### Partnership Leverage (0-25)

| Range | Description | Example |
|---|---|---|
| 20-25 | Neither can build this alone; creates deep mutual dependency | Identity layer that's both Theseus-native and cross-chain portable |
| 15-19 | Theseus would need 3+ months and ZKP expertise to replicate | ZKP delegation circuits with on-chain enforcement |
| 10-14 | Theseus could build a simpler version, but Bolyra's is better | Permission scoping (ACLs vs ZKP-enforced cumulative bits) |
| 5-9 | Bolyra accelerates but isn't essential | SDK integration helpers |
| 0-4 | Theseus can trivially build this themselves | Basic key registration |

## What Makes a Good Proposal

- **Specific:** names the exact capability, not a vague category
- **Grounded:** references a real problem agents face, not a theoretical concern
- **Actionable:** describes what Bolyra would build or integrate, not just "identity is important"
- **Differentiated:** explains why ZKP/Bolyra specifically, not just any identity solution

## What Makes a Bad Proposal

- "Bolyra provides identity for Theseus agents" — too vague, no specific capability
- "ZKP everything" — not every problem needs ZKP; be honest about where standard crypto suffices
- "Build a cross-chain bridge" — scope too large for a partnership integration
- Proposals that score high on Agent Need but 0 on ZKP Edge — Theseus can build these without Bolyra
