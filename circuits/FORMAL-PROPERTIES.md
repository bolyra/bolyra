# Formal Verification Properties for Bolyra Circuits

**Protocol**: Bolyra (formerly IdentityOS)
**Version**: Phase 1 (Proof of Enrollment)
**Date**: 2026-04-20
**Status**: Research artifact — academic analysis of circuit security properties

---

## Notation

| Symbol | Meaning |
|--------|---------|
| `P(x1, ..., xn)` | Poseidon hash with n inputs |
| `BabyPbk(s)` | Baby Jubjub scalar multiplication: s * G → (Ax, Ay) |
| `EdDSA.Verify(pk, m, sig)` | EdDSA signature verification over Poseidon |
| `MerkleRoot(leaf, proof)` | Binary Merkle root computation |
| `bits(x)` | Bit decomposition of field element x |
| `x[i]` | The i-th bit of x |
| `F_p` | BN254 scalar field (≈254 bits) |
| `l` | Baby Jubjub subgroup order (≈251 bits) |

## 1. HumanUniqueness(MAX_DEPTH)

**Proving system**: Groth16 (reuses Semaphore v4 ceremony for depth 20)
**Public inputs**: `scope`, `sessionNonce`
**Public outputs**: `humanMerkleRoot`, `nullifierHash`, `nonceBinding`

### 1.1 Soundness Properties

**S1.1** (Membership): For any valid proof π with outputs (root, nH, nB):
```
∀π valid: ∃ secret ∈ [0, 2^251) such that
  let (Ax, Ay) = BabyPbk(secret)
  let leaf = P(Ax, Ay)
  MerkleRoot(leaf, proof) = humanMerkleRoot
```

**S1.2** (Nullifier determinism): The nullifier is uniquely determined by (secret, scope):
```
∀ secret, scope:
  nullifierHash = P(scope, secret)
```
Corollary: Two proofs with the same (secret, scope) produce identical nullifiers regardless of sessionNonce or Merkle proof path.

**S1.3** (Nonce binding integrity): The nonceBinding output commits to both the nullifier and the session:
```
∀ valid proofs:
  nonceBinding = P(nullifierHash, sessionNonce)
```

**S1.4** (Key-commitment binding): The identity commitment is deterministically derived from the secret:
```
∀ secret: identityCommitment = P(BabyPbk(secret).Ax, BabyPbk(secret).Ay)
```
No two distinct secrets in [0, l) produce the same (Ax, Ay) pair (discrete log hardness on Baby Jubjub).

### 1.2 Completeness Properties

**C1.1**: For any enrolled human with secret s where:
- `s ∈ [0, 2^251)`
- `P(BabyPbk(s).Ax, BabyPbk(s).Ay)` is a leaf in the Merkle tree of depth ≤ MAX_DEPTH

a valid proof exists for any (scope, sessionNonce) ∈ F_p.

**C1.2**: All secrets generated by `@semaphore-protocol/identity` satisfy the range check (since Semaphore uses 32 random bytes → value < 2^256, but reduces mod l, yielding < l < 2^251).

### 1.3 Privacy Properties

**P1.1** (Identity hiding): The proof reveals NOTHING about:
- The secret scalar
- The public key (Ax, Ay)
- The identity commitment (leaf value)
- The Merkle proof path (index, siblings, depth)

**P1.2** (Anonymity set): The prover is anonymous within the set of all leaves in the tree whose root equals `humanMerkleRoot`. The anonymity set size = number of enrolled identities.

**P1.3** (Unlinkability across scopes): Given proofs π₁ with scope₁ and π₂ with scope₂ from the same identity:
```
nullifierHash₁ = P(scope₁, secret) ≠ P(scope₂, secret) = nullifierHash₂
```
An observer cannot link π₁ and π₂ to the same identity (under Poseidon PRF assumption).

**P1.4** (Session unlinkability): Two proofs with the same (secret, scope) but different sessionNonce values produce the SAME nullifierHash but different nonceBinding values. The nullifier enables sybil detection; the nonce binding enables replay detection.

### 1.4 Security Invariants

| Attack | Prevention mechanism | Constraint reference |
|--------|---------------------|---------------------|
| Sybil (multiple identities per human) | Nullifier deduplication per scope | Step 4: `nullifierHash = P(scope, secret)` |
| Replay (reuse proof in different session) | Nonce binding checked by verifier | Step 5: `nonceBinding = P(nullifierHash, sessionNonce)` |
| Non-member proof | Merkle inclusion proof against on-chain root | Step 3: `BinaryMerkleRoot` |
| Secret outside valid range | Num2Bits(251) range check | Step 6: `secretRange` |
| Identity forgery | BabyPbk one-wayness (DLP hardness) | Step 1: `BabyPbk()` |

### 1.5 Range Check Coverage

| Signal | Component | Bit width | Purpose |
|--------|-----------|-----------|---------|
| `secret` | `secretRange = Num2Bits(251)` | 251 | Ensures secret < 2^251 < l (subgroup order) |

Note: `scope`, `sessionNonce` have NO range checks — they are public inputs and can be any field element. This is by design: the verifier controls them.

### 1.6 Underconstraint Analysis

**UC1.1** (merkleProofSiblings beyond depth): For indices `i ≥ merkleProofLength`, sibling values are unused by `BinaryMerkleRoot` (it multiplexes them out). These signals are technically unconstrained in their effect on the output. However, they are private inputs that do not influence any public output when `i ≥ depth`, so this is NOT a vulnerability — it is a standard pattern in variable-depth Merkle proofs.

**UC1.2** (No explicit constraint that secret ≠ 0): If `secret = 0`, then `BabyPbk(0)` produces the identity point. The circuit does not reject this. Whether the identity point is enrolled in the tree is an application-layer concern. This is a minor observation, not a vulnerability.

### 1.7 Known Limitations

1. **No revocation**: Once enrolled, an identity cannot be removed from the tree without a root update. The circuit has no revocation check.
2. **Phase 1 stub**: Sybil resistance relies entirely on the enrollment mechanism (off-chain). The circuit proves enrollment, not humanness.
3. **Approximate range check**: The secret range check uses 2^251 rather than the exact subgroup order l. The gap [l, 2^251) is empty (l > 2^251 is false; actually l < 2^251), so no valid secret is rejected. Values in [2^251, p) are correctly rejected by Num2Bits(251).
4. **Scope is verifier-controlled**: If the verifier allows arbitrary scope values, the same identity can generate unlimited distinct nullifiers. Sybil resistance requires scope to be fixed per context.
5. **No timestamp/expiry**: Human identity proofs do not expire within the circuit. Expiry must be enforced by root rotation at the contract layer.

## 2. AgentPolicy(MAX_DEPTH)

**Proving system**: PLONK (universal setup)
**Public inputs**: `requiredScopeMask`, `currentTimestamp`, `sessionNonce`
**Public outputs**: `agentMerkleRoot`, `nullifierHash`, `scopeCommitment`

### 2.1 Soundness Properties

**S2.1** (Credential authenticity): For any valid proof:
```
∀π valid: ∃ (modelHash, opAx, opAy, permBitmask, expiry, sig) such that
  let cred = P(modelHash, opAx, opAy, permBitmask, expiry)
  EdDSA.Verify((opAx, opAy), cred, sig) = true
```
The credential is signed by the operator whose public key is embedded in it.

**S2.2** (Tree membership):
```
∀π valid: MerkleRoot(credentialCommitment, proof) = agentMerkleRoot
```

**S2.3** (Permission satisfaction):
```
∀π valid, ∀i ∈ [0,64):
  requiredScopeMask[i] = 1 → permissionBitmask[i] = 1
```
Equivalently: `requiredScopeMask & ~permissionBitmask = 0`

**S2.4** (Temporal validity):
```
∀π valid: currentTimestamp < expiryTimestamp
```

**S2.5** (Nullifier determinism):
```
∀π valid: nullifierHash = P(credentialCommitment, sessionNonce)
```

**S2.6** (Scope commitment binding):
```
∀π valid: scopeCommitment = P(permissionBitmask, credentialCommitment)
```
This binds scope to identity — two agents with the same permissions but different credentials produce different scopeCommitments.

**S2.7** (Cumulative bit encoding):
```
∀π valid:
  permissionBitmask[4] = 1 → permissionBitmask[3] = 1 ∧ permissionBitmask[2] = 1
  permissionBitmask[3] = 1 → permissionBitmask[2] = 1
```

### 2.2 Completeness Properties

**C2.1**: For any agent with:
- Valid EdDSA signature over credential commitment
- Credential commitment enrolled in tree of depth ≤ MAX_DEPTH
- `permissionBitmask ∈ [0, 2^64)` satisfying cumulative encoding
- `expiryTimestamp ∈ [0, 2^64)` with `currentTimestamp < expiryTimestamp`
- `(requiredScopeMask & ~permissionBitmask) = 0`

a valid proof exists for any `sessionNonce ∈ F_p`.

### 2.3 Privacy Properties

**P2.1** (Model hiding): The proof reveals NOTHING about `modelHash`. An observer cannot determine which AI model backs the agent.

**P2.2** (Operator hiding): The operator's public key `(opAx, opAy)` is not revealed. The observer cannot identify which operator issued the credential.

**P2.3** (Exact permissions hiding): Only the satisfaction of `requiredScopeMask` is proven. The agent's actual `permissionBitmask` may have MORE bits set than required, and the observer cannot determine which additional permissions exist.

**P2.4** (Expiry hiding): The exact expiry timestamp is hidden. The observer knows only that the credential is not yet expired relative to `currentTimestamp`.

**P2.5** (Signature hiding): The EdDSA signature `(R8x, R8y, S)` is not revealed.

**P2.6** (Credential identity hiding): The `credentialCommitment` (leaf) is hidden. The agent is anonymous within the tree.

**P2.7** (Scope commitment partial linkability): `scopeCommitment` IS public. Two proofs from the same agent (same credential) with the same permission bitmask produce the same `scopeCommitment`. This is intentional — it enables delegation chain linking.

### 2.4 Security Invariants

| Attack | Prevention mechanism | Constraint reference |
|--------|---------------------|---------------------|
| Expired credential usage | `LessThan(64)` comparing timestamps | Step 6: `notExpired` |
| Privilege escalation | Bitwise subset check | Step 5: `scopeViolation[i] === 0` |
| Forged credential | EdDSA signature verification | Step 3: `sigVerify` |
| Non-enrolled agent | Merkle inclusion against on-chain root | Step 4: `BinaryMerkleRoot` |
| Replay attack | Nullifier = P(cred, nonce), checked on-chain | Step 7: `nullifier` |
| Field overflow (uint64 wrap) | Num2Bits(64) on bitmask, expiry, timestamp | Step 1: range checks |
| Impersonation in delegation chain | scopeCommitment binds identity to scope | Step 8: `scopeHash` |
| Inconsistent permission hierarchy | Cumulative bit encoding enforcement | Step 5b: `agent_bit4_requires_bit3`, etc. |

### 2.5 Range Check Coverage

| Signal | Component | Bit width | Purpose |
|--------|-----------|-----------|---------|
| `permissionBitmask` | `bitmaskRange = Num2Bits(64)` | 64 | Prevent field-overflow on Solidity side |
| `expiryTimestamp` | `expiryRange = Num2Bits(64)` | 64 | Ensure valid uint64 for comparison |
| `currentTimestamp` | `timestampRange = Num2Bits(64)` | 64 | Ensure valid uint64 for comparison |
| `requiredScopeMask` | `requiredBits = Num2Bits(64)` | 64 | Decompose for bitwise check |

Note: `modelHash`, `operatorPubkeyAx`, `operatorPubkeyAy`, `sigR8x`, `sigR8y`, `sigS` have NO explicit range checks. They are constrained indirectly through EdDSA verification (which internally constrains curve points) and Poseidon hashing (which operates in F_p natively).

### 2.6 Underconstraint Analysis

**UC2.1** (modelHash independence from scope): `modelHash` contributes to `credentialCommitment` (and thus `agentMerkleRoot` and `nullifierHash`), but does NOT independently appear in any public output. It is fully constrained through the credential commitment hash and Merkle inclusion. NOT a vulnerability.

**UC2.2** (Operator key not independently constrained to a curve point): The operator public key `(opAx, opAy)` is not explicitly checked to lie on Baby Jubjub. However, `EdDSAPoseidonVerifier` internally performs this check (it verifies the public key is a valid curve point). NOT a vulnerability.

**UC2.3** (sessionNonce is public and unconstrained in range): `sessionNonce` is a public input with no range check. It can be any field element. This is by design — the verifier controls it. However, if `sessionNonce` is reused across sessions, the same nullifier would be produced for the same agent, which is the expected replay-detection mechanism.

### 2.7 Known Limitations

1. **No credential revocation in-circuit**: A credential can only be invalidated by removing it from the Merkle tree (root rotation). There is no in-circuit revocation list.
2. **Timestamp oracle trust**: `currentTimestamp` is a public input provided by the verifier/relayer. The circuit trusts this value. A malicious verifier could provide a timestamp far in the past to accept expired credentials.
3. **No binding between operator key and tree membership**: The circuit proves the operator signed the credential AND the credential is in the tree, but does not prove the operator is authorized to enroll credentials. This is an application-layer concern.
4. **Nullifier is per-session, not per-scope**: Unlike HumanUniqueness, the agent nullifier uses `sessionNonce` (not a fixed scope). The same agent can produce unlimited proofs across sessions. Rate limiting must be enforced externally.
5. **requiredScopeMask = 0 is valid**: If the verifier sets `requiredScopeMask = 0`, no permissions are checked. The proof only demonstrates enrollment and non-expiry.
6. **Cumulative encoding is partial**: Only bits 2, 3, 4 have hierarchy enforcement. Bits 0-1 and 5-63 have no structural constraints.

## 3. Delegation(MAX_DEPTH)

**Proving system**: PLONK (universal setup)
**Public inputs**: `previousScopeCommitment`, `sessionNonce`
**Public outputs**: `newScopeCommitment`, `delegationNullifier`, `delegateeMerkleRoot`

### 3.1 Soundness Properties

**S3.1** (Chain integrity): The delegation is linked to a previous hop's output:
```
∀π valid:
  P(delegatorScope, delegatorCredCommitment) = previousScopeCommitment
```
This is the critical chain-linking constraint. It proves the delegator possesses the private preimage of the previous hop's public commitment.

**S3.2** (Scope narrowing / monotone attenuation):
```
∀π valid, ∀i ∈ [0,64):
  delegateeScope[i] = 1 → delegatorScope[i] = 1
```
Equivalently: `delegateeScope ⊆ delegatorScope` (bitwise subset).

**S3.3** (Expiry narrowing):
```
∀π valid: delegateeExpiry ≤ delegatorExpiry
```
Delegation cannot extend lifetime beyond what the delegator holds.

**S3.4** (Delegation authorization): The delegator signed the delegation token:
```
∀π valid:
  let token = P(previousScopeCommitment, delegateeCredCommitment, delegateeScope, delegateeExpiry)
  EdDSA.Verify((delegatorPubkeyAx, delegatorPubkeyAy), token, (sigR8x, sigR8y, sigS)) = true
```

**S3.5** (Delegatee enrollment — CIP-1):
```
∀π valid: MerkleRoot(delegateeCredCommitment, proof) = delegateeMerkleRoot
```
The delegatee must be an enrolled agent. Prevents phantom delegatee attacks.

**S3.6** (Output commitment binding):
```
∀π valid: newScopeCommitment = P(delegateeScope, delegateeCredCommitment)
```

**S3.7** (Nullifier determinism):
```
∀π valid:
  let token = P(previousScopeCommitment, delegateeCredCommitment, delegateeScope, delegateeExpiry)
  delegationNullifier = P(token, sessionNonce)
```

**S3.8** (Cumulative bit encoding on delegatee):
```
∀π valid:
  delegateeScope[4] = 1 → delegateeScope[3] = 1 ∧ delegateeScope[2] = 1
  delegateeScope[3] = 1 → delegateeScope[2] = 1
```

### 3.2 Completeness Properties

**C3.1**: For any delegation where:
- `delegatorScope, delegateeScope ∈ [0, 2^64)`
- `delegateeScope ⊆ delegatorScope` (bitwise)
- `delegateeExpiry ≤ delegatorExpiry`, both in `[0, 2^64)`
- Delegatee's cumulative bit encoding is valid
- `P(delegatorScope, delegatorCredCommitment) = previousScopeCommitment`
- Delegator signed `P(previousScopeCommitment, delegateeCredCommitment, delegateeScope, delegateeExpiry)`
- `delegateeCredCommitment` is in the agent tree of depth ≤ MAX_DEPTH

a valid proof exists for any `sessionNonce ∈ F_p`.

### 3.3 Privacy Properties

**P3.1** (Delegator scope hiding): The delegator's actual `delegatorScope` bits are NOT revealed. Only `previousScopeCommitment = P(delegatorScope, delegatorCredCommitment)` is public (from the previous hop).

**P3.2** (Delegatee scope hiding): The delegatee's actual `delegateeScope` bits are NOT revealed. Only `newScopeCommitment = P(delegateeScope, delegateeCredCommitment)` is output.

**P3.3** (Credential hiding): Both `delegatorCredCommitment` and `delegateeCredCommitment` are hidden. The delegation graph topology is not observable.

**P3.4** (Delegator identity hiding): The delegator's public key and signature are not revealed. An observer cannot determine WHO delegated.

**P3.5** (Expiry hiding): Neither `delegatorExpiry` nor `delegateeExpiry` is revealed. Only the validity of the narrowing relationship is proven.

**P3.6** (Subset relationship hiding): The observer learns that delegation occurred (a new scopeCommitment is produced) but cannot determine HOW MUCH narrowing occurred — whether 1 bit was removed or 60.

**P3.7** (Chain position hiding): The circuit itself does not reveal whether this is hop 1, 2, or 3. All hops produce identical proof structures. Chain depth is enforced by the on-chain contract.

### 3.4 Security Invariants

| Attack | Prevention mechanism | Constraint reference |
|--------|---------------------|---------------------|
| Privilege escalation | Bitwise subset check on scope | Step 3: `scopeViolation[i] === 0` |
| Lifetime extension | `LessEqThan(64)` on expiry | Step 5: `expiryCheck` |
| Unauthorized delegation | EdDSA signature over token | Step 6: `sigVerify` |
| Chain impersonation | Identity-bound scope commitment | Step 2: `delegatorScopeHash.out === previousScopeCommitment` |
| Phantom delegatee (CIP-1) | Delegatee Merkle inclusion proof | Step 6b: `delegateeMerkle` |
| Replay attack | Nullifier = P(token, nonce) | Step 7: `nullifier` |
| Field overflow | Num2Bits(64) on all scope/expiry signals | Step 1: range checks |
| Permission hierarchy violation | Cumulative bit encoding on delegatee | Step 4: `bit4_requires_bit3`, etc. |

### 3.5 Range Check Coverage

| Signal | Component | Bit width | Purpose |
|--------|-----------|-----------|---------|
| `delegatorScope` | `delegatorScopeBits = Num2Bits(64)` | 64 | Decompose for subset check |
| `delegateeScope` | `delegateeScopeBits = Num2Bits(64)` | 64 | Decompose for subset check + cumulative |
| `delegateeExpiry` | `delegateeExpiryRange = Num2Bits(64)` | 64 | Valid uint64 for comparison |
| `delegatorExpiry` | `delegatorExpiryRange = Num2Bits(64)` | 64 | Valid uint64 for comparison |

Note: `delegatorCredCommitment`, `delegateeCredCommitment`, public keys, and signature components have NO explicit range checks. They are constrained through Poseidon hashing and EdDSA verification.

### 3.6 Underconstraint Analysis

**UC3.1** (Delegator public key not bound to delegatorCredCommitment): The circuit verifies that the delegator's public key signed the token, and that `P(delegatorScope, delegatorCredCommitment)` equals the previous scope commitment. However, there is NO constraint that `delegatorPubkeyAx/Ay` is the key embedded within `delegatorCredCommitment`.

**Implication**: If the credential commitment scheme is `P(modelHash, Ax, Ay, perm, expiry)` (as in AgentPolicy), then the binding between the signing key and the credential is established by the PREVIOUS hop's AgentPolicy proof (which verifies the signature over the credential). The Delegation circuit trusts that the `previousScopeCommitment` was produced by a valid AgentPolicy proof. This is secure IF the on-chain verifier checks the chain linkage properly. If `previousScopeCommitment` is not validated against a known-good AgentPolicy output, a malicious prover could fabricate a delegatorCredCommitment that hashes to match `previousScopeCommitment` but uses a different key pair.

**Severity**: Medium. Security depends on correct on-chain chain verification.

**UC3.2** (delegatorExpiry not bound to any public output): `delegatorExpiry` is a private input that participates only in the `LessEqThan` check against `delegateeExpiry`. It is NOT committed to in `previousScopeCommitment` (which only binds scope and credential).

**Implication**: A delegator could claim any `delegatorExpiry` value, as long as it is ≥ `delegateeExpiry`. The actual expiry enforcement must come from the AgentPolicy proof that generated the original scope commitment (which checks `currentTimestamp < expiryTimestamp`).

**Severity**: Low-Medium. The delegator can artificially raise their claimed expiry within the delegation circuit. However, the chain is secure if the original AgentPolicy proof was valid at its time of creation. The delegatee's expiry is still bounded by whatever the delegator claims.

**Mitigation**: Consider including expiry in the scope commitment: `P(scope, credCommitment, expiry)`. This would require a protocol change.

**UC3.3** (Delegatee Merkle siblings beyond depth): Same pattern as UC1.1 — siblings beyond `delegateeMerkleProofLength` are unconstrained but do not affect the output. Standard variable-depth Merkle proof pattern. NOT a vulnerability.

### 3.7 Known Limitations

1. **No current-time check**: Unlike AgentPolicy, the Delegation circuit does NOT verify that `delegateeExpiry > currentTimestamp`. It only enforces `delegateeExpiry ≤ delegatorExpiry`. Liveness checking (is the delegatee's credential currently valid?) must be done by a separate AgentPolicy proof or on-chain logic.

2. **Chain depth not enforced in-circuit**: The maximum delegation depth (3 hops) is enforced by the on-chain contract, not by the circuit. The circuit itself would accept arbitrarily deep chains.

3. **No revocation of delegations**: Once a delegation signature is issued, it cannot be revoked within the circuit. Revocation requires removing the delegatee from the agent tree (which invalidates CIP-1 Merkle proof).

4. **Delegator expiry is self-asserted** (see UC3.2): The delegator's expiry is not bound to the scope commitment chain. A malicious delegator could use an inflated expiry value.

5. **Single-use nonce semantics**: The nullifier is `P(token, sessionNonce)`. If the same delegation token is used with different nonces, different nullifiers are produced. The on-chain contract must track nonces to prevent reuse.

6. **No delegator Merkle proof**: The circuit does NOT prove the delegator is enrolled. It proves only that the delegator's scope commitment matches the previous hop. The delegator's enrollment is guaranteed transitively through chain verification back to an AgentPolicy proof.

## Cross-Circuit Properties

### Composition Soundness

**CC1** (AgentPolicy → Delegation chain linkage):
```
AgentPolicy.scopeCommitment = P(permissionBitmask, credentialCommitment)
Delegation.previousScopeCommitment must equal this value for hop 0
```
The on-chain verifier MUST enforce that `Delegation.previousScopeCommitment` equals a previously-verified `AgentPolicy.scopeCommitment`.

**CC2** (Delegation → Delegation chain linkage):
```
Delegation_n.newScopeCommitment = Delegation_{n+1}.previousScopeCommitment
```

**CC3** (Monotone scope attenuation across full chain):
```
∀ chain of length k:
  finalScope ⊆ ... ⊆ scope_1 ⊆ originalScope
```
By transitivity of the bitwise subset relation enforced at each hop.

**CC4** (Monotone expiry attenuation):
```
∀ chain: delegateeExpiry_k ≤ ... ≤ delegateeExpiry_1 ≤ originalExpiry
```
Note: This is only guaranteed if delegatorExpiry at each hop is honestly provided (see UC3.2).

### Cross-Circuit Privacy Boundary

| Information | HumanUniqueness | AgentPolicy | Delegation |
|-------------|:-:|:-:|:-:|
| Identity/secret | Hidden | N/A | N/A |
| Model identity | N/A | Hidden | N/A |
| Operator identity | N/A | Hidden | Hidden |
| Exact permissions | N/A | Hidden (partial: satisfaction proven) | Hidden |
| Expiry timestamp | N/A | Hidden (validity proven) | Hidden |
| Merkle path | Hidden | Hidden | Hidden |
| Delegation graph topology | N/A | N/A | Hidden |
| Scope commitment | N/A | **Public** | **Public** (new + previous) |
| Nullifier | **Public** | **Public** | **Public** |
| Merkle root | **Public** | **Public** | **Public** (delegatee) |

## Summary of Findings

### Critical Design Decisions (Correct)
1. Identity-bound scope commitments prevent impersonation across the delegation chain
2. CIP-1 (delegatee Merkle inclusion) closes the phantom delegatee attack
3. Range checks on all uint64 fields prevent field overflow attacks at the Solidity boundary
4. Cumulative bit encoding is enforced at both enrollment (AgentPolicy) and delegation time

### Findings Requiring Protocol-Level Consideration
1. **UC3.2**: `delegatorExpiry` is not committed to in the scope commitment, allowing self-assertion. Consider `scopeCommitment = P(scope, cred, expiry)`.
2. **UC3.1**: Delegator key binding depends on correct on-chain chain verification. Document this trust assumption explicitly in the verifier contract.
3. **Limitation 3.1**: No current-time check in Delegation means expired delegatees can still produce valid delegation proofs if the Merkle root hasn't been rotated.

### Verification Recommendations
1. Implement automated constraint counting to verify no signal is unconstrained
2. Use `circom --inspect` to check for signals not appearing in any constraint
3. Fuzz the Num2Bits boundaries: test values at 2^64 - 1, 2^64, and 2^251 - 1, 2^251
4. Test the delegation chain with fabricated `previousScopeCommitment` values to verify on-chain rejection
5. Verify EdDSAPoseidonVerifier rejects malformed curve points for operator keys
