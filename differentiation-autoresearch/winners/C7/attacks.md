### Attack 4: PLONK Achieves HVZK — Malicious Verifier Breaks the Zero-Knowledge Claim

**Attack:** The construction lists `sessionNonce`, `currentTimestamp`, and `requiredScopeMask` as public inputs supplied by the verifier (Section 2, public signal layout, indices 5–7). A malicious verifier — one who deviates from the protocol — can adaptively choose these inputs across multiple proof queries for the same credential. By choosing `sessionNonce` values strategically and observing the resulting `nullifierHash = Poseidon2(credentialCommitment, sessionNonce)`, the malicious verifier can extract information about `credentialCommitment` in the random oracle model. Similarly, setting `currentTimestamp` near `expiryTimestamp` leaks the range of `expiryTimestamp`.

**Why it works:** PLONK achieves honest-verifier zero-knowledge (HVZK) — the simulator works only for verifiers who use the prescribed public inputs. The security argument in Section 4 names A3 as "knowledge soundness of PLONK in the algebraic group model + ROM" and provides a reduction sketch for forgery games. It does not provide a ZK simulator. Without a simulator, the zero-knowledge claim is an assertion, not a proof. For the UC composition framework — required if this construction is to be composed with other protocols (e.g., a token exchange protocol, as discussed in Section 8 regarding RFC 8693) — simulation-extractable ZK is needed, not HVZK.

**Concretely:** If the verifier queries the prover with `sessionNonce_1, sessionNonce_2, ..., sessionNonce_k` for the same credential and observes `nullifierHash_i = Poseidon2(credentialCommitment, sessionNonce_i)`, the verifier has k equations in one unknown (`credentialCommitment`) in the Poseidon hash function. In the ROM, Poseidon queries are random oracle queries — learning k (input, output) pairs for the same hidden input leaks information that a simulator without the witness cannot reproduce. This violates simulation-based ZK.

**Why it fails to be addressed:** Section 3 states adversary capabilities and winning conditions for forgery games but never defines a distinguisher game for ZK. The claim "verifier learns ONLY {model_hash, operator_pk, permission_bitmask, message_hash}" (from `gap_to_close`) is a ZK claim that requires a simulator. The construction provides no simulator description — not even a sketch. Section 4's security argument is entirely about soundness (Cases 1–5), with ZK treated as inherited from PLONK by reference to "Section 3.3." But PLONK's ZK in the context of verifier-supplied public inputs and multi-query settings is not addressed by a reference — it requires proof.

**In-threat-model?** No. The threat model defines only the forgery game (MODEL-BIND-FORGE). It does not define a ZK distinguisher game or specify whether the zero-knowledge guarantee holds against malicious verifiers, adaptive public-input selectors, or in the multi-proof setting. For the construction to support UC composition — which the NCUA audit and cross-institution scenarios implicitly require — the authors must provide a ZK simulator and prove either malicious-verifier ZK or simulation-extractable ZK. HVZK alone is insufficient for any setting where the verifier is not assumed to follow the protocol honestly.

Specifically: in Bolyra, **the verifier cannot contact Anthropic to deanonymize a proof** — the provider key is a private input, and `providerRegistryRoot` reveals only that *some* enrolled provider attested the credential, not which one or which model version. ### Attack 4: AS-Side Model Attestation Is Operator-Asserted in Your Construction Too — The Trust Root Is Symmetric

- **Attack:** Your Section 8(a) reads: "SPIFFE SVIDs bind workload identity to a SPIRE-attested process, but the SPIRE server is operator-controlled. An operator running their own SPIRE deployment can register `spiffe://anthropic.com/models/sonnet-4-6` for a process actually running Opus." I accept this. Now I apply the same argument to your construction. Your `modelHash` is a hash of "model weights/identifier" (Section 2, private inputs). Who computes this hash? The operator, when they construct `credentialCommitment = Poseidon5(modelHash, ...)`. Anthropic signs `deploymentAuthorization = Poseidon3(modelHash, operatorPkAx, operatorPkAy)` — but over a *hash the operator supplies*. The construction proves "the prover committed to *some* `modelHash` that Anthropic signed for this operator" — it does not prove "that `modelHash` is a correct hash of Sonnet 4.6 weights." No — construction must address this.** The required addition: either (a) a transparency log binding `modelHash → published weights checksum` that any party can independently verify (analogous to Certificate Transparency), or (b) explicit acknowledgment that the trust boundary is Anthropic's organizational correctness in issuing `deploymentAuthorization`, not a cryptographic guarantee about weight identity, and scope the claim accordingly.

### Attack 1: The ZK Attestor Plugin Exists — You Built a Protocol When a Plugin Would Do

- **Attack:** Section 8(a) dismisses SPIFFE by saying "the SPIRE server is operator-controlled." This is a deployment argument, not an architectural one. SPIRE's node attestation is plugin-based: `NodeAttestor` and `WorkloadAttestor` are extension points. A production-ready pattern is: Anthropic ships a SPIRE WorkloadAttestor plugin that accepts a ZK proof from the workload process, verifies the proof inside the attestor against Anthropic's public key (the same `providerPk` your circuit needs), and issues the SVID only if the proof passes. The resulting X.509 SVID is signed by the SPIRE CA for `spiffe://anthropic.com/models/sonnet-4-6/operator/acme-corp`. No new protocol. No on-chain governance. No PLONK circuit. Just a SPIRE plugin, a transparency log for the provider key, and mTLS.

- **Why it works / why it fails:** The construction never engages with the attestor extension model. It argues at the level of "SPIRE server is operator-controlled" (Section 8a) but doesn't show why a ZK proof verification cannot be moved into a SPIRE node attestor running on Anthropic-controlled infrastructure. The counterargument would be that SPIRE node attestation requires the attestor to be online at attestation time and doesn't produce a portable, offline-verifiable artifact. That is a real difference — but the construction never makes this argument. It just asserts SPIFFE is insufficient.

- **In-threat-model?** No. Section 8 does not address the SPIRE attestor extension model. The construction must show why ZK-in-circuit is necessary versus ZK-in-attestor-plugin, and why the resulting SVID is insufficient for the offline/archival verifier use case.

### Attack 2: The `modelHash` Private Input Proves a Pre-Commitment, Not Runtime Execution

- **Attack:** The construction's circuit takes `modelHash` as a **private input** and verifies that Anthropic signed a binding `Poseidon3(modelHash, operatorPkAx, operatorPkAy)`. This proves: "Anthropic authorized operator ACME to use the model whose weights hash to H." It does not prove: "at proof-generation time, the agent was running the model whose weights hash to H." An operator who holds a valid deployment authorization for Sonnet (`modelHash_sonnet`) can run Opus internally, observe any message, and generate a syntactically valid PLONK proof binding that message to `modelHash_sonnet`. The circuit has no runtime execution witness — no TEE measurement, no hardware attestation nonce, no kernel-reported binary hash. The `modelHash` private input is set by the prover, checked only against a pre-signed credential, not against the live execution environment.

- **Why it works / why it fails:** The threat model (Section 3) defines the adversary as one who has credentials for M_adv and tries to forge proofs for M_target. It does not model an adversary who has legitimate credentials for M_target but is running a different model at runtime. This is outside the stated game — but it is the most operationally realistic attack: an operator licensed for Sonnet runs a cheaper model and generates valid Sonnet proofs. SPIFFE's node attestation solves this by tying the SVID to a TPM-attested or hypervisor-measured process, which the construction has no equivalent of. The `modelHash` is asserted by the prover, not measured by a trusted third party at runtime.

- **In-threat-model?** No. The construction must address whether `modelHash` is self-reported or externally measured. If self-reported, the binding claim (Section 1) is weakened to "the operator was authorized to use this model" not "this specific execution used this model."

### Attack 3: WIMSE Token Exchange Does Not Require Per-Call AS Visibility

- **Attack:** Section 8(c) argues: "Adding an AS to enforce the model-identity claim at issuance re-introduces the correlation problem (AS sees every call)." This conflates token issuance with token exchange frequency. WIMSE (draft-ietf-wimse-arch, Section 5.2) explicitly supports pre-issuance of workload tokens at deployment time. The AS issues a workload bearer token (WBT) once per deployment, binds it to the workload via DPoP, and the workload carries that token for its lifetime. The AS does not participate in subsequent calls. This is structurally identical to Bolyra's `deploymentAuthorization`: Anthropic signs once per `(modelHash, operatorPk)`, and that credential travels with the operator. The construction's claim that WIMSE requires per-call AS participation misreads the draft architecture.

- **Why it works / why it fails:** The construction's Section 8(c) is its core differentiation argument against the baseline. If WIMSE token exchange can pre-issue a bound token without per-call AS correlation, then the "no AS in the verification path" property does not distinguish Bolyra from WIMSE+DPoP. The remaining distinction would be the selective-disclosure property (verifier learns only `{model_hash, operator_pk, permission_bitmask, message_hash}`) — but BBS+ selective disclosure on WIMSE tokens is already in scope for the WIMSE working group.

- **In-threat-model?** No. The construction must engage with WIMSE's deployment-time token issuance model specifically, not the generic "AS sees every call" straw man. The gap-to-close (as stated in the candidate metadata) requires proving that RFC 8693 token exchange cannot achieve (a)+(c) simultaneously — but the construction's argument in Section 8(c) does not actually prove this; it asserts it based on a mischaracterization of when the AS participates.

### Attack 4: On-Chain Provider Registry Governance Is Not Stronger Than SPIFFE Bundle Federation

- **Attack:** The construction's root of trust is an on-chain Merkle tree controlled by "governance-multisig or DAO" (Section 2, enrollment protocol). SPIFFE trust domain federation (SPIFFE Federation spec, Section 4) uses signed bundle endpoints: each trust domain publishes a JWKS-style bundle at a well-known HTTPS endpoint, signed by the trust domain's CA. Cross-domain federation is established by pinning the peer bundle endpoint certificate. Rotation is handled by re-fetching the bundle. There is no on-chain component, no smart contract governance, and no DAO attack surface. The construction asserts its provider registry is the root of trust but offers no argument for why a governance-controlled smart contract is more tamper-resistant than a CA-signed SPIFFE bundle endpoint backed by an HSM. Smart contract governance attacks — 51% multisig compromise, DAO proposal manipulation, front-running during key rotation — are well-documented and are not addressed in the threat model.

- **Why it works / why it fails:** Section 3 (threat model) states the adversary "does NOT control the on-chain provider registry (governance-multisig or DAO-controlled)" as an axiom, not a design property. The construction inherits exactly the trust assumption it is trying to eliminate from the baseline: a trusted operator controls the authoritative registry. The difference is whether that operator is Anthropic's HSM (SPIFFE) or a multisig committee (Bolyra). The threat model should either (a) argue why the on-chain governance is more attack-resistant than SPIFFE bundle federation, or (b) acknowledge this as an equivalent trust assumption and scope the claim to the ZK-specific properties that SPIFFE cannot provide (offline verifiability, selective disclosure, bitmask predicates).

- **In-threat-model?** No. The construction does not model compromise of the provider registry governance. Any realistic deployment of this system must treat registry governance compromise as a threat, and the construction should either address it or explicitly bound the claim to "assuming registry integrity."
