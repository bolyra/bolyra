### Attack 1: Adaptive Bitmask Recovery — SCOPE-HIDE Is a Single-Shot Game Against a Multi-Shot Adversary

**Attack**: An adversarial RS is not bound to a single interaction. It issues 64 sequential challenges, one per bit position: `requiredScopeMask = 2^0`, then `2^1`, …, `2^63`. For each challenge, it observes whether the agent produces a valid proof. By knowledge soundness (A1b), the agent **cannot** produce a valid proof if bit `i` is not set. The presence or absence of a proof is perfectly observable. After 64 rounds, the RS knows `permissionBitmask` exactly.

**Why it works**: Game SCOPE-HIDE (§3, Game 2) as written is single-proof:

```
3. Challenger generates proof π with requiredScopeMask = M.
4. A receives (π, all public signals) and outputs guess b'.
```

The game freezes the adversary to a single mask choice. In deployment there is no such restriction. The adaptive game is:

```
Game ADAPTIVE-SCOPE-HIDE(A, λ):
  A queries (M₁), (M₂), …, (M_q) adaptively.
  For each Mⱼ: if B_b & Mⱼ = Mⱼ, A receives a proof; else A receives ⊥.
  A outputs b'.
```

Under this game, `Adv[ADAPTIVE-SCOPE-HIDE](A) = 1` for any `B₀ ≠ B₁` differing in at least one bit. This is not a reduction — it is a direct win. The ZK property of Groth16 is irrelevant because the information leaks through the **protocol oracle** (proof present/absent), not through proof contents.

**In-threat-model?** No — §3 restricts A to a single mask and does not model adaptive querying. The construction must address this, either by explicitly accepting it as an inherent information-theoretic limitation (1 bit per interaction is necessarily disclosed), bounding the adaptive leakage in the security argument, or introducing a protocol-level rate constraint (which is not a cryptographic control). The current reduction sketch for SCOPE-HIDE is valid only in the single-query model it was written for, not in any realistic deployment.

### Attack 2: Delegation Privacy Leak — Delegatee Learns Delegator's Full Permission Bitmask

**Attack**: The chain-linking constraint in the Delegation circuit is (§2, "Delegation chain linking"):

```
Poseidon3(delegatorScope, delegatorCredCommitment, delegatorBlindedNonce) == previousScopeCommitment
```

For the delegatee to satisfy this constraint as the prover, they must supply `delegatorScope`, `delegatorCredCommitment`, and `delegatorBlindedNonce` as private inputs. Tracing the mapping in §5, `delegatorScope` corresponds to `permissionBitmask` (the delegator's full 64-bit bitmask) — since `scopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, blindedNonce)` in the AgentPolicy circuit. So the delegatee receives the delegator's unmasked `permissionBitmask` out-of-band to run the delegation circuit.

**Why it matters**: The entire construction's claimed property is that neither the AS nor the RS learns which bits beyond the predicate are set. In a delegation chain, the delegatee (who may be a less-trusted sub-agent) learns the delegator's full scope exactly. The selective-scope privacy guarantee collapses at every hop in the delegation chain. The "scope narrowing is one-way" enforcement (§CLAUDE.md) is preserved cryptographically, but the privacy of the narrowed-from set is not.

**Why it structurally cannot be fixed within the current circuit design**: To verify `Poseidon3(delegatorScope, ...)`, the prover must know `delegatorScope`. To commit without revealing, you would need a separate proof that the delegator's scope satisfies some predicate — i.e., a recursive SNARK composition (Groth16 recursion or IVC) or an inner proof of the delegator's AgentPolicy circuit output. Neither is present in the current construction.

**In-threat-model?** No — §3 only models the RS as the observer. The delegatee is not in the adversary model, and the privacy game SCOPE-HIDE considers only single-hop scenarios. In any multi-hop delegation, the delegatee becomes a side-channel into the delegator's full permission set. The construction must either add a recursion layer, replace `delegatorScope` with a commitment whose preimage is not required by the delegatee, or explicitly state that delegation breaks the privacy guarantee.

### Attack 3: Nullifier Futility — Same Credential Can Double-Present Against the Same RS

**Attack**: The nullifier is defined as `nullifierHash = Poseidon2(credentialCommitment, blindedNonce)` where `blindedNonce = Poseidon2(sessionNonce, agentBlind)`. The construction requires `agentBlind` to be **fresh per proof** (§2: "The agent MUST sample `agentBlind` uniformly at random from F_p for each proof"). Given a fixed RS-provided `sessionNonce = n` and a fixed credential, the agent can generate:

- Proof 1: `blindedNonce₁ = Poseidon2(n, blind₁)` → `nullifierHash₁ = Poseidon2(cred, blindedNonce₁)`
- Proof 2: `blindedNonce₂ = Poseidon2(n, blind₂)` → `nullifierHash₂ = Poseidon2(cred, blindedNonce₂)`

Both proofs are valid (different private `agentBlind`, same public `sessionNonce`). The RS records `nullifierHash₁` after proof 1 and does not recognize `nullifierHash₂` as a replay of the same credential. The RS sees two distinct nullifiers and allocates resources, grants access, or counts quota for what it believes are two different agents.

**Why the construction cannot prevent this**: The `agentBlind` freshness requirement is precisely what makes nullifiers non-repeating. Replay of an intercepted proof is prevented by the RS generating a fresh nonce at each request initiation — the nullifier provides no independent protection. The sentence in §3 ("RS records `nullifierHash` for replay detection within the session scope") is vacuous under the same construction that mandates fresh `agentBlind`. These two design requirements are in direct conflict.

**Consequences**: Rate-limit bypass, quota doubling, billing fraud in any system that counts resource accesses by nullifier uniqueness. The construction's §7 scenario (credit union AI agent controlling wire transfers) is particularly sensitive — an agent with `FINANCIAL_MEDIUM` could present multiple times in a single `sessionNonce` window if the RS doesn't impose nonce expiration with strict single-use semantics at the application layer (not specified in the protocol).

**In-threat-model?** No — the adversary model (§3) models replay as "A intercepts (π, pubSignals) and replays them." It does not model the agent itself as the entity generating multiple valid proofs against the same RS nonce. The nullifier's stated purpose is broken by the protocol's own freshness requirement.

### Attack 4: A4 Is Not "Unconditional" — Groth16 ZK Lives in the Generic Group Model

**Attack**: §4, Assumption A4 states: "Perfect (simulation-based) in the generic group model. Crucially, this holds unconditionally — it does not depend on A1a." The claim is repeated throughout: "ZK is unconditional with respect to ceremony trust," "this bound holds even if A1a is violated."

This is formally incorrect in two ways:

**First**, "unconditional" in cryptography means the guarantee holds without any computational hardness assumption, information-theoretically. Groth16 ZK is not unconditional in this sense. It is *perfect* only in the generic group model (GGM) — itself an idealized model that assumes adversaries cannot exploit the algebraic structure of BN254 beyond generic group operations. The GGM is an assumption about the hardness of finding algebraic relations in the group, comparable in nature (though not strength) to discrete log hardness. Equivalently, if a concrete non-generic algorithm exists for BN254 that breaks the GGM assumption, ZK fails regardless of the ceremony.

**Second**, the assertion that "even an adversary who possesses the SRS toxic waste cannot distinguish real proofs from simulated ones" is correct for privacy (SCOPE-HIDE, CROSS-RS-UNLINK) as long as the GGM holds. But this is not "unconditional" — it is "GGM-conditional," and the GGM is a stronger-than-standard assumption. Concrete attacks on BN254 at sub-group level have been studied (e.g., small-subgroup attacks, twist-security bounds); the GGM does not account for these.

**Practical consequence**: The separation between SCOPE-FORGE (broken by toxic waste) and SCOPE-HIDE (claimed independent) relies on the GGM for the ZK direction. If an adversary finds a distinguisher between real and simulated Groth16 proofs on BN254 without breaking DL — an open research question at BN254's security level — SCOPE-HIDE and CROSS-RS-UNLINK both degrade. The "unconditional" language actively misleads a reader evaluating whether PLONK provides any ZK improvement over Groth16: it does not for privacy, but the reason is not "unconditional ZK" — it is "both rely on GGM for ZK."

**In-threat-model?** Partially — the construction correctly identifies the separation between soundness (ceremony-dependent) and ZK (ceremony-independent). The substance is sound; the formalization is wrong. An IACR reviewer would require A4 to be restated as: "Perfect ZK in the generic group model, unconditional with respect to SRS toxic waste knowledge but conditional on the GGM holding for BN254." The SRS trust table (§4) is misleading in its current form.

### Attack 1: Multi-round bit-oracle collapses SCOPE-HIDE to trivial

**Attack:**

- Agent holds bit i → produces a valid proof → RS observes "success"
- Agent lacks bit i → cannot satisfy the constraint `requiredBits[i] * (1 - permBits[i]) === 0` → produces no valid proof → RS observes "failure"

After at most 64 adaptive queries, the RS has reconstructed the full 64-bit bitmask exactly. Agent-blinding (§2, Modification 2) addresses *cross-RS unlinkability* across independently operating RSes; it does not address a *single RS issuing sequential single-bit challenges*. **Why it lands:** The construction's single-proof SCOPE-HIDE game and its supporting reduction (§4) provide no adaptive security bound. There is no mechanism that rate-limits mask queries, commits the RS to a fixed mask before the agent generates a proof, or prevents the RS from issuing 64 sequential challenges. This is a protocol design gap, not a cryptographic one.

**RFC 7662 comparison:** The AS reveals the full scope set in one introspection response. The Bolyra construction reveals the same information across 64 sequential binary oracle queries. For a sufficiently curious RS, information-theoretic leakage is identical; the only difference is latency.

### Attack 2: Axis 2 proof vs. BBS+ is a strawman — per-attribute disclosure is not exponential

**Attack:**
§8 Axis 2 dismisses BBS+ with: *"The RS would need to pre-enumerate all valid permission combinations and request BBS+ equality proofs for each — exponential in bitmask width."* This conflates two distinct BBS+ disclosure modes:

- **Subset-attribute disclosure (claimed exponential):** proving "my scope string includes READ+WRITE+FINANCIAL_SMALL" by exhaustively enumerating credential variants. This *is* exponential if done naively.
- **Per-attribute disclosure (not exponential):** encode 64 permissions as 64 boolean attributes in a single BBS+ credential. The RS specifies `{bit_0: true, bit_2: true}`. The holder generates a BBS+ proof of knowledge disclosing *exactly those two attributes* and hiding all others. Proof size: O(number of required attributes), not O(2^64). This is standard BBS+ selective disclosure, RFC-track in the W3C VC data model.

The cumulative-bit implication constraint (§2, gadget 4) is a genuine differentiator — BBS+ cannot *enforce* that `bit_4 → bit_3 → bit_2` holds without either revealing those bits to the verifier or encoding the predicate as a separate committed claim. But the construction's framing ("exponential enumeration") is the wrong comparison.

**Why it lands:** Axis 2 is the weakest part of §8. The "constant-size" claim (192-byte Groth16 vs. O(required_bits) BBS+ proof) is real and valid. The "inexpressible predicate" claim for bitwise AND is overstated — BBS+ expresses it with per-attribute selective disclosure. The *accurate* differentiator is: (a) BBS+ proof size grows with required bit count; (b) BBS+ cannot enforce implication closure without revealing implication witnesses; (c) AS-side issuance is required. Only (c) and (b) are fundamental; (a) is a practical performance gap.

**In-threat-model?** The attack succeeds at narrowing the claimed gap, but does not fully close it — the cumulative-bit implication constraint remains genuinely novel in BBS+. The construction should replace "exponential enumeration" with the accurate comparison. Not closing the gap entirely, but weakening the stated argument.

This means the delegatee possesses `delegatorBlindedNonce = Poseidon2(sessionNonce, agentBlind_delegator)`. Given that `sessionNonce` is a public input (known to everyone), the delegatee can attempt offline dictionary attacks on `agentBlind_delegator` if the delegator samples from any predictable distribution (e.g., a 128-bit CSPRNG output rather than a full F_p element). The construction mandates "uniform from F_p" (§2), but provides no in-circuit proof of this — `agentBlind` is a private input with no range constraint enforced by the circuit.

More concretely: the delegatee now knows `delegatorBlindedNonce`. For any candidate pair `(delegatorScope, delegatorCredCommitment)` it can guess, it can compute `Poseidon3(scope, cred, delegatorBlindedNonce)` and check against the on-chain `previousScopeCommitment`. If the delegatee is curious about the delegator's scope (e.g., to learn how much permission the delegator is withholding), and if the scope space is small (8-bit cumulative encoding has at most 256 valid values per the implication rules), the delegatee can enumerate all valid scopes in 256 Poseidon evaluations and recover the delegator's exact scope.

**Why it lands:** The threat model (§3, "Adversary model") lists colluding *RSes* as the adversary but does not model an honest-but-curious *delegatee*. The unlinkability claim in §2 ("it is never public, preserving unlinkability") applies to external observers, not to the delegatee who is explicitly given the value. The 8-bit scope space means the delegatee needs only ≤256 Poseidon queries to break the delegator's scope privacy, regardless of A2 (Poseidon collision resistance does not imply preimage resistance when the preimage space is bounded).

**In-threat-model?** The delegatee receiving `delegatorBlindedNonce` is a necessary protocol step. The construction must either: (a) formally exclude delegatee-as-adversary from scope privacy claims, or (b) add a range proof that `delegatorBlindedNonce` is uniformly distributed and limit what the delegatee learns. As written, delegation chain privacy against a curious delegatee is unaddressed.

### Attack 4: RFC 8693 + DPoP narrows the "adversarial AS" gap to a single non-repudiable claim — and the on-chain Merkle root has the same attack surface

**Attack:**
§8 Axis 3 claims AS trust is "implicit, monolithic, and inescapable" in the baseline. But RFC 8693 Token Exchange with a scope-restriction policy and RFC 9449 DPoP provides:

- **DPoP**: The token is cryptographically bound to an ephemeral public key the agent controls. The AS cannot issue a DPoP-bound token that a different agent can present. The RS verifies the DPoP proof at the HTTP layer — the AS is not in the verification path for freshness.
- **RFC 8693 scope narrowing**: Token exchange can reduce scope at delegation time. The exchange server (which may be operated independently of the issuing AS) attests only that the incoming token had ≥ the outgoing scope. A compromised primary AS cannot expand scope post-issuance if the exchange server is honest.
- **Signed jwt-introspection-response (draft-ietf-oauth-jwt-introspection-response)**: The RS can verify the introspection response offline via the AS's signing key. A network-level attacker cannot alter it; only the AS can forge it.

The construction's §3 Game 3 (AS-BLIND) correctly identifies that a *fully adversarial AS* breaks all of the above. But the reduction then requires the adversary to "corrupt the Merkle tree contract" — and the Merkle contract is owned by a deployer whose private key must be protected. In the credit union scenario (§7), the Merkle tree operator's EdDSA key is the true trust anchor. If that key is compromised (supply-chain attack on the operator's key management system), the adversary can enroll arbitrary credentials with arbitrary bitmasks — the same attack surface as a compromised AS minting tokens with forged scopes.

**Sharpened question**: The construction shifts trust from "AS is honest" to "operator EdDSA key + SRS ceremony + on-chain contract are honest." In the specific credit union deployment, these are all controlled by the same organizational unit as the AS. The trust decomposition is more explicit (a genuine improvement) but not more distributed — and the threat of "supply chain attack on the AS vendor" applies with equal force to "supply chain attack on the Merkle tree contract deployer or the PLONK SRS updater."

**In-threat-model?** Not fully. Axis 3 is a genuine differentiator when the AS and the on-chain infrastructure have genuinely distinct trust domains (e.g., an AS operated by a cloud vendor vs. an on-chain contract the operator controls directly). In the §7 credit union scenario, both are controlled by the same entity, so the adversarial-AS model is not the primary risk driver. The construction should qualify Axis 3 with explicit conditions on when the trust domains diverge enough to matter.

### Attack 3: The "Runtime-Adaptive Predicate" Is Solvable with Per-RS SVID Scoping

### Attack 4: The On-Chain Merkle Root Is a Smart Contract PKI — Operationally Worse Than SPIFFE Trust Bundles

- **Attack**: The construction's §7 deployment scenario anchors trust in a "30-entry circular root history buffer" maintained by a Solidity contract on Base Sepolia. The RS verifies `agentMerkleRoot ∈ root history buffer` (§2, step 3). This is cryptographically equivalent to SPIFFE's trust bundle distribution — the RS trusts roots published by an authority. But the operational comparison is unfavorable:

    1. **Liveness dependency**: The RS must maintain an RPC connection to Base Sepolia or a cached copy of the root history buffer. SPIFFE trust bundles are distributed via authenticated gRPC and cached. A blockchain RPC outage kills verification; a SPIRE outage with cached bundles does not (SVIDs are valid until `exp`).
    2. **30-root window creates a revocation race**: If an operator key is compromised, the operator must be removed from the Merkle tree. The 30-root history buffer means the compromised agent can still prove against the prior root for the duration of the buffer window. SPIFFE X.509 SVIDs expire in 1 hour by default and can be revoked via CRL. The construction has no revocation mechanism — §2 and §7 do not address it.
    3. **Contract upgrade is a governance event**: Changing the Merkle tree contract requires a deployment. Adding a new trust anchor in SPIFFE is a SPIRE config file edit. For a credit union operating under NCUA examination (§7 deployment scenario), the NCUA examiner accepts certificate-based PKI as a standard control; blockchain smart contract governance is novel compliance territory.

- **Why it works / fails**: The attack is operational, not cryptographic. It doesn't break soundness or privacy. It does challenge the deployment viability of §7, specifically for the NCUA-regulated credit union scenario. The construction's §8 Axis 3 argument ("AS compromise resilience") is real, but it trades one operational risk (AS compromise) for a different one (contract liveness + revocation gap + blockchain governance). This is not acknowledged in the threat model.

- **In-threat-model?** **No — construction must address this.** The threat model (§3) scopes the adversary to network observers and colluding RSes but does not model **credential revocation** (compromised operator key, agent decommission). The 30-root history buffer is the only mechanism, and it provides a forward-window of continued validity after removal — potentially covering multiple minutes of blockchain finality plus proof generation time. A revocation game should be added, or the threat boundary must explicitly exclude post-compromise forward security.
