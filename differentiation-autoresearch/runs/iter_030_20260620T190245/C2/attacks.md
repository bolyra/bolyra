### Attack 1: The Operator-Independence Invariant Collapses in the Primary Deployment Scenario

**Attack:**
Section 2.4 states the entire IND-UNL-AS guarantee rests on one invariant: "the `scopeBlindingSecret` is generated by the agent's local execution environment" and "never leaves the agent's local environment." Section 3.1 then acknowledges the game immediately void if "the adversary compromises the agent's local storage." Now look at the primary deployment scenario in §7: Navy Federal Credit Union (NFCU) is simultaneously (a) the credential operator, (b) the AS, and (c) the entity that deploys and operates the agent runtime. The "agent" runs on NFCU's AWS Lambda functions, GCP Cloud Run instances, or Kubernetes pods. NFCU wrote the agent code, provisions its secrets, and controls the filesystem where `scopeBlindingSecret` is stored. The threat model (§3.1) explicitly places "the AS and/or the credential operator" inside the adversary's control — but then excludes "the agent's local execution environment." These two conditions are mutually exclusive in the NFCU scenario.

**Why it works / why it fails:**
This isn't a theoretical attack — it's the normal enterprise deployment topology. When a credit union deploys an agent to serve its members, it runs that agent on its own infrastructure. The IND-UNL-AS game (§3.2) gives the adversary all operator private keys but withholds the blinding secrets. That adversary model describes a third-party operator who hands credentials to an agent running on *the member's device*. The moment the operator runs the agent infrastructure, the adversary trivially reads `scopeBlindingSecret` from environment variables, mounted secrets, or HSM logs — and wins IND-UNL-AS with advantage 1. The construction is a privacy guarantee for user-local agents (mobile wallet, browser extension). Framed as "CU-as-AS must not see member merchant graph," the threat model is broken by construction for the named scenario.

**In-threat-model?** **No.** Construction must either (a) restrict deployment scope to user-local agents with hardware enclaves or TEEs, (b) add an explicit statement that CU-operated agent runtimes break the guarantee, or (c) redesign `scopeBlindingSecret` generation using hardware-attested randomness with remote attestation that the operator cannot inspect — a substantially harder engineering problem than anything in this document.

### Attack 2: Proving Latency + Epoch Boundary Failures Create Availability Gaps with No Fallback

**Attack:**
Section 6 targets "< 3s" PLONK proving for ~17,124 constraints on "commodity hardware." The construction claims PLONK proving is "constant-time for fixed circuit size" (§3.5). Now apply this to actual enterprise MCP deployments: agents run on cloud functions with variable CPU allocations (shared tenancy, noisy neighbors), mobile SDKs on Android/iOS, and embedded IoT devices for industrial credit union use. Rapidsnark benchmarks on an M2 MacBook Pro hit ~2s for 17K constraints; the same workload on a constrained cloud function (512MB, shared vCPU) routinely takes 8–15s. The construction does not specify a minimum hardware profile, a timeout policy, or a retry protocol. The `sessionNonce` is per-request and the `epochId` is time-bucketed at 300s — if proof generation spans an epoch boundary, the `epochBinding` in the completed proof is stale relative to the RS's `currentTimestamp` check. The RS rejects the proof. What does the agent do? Retry? With a fresh nonce? The construction has no answer. WorkOS issues a signed JWT in <80ms with a documented retry policy, a rate limit, and a 99.95% SLA. Every Bolyra proof is a ~3s gamble with no fallback for failure.

**Why it works / why it fails:**
The construction partially addresses timing via "agent MAY add random delay within epoch" (§3.5), but this is the opposite of the problem — the concern is proof generation *taking longer than* a fraction of the epoch, not the agent choosing to wait. An epoch of 300s sounds generous, but proof generation latency variance of 2–15s on real infrastructure means agents will routinely start proving in one epoch and submit in the next. The RS must then accept proofs with an `epochBinding` from the prior epoch (requiring a history buffer of epoch bindings per nullifier), or reject them (breaking the agent). Neither the circuit spec nor the RS verification protocol in §2.6 mentions epoch-rollover handling. Meanwhile, there's no documented degradation path — the construction removes the AS entirely ("the AS is never contacted," §2.6), so there's no token fallback when ZK proving fails.

**In-threat-model?** **No.** The proving time claim is asserted without hardware profile bounds, and the epoch-boundary edge case is unaddressed in the verification flow. The construction must specify: minimum hardware requirements, epoch grace window for proof submission, RS-side epoch history depth, and what the agent does when proving times out.

The answer under Bolyra is "we have the enrollment event and nothing else." A BSA officer at NFCU hearing "the protocol is designed so you cannot reconstruct the merchant graph" is hearing "we cannot comply with BSA record-keeping requirements." Bolyra's strongest privacy property is its biggest regulatory blocker for the named customers.

**Why it works / why it fails:**
The construction does not mention 31 CFR 1020 (Bank Secrecy Act), FFIEC guidance on AI systems, or HIPAA audit log requirements. The healthcare scenario in §7 describes a patient agent accessing Quest Diagnostics such that "Kaiser cannot determine whether the patient visited Quest, Cedars-Sinai, or both." **In-threat-model?** **No.** The construction must either (a) document that regulated deployments require a separate audit layer that reintroduces an AS-equivalent (collapsing the unlinkability claim), or (b) explicitly scope the construction to unregulated deployments and remove the CU and healthcare scenarios from §7.

### Attack 4: Nullifier Set Management at the RS Is an Unspecified Operational Problem That Incumbents Already Solve

**Attack:**
Section 2.6 specifies that the RS "checks `scopedNullifier` against its local double-spend set." Section 2.5 notes the `epochBinding` enables "double-authorization detection" within an epoch. But the construction is silent on: (a) how large this nullifier set grows, (b) how it is purged, (c) what happens when two RSes in the same organization (e.g., Amazon.com and Amazon Pay) share scope but not infrastructure, and (d) what the RS does when it receives a valid proof with a nullifier it has never seen — this is the normal case for a first-time authorization. The RS cannot distinguish a legitimate new agent from a replayed proof with a freshly manufactured `scopedNullifier` for a scope it has never served. The construction relies on the PLONK verifier to catch invalid witnesses, but the RS must also maintain the nullifier set correctly to prevent replay within a scope. At scale — an RS serving 10M agent authorizations per day — the nullifier set is a 10M-entry append-only database with no specified expiry policy, no cross-RS synchronization protocol, and no monitoring surface. WorkOS and Auth0 solve this with a centralized token store, revocation list, and Redis-backed introspection endpoint that RS operators get for free. Bolyra tells each RS to build and maintain its own nullifier database with no tooling.

**Why it works / why it fails:**
The `epochBinding` provides time-bucketing, but the `scopedNullifier` itself is permanent per (agent, scope) — an RS must retain every nullifier it has ever seen to detect future replays. The construction notes "the `scopedNullifier` is deterministic per (agent, scope)" (§2.2), which means the RS's nullifier set is the only replay protection. The 30-entry Merkle root history buffer (§3.5, §7) addresses root staleness but not nullifier database growth. For the NFCU member-agent scenario with millions of members each having per-merchant nullifiers, this is a multi-hundred-million-entry nullifier store per RS — with no specified schema, expiry, or consistency protocol across RS shards. The construction offloads an infrastructure problem that OAuth/OIDC centralizes at the AS to every RS independently, with no guidance.

**In-threat-model?** **No (operationally).** The nullifier management problem doesn't break the cryptographic claims, but it means no enterprise RS will self-operate this correctly. The construction needs a specified nullifier retention policy (tied to credential expiry or epoch count), a reference implementation, and ideally a shared nullifier service — which reintroduces a centralized component, partially collapsing the "AS-free" architecture claim.

### Attack 1: `DelegationEntry` `entryNullifier` Preimage Lookup — AS Fully Deanonymizes Every Delegation-Using Agent

**Circuit:** `DelegationEntry`, §2.3. Public outputs stored on-chain: `(agentMerkleRoot, scopeCommitment, entryNullifier)`. Public input on-chain: `sessionNonce`.

**Attack:**

The AS, acting as credential operator, holds every enrolled agent's credential fields `(modelHash, operatorPubkeyAx/Ay, permissionBitmask, expiryTimestamp)` and can therefore compute `credentialCommitment_j = Poseidon5(...)` for every agent `j`. The on-chain record exposes `(sessionNonce*, entryNullifier*)` in cleartext. The attack is a forward evaluation, not an inversion: for each `j`, the AS checks whether `Poseidon2(credentialCommitment_j, sessionNonce*) == entryNullifier*`. Complexity: O(N) Poseidon evaluations, where N is enrollment count — trivial for any realistic deployment.

The PRF argument in §4.1 (A-PRF) covers `scopedNullifier = Poseidon2(scopeId, scopeBlindingSecret)` because `scopeBlindingSecret` is the unknown key. The `entryNullifier` uses `credentialCommitment` as the first argument — a value the AS can compute from known credential fields. There is no analogue of `scopeBlindingSecret` here; the PRF key is known to the adversary. The construction's own §3.4 explains why operator-derived values collapse security: this is exactly that scenario applied to `DelegationEntry`.

**Why it fails against the construction:**

It does not fail — the current circuit has no `scopeBlindingSecret` in the `entryNullifier` derivation. The fix requires replacing `Poseidon2(credentialCommitment, sessionNonce)` with `Poseidon2(scopeBlindingSecret, sessionNonce)`, bringing `entryNullifier` under the same PRF argument as `scopedNullifier`. The current formulation is inconsistent with the design rationale of §2.4 and §3.4.

**In-threat-model?**

**No.** The construction's §1 states "unlinkability holds across scopes, across time" without qualifying "except for agents using delegation." The IND-UNL-AS game (§3.2) gives the adversary all operator keys — exactly the material needed for this preimage check. Any agent that submits a `DelegationEntry` is immediately identified to the AS with zero cryptographic cost. The scenarios in §7 (healthcare delegation, cross-CU merchant graph) both use delegation and are therefore fully broken for the purpose of hiding agent identity from the AS. This is a circuit-level error, not a proof-level gap.

### Attack 2: IND-UNL-AS Is a Two-Proof Game; Colluding RSes Mount a Multi-Probe Co-Occurrence Distinguisher

**Relevant sections:** §3.2 (IND-UNL-AS game), §2.5 (epoch bucketing), §3.5 (side-channel table).

**Attack:**

The proven bound — $2 \cdot \text{Adv}^{\text{PRF}}_{\text{Poseidon}} + \text{Adv}^{\text{KS}}_{\text{PLONK}} + \text{negl}(\lambda)$ — applies to the two-proof challenge in the IND-UNL-AS game. Real deployments produce unbounded transcripts. Consider $m$ colluding RSes accumulating `(scopeId_i, epochBinding_i, epochId, receivedTime_i)` over a period $T$. Within any 300s epoch window, `epochBinding = Poseidon2(scopedNullifier, epochId)` is deterministic per `(agent, scope, epoch)`. The adversary runs the following statistical test: for all pairs of RS-received proofs `(p_A, p_B)` sharing the same `epochId`, count co-occurrence frequency. For the same agent visiting $k$ scopes uniformly within the epoch, expected co-occurrences scale as $O(k^2 / W)$ where $W$ is epoch window count. For independent agents, expected co-occurrences scale as $O(k^2 N / W^2)$. At sufficient $k$ (many scope accesses per agent per epoch), this distinguisher achieves non-negligible advantage without inverting any PRF or breaking any named assumption — it exploits the *deterministic timing correlation* introduced by epoch bucketing itself.

The §3.5 timing side-channel table claims "Epoch bucketing: all proofs within a 300s window share the same `epochId`. Agent MAY add random delay within epoch." The word "MAY" is doing enormous work here. Without a mandatory uniform distribution over the epoch window, co-occurrence timing is an exploitable signal.

**Why it fails against the construction:**

The two-proof PRF bound remains tight for the individual challenge. The construction has no composition theorem extending unlinkability to $k$-probe transcripts. The game's query phase permits the adversary to "accumulate an arbitrary transcript of (agent, scope, proof-outputs) tuples" — but the proven bound covers only the two-world challenge distinction, not the reconstruction of the agent graph from the accumulated transcript. The security of the transcript as a whole is unproven.

**In-threat-model?**

**No.** The claim "Unlinkability holds across scopes, across time, and against collusion between AS and any subset of RSes" requires a $k$-proof composition theorem or an explicit restriction of the claim to the two-proof setting. Neither is provided. The construction must either prove that the Poseidon PRF outputs at `(scopeId_A, s_b)` and `(scopeId_B, s_b)` are jointly indistinguishable from independent randoms across an unbounded query sequence — a stronger PRF-to-PRF composition argument — or bound the timing advantage and require mandatory uniform jitter (not optional).

### Attack 3: PLONK Proof Simulation in the Reduction Is Trapdoor-Dependent — the Formal Proof Has an Unverified Link

**Relevant section:** §4.2, hybrid reduction, "Simulates valid PLONK proofs via the PLONK zero-knowledge simulator."

**Attack:**

The hybrid reduction requires $\mathcal{B}$ to simulate valid PLONK proofs during the query phase for agent $j_0$ using oracle-provided nullifiers rather than the real blinding secret. PLONK's zero-knowledge property in the common reference string (CRS) model is realized by a *trapdoor simulator* $\mathcal{S}(\tau)$: given the SRS trapdoor $\tau$ (the secret scalar from the powers-of-tau ceremony), $\mathcal{S}$ can produce proofs accepting any public input/output pair by programming polynomial commitments under the KZG hiding property. Without $\tau$, no black-box simulator exists for PLONK in the plain CRS model.

The construction uses `pot16.ptau` with $\tau$ assumed destroyed post-ceremony. $\mathcal{B}$ therefore has no access to $\tau$ and cannot run the PLONK ZK simulator. The reduction sketch states "simulates valid PLONK proofs via the PLONK zero-knowledge simulator" without resolving this.

Two resolutions each carry a cost: (1) Operate in the *programmable random oracle model* (PROM) — Fiat-Shamir hashes are programmed by $\mathcal{B}$, enabling proof extraction via rewinding, but PLONK simulation in PROM still requires either the trapdoor or the full witness. (2) Give $\mathcal{B}$ a freshly generated SRS with known trapdoor — but then the reduction applies to a *trapdoor-simulation* model where the adversary interacts with a simulated SRS, not the ceremony-generated `pot16.ptau`, and the proof must argue the adversary cannot distinguish the two (which requires an additional SRS-indistinguishability argument not present in the construction).

**Why it fails against the construction:**

This does not constitute an attack on the deployed protocol — assuming honest ceremony, the construction is likely sound. The issue is that the claim "For any PPT adversary $\mathcal{A}$, $\text{Adv}^{\text{IND-UNL-AS}}_{\mathcal{A}} \leq ...$" overstates the proof. The bound is informally argued, not formally established. The simulation step is where the argument breaks: if $\mathcal{B}$ cannot simulate proofs indistinguishably, the hybrid is not valid.

**In-threat-model?**

**Partially.** Not a direct deployment attack. But a construction claiming a "concrete reduction" (§3.2) to named assumptions must close this gap to be publication-ready. The current reduction sketch is at best a proof sketch requiring the AGM + PROM + trapdoor-simulation model, none of which are stated.

### Attack 4: `scopeBlindingSecret` Is a Persistent Master Key — No Forward Secrecy, Rotation Requires Trusting the Threat

**Relevant sections:** §2.4 (Blinding secret lifecycle), §3.1 (adversary does NOT control local environment).

**Attack:**

The entire cross-scope unlinkability guarantee is keyed on a single 254-bit value `scopeBlindingSecret = s_b`, sampled once at credential creation and reused indefinitely. `scopedNullifier_i = Poseidon2(scopeId_i, s_b)` for every scope `i`, at every epoch, for the entire credential lifetime. If `s_b` is leaked at any time $T_0$, the adversary evaluates `Poseidon2(scopeId_i, s_b)` for every observed scope, recovers all past nullifiers back to credential creation time, and reconstructs the complete historical traffic graph. The construction's statement "unlinkability holds across time" has no temporal bound — yet it admits a single stored secret as a point of total retroactive failure.

The rotation mechanism (§2.4: "generate a new credential with a fresh `scopeBlindingSecret`") requires the *operator* to co-sign a new credential. In the primary threat scenario — the AS is the operator — re-enrollment requires the agent to trust the exact entity that constitutes its threat model to cooperate with recovery. This is not a procedural inconvenience; it is a structural property: an agent that has been compromised cannot independently restore its unlinkability guarantees without the adversary's participation.

Furthermore, the IND-UNL-AS game (§3.2) models a static experiment: blinding secrets are fixed throughout setup, query, and challenge phases. There is no corruption oracle, no key-update phase, no forward-secrecy game (as in ACCE or post-compromise security models). The game therefore says nothing about the security of transcripts observed before or after a local storage compromise. The claim "unlinkability holds across time" is unqualified with respect to this window.

**Why it fails against the construction:**

The construction explicitly scopes out `s_b` compromise: "If the adversary compromises the agent's local storage, all unlinkability guarantees are void." This is an acknowledged trust assumption. However, the stated claim in §1 — "unlinkability holds across scopes, across time" — is not restricted to "while local storage is uncompromised." The claim is weaker than stated. The rotation-requires-operator-trust issue is unaddressed anywhere in the construction and is the sharpest contradiction: the threat model explicitly models the operator as the adversary (§3.1, "The AS may be the same entity as the operator"), yet the only recovery path from `s_b` compromise routes through the operator.

**In-threat-model?**

**Partially no.** The static game does not model this. A hardened security statement would either: (1) define a forward-secrecy game with a corruption oracle and prove that unlinkability holds for transcripts preceding the corruption epoch; or (2) introduce a key-update mechanism (`s_b` is rotated per epoch without re-enrollment, using a key-derivation ratchet) so that past epochs remain unlinkable after future compromise. Neither is currently specified.

### Attack 1: `epochId` as a Public Population-Reduction Oracle

- **Attack:** `epochId` is a **public input** to `ScopedAgentAuth` (§2.2), bucketing all proofs into 300-second windows. Colluding RSes (permitted by the threat model, §3.1) share their received proof transcripts. Each transcript entry includes `(agentMerkleRoot, scopedNullifier, epochBinding, epochId, ...)`. Two colluding RSes — RS-A (Amazon) and RS-B (Costco) — intersect their "proofs received in epoch $E$" sets. The intersection is indexed by `(agentMerkleRoot, epochId)`. In a small-population deployment (e.g., 100 NFCU member agents), the set of agents that touched *both* RS-A and RS-B within a single 5-minute epoch is plausibly a set of 1. The adversary wins not by inverting the Poseidon PRF, but by set intersection over a population that the IND-UNL-AS game doesn't bound. The construction (§3.5) notes "Agent MAY add random delay within epoch" — the word MAY is load-bearing, and this constraint is **not enforced by any circuit signal**. A circuit cannot enforce a wall-clock delay.

- **Why it works / why it fails:** The IND-UNL-AS game (§3.2) gives the adversary all public signals including `epochId`. The game asks: can the adversary distinguish "same agent at two scopes" from "different agents at two scopes"? If the population is large and epochs are densely populated, the epoch signal carries minimal information. But the game makes no population-size assumption. In a NFCU deployment of 100 agents where each agent accesses at most 3 merchants per day, the epoch intersection is often unique. This isn't a break of Poseidon PRF — it's a traffic analysis attack that the formal reduction in §4.2 doesn't account for because the hybrid argument only bounds PRF-breaking, not population-inference.

- **In-threat-model?** **Yes — construction must address.** The adversary in §3.1 controls a coalition of RSes and observes on-chain state. Set intersection over public `epochId` values is within this capability. The construction must either (a) bound the epoch-level de-anonymization probability as a function of population size, or (b) enforce epoch-decoupling at the circuit level (e.g., requiring agents to add a deterministically-computed per-scope delay that is provably bounded in circuit). The current "MAY add random delay" is not a cryptographic mitigation.

### Attack 2: `scopeCommitment` Is a Stable Cross-Session Fingerprint for Delegating Agents

- **Attack:** For agents that use delegation, the `DelegationEntry` circuit (§2.3) outputs `scopeCommitment = Poseidon2(permissionBitmask, credentialCommitment)` as a **public output stored on-chain**, indexed by `sessionNonce` in the `lastScopeCommitment` mapping. Critically, `credentialCommitment = Poseidon5(modelHash, opPubAx, opPubAy, permissionBitmask, expiryTimestamp)` — this value is **identical for every delegation chain this agent initiates**, across all sessions, until credential rotation. NFCU (adversarial AS/operator) controls on-chain state observation (§3.1). It can enumerate all `DelegationEntry` events and cluster them by `scopeCommitment`. Every delegation chain from agent #4821 produces the same `scopeCommitment`. NFCU observes: "This `scopeCommitment` appeared on-chain 23 times over 6 months — meaning this agent initiated 23 delegation chains." The agent's delegation frequency, delegation timing, and even delegation session correlation (since `sessionNonce` is also on-chain) are fully visible. The construction's §3.3 only argues that `scopeCommitment` doesn't link to RS-facing nullifiers — it never argues that `scopeCommitment` doesn't link *to itself* across sessions.

- **Why it works / why it fails:** The construction explicitly acknowledges (§2.3, §3.3) that `scopeCommitment` is on-chain visible. It claims: "it reveals only that the agent is capable of delegating, not where it authorizes." This is literally true but misleading. The adversary doesn't need to know *where* — it can reconstruct *whether, when, and how often* the agent delegates by fingerprinting the stable `scopeCommitment`. In the healthcare scenario (§7), Kaiser observes the patient agent's `scopeCommitment` appearing 8 times in 6 months — inferring 8 referral or specialist delegation events, reconstructing a coarse referral graph even without knowing *which* providers were involved. The construction's §3.3 argument that "the adversary cannot link `scopeCommitment` to RS-facing `scopedNullifier`" is correct but irrelevant to this attack.

- **In-threat-model?** **Yes — construction must address.** The adversary controls on-chain observation (§3.1). The IND-UNL-AS game's claim is "cryptographically unlinkable authorizations even under adversarial AS." A stable on-chain fingerprint that reveals delegation frequency and session count is a partial de-anonymization. The construction should either (a) randomize `scopeCommitment` per delegation-chain initiation using an additional blinding factor (analogous to how `scopeBlindingSecret` randomizes nullifiers), or (b) explicitly scope the unlinkability claim to exclude delegation flows and document this as a known limitation.

### Attack 3: The "Agent's Local Execution Environment" Trust Assumption Is Vacuous in the Primary Threat Scenario

- **Attack:** The entire IND-UNL-AS security argument rests on one boundary condition stated in §3.1: "The adversary does NOT control: The agent's local execution environment." The construction explicitly models NFCU as simultaneously the operator (holding `operatorPrivKey`) and the adversarial AS. But in every realistic AI agent deployment — Lambda functions, containerized microservices, managed Kubernetes pods — the **operator controls the execution environment**. NFCU doesn't just sign the agent credential; NFCU runs the agent on AWS infrastructure provisioned by NFCU. The agent process, its CSPRNG state, its filesystem (where `scopeBlindingSecret` is stored per §2.4), and its memory are all within NFCU's control plane. The `scopeBlindingSecret` that §2.4 says "never leaves the agent's local environment" is stored in an environment NFCU administers. I don't need to break Poseidon. I read `/var/agent/secrets/scope_blinding.bin` from my own Lambda function's filesystem.

- **Why it works / why it fails:** The construction pre-empts this (§3.1): "This is explicitly documented as the trust assumption, not a limitation — it mirrors the standard assumption that a user's secret key is not leaked to the adversary." The analogy to a user's secret key is the weakest part of this defense. When a *user* holds a secret key, the user's device is typically outside the issuer's control plane — the user bought a phone, the bank didn't manufacture it. When an *AI agent* holds a `scopeBlindingSecret`, the agent runs on infrastructure that is overwhelmingly likely to be operated by the same entity that issued the credential. The trust model inverts the realistic deployment topology. The RFC 7662 baseline does not require this inversion — it makes no assumption about who operates the client software. The construction's primary differentiator (§7: "NFCU cannot compute the mapping between its known credentials and the RS-facing nullifiers") is only true if NFCU doesn't control the agent's runtime — an assumption that fails in the exact scenario described.

- **In-threat-model?** **No — construction must address.** This is not a cryptographic break; the IND-UNL-AS game is well-formed on its own terms. But the construction markets itself against the RFC 7662 baseline using a threat scenario (§7: NFCU as both operator and AS) in which the construction's core security assumption is typically violated. The construction should (a) explicitly characterize which agent deployment architectures preserve the trust boundary (TEE-based execution, user-device agents, hardware-enclaved agents), (b) compare these deployment models against what RFC 7662 baseline requires, and (c) not present the NFCU scenario as illustrative of the construction's strength when NFCU-operated agents are exactly the case where it fails.

### Attack 4: JWT Introspection Response + DPoP Collapses the "AS on Hot Path" Differentiation

- **Attack:** Table 8 (§8) states "AS must be contacted for every token issuance. JWT introspection response is optional and still requires initial AS contact." This misrepresents `draft-ietf-oauth-jwt-introspection-response`. With signed JWT introspection responses: (1) the AS issues a signed JWT bearing the introspection result, (2) the RS validates the JWT locally using the AS's public key — **no AS contact at verification time**. Combined with RFC 9449 DPoP (sender constraint bound to the client's proof-of-possession key), RFC 8707 (audience binding to specific RS resource indicators), and OIDC PPID (pairwise `sub` so RS-A and RS-B see different subject identifiers for the same user), the baseline AS is contacted **once per token issuance**, not once per RS request. If tokens are issued with 1-hour lifetimes and the agent caches them, the AS sees: "Agent #4821 requested `financial_small` for the Amazon RS and Costco RS at 14:00." This is a coarser view than per-request — not eliminated, but significantly reduced. The construction's claim that "AS removal from the hot path" is a unique property of Bolyra is overstated relative to this baseline.

- **Why it works / why it fails:** The construction's real advantage is not that the AS is contacted less frequently — it's that the AS is *never contacted at all* after enrollment (§2.6). The JWT introspection baseline still requires the AS at issuance time, and the AS's issuance log reveals which (agent, scope, RS) tuples were requested. In the IND-UNL-AS threat model, an adversarial AS that issues tokens trivially logs traffic graphs from issuance records, regardless of whether RSes contact it later. But the construction's §8 table oversimplifies by describing the baseline as "AS must be contacted for every token issuance" — which elides the distinction between issuance-time contact and per-request contact. A more accurate table entry would state: "Baseline AS-contact is per-issuance (not per-request with JWT introspection), but issuance-time logging gives the adversarial AS full scope × agent × time data, whereas Bolyra exposes nothing post-enrollment." The weaker framing is technically defensible; the current framing invites a well-deserved RFC citation in peer review.

- **In-threat-model?** **No — construction survives**, but §8 must be amended. The formal IND-UNL-AS argument is unaffected by this attack — a baseline AS that logs issuance requests wins the game trivially, confirming the construction's advantage. The issue is presentational accuracy: the construction should replace "AS must be contacted for every token issuance" with "AS observes full (scope × agent × RS) tuple at issuance time regardless of JWT caching, eliminating any unlinkability against adversarial AS" — this is both accurate and stronger than the current wording.

### Attack 3: Per-Scope HKDF Achieves IND-UNL-AS Without 17K Constraints — Wrong Layer

**Attack:** Strip out the ZK claim and examine what cryptographic work the construction actually does. The unlinkability reduction (§4.2) says:

> If $\mathcal{A}$ wins IND-UNL-AS with non-negligible advantage, then $\mathcal{B}$ breaks Poseidon PRF security.

The Poseidon PRF security of `scopedNullifier = Poseidon2(scopeId, scopeBlindingSecret)` is exactly the security of HKDF with input key material `scopeBlindingSecret` and info `scopeId`. HKDF is proven in the standard model under HMAC-SHA-256 assumptions that are better understood than Poseidon. The nullifier construction is a PRF evaluation. The ZK proof wraps around this PRF evaluation and adds:

1. Merkle membership proof that the credential is enrolled.
2. EdDSA signature verification from the operator.
3. Permission bitmask range checks.

A SPIFFE/WIMSE architecture delivers (1) and (2) via SVID issuance: the SPIRE server verifies node attestation, issues an X.509 SVID containing the credential fields (modelHash, permBitmask, expiry) as extensions, signed by the SPIRE CA. The RS verifies the SVID chain against the SPIFFE trust bundle (analogous to `agentMerkleRoot` for credential membership). The agent proves scope eligibility via SVID attributes.

For unlinkability, the SPIRE Workload API delivers a per-agent master secret to the agent just-in-time (no at-rest storage). The agent derives per-scope ephemeral keys: `scopedKey = HKDF(masterSecret, "bolyra-scope", scopeId)`. The agent sends `(SVID, MAC(scopedKey, request))` to the RS. The RS verifies the SVID and the MAC. The AS never sees which RS the agent contacts. Collusion resistance among RSes is bounded by HKDF PRF security — identical to the Poseidon PRF reduction in §4.2. The operator cannot recompute `masterSecret` because it is delivered via the Workload API and never persisted.

Total constraint count: 0. Trusted setup: not required. Proof generation time: <1ms (MAC). Verifier complexity: HMAC + X.509 chain verification.

**Why it works / fails:** The SPIFFE engineer is correct that the PRF unlinkability property does not require ZK. The ZK proof in `ScopedAgentAuth` serves two roles: (a) proving Merkle membership and operator authorization without revealing the credential, and (b) binding the PRF evaluation to the credential. Role (a) is the genuine ZK contribution — it hides `credentialCommitment` from the RS while proving membership. Role (b) is just constraint 9 (`blindingCommitment = Poseidon2(scopeBlindingSecret, credentialCommitment)`) which the HKDF approach handles by making `masterSecret` an SVID-attested value. The construction must defend why hiding `credentialCommitment` from the RS is necessary and achievable, given that the SVID would reveal credential fields (or their commitments) to the RS in the SPIFFE model.

The actual gap ZK closes: the SVID approach reveals `credentialCommitment` (or equivalent) to the RS. The RS then becomes a potential correlation point — two RSes can compare credential hashes. The `ScopedAgentAuth` circuit hides `credentialCommitment` entirely, with only the blinded nullifier appearing in the public outputs. This is a real ZK contribution the HKDF approach does not match.

**In-threat-model?** **Yes — the construction survives this attack**, but incompletely. The construction must add an explicit section explaining why hiding `credentialCommitment` from RSes is in scope (it's not currently stated as a goal — only AS-unlinkability is formalized). If the RS-privacy goal is dropped, the HKDF approach is a simpler replacement. The construction currently does not argue for RS-credential-privacy, making the ZK overhead appear unjustified to a SPIFFE engineer auditing the threat model.

### Attack 4: The IND-UNL-AS Query Restriction Is Too Weak — Adaptive Nullifier Precomputation

**Attack:** The IND-UNL-AS game (§3.2, step 2) restricts the adversary from querying the challenge (agent, scope) pair before the challenge. But the adversary controls the AS and holds all operator private keys. In a real deployment:

1. The adversary (NFCU-as-AS) can enumerate all possible `scopeId` values — they are public RS identifiers (e.g., `Poseidon("CU-A:merchant-read")`), published by RSes on-chain or in well-known endpoints (§2.6, step 1). The set of RSes an agent might contact is finite and enumerable from public on-chain state.

2. The adversary cannot precompute `scopedNullifier = Poseidon2(scopeId, scopeBlindingSecret)` without knowing `scopeBlindingSecret`. But the adversary CAN, after observing the challenge proof $\pi_a$ with public output `scopedNullifier_a`, mount an offline test: for each enrolled agent $j$, if the adversary could query agent $j$ at ANY scope $S_i \neq S_a$ during the query phase, it receives `Poseidon2(S_i, s^j_b)`. It then tests whether `Poseidon2(S_a, s^j_b)` matches `scopedNullifier_a` — but it cannot evaluate this without $s^j_b$.

   However, consider the epoch binding: `epochBinding = Poseidon2(scopedNullifier, epochId)`. The adversary observes `epochBinding_a` in $\pi_a$. From the query phase, the adversary holds `scopedNullifier_q = Poseidon2(S_q, s^j_b)` for some query scope $S_q$ and the same epoch. It can compute `epochBinding_q = Poseidon2(scopedNullifier_q, epochId)`. If `epochBinding_a = epochBinding_q`, the adversary wins — but this requires `scopedNullifier_a = scopedNullifier_q`, which means $S_a = S_q$ and same agent — but $S_a \neq S_q$ by assumption. No direct collision.

   The real attack: for large deployments with many agents and sparse scope coverage, the `agentMerkleRoot` is the same for all agents. An adversary who receives `scopedNullifier_a` at scope $S_a$ and has query transcripts `{(scopeId_i, scopedNullifier_i) : agent j, many scopes i}` can build a candidate set. Across epochs, the `epochBinding` changes but `scopedNullifier` is stable per (agent, scope). If the adversary can induce the agent to authorize at a scope they did NOT query during the pre-challenge phase — by controlling an RS and presenting a fresh `scopeId` that is a one-way transform of a known value — they can expand the query transcript adaptively post-challenge.

**Why it works / fails:** The query restriction in §3.2 prevents direct precomputation for challenge (agent, scope) pairs. The construction is correct that no cross-scope stable identifier appears in the public outputs. However, WIMSE's workload attestation model highlights a practical gap: the set of `scopeId` values an agent might use is small and enumerable in production (NFCU's members interact with a finite merchant set). The anonymity set shrinks to agents with matching permission bitmasks who have previously authorized at the non-challenge scopes. If NFCU issues 1,000 agent credentials with `FINANCIAL_SMALL` and the member's agent contacts 5 specific merchants, NFCU's merchant list is 50 large — the anonymity set may be small enough for traffic analysis even without breaking the PRF.

The construction's epoch bucketing (§2.5, §3.5) addresses *timing* side channels but not *scope graph* side channels from a small universe of RSes. The formal game assumes the adversary cannot correlate across scopes, but in practice the scope universe is public and finite. This does not break IND-UNL-AS formally (the game is sound), but it means the practical unlinkability guarantee degrades in deployments with small RS populations — a concern a WIMSE co-author who has run federation across 40 trust domains would immediately raise.

**In-threat-model?** **Partially yes.** The IND-UNL-AS game is correctly formalized for the stated attack. The construction survives the formal attack. But it must address the practical anonymity-set concern: with a small, enumerable scope universe, the PRF security bound is tight but the concrete security is weak because the effective anonymity set collapses from $N$ agents to the few agents with matching permissions who have visited overlapping RS subsets. This is an empirical gap, not a formal one — the §5 scenarios need a concrete anonymity set analysis, not just a PRF reduction.
