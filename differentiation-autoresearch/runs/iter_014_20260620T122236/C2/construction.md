# Construction

## 1. Statement of claim

The same Bolyra agent accessing *k* distinct Resource Servers produces *k* authorization proofs that are computationally indistinguishable from proofs generated by *k* independent agents, even to an adversarial Authorization Server (AS) that controls the agent Merkle tree, observes all proof submissions, and colludes with up to *k−1* of the *k* RSes. Cross-scope unlinkability holds under the Poseidon PRF assumption on BN254 and the knowledge soundness of Groth16/PLONK.

## 2. Construction (gadgets, circuits, public/private inputs)

### New circuit: `ScopeBlindedAgent(MAX_DEPTH)`

This circuit extends `AgentPolicy` with a scope-blinding layer. The agent proves credential validity and scope satisfaction while producing a **scope-specific pseudonym** that is unlinkable across scopes.

**Private inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `modelHash` | F_p | Hash of model identifier |
| `operatorPubkeyAx`, `operatorPubkeyAy` | F_p | Operator EdDSA public key |
| `permissionBitmask` | 64-bit | Permission bitfield |
| `expiryTimestamp` | 64-bit | Credential expiration |
| `sigR8x`, `sigR8y`, `sigS` | F_p | Operator EdDSA signature |
| `merkleProofLength`, `merkleProofIndex`, `merkleProofSiblings[MAX_DEPTH]` | — | Merkle inclusion proof |
| `blindingScalar` | F_p | Fresh random blinding value per proof |

**Public inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `scopeId` | F_p | RS-specific scope identifier (known to agent + RS, NOT to AS) |
| `requiredScopeMask` | 64-bit | Required permission bits |
| `currentTimestamp` | 64-bit | Current time |
| `sessionNonce` | F_p | Fresh per-session nonce |

**Public outputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `agentMerkleRoot` | F_p | Computed Merkle root |
| `scopePseudonym` | F_p | `Poseidon2(credentialCommitment, scopeId)` — unlinkable across scopes |
| `blindedScopeCommitment` | F_p | `Poseidon3(permissionBitmask, credentialCommitment, blindingScalar)` — rerandomized per proof |
| `sessionNullifier` | F_p | `Poseidon3(scopePseudonym, sessionNonce, blindingScalar)` — replay prevention per session×scope |

**Circuit constraints (10 steps):**

1. **Range checks:** `Num2Bits(64)` on `permissionBitmask`, `expiryTimestamp`, `currentTimestamp`.
2. **Credential commitment:** `credentialCommitment = Poseidon5(modelHash, operatorPubkeyAx, operatorPubkeyAy, permissionBitmask, expiryTimestamp)`.
3. **EdDSA signature:** `EdDSAPoseidonVerifier(operatorPubkey, credentialCommitment, sig)`.
4. **Merkle membership:** `BinaryMerkleRoot(MAX_DEPTH, credentialCommitment, proof) == agentMerkleRoot`.
5. **Scope satisfaction:** For each bit *i* in [0, 64): `requiredBits[i] * (1 - permBits[i]) === 0`.
6. **Cumulative bit encoding:** Standard Bolyra tier implication constraints on bits 2–4.
7. **Expiry check:** `currentTimestamp < expiryTimestamp` via `LessThan(64)`.
8. **Scope pseudonym:** `scopePseudonym = Poseidon2(credentialCommitment, scopeId)`.
9. **Blinded scope commitment:** `blindedScopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, blindingScalar)`.
10. **Session nullifier:** `sessionNullifier = Poseidon3(scopePseudonym, sessionNonce, blindingScalar)`.

**Constraint estimate:** ~12,400 constraints (AgentPolicy baseline ~9,800 + 3 additional Poseidon hashes at ~800 each + Num2Bits for no additional range checks beyond existing). Fits within `pot16.ptau` (2^16).

### Protocol flow (scope-blinded handshake)

1. RS generates `(sessionNonce, scopeId)` where `scopeId = Poseidon2(RS_domain, RS_public_key)`. The RS sends both to the agent over a direct channel. The AS never sees `scopeId`.

2. Agent generates `ScopeBlindedAgent` proof with fresh `blindingScalar ← random(F_p)`.

3. Agent submits proof to the on-chain registry. The registry sees `(agentMerkleRoot, scopePseudonym, blindedScopeCommitment, sessionNullifier, scopeId, requiredScopeMask, currentTimestamp, sessionNonce)`.

4. Registry verifies:
   - `sessionNonce` freshness (not reused).
   - `agentMerkleRoot` in root history buffer.
   - Groth16/PLONK proof validity.
   - `sessionNullifier` not previously recorded (replay prevention).

5. RS verifies that the `scopeId` public input matches its own computed scope identity.

### Batched submission (timing side-channel mitigation)

To prevent the AS from correlating proof submissions by timing, agents MAY use a **batch relay**:
- A relay collects proofs from multiple agents over a mixing window *Δt* (e.g., 30 seconds).
- The relay shuffles and submits proofs in a single multicall transaction.
- The AS sees *n* proofs arriving simultaneously with no per-agent timing signal.

This is an application-layer defense; the cryptographic unlinkability holds without it.

## 3. Threat model (adversary capabilities, game definition)

### Adversary capabilities

The adversary **A** controls:
- The Authorization Server (AS): full read access to the agent Merkle tree, all proof submissions, all public signals.
- Up to *k−1* of *k* target RSes: A learns the `scopeId`, `requiredScopeMask`, and proof for those RSes.
- Timing of proof submissions (unless batch relay is used).

The adversary **A** does NOT control:
- The agent's private key or `blindingScalar`.
- The ZK proving process (no malicious prover).
- At least one RS that remains honest (the one whose traffic A tries to link).

### IND-UNL-AS game

**Game IND-UNL-AS(λ):**

1. **Setup:** Challenger generates two agents `(cred₀, cred₁)` with identical permission bitmasks and enrolls both in the Merkle tree. Challenger generates *k* RS scope identifiers `{scopeId_1, …, scopeId_k}`.

2. **Challenge:** Challenger flips coin `b ← {0, 1}`. For each `scopeId_j` (j = 1…k), challenger generates a valid `ScopeBlindedAgent` proof using agent `cred_b` with fresh `blindingScalar_j` and `sessionNonce_j`.

3. **Oracle access:** A may:
   - Request proofs for any `scopeId` of A's choosing for either agent (adaptive queries, up to poly(λ) queries).
   - Observe all public signals from all proofs.
   - Corrupt up to *k−1* RSes, learning their `scopeId` values.

4. **Restriction:** A may NOT query both agents on the same `scopeId` as the challenge scope (trivial distinguisher via Merkle root + known tree).

5. **A outputs** guess `b'`.

**Advantage:** `Adv_A = |Pr[b' = b] - 1/2|`.

**Claim:** For all PPT adversaries A, `Adv_A ≤ negl(λ)` under the Poseidon PRF assumption on BN254 and knowledge soundness of Groth16/PLONK.

## 4. Security argument (named assumption + reduction sketch)

### Named assumptions

- **A1 — Poseidon PRF:** `Poseidon2(·, ·)` is a pseudorandom function when keyed on its first argument over the BN254 scalar field. Specifically, `f_k(x) = Poseidon2(k, x)` is computationally indistinguishable from a random function for any fixed key `k` unknown to the adversary.

- **A2 — Knowledge soundness of Groth16 (ROM):** In the random oracle model with the BN254 pairing, Groth16 satisfies knowledge soundness: a valid proof implies the prover knows a satisfying witness.

- **A3 — Discrete log hardness on Baby Jubjub:** Given `(Ax, Ay) = BabyPbk(s)`, recovering `s` is computationally intractable.

### Reduction sketch

**Theorem:** If there exists a PPT adversary A winning IND-UNL-AS with non-negligible advantage, then there exists a PPT distinguisher D breaking the Poseidon PRF assumption.

**Proof sketch:**

1. **Scope pseudonym unlinkability.** The scope pseudonym is `Poseidon2(credentialCommitment, scopeId)`. For two different scopes `scopeId_i ≠ scopeId_j`, the pseudonyms are `Poseidon2(cc, scopeId_i)` and `Poseidon2(cc, scopeId_j)`. Under A1, keying Poseidon on `cc` (which is hidden as a private input), the outputs are indistinguishable from random. D simulates IND-UNL-AS for A: when A distinguishes two pseudonyms as coming from the same `cc`, D uses this to distinguish Poseidon from random, contradicting A1.

2. **Blinded scope commitment unlinkability.** The blinded scope commitment includes a fresh `blindingScalar` per proof. Even if `permissionBitmask` and `credentialCommitment` are identical across proofs, the output `Poseidon3(permissionBitmask, cc, blindingScalar)` is fresh each time. Under A1, this is indistinguishable from random for each new `blindingScalar`.

3. **Session nullifier unlinkability.** The session nullifier `Poseidon3(scopePseudonym, sessionNonce, blindingScalar)` depends on the already-unlinkable `scopePseudonym` and a fresh `blindingScalar`. Composing two PRF applications preserves pseudorandomness under A1.

4. **Witness hiding via ZK.** By the zero-knowledge property of Groth16/PLONK (A2), the proof transcript reveals nothing about private inputs (`credentialCommitment`, `blindingScalar`, operator key, Merkle path). The adversary's view is limited to public signals, which are unlinkable per (1)–(3).

5. **Collusion resistance (k−1 RSes).** Corrupting RS_j reveals `scopeId_j` to A. A can then compute `Poseidon2(cc_guess, scopeId_j)` for candidate credential commitments. But `cc` is never revealed (it is a private input), and the Merkle tree contains *N* enrolled agents. A must guess `cc` from the tree of size *N*, each guess requiring a Poseidon evaluation. For the honest RS (RS_h), A does not know `scopeId_h` and cannot test candidates. The advantage reduces to `1/N` per corrupted RS, which is negligible for `N = poly(λ)`.

**QED (sketch).**

### Side-channel treatment

- **Timing:** Batch relay (§2) ensures proof submission times are shuffled within a mixing window. Without batch relay, timing correlation is possible but does not break cryptographic unlinkability — it is a traffic-analysis attack mitigated at the network layer.
- **Nonce freshness:** Each `sessionNonce` is unique per handshake. The `sessionNullifier` binds the proof to a specific `(scopePseudonym, sessionNonce, blindingScalar)` triple. Replaying a proof with a stale nonce fails the freshness check.
- **Merkle root correlation:** If the agent tree is small, the AS could narrow the anonymity set by observing which Merkle root is used. The 30-entry root history buffer and the shared tree across all agents mitigate this — every agent references the same tree.

## 5. Bolyra primitive mapping

| Construction element | Bolyra primitive | Source |
|---|---|---|
| Credential commitment | `Poseidon5(modelHash, Ax, Ay, permissionBitmask, expiryTimestamp)` | AgentPolicy circuit (spec §4.2) |
| Scope pseudonym | `Poseidon2(credentialCommitment, scopeId)` | NEW — analogous to human nullifier `Poseidon2(scope, secret)` |
| Blinded scope commitment | `Poseidon3(permissionBitmask, credentialCommitment, blindingScalar)` | NEW — rerandomized variant of `scopeCommitment` |
| Session nullifier | `Poseidon3(scopePseudonym, sessionNonce, blindingScalar)` | NEW — extends agent nullifier pattern |
| Operator signature | EdDSA on Baby Jubjub over `credentialCommitment` | AgentPolicy circuit (spec §4.2) |
| Merkle membership | Lean Incremental Merkle Tree, depth 20, Poseidon2 node hash | Shared agent tree (spec §3.1) |
| Scope satisfaction | Bitwise AND check: `requiredBits[i] * (1 - permBits[i]) === 0` | AgentPolicy circuit (spec §4.2) |
| Cumulative encoding | Tier implication constraints on bits 2–4 | Permissions model (CLAUDE.md) |
| Proving system | PLONK (universal setup, no per-circuit ceremony) for agent circuits | Spec §3.3 (OPTIONAL for agent) |
| Delegation chain entry | `blindedScopeCommitment` replaces `scopeCommitment` as chain seed | Delegation circuit (spec §5) — requires adapter |

**Delegation compatibility:** The existing Delegation circuit links via `previousScopeCommitment = Poseidon2(delegatorScope, delegatorCredCommitment)`. To chain from a `ScopeBlindedAgent` proof, the delegator must reveal `credentialCommitment` to the delegation circuit as a private input and prove consistency with the `blindedScopeCommitment`. This requires a one-constraint adapter: `Poseidon3(delegatorScope, delegatorCredCommitment, blindingScalar) == blindedScopeCommitment` (blinding scalar also private to the delegator). The delegation circuit then proceeds with standard chain linking.

- Verification: **< 250ms** on-chain (single pairing check for Groth16; slightly more for PLONK).

## 7. Concrete deployment scenario

### Credit union member agent — merchant graph privacy

**Stakeholder:** Desert Financial Credit Union (DFCU), a $9B credit union in Arizona with 400K members.

**Setup:** DFCU operates an Authorization Server for its members' AI agents. Members deploy agents to interact with merchants (Target, Costco, local pharmacies) for price comparison, coupon redemption, and loyalty point management. Each merchant operates a Resource Server.

**Problem under baseline:** DFCU-as-AS sees every token issuance: "Member #4821's agent requested `merchant-read` scope for Target at 14:02, then `pharmacy-read` scope for CVS at 14:07." DFCU reconstructs the member's complete merchant graph. Under NCUA Regulation §748 (member data protection), this is a compliance liability. Under the CFPB's proposed open-banking rules (§1033), the AS must not use authorization metadata to build behavioral profiles.

**Bolyra deployment:**

1. DFCU enrolls member agents in the Bolyra agent Merkle tree. Each agent has an operator-signed credential with `permissionBitmask = 0b00000101` (READ_DATA + FINANCIAL_SMALL).

2. Target's RS has `scopeId_Target = Poseidon2("target.com", Target_RS_pubkey)`. CVS has `scopeId_CVS = Poseidon2("cvs.com", CVS_RS_pubkey)`. These are computed locally by each RS and shared with the agent over a direct TLS channel.

3. Member's agent generates `ScopeBlindedAgent` proof for Target:
   - `scopePseudonym_Target = Poseidon2(credCommitment, scopeId_Target)` — visible on-chain.
   - `blindedScopeCommitment_Target = Poseidon3(0b00000101, credCommitment, blinding₁)` — unique to this proof.

4. Same agent generates `ScopeBlindedAgent` proof for CVS:
   - `scopePseudonym_CVS = Poseidon2(credCommitment, scopeId_CVS)` — different from Target's pseudonym.
   - `blindedScopeCommitment_CVS = Poseidon3(0b00000101, credCommitment, blinding₂)` — fresh blinding.

5. DFCU (AS) observes two proofs on-chain. Both reference the same `agentMerkleRoot` (shared tree with 400K agents). The `scopePseudonym` values differ. The `blindedScopeCommitment` values differ. DFCU cannot determine whether these are the same agent or two different agents.

6. Target verifies: `scopeId` in the proof matches `scopeId_Target`. Target sees a valid authorization with satisfied scope. Target does NOT learn anything about the agent's CVS activity.

**Quantified privacy:** DFCU's advantage in IND-UNL-AS is bounded by `negl(λ)` under Poseidon PRF. The anonymity set is the full agent tree (400K members). Even if DFCU colludes with Target (corrupting 1 of 2 RSes), it cannot link the CVS proof to the Target proof because `scopeId_CVS` is unknown to DFCU.

### Healthcare delegation — referral network privacy

**Stakeholder:** Kaiser Permanente (KP), a healthcare system with 12.6M members.

**Setup:** A primary care physician (PCP) delegates a scoped credential to a specialist referral agent. The agent interacts with Lab Corp (lab orders) and Walgreens (prescription fulfillment). KP operates the AS.

**Problem:** Under the baseline, KP sees the full referral chain: PCP → specialist → Lab Corp → Walgreens. This reveals the member's condition (specialist type), diagnostic path (lab orders), and treatment (prescriptions). HIPAA's Minimum Necessary Standard (45 CFR §164.502(b)) requires limiting data exposure to what is necessary for the specific purpose.

**Bolyra deployment:** The PCP's agent uses `ScopeBlindedAgent` with `permissionBitmask = 0b10000101` (READ_DATA + FINANCIAL_SMALL + ACCESS_PII). It delegates to the specialist agent via the Delegation circuit, narrowing to `0b00000101`. The specialist's agent generates scope-blinded proofs for Lab Corp and Walgreens with independent `scopeId` values. KP-as-AS cannot reconstruct the referral topology.

## 8. Why the baseline cannot match

| Capability | Baseline (PPID + RFC 8707 + DPoP + BBS+) | ScopeBlindedAgent |
|---|---|---|
| **AS sees token issuance per RS** | Yes — every token is AS-issued, AS logs `(agent, RS, scope, time)` | No — agent generates ZK proof locally; AS sees only unlinkable pseudonyms |
| **AS can correlate scope sequences** | Yes — RFC 8707 `resource` parameter visible at issuance | No — `scopeId` is a private input to the circuit; AS never learns which RS the proof targets |
| **Collusion (AS + k−1 RSes)** | Trivial correlation — AS holds the PPID mapping table | Bounded by `negl(λ)` — corrupted RSes reveal their `scopeId` but cannot link to honest RS's proof |
| **Issuer anonymity** | Absent — BBS+ exposes issuer public key | N/A — agent Merkle tree is shared; no issuer key in public signals |
| **Formal security definition** | None — no IND-UNL-AS game exists in OAuth/OIDC/BBS+ specs | IND-UNL-AS game defined (§3) with reduction to Poseidon PRF (§4) |
| **Timing side-channel** | No mitigation — DPoP `jti` and timestamps leak request timing to AS | Batch relay mixing window; cryptographic unlinkability holds even without it |
| **Delegation chain privacy from AS** | Absent — RFC 8693 every hop is AS-visible | Blinded scope commitment chains through Delegation circuit; AS sees only rerandomized outputs |
| **Rerandomization per proof** | Absent — same BBS+ credential, same issuer key | `blindingScalar` ensures each proof's public signals are fresh even for identical inputs |

**The structural impossibility:** The baseline requires the AS to issue every token. This is not a configuration flaw — it is the definition of OAuth 2.0. An AS that issues a token necessarily knows the requesting agent, the target RS, and the requested scope. No combination of PPIDs, DPoP, or BBS+ changes this because they operate *after* issuance. Bolyra's `ScopeBlindedAgent` eliminates the issuance step entirely: the agent proves its credential validity in zero knowledge without any AS interaction at proof time. The AS is reduced to a tree maintainer (enrollment) and cannot observe per-session authorization events. This is a categorical, not incremental, separation.
