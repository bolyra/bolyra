# Construction

## 1. Statement of claim

Same agent accessing different Resource Server scopes produces cryptographically unlinkable authorizations ‚ÄĒ even when the Authorization Server (AS) is adversarial and colludes with any subset of RSes to correlate per-agent traffic graphs. Formally: no PPT adversary controlling the AS and up to N‚ąí1 of N RSes can distinguish whether two authorization proofs at distinct scopes were generated by the same agent or two independent agents, with advantage better than negligible in the security parameter.

## 2. Construction (gadgets, circuits, public/private inputs)

### Core idea

Replace the single `nullifierHash = Poseidon2(credentialCommitment, sessionNonce)` in AgentPolicy with a **scope-bound nullifier** `Poseidon2(scopeId, agentSecret)` where `agentSecret` is a fresh per-agent secret scalar (distinct from the operator key) committed inside the credential. The AS never sees `scopeId` ‚ÄĒ it is hashed into the nullifier before leaving the circuit. Two authorizations at different scopes produce independent nullifiers that are unlinkable under the Poseidon PRF assumption.

### New circuit: `AgentPolicyScopeBound`

Replaces `AgentPolicy` for scope-unlinkable deployments. PLONK proving system (agent circuits use PLONK per Bolyra convention).

**Private inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `agentSecret` | F_p, range-checked 251 bits | Per-agent secret scalar on Baby Jubjub |
| `modelHash` | F_p | Hash of model identifier |
| `operatorPubkeyAx`, `operatorPubkeyAy` | F_p | Operator EdDSA public key |
| `permissionBitmask` | 64-bit | Permission bitfield |
| `expiryTimestamp` | 64-bit | Credential expiration |
| `sigR8x`, `sigR8y`, `sigS` | F_p | Operator EdDSA signature |
| `merkleProofLength`, `merkleProofIndex`, `merkleProofSiblings[20]` | - | Merkle inclusion proof |
| `blindingNonce` | F_p | Per-proof randomness for scope commitment blinding |

**Public inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `scopeId` | F_p | RS-specific scope identifier (e.g., `Poseidon("cu-merchant-payments")`) |
| `requiredScopeMask` | 64-bit | Policy requiring specific permission bits |
| `currentTimestamp` | 64-bit | Verifier-provided current time |
| `sessionNonce` | F_p | Fresh per-session nonce |

**Public outputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `agentMerkleRoot` | F_p | Computed Merkle root |
| `scopeNullifier` | F_p | `Poseidon2(scopeId, agentSecret)` ‚ÄĒ unlinkable across scopes |
| `blindedScopeCommitment` | F_p | `Poseidon3(permissionBitmask, credentialCommitment, blindingNonce)` |
| `nonceBinding` | F_p | `Poseidon2(scopeNullifier, sessionNonce)` ‚ÄĒ replay prevention |

**Constraints (in order):**

1. **Agent secret range**: `Num2Bits(251)` on `agentSecret` ‚ÄĒ ensures `agentSecret ‚ąą [0, 2^251)`.
2. **Agent identity derivation**: `(agentAx, agentAy) = BabyPbk(agentSecret)` ‚ÄĒ deterministic public key from secret.
3. **Extended credential commitment**: `credentialCommitment = Poseidon6(modelHash, operatorPubkeyAx, operatorPubkeyAy, permissionBitmask, expiryTimestamp, Poseidon2(agentAx, agentAy))` ‚ÄĒ binds the agent's identity commitment into the credential. The `Poseidon2(agentAx, agentAy)` sub-hash follows the same pattern as human identity commitments.
4. **EdDSA signature verification**: `EdDSAPoseidonVerifier(operatorPubkeyAx, operatorPubkeyAy, sigR8x, sigR8y, sigS, credentialCommitment)` ‚ÄĒ operator signed the credential.
5. **Merkle membership**: `BinaryMerkleRoot(20, credentialCommitment, merkleProofIndex, merkleProofSiblings) == agentMerkleRoot`.
6. **Range checks**: `Num2Bits(64)` on `permissionBitmask`, `expiryTimestamp`, `currentTimestamp`.
7. **Scope satisfaction**: For each bit `i ‚ąą [0, 64)`: `requiredBits[i] * (1 - permBits[i]) === 0`.
8. **Cumulative bit encoding**: Same three constraints as standard AgentPolicy (bits 4‚Üí3, 4‚Üí2, 3‚Üí2).
9. **Expiry**: `LessThan(64)(currentTimestamp, expiryTimestamp)`.
10. **Scope-bound nullifier**: `scopeNullifier = Poseidon2(scopeId, agentSecret)`.
11. **Nonce binding**: `nonceBinding = Poseidon2(scopeNullifier, sessionNonce)`.
12. **Blinded scope commitment**: `blindedScopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, blindingNonce)`.

### Modified handshake flow

1. RS (not AS) provides `scopeId` to the agent. The AS sees only the proof submission ‚ÄĒ it never learns the mapping between `scopeId` and the RS identity.
2. Agent generates `AgentPolicyScopeBound` proof locally.
3. Proof is submitted on-chain (or to a relayer). The AS can observe the transaction but sees only `(agentMerkleRoot, scopeNullifier, blindedScopeCommitment, nonceBinding, scopeId, requiredScopeMask, currentTimestamp, sessionNonce)`.
4. Sybil detection: within the same `scopeId`, duplicate `scopeNullifier` values indicate the same agent. Across different `scopeId` values, nullifiers are independent.

### Delegation extension

When `AgentPolicyScopeBound` feeds into the Delegation circuit, the chain-linking value uses `blindedScopeCommitment` instead of the standard `scopeCommitment`. The delegatee receives a fresh `blindingNonce` and produces a new `blindedScopeCommitment`. The delegation circuit additionally proves:

- `Poseidon3(delegatorScope, delegatorCredCommitment, delegatorBlindingNonce) == previousBlindedScopeCommitment`
- Scope subset (standard bitwise constraint)
- New output: `Poseidon3(delegateeScope, delegateeCredCommitment, delegateeBlindingNonce)`

This ensures each hop in a delegation chain is unlinkable to external observers.

## 3. Threat model (adversary capabilities, game definition)

### Adversary capabilities

The adversary A controls:
- The Authorization Server (AS) ‚ÄĒ full view of all token issuance requests, timing, metadata
- Up to N‚ąí1 of N Resource Servers ‚ÄĒ can collude to share received proofs
- Network-level observation of on-chain transactions (public blockchain)
- A cannot break DL on Baby Jubjub, Poseidon collision resistance, or knowledge soundness of PLONK

### IND-UNL-AS game

**Setup**: Challenger enrolls M agents with independent `agentSecret` values into the Merkle tree. Adversary A is given the AS's full state and the public keys of all RSes.

**Challenge phase**:
1. A selects two distinct scope identifiers `scopeId_0`, `scopeId_1` and a challenge bit `b ‚ąą {0, 1}`.
2. A selects two agents `agent_0`, `agent_1` from the enrolled set.
3. Challenger generates two proofs:
   - `ŌÄ_A` = AgentPolicyScopeBound proof for `agent_b` at `scopeId_0`
   - `ŌÄ_B` = AgentPolicyScopeBound proof for `agent_{1-b}` at `scopeId_1` (if `b=0`) or `agent_b` at `scopeId_1` (if `b=1`)
4. A receives `(ŌÄ_A, ŌÄ_B)` and all public signals.

**Win condition**: A outputs `b'`. A wins if `b' = b`.

**Advantage**: `Adv[A] = |Pr[b' = b] - 1/2|`

**Claim**: For all PPT adversaries A, `Adv[A] ‚Č§ negl(őĽ)` under Poseidon PRF and PLONK knowledge soundness assumptions.

### Side-channel threat model

- **Timing**: Adversary observes proof submission timestamps on-chain. Mitigation: agents batch proof submissions or use a relayer with random delay (application-layer, not circuit-enforced).
- **Nonce freshness**: `sessionNonce` is public but does not leak scope or agent identity ‚ÄĒ it binds proofs to sessions, not to agent identifiers.
- **Merkle root correlation**: If the agent tree is small, root changes may leak enrollment timing. Mitigation: the 30-entry root history buffer smooths this.

## 4. Security argument (named assumption + reduction sketch)

### Named assumptions

1. **Poseidon PRF**: Poseidon2 is a pseudorandom function over F_p. Specifically, for a random key `k`, the function `f_k(x) = Poseidon2(x, k)` is computationally indistinguishable from a random function.
2. **DL on Baby Jubjub**: Given `(Ax, Ay) = BabyPbk(s)`, finding `s` is hard.
3. **PLONK knowledge soundness**: From any accepting proof, an extractor can recover a valid witness.
4. **Poseidon collision resistance**: Finding `(x, y) ‚Č† (x', y')` with `Poseidon2(x, y) = Poseidon2(x', y')` is hard.

### Reduction sketch

**Theorem**: If Poseidon2 is a PRF, then no PPT adversary wins the IND-UNL-AS game with non-negligible advantage.

**Proof sketch**:

1. The only public values linking a proof to an agent identity are `(scopeNullifier, blindedScopeCommitment, nonceBinding, agentMerkleRoot)`.

2. `scopeNullifier = Poseidon2(scopeId, agentSecret)`. By the PRF assumption, for two different `scopeId` values, the outputs are computationally indistinguishable from independent random values ‚ÄĒ even if both proofs use the same `agentSecret`. This is the Poseidon PRF keyed on `agentSecret` evaluated at `scopeId`.

3. `blindedScopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, blindingNonce)`. Since `blindingNonce` is fresh per proof and private, this output is indistinguishable from random under the PRF assumption (keyed on `blindingNonce`).

4. `nonceBinding = Poseidon2(scopeNullifier, sessionNonce)`. Since `scopeNullifier` is already pseudo-random (from step 2), and `sessionNonce` is fresh, this adds no linkability.

5. `agentMerkleRoot` is shared across all agents in the tree ‚ÄĒ it identifies the tree state, not a specific agent.

6. Suppose adversary A wins IND-UNL-AS with advantage őĶ. Construct a PRF distinguisher B: B receives an oracle O that is either `Poseidon2(¬∑, k)` for random `k` or a truly random function. B embeds O as the nullifier computation for the challenge agent, using `O(scopeId_0)` and `O(scopeId_1)` as the two nullifiers. If O is a PRF, this perfectly simulates the game. If O is random, A's advantage is 0 (the nullifiers are independent random values). Therefore B's distinguishing advantage ‚Č• őĶ, contradicting the PRF assumption if őĶ is non-negligible. ‚Ė°

### Why `agentSecret` is necessary

The standard AgentPolicy circuit uses `nullifierHash = Poseidon2(credentialCommitment, sessionNonce)`. The `credentialCommitment` is deterministic per agent ‚ÄĒ it does not vary by scope. An adversarial AS that sees two proofs with different `scopeId` values can check whether the underlying `credentialCommitment` is the same by observing cross-session nullifier patterns (if two sessions share a nonce collision, or by colluding with RSes that reveal `scopeId`). The `agentSecret` construction eliminates this: the secret is never revealed, and the PRF hides which secret produced which nullifier.

## 7. Concrete deployment scenario

### Cross-credit-union member agent ‚ÄĒ merchant graph privacy

**Stakeholder**: Pacific Federal Credit Union (PFCU), a NCUA-insured credit union with 45,000 members.

**Setting**: PFCU operates as the Authorization Server for its members' AI agents. Members deploy agents that autonomously interact with merchant RSes (grocery stores, gas stations, subscription services) to execute payments, negotiate prices, and manage recurring billing.

**Problem under baseline**: PFCU-as-AS sees every token issuance: "Member #12847's agent requested a token for Costco at 14:02, then Chevron at 14:15, then Netflix at 20:30." PPID hides the member from Costco, but PFCU reconstructs the full merchant graph. Under NCUA Reg V (¬ß1016.4), the credit union may share this data with affiliates unless the member opts out ‚ÄĒ but the member cannot prevent PFCU from *collecting* it in the first place.

**Bolyra deployment**:

1. PFCU enrolls each member's agent into the Bolyra agent Merkle tree. The `agentSecret` is generated client-side and never leaves the member's device. The `credentialCommitment` (which embeds the `agentIdentityCommitment` derived from `agentSecret`) is the only value sent to PFCU for enrollment.

2. Each merchant RS registers a unique `scopeId` (e.g., `Poseidon("costco-payments-2026")`). The `scopeId` is provided directly by the RS to the agent ‚ÄĒ PFCU never sees it.

3. When the agent transacts with Costco, it generates an `AgentPolicyScopeBound` proof with `scopeId = Poseidon("costco-payments-2026")`. The resulting `scopeNullifier` is `Poseidon2(scopeId, agentSecret)`.

4. When the same agent transacts with Chevron, it generates a proof with `scopeId = Poseidon("chevron-fuel-2026")`. The resulting `scopeNullifier` is `Poseidon2(scopeId', agentSecret)` ‚ÄĒ a completely different value.

5. PFCU observes both on-chain transactions but sees two unrelated `scopeNullifier` values. Under the Poseidon PRF assumption, PFCU cannot determine whether these were produced by the same agent or two different members' agents.

6. Costco can detect repeat visits from the same agent (same `scopeNullifier` within `scopeId = costco-*`), enabling loyalty tracking. But Costco cannot link the agent to its Chevron activity.

**Regulatory alignment**: This satisfies NCUA Reg V ¬ß1016.10 (opt-out right) by making collection technically impossible ‚ÄĒ PFCU cannot build the graph even if it wanted to. It also aligns with the CFPB's 2025 open banking rule (¬ß1033) where data minimization is a design principle.

### Healthcare delegation ‚ÄĒ referral network privacy

**Stakeholder**: Ascension Health, a 140-hospital Catholic health system.

**Setting**: A primary care physician (PCP) delegates a scoped agent credential to a specialist's agent for referral processing. The hospital system acts as AS.

**Bolyra deployment**: The PCP's agent holds a credential with `permissionBitmask = 0b00000111` (READ_DATA + WRITE_DATA + FINANCIAL_SMALL). It delegates to the cardiologist's agent with `delegateeScope = 0b00000001` (READ_DATA only). The delegation circuit uses `blindedScopeCommitment` ‚ÄĒ Ascension-as-AS sees the delegation proof on-chain but cannot determine which specialist received the delegation (the `delegateeCredCommitment` is private). The referral network topology is hidden from the AS.

## 8. Why the baseline cannot match

The baseline's fundamental deficiency is **architectural, not configurational**. Five specific gaps are unbridgeable without zero-knowledge proofs:

**Gap 1: AS omniscience is definitional in OAuth/OIDC.** Every token issuance request is visible to the AS. PPID hides the subject from RSes but the AS holds the PPID mapping table. No combination of OAuth RFCs removes the AS from the issuance path ‚ÄĒ the AS *is* the issuance path. Bolyra's construction moves authorization proving to the client side: the agent generates a ZK proof locally, and the AS (if it observes the on-chain submission) sees only `scopeNullifier`, which is unlinkable across scopes by the PRF property.

**Gap 2: No scope blinding exists in the standards stack.** RFC 8707 Resource Indicators require the AS to see the requested `resource` URI at token issuance time. The AS logs `(agent, resource, timestamp)` for every request. Even BBS+ selective disclosure does not help here ‚ÄĒ the scope request itself (not the credential content) is the leaking channel. Bolyra's `scopeId` is provided by the RS directly to the agent and enters the circuit as a public input; the AS never observes the `scopeId ‚ÜĒ RS` mapping.

**Gap 3: Cross-scope nullifier independence has no analogue.** In the baseline, the agent's `sub` (or `client_id`, or DPoP key thumbprint) is constant across RS interactions at the AS layer. Even with PPID, the AS sees the same internal subject identifier requesting tokens for different RSes. Bolyra produces `scopeNullifier = Poseidon2(scopeId, agentSecret)` ‚ÄĒ a deterministic but scope-specific value that enables per-RS sybil detection while being provably unlinkable across scopes under the PRF assumption.

**Gap 4: Delegation chain topology is AS-observable in RFC 8693.** Token Exchange requires the AS to see actor token, subject token, and requested scope at every delegation hop. The AS reconstructs the full delegation graph. Bolyra's blinded scope commitment chain (`Poseidon3(scope, credCommitment, blindingNonce)`) hides the delegation topology: the AS sees a new `blindedScopeCommitment` at each hop but cannot link them (fresh `blindingNonce` per hop, PRF assumption).

**Gap 5: No formal security definition exists in the baseline.** The baseline has no IND-UNL-AS game, no adversarial model for AS-level correlation, and no security proof. BBS+ multi-show unlinkability is defined only at the holder-verifier layer (RS-to-RS), not at the issuer layer (AS-to-RS). Bolyra's construction comes with a concrete game definition and a reduction to the Poseidon PRF assumption, providing a falsifiable security claim that can be peer-reviewed and formally verified.
