### Attack 1: Nullifier–Scope Decoupling (Soundness Break)

- **Attack**: The circuit accepts `(scope[i], credCommitment[i], delegationNullifier[i])` as *independent* private inputs with no constraint binding them together. On-chain, each `DelegationVerified` event presumably records a tuple `(nullifier_i, newScopeCommitment_i)`. But the circuit never constrains `delegationNullifier[i]` to equal any function of `scope[i]` or `credCommitment[i]`. The only in-circuit anchor is `Poseidon2(scope[0], credCommitment[0]) === rootScopeCommitment` (hop 0) and the publicly output `terminalScopeCommitment` (verified by the auditor off-circuit). For all interior hops 1 ≤ i ≤ n−2, the prover can freely choose any `(scope[i], credCommitment[i])` that satisfies narrowing relative to neighbors, while supplying any on-chain nullifier `delegationNullifier[i]` — including one from an unrelated delegation session or a delegation that actually *expanded* scope. The `chainAnchor = PoseidonN(n_0, …, n_7)` proves those nullifiers exist somewhere on-chain, but proves nothing about which scope commitments they correspond to.

    Concretely: the adversary has a real 3-hop chain where hop 1 expands permissions (non-narrowing). They:
    1. Pick fresh `scope[1]' ⊊ scope[0]` (narrowing, different from the real scope).
    2. Pick arbitrary `credCommitment[1]'`.
    3. Set `delegationNullifier[1]` to the real on-chain nullifier for the non-narrowing delegation at hop 1.
    4. Generate a PLONK proof. The circuit accepts: narrowing holds in the witness; the anchor passes because the nullifier is real.

- **Why the reduction fails**: §4 says "by PLONK knowledge soundness, extract witness `scope[0..n]`… the extracted witness satisfies constraint 5 (monotonic narrowing)." True — but the extracted scopes are *the prover's witness values*, not the *actual delegated scopes* recorded on-chain. The game definition in §3 requires "there exists some hop i where scope[i] ⊄ scope[i-1]" with respect to **actual scopes**. The extracted witness satisfies narrowing by construction; the actual delegation didn't. The soundness game and the reduction argue about different objects. This is the central semantic gap.

- **In-threat-model?** **No — construction must address.** Every interior hop needs an in-circuit commitment: `delegationNullifier[i]` must be derived as `Poseidon2(scopeCommit[i], sessionNonce)` (or equivalent), and that derivation must match the on-chain DelegationVerified event format. Without this, the audit circuit proves narrowing of a fabricated witness, not the actual delegation chain.

### Attack 2: Chain Truncation via Adversary-Controlled `chainLength`

- **Attack**: `chainLength` is a **public input** — the prover sets it. The circuit enforces `sum(hopActive[i]) === chainLength` but nothing else. Consider a real 4-hop chain where hop 2 grants `FINANCIAL_MEDIUM` when hop 1 only granted `FINANCIAL_SMALL` (a genuine expansion). The adversary:
    1. Sets `chainLength = 2` (public input).
    2. Sets `hopActive = [1, 1, 0, 0, 0, 0, 0, 0]`.
    3. Uses real scopes at hops 0–1 (which do narrow).
    4. Uses the real `delegationNullifier[0]`, `delegationNullifier[1]`; pads hops 2–7 with nullifier = 0.

    `chainAnchor = PoseidonN(n_0, n_1, 0, 0, 0, 0, 0, 0)`. The proof verifies. The auditor sees `narrowingValid = 1`, `chainLength = 2`, and cross-references `n_0`, `n_1` on-chain — both exist. The auditor has no in-protocol mechanism to know the session involved 4 hops, not 2, unless the on-chain registry independently records `chainLength` per `sessionNonce` — which §2 and §5 do not describe.

- **Why it works**: §7 says "Examiner verifies: chainAnchor matches delegation nullifiers found in on-chain DelegationVerified events (chain is real, not fabricated)." This only verifies that the *included* nullifiers are real. It does not verify that the included nullifiers are *all* nullifiers for this `sessionNonce`. The NCUA deployment scenario (§7) is the exact risk: NFCU could present a 2-hop proof omitting the third-party agent hop where `FINANCIAL_MEDIUM` was improperly granted.

- **In-threat-model?** **No — construction must address.** Fixing this requires either (a) binding `chainLength` to on-chain state — the registry records the total hop count per session, and the public input must match — or (b) the chain anchor includes a Merkle root over *all* on-chain DelegationVerified events for this `sessionNonce`, so any omission changes the anchor.

### Attack 3: HVZK Does Not Protect Against an Adversarial Auditor (Privacy Game)

- **Attack**: The `AuditPrivacy` game (§3) requires the proof transcript to be computationally indistinguishable across two chains C₀, C₁. The §4 privacy reduction says "by the honest-verifier zero-knowledge property of PLONK, the proof transcript is simulatable given only public inputs/outputs." HVZK guarantees simulatability only when the verifier's challenges are drawn honestly (in the interactive setting) or when the Fiat–Shamir hash is modeled as a random oracle and the adversary cannot influence the challenge computation. In the journalist/source scenario (§7), the "editor" is explicitly an *auditing* party — potentially adversarial, representing e.g. a government subpoena served to the publication. If the auditor is malicious, HVZK gives no security: a malicious verifier can choose challenges adaptively (in the interactive variant) or mount selective-failure attacks against the Fiat–Shamir instantiation if the hash domain is not carefully bound.

    More precisely: the construction must achieve **simulation-extractability** (SE) or **full ZK**, not merely HVZK. Standard PLONK (Marlin 2019, PLONK 2019) achieves HVZK. SE-PLONK requires either (a) an additional "knowledge" assumption in the algebraic group model, or (b) using a hash-based Fiat–Shamir with a domain-separated sponge and a careful rewinding argument. Neither is specified.

- **Why it matters for the journalist scenario**: The privacy game sets up C₀ and C₁ with "different intermediate participants." The zero-knowledge property must hold even if the editor mounts a distinguishing attack using auxiliary information (e.g., timing, proof structure, or non-standard challenges). With HVZK only, a distinguisher who knows one of the chains may be able to break privacy against a modified challenge strategy.

- **In-threat-model?** **No — construction must address.** §3 must state whether the adversary in `AuditPrivacy` is honest-but-curious or fully malicious. If the journalist/source scenario requires protection against a malicious auditor, the construction must specify SE-PLONK or a full ZK argument, and the reduction sketch must be rewritten accordingly.

### Attack 4: Scope Commitment Chain Is Not a Linked List (Constraint 4 Tautology)

- **Attack**: Constraint 4 ("Chain linking") is the crux of the construction's claim that the scope commitments form a verified linked chain. The stated constraint is:

    ```
    select(hopActive[i], scopeCommit[i-1], scopeCommit[i-1]) ===
    select(hopActive[i], previousExpected, previousExpected)
    ```

    Both branches of each `select` call return the *same* value (`scopeCommit[i-1]` on the left, `previousExpected` on the right). This is equivalent to the unconditional constraint `scopeCommit[i-1] === previousExpected`. But `previousExpected` is never defined as a separate signal — it appears to be computed from the same private `scope[i-1]` and `credCommitment[i-1]` that generated `scopeCommit[i-1]`. If `previousExpected = scopeCommit[i-1]` (which it must be, since both are Poseidon2 of the same private inputs), the constraint is `scopeCommit[i-1] === scopeCommit[i-1]` — a tautology.

    The *intended* constraint is: "the `scopeCommit[i-1]` used in the delegation proof at hop `i` (as `previousScopeCommitment`) matches the `scopeCommit[i-1]` in the audit circuit." But enforcing this requires `scopeCommit[i-1]` to be a public input at every hop (not just hop 0) or an in-circuit re-derivation of the on-chain registry value. As written, the circuit only anchors hop 0 via the public `rootScopeCommitment` and the terminal hop via `terminalScopeCommitment`. Interior scope commitments are self-consistently computed from unconstrained private inputs — they are not required to match anything on-chain.

    This compounds Attack 1: not only are nullifiers uncoupled from scopes, but the sequential linking of scope commitments across hops is itself a tautology for interior hops.

- **In-threat-model?** **No — construction must address.** A correct linked-list constraint requires that for each active hop `i > 0`, the `scopeCommit[i-1]` in the audit circuit equals the `previousScopeCommitment` public output recorded on-chain by the per-hop `Delegation` circuit proof at hop `i`. This means either (a) each `scopeCommit[i]` must be a public input (breaking privacy), or (b) an in-circuit Merkle/map lookup against on-chain state must verify the binding — neither of which is described in §2.

### Attack 2: Nullifier–Scope Commitment Decoupling (Soundness Gap)

- **Attack**: The `DelegationChainAudit` circuit proves two structurally independent claims in parallel:

  1. `scope[0..n]` narrows monotonically and `scopeCommit[i] = Poseidon2(scope[i], credCommitment[i])` (constraints 3, 5).
  2. `chainAnchor = PoseidonN(delegationNullifier[0..7])` and each nullifier exists in an on-chain `DelegationVerified` event (constraint 8 + auditor off-chain step).

  The circuit contains **no constraint binding `delegationNullifier[i]` to `scopeCommit[i-1]` → `scopeCommit[i]`**. A malicious prover can construct a proof using: (a) genuine on-chain nullifiers harvested from *completely different* delegation sessions, and (b) fabricated `scope[i]` / `credCommitment[i]` values chosen to satisfy narrowing, anchored only at `rootScopeCommitment` (the handshake) and `terminalScopeCommitment` (the on-chain final commitment). As long as these two public endpoints match real on-chain state, and the intermediate scopes happen to narrow, the circuit accepts — even though the nullifiers bear no cryptographic relationship to the claimed scope commitments.

  The auditor's off-chain step (Section 2, constraint 8 prose) says "check each non-zero nullifier exists in on-chain `DelegationVerified` events." If `DelegationVerified` events record only `(nullifier, timestamp)` and not `(nullifier, previousScopeCommitment, newScopeCommitment)`, the auditor cannot detect the substitution. The construction never specifies `DelegationVerified` event schema.

- **Why it works**: This is a classical linkage failure: proving statement A and statement B separately does not prove they refer to the same underlying data. RFC 8693 at least has referential integrity — the `act` chain in a JWT ties the `sub` of the exchange to the exact `token_type` and `scope` granted.

- **In-threat-model?** **Yes — construction must address.** Either (a) the `DelegationVerified` event schema must include the scope commitment pair `(previousScopeCommitment, newScopeCommitment)` alongside the nullifier, and the circuit must prove that each `delegationNullifier[i]` was derived from a delegation that used `scopeCommit[i-1]` and `scopeCommit[i]`, or (b) the per-hop `Delegation` circuit proof must embed the scope commitment transition and the `DelegationVerified` event must log it, with the audit circuit enforcing consistency. As written, the soundness reduction in Section 4 only argues about extracted `scope[i]` values satisfying constraints — it never addresses whether the extracted witness binds to the on-chain nullifier record.

### Attack 3: PPIDs + Audience Binding Already Break the Linkability Claim — Without ZK

- **Attack**: Section 8 claims the baseline cannot hide intermediate participants. But:

  - **RFC 9449 DPoP** sender-constrains each token to a per-session ephemeral key. An intermediary agent generates a fresh DPoP key per hop — the receiving RS sees only that hop's DPoP key, not a stable agent identity.
  - **OIDC Pairwise Pseudonymous Identifiers (PPIDs)** assign a different `sub` to each (issuer, RS) pair. When agent A delegates to B, the `act.sub` in the RFC 8693 token is B's PPID for *this* RS. The auditor sees a PPID, not B's stable identifier.
  - **RFC 8707 Resource Indicators** bound tokens are non-transferable to other RSes, eliminating cross-RS correlation.

  Combining these: each hop's token has an ephemeral DPoP key, a per-RS PPID, and audience binding. The auditor sees `chainLength` pseudonymous hops with ephemeral keys. The "intermediate participants" are already unlinkable across sessions and across RSes — matching the construction's privacy game definition.

- **Why it fails against the construction**: The construction's journalist/source scenario (Section 7) requires hiding participants **from the auditor on the same audit chain**, not just from other RSes. PPIDs hide identities from cross-RS correlation but the AS *still knows* the stable identity behind each PPID. If the AS is compelled (subpoena, government order), it can deanonymize every PPID. The construction's `credCommitment[i]` values never leave the circuit — not even the AS knows the private input that produced them. DPoP + PPIDs are AS-deanonymizable; Bolyra's intermediate commitments are cryptographically opaque even to the AS.

  However, the construction undermines itself: the on-chain `DelegationVerified` event records something associated with each hop (at minimum the nullifier). If the nullifier correlates to observable behavior (transaction timing, gas payer, RPC origin), the "journalist source" is deanonymizable via metadata analysis. The construction offers in-circuit anonymity but no network-layer anonymity guarantee.

- **In-threat-model?** Construction survives the AS-coercion angle but **must address** the gap: DPoP + PPID closes the cross-RS linkability problem the construction implies is unique to ZK. The differentiation should be narrowed to "AS-coercion resistance" and "off-chain metadata leakage" should be explicitly acknowledged as out of scope.

### Attack 4: The "No Trusted Third Party" Claim is Falsified by the On-Chain Registry

- **Attack**: Section 8 asserts: "PLONK proof is self-verifiable. No AS, no policy engine, no federation trust anchor. The BN254 pairing check is the only trust assumption." This is false.

  The construction requires the auditor to trust that:
  1. The Bolyra registry contract at a given address correctly implemented `verifyProof` for each hop and emitted `DelegationVerified` only on valid proofs.
  2. The EVM execution of the registry was not reorg'd or manipulated between delegation and audit time.
  3. The `lastScopeCommitment[sessionNonce]` storage slot reflects the canonical chain state.

  The smart contract IS a trusted aggregator. It is analogous to an AS in one critical way: if it is compromised (exploited, upgraded via a malicious DAO proposal, or the chain experiences a 51% attack), an adversary can insert fraudulent `DelegationVerified` events. The construction provides no protection against this — the `chainAnchor` verification depends entirely on those events being authentic.

  Contrast: an AS with an HSM-protected signing key and append-only audit log in a different threat model may offer *stronger* integrity guarantees than a public chain with finality delays. FFIEC guidance (invoked in Section 7) does not recognize on-chain event logs as authoritative audit records — the NCUA examiner in the deployment scenario would need to establish that the Base Sepolia events are admissible evidence, which is a legal and regulatory gap the construction leaves entirely open.

- **Why it partially fails**: For the specific adversary model in Section 3 (adversary controls up to `MAX_HOPS - 1` participants, not the chain itself), the on-chain registry is a valid trust anchor. The reduction sketch is internally consistent. The problem is the claim in Section 8 that there is "no trusted third party" — the chain validators and the contract deployer are trusted third parties.

- **In-threat-model?** The construction's security argument (Section 4) survives if the threat model is stated correctly — but Section 8's "no trusted third party" row in the comparison table is factually inaccurate. The construction must be amended to read: "No *application-layer* trusted third party — trust is delegated to the consensus layer and the contract bytecode, which is auditable and immutable post-deployment." This is a meaningful narrowing of the claim, not its elimination.

### Attack 1: Nullifier–Scope Decoupling — The Soundness Proof Proves the Wrong Thing

**Attack:**

In Section 2, `delegationNullifier[i]` and `scopeCommit[i] = Poseidon2(scope[i], credCommitment[i])` are *independent* private inputs. The chain anchor (constraint 8) commits to the nullifiers. The auditor checks each non-zero nullifier against on-chain `DelegationVerified` events. But the audit circuit never asserts:

```
delegationNullifier[i] was emitted by the same on-chain tx that recorded scopeCommit[i]
```

The `DelegationVerified` event presumably records `(nullifier, previousScopeCommitment, newScopeCommitment)`. The audit circuit does **not** read those event values. The prover provides `scope[i]` and `credCommitment[i]` freely — constrained only to form a self-consistent sequence linking back to `rootScopeCommitment` (public, anchored to hop 0) and forward to `terminalScopeCommitment` (public output, anchored to the last hop).

**Concrete exploit:** Adversary controls hops 1–2 in a 4-hop chain. Real scopes: `0x87 → 0x07 → 0x87 → 0x01` (expansion at hop 2). Adversary produces an audit proof with fabricated private witness `scope[2] = 0x05` (narrows correctly), using the real nullifiers from the on-chain events. The circuit verifies the fabricated scopes narrow. `narrowingValid = 1`. The examiner cannot detect the expansion because `scopeCommit[2]` is private and is never cross-referenced against the `DelegationVerified` event that recorded it.

**Why it works:** The soundness game in Section 3 defines A as winning if "there exists some hop `i` where `scope[i] ⊄ scope[i-1]`" where `scope[i]` refers to the *actual* chain scopes. The Section 4 reduction shows the *extracted witness* narrows — but the extracted witness may be entirely fabricated. The reduction conflates "the prover's PLONK witness satisfies narrowing" with "the actual delegation chain narrowed." These are identical only if the circuit cryptographically binds each `nullifier[i]` to `scopeCommit[i]` — which it does not.

The fix would require proving, inside the circuit, that `(delegationNullifier[i], scopeCommit[i])` matches a public on-chain record — but that would reveal intermediate scope commitments, defeating the construction's privacy goal. There is no free lunch here: you cannot simultaneously hide intermediate scope commitments from the auditor and prove they match on-chain records without another layer of ZKP (e.g., proving membership in a Merkle tree of `(nullifier, scopeCommit)` pairs).

**In-threat-model?** No. This is a soundness break against the stated security game. The construction must address it.

### Attack 2: WIMSE Token Chains Are Offline-Verifiable — The Baseline Comparison in Section 8 Is Incorrect

**Attack:**

Section 8 claims "RFC 7662 introspection requires live AS queries" and "RFC 8693 narrowing enforcement requires the Authorization Server. Auditor who cannot query or trust the AS has no narrowing guarantee." This is the baseline the construction is measured against.

This characterization applies only to *reference tokens* (opaque token + RFC 7662 introspection endpoint). WIMSE (`draft-ietf-wimse-arch`) mandates *self-contained workload tokens* — JWS-signed with public keys bound to SPIFFE SVIDs. The full delegation chain is encoded in nested `act` claims, each JWS-signed by the delegating workload's SVID private key. An auditor with the trust domain bundle (a static HTTPS document, no live AS) can verify the complete chain offline:

```
JWT₃.act = JWT₂.act = JWT₁ (signed by SVID₁)
JWT₁.scope ⊇ JWT₂.scope ⊇ JWT₃.scope  ← verifiable by inspection
```

No live AS. No real-time query. Narrowing is cryptographically enforced at issuance (AS rejects expansions) and visible at audit time.

The Bolyra claim reduces to: "we hide the intermediate scope values from the auditor." For the NFCU scenario, this is audit opacity, not audit privacy. **In-threat-model?** Partially. The Section 8 baseline comparison overstates WIMSE's limitations, inflating the apparent delta. The construction should restate its advantage specifically: "hides intermediate scope *values* (not just identities) from a legitimate auditor, in cases where the auditor's mandate is to verify structural narrowing, not inspect quantum of permission." This is a real but narrower claim.

### Attack 3: Substituting Blockchain Trust for AS Trust Is Not Trustless

**Attack:**

Section 8 states: "No trusted third party — PLONK proof is self-verifiable... The BN254 pairing check is the only trust assumption." The construction's Section 4 threat model lists what the adversary does NOT control: the Poseidon hash function, the BN254 pairing, and the PLONK universal setup.

Missing from that list, but required for the construction to function:

1. **Base L2 consensus and finality.** The auditor cross-references `chainAnchor` against on-chain `DelegationVerified` events (Section 2). Base mainnet uses an optimistic rollup with a ~7-day challenge window. A determined adversary who controls the sequencer can censor, reorder, or delay event emission. A finalized event on Base requires trusting the sequencer + L1 Ethereum + the bridge contract.

2. **Smart contract correctness and upgradeability.** `HandshakeVerified` and `DelegationVerified` events are emitted by Bolyra registry contracts. If those contracts have an admin key (standard for upgradeable proxies), an adversary controlling the key can emit fabricated events. Base's admin multisig can upgrade contracts. The Section 3 threat model says nothing about contract upgrade authority.

3. **PLONK universal trusted setup.** The Section 2 claim "no per-circuit ceremony" is true, but the universal SRS (powers-of-tau) requires trusting that at least one participant in the ceremony was honest. Compared to SPIFFE: an AS trust anchor requires trusting the CA that signed the SPIRE bundle + TLS PKI for the bundle endpoint. These are well-understood, institutionally-governed trust assumptions (WebPKI, corporate CA). Blockchain trust assumptions (sequencer liveness, contract upgrade authority, ZK ceremony participants) are structurally similar in risk profile but less familiar to enterprise security teams and less regulated.

The NFCU CISO already trusts SPIRE's CA infrastructure. Asking them to additionally trust Base L2 sequencer liveness, a PLONK ceremony, and a smart contract with an upgradeable proxy is an expansion of the trusted computing base, not a reduction.

**In-threat-model?** No. The threat model explicitly excludes the pairing and hash function but is silent on blockchain/contract trust. The construction must define its full trusted computing base and compare it honestly to the SPIFFE baseline — or restrict the "no trusted third party" claim to the ZK layer only.

### Attack 4: `chainLength` Leakage Breaks Privacy in Thin Anonymity Sets

**Attack:**

`chainLength` is a public input (Section 2). `sessionNonce` is also public. In the journalist/source scenario, this combination is a near-unique fingerprint: a 3-hop chain at timestamp T with session nonce N, where the auditor knows hop 0 is the journalist's agent.

The AuditPrivacy game in Section 3 correctly requires C₀ and C₁ to have "equal length" — acknowledging that chain length is revealed. But the construction does not call this out as a limitation, propose padding, or analyze the anonymity set.

In practice, AI agent pipelines have distinctive chain topologies. A 4-hop NFCU loan pipeline is distinguishable from a 2-hop payment authorization pipeline. Traffic analysis on `(chainLength, sessionNonce, timestamp)` tuples, correlated with known participants at hop 0 and hop N-1, can probabilistically de-anonymize intermediate hops without breaking any cryptographic primitive:

- Adversary knows hop 0 identity (public in handshake) and terminal scope commitment (public output)
- Adversary observes chainLength = 4, timestamp, session nonce
- Adversary has a prior distribution over who runs 4-hop pipelines at that time
- With even moderate side-channel information (tool call timing, network metadata), the intermediate participant set is small

For the journalist scenario: if the adversary knows 5 journalists use Bolyra, each with characteristic chain lengths for their relay networks, chain length alone may identify the journalist. The ZK proof hides intermediate identities but leaks the skeleton.

SPIFFE doesn't have this problem because it doesn't claim to hide intermediate participants — the `act` chain is explicit. Bolyra's partial hiding (participants hidden, structure revealed) can be worse than full disclosure if it creates a false sense of anonymity while retaining a deanonymizable fingerprint.

The fix — padding all chains to MAX_HOPS — is structurally available but not mandated by the construction. Section 2's `hopActive` padding exists but is optional; `chainLength` is still exposed as the true count.

**In-threat-model?** No. The AuditPrivacy game acknowledges length leakage as a design parameter constraint (equal-length challenge pairs) but does not quantify the privacy loss or bound the anonymity set. For the journalist/whistleblower scenarios in Section 1, chain length leakage can be catastrophic. The construction must either mandate MAX_HOPS padding (and expose only `chainLength = MAX_HOPS`) or formally bound the anonymity set under the leakage.
