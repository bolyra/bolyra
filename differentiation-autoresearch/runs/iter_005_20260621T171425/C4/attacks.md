### Attack 1: Don't-Care Clause Encoding Contradiction — Predicate Hash is Prover-Dependent

**Attack:** Consider the canonical predicate `chartered_by_NCUA == 1`. The construction encodes seven don't-care clause slots with `(attrIndex=0, comparator=EQ, threshold=0)`, yielding a fixed verifier-published `predicateHash = Poseidon25(1, 0, 1, 0, 0, 0, ..., AND_ALL)`. The circuit evaluates don't-care clause `c_1` as: `Mux(attrValues, 0) == 0`, i.e., `attrValues[0] == 0`. If the actual credential has `attrValues[0] = charter_number = 12345` (as shown in the NCUA deployment scenario, §7), this equality fails, the clause evaluates to 0, `predicateResult = 0`, and the circuit rejects the proof.

The construction attempts to escape this by stating (§2, Gadget 4b): *"the prover sets the threshold to match the actual value at index 0, making the clause trivially true."* But this makes `clauseThreshold[1] = attrValues[0]` a private, credential-specific value. The predicate hash therefore becomes `Poseidon25(..., attrValues[0], ...)` — a different field element for every prover. The verifier cannot pre-publish a fixed `predicateHash` for the NCUA membership predicate; it would depend on each prover's private attribute values. This directly contradicts the verifier-controlled `predicateHash` design and the statement in §2 that "one clause vector → one hash."

**Why it works / fails:** The construction tries to have it both ways: a fixed, verifier-publishable `predicateHash` AND don't-care clauses that are tautological against arbitrary private attribute values. These are mutually exclusive. Either (a) the threshold is fixed (e.g., 0), in which case most real credentials will fail the don't-care clause; or (b) the threshold is prover-supplied (matching private attributes), in which case the verifier's `predicateHash` is credential-dependent and cannot be pre-committed. The suggested alternative (a dedicated `ALWAYS_TRUE` comparator type 6) was dismissed as having "no security benefit," but it is the *only* design that resolves this contradiction.

**In-threat-model?** No. This is a soundness failure at the predicate-integrity layer. A compliant prover may be unable to produce a proof for a verifier-published `predicateHash` unless the verifier has out-of-band knowledge of the prover's private attribute at index 0 (required for the don't-care threshold). The construction must define and enforce a canonical don't-care encoding — the simplest correct fix is reserving comparator type 6 as `ALWAYS_TRUE` and constraining `booleanTree` to short-circuit unused clauses before the AND reduction.

### Attack 2: PRF Security is Not a Consequence of Collision Resistance — The Named Assumption is False

**Attack:** Section 4 states assumption (4): *"PRF security of Poseidon (keyed by the first argument)... This is a standard consequence of CR + preimage resistance in the ideal permutation model for Poseidon."* This is false as a matter of cryptographic theory. Collision resistance and preimage resistance are properties of the output distribution; PRF security is a property of the function's behavior under a secret key. No implication holds in general: a hash function can be CR and preimage-resistant while being trivially distinguishable from a random function when the key is known or partially observable.

Concretely: suppose Poseidon2 satisfies CR and preimage resistance but its round-key schedule leaks one bit of `credentialId` to any adversary who queries `Poseidon2(credentialId, ·)` at two chosen points. CR and preimage resistance would still hold, but the PRF game would be broken. The "ideal permutation model" cited is not a standard reduction target for PRF security — it is an informal heuristic. The construction's unlinkability claims for both `blindedCredCommitment = Poseidon2(credentialId, proofSalt)` and `blindNullifier = Poseidon2(credentialId, sessionNonce)` rest entirely on this assumption. If Poseidon2 is not a PRF keyed by the first argument, both outputs may leak `credentialId` to an adversary who sees proofs across sessions.

**Why it works / fails:** The adversary `A_AS` (§3) explicitly accumulates cross-session observations. If Poseidon PRF security fails, `A_AS` with oracle access to `f(·) = Poseidon2(credentialId, ·)` via adaptive `sessionNonce` choices (the nonce is adversary-controlled per the IND-ISS game, §3, step 3) could distinguish the function from random and recover `credentialId`, breaking issuer anonymity. The hybrid argument in §4 collapses entirely: Hybrid 0→1 and Hybrid 3→4 are not justified by CR+preimage alone.

**In-threat-model?** This attack is in-threat-model for both `A_RS` and `A_AS`. The correct fix is to either (a) explicitly add PRF security of Poseidon as a *standalone named assumption* (not derived from CR), with a citation to the Poseidon2 security analysis that treats the first argument as a key, or (b) replace the nullifier and commitment construction with a scheme whose PRF security reduces to a standard assumption (e.g., AES-based PRF for the commitment, accepting higher constraint cost, or a dedicated keyed-hash gadget with a domain separator that admits a tighter reduction).

### Attack 3: Hybrid 2→3 Transition Requires Simulating Proofs for Infeasible Witnesses

**Attack:** The reduction sketch in §4 proceeds as follows: Hybrid 2 replaces both issuer-dependent public outputs with uniform random field elements `(r_1, r_2)`. Hybrid 3 then "switch[es] from issuer `j_0` to issuer `j_1` inside the PLONK proof witness," with indistinguishability argued from PLONK zero-knowledge. The gap: PLONK zero-knowledge (HVZK) guarantees that for a *valid* instance `(x, w)` with `R(x, w) = 1`, a simulator can produce a transcript indistinguishable from a real proof. However, after Hybrid 2, the public outputs are `blindedCredCommitment = r_1` and `blindNullifier = r_2` for uniform random `r_1, r_2`. The circuit constrains: `r_1 = Poseidon2(credentialId, proofSalt)` and `r_2 = Poseidon2(credentialId, sessionNonce)`. For a uniformly random `r_1`, there exist `(credentialId, proofSalt)` pairs that satisfy this (Poseidon2 is surjective over `F_p`), but the SAME `credentialId` must also (a) be a leaf-preimage in the issuer Merkle tree, (b) match a valid EdDSA signature chain, (c) produce `r_2` via `Poseidon2(credentialId, nonce)`, and (d) pass the revocation non-membership check. For uniformly random `(r_1, r_2)`, the probability that a consistent full witness exists is negligible without the SRS trapdoor `τ`.

The PLONK ZK simulator requires the trapdoor to generate proofs for arbitrary (not necessarily satisfiable) public statements. Using the trapdoor is exactly the subverted-setup concern the construction does not address. Without the trapdoor, the simulator cannot generate valid PLONK proofs for the modified public outputs in Hybrid 2, and the entire hybrid chain breaks.

**Why it works / fails:** The correct approach is to apply a *programmable random oracle* (for the Poseidon hash) or to use *simulation extractability* (SE-SNARK) instead of plain ZK. SE-SNARKs (e.g., Groth16 with simulation trapdoor, or PLONK with a preprocessing simulator) allow the hybrid argument to go through without requiring satisfiable witnesses in intermediate hybrids, but they require explicitly modeling the CRS generation and simulator access. The construction's proof sketch skips this, leaving the Hybrid 2→3 step unjustified.

**In-threat-model?** This is a proof-level gap, not a direct cryptanalytic attack — a reduction flaw rather than a construction break. However, without a complete reduction, the IND-ISS theorem is unproven. A construction claiming formal IND-ISS security under PLONK ZK + Poseidon PRF must close this gap, either by using an SE-SNARK formulation or by restructuring the hybrids so intermediate statements are always satisfiable (e.g., by applying the PRF replacement *before* switching the witness, using fresh random `proofSalt` chosen by the simulator).

### Attack 4: Revocation Issuer-Anonymity Degrades from "Cryptographic" to "SHOULD" — No Circuit Enforcement

**Attack:** Section 3 introduces `A_REV` (revocation-tree observer) and argues they cannot link a revoked `credentialId = Poseidon3(credentialDigest, Ax, Ay)` to a specific issuer under Poseidon preimage resistance. The defense against the low-entropy brute-force is: *"issuers SHOULD include a per-credential secret nonce as one of the attribute slots (e.g., `attrValues[15] = random_nonce`)."* This is explicitly labeled an "application-layer mitigation, not a circuit change." The circuit does not enforce the presence of a nonce.

Consider an attacker scenario: NCUA mandates a fixed attribute schema for all 4,600 federally-insured credit unions (no optional fields, no random nonces — regulatory schemas are often rigid). The credential has `attrValues = [charter_number, chartered_by_NCUA=1, total_assets_tier, jurisdiction_code, enforcement_action_count, 0, 0, ..., 0]`. All non-Boolean fields are public record (NCUA publishes charter numbers, asset tiers, and enforcement actions). The adversary `A_REV` (e.g., an on-chain indexer) sees a revoked `credentialId` in the sparse Merkle tree. They enumerate: for each of 4,600 issuer keys `(Ax_j, Ay_j)` in the registry AND each plausible attribute tuple (bounded by public NCUA data), compute `Poseidon3(credentialDigest, Ax_j, Ay_j)` and compare. The "preimage resistance" argument fails because `credentialDigest` is not a random element — it is a Poseidon hash of known or enumerable public-record values. The adversary finds the matching `(j, attrValues)` tuple with work proportional to `|issuers| × |attribute_domain| = 4600 × (small_product_of_public_ranges)`.

In the §7 NCUA scenario, the construction explicitly includes `credential_nonce = random` in `attrValues[15]`. But this is not part of the circuit, not part of the IETF draft schema (§5 maps to existing primitives; no nonce field appears in the formal spec), and violates the "Property 3: Revocation-tree issuer hiding" claim in §3, which is presented as a cryptographic guarantee rather than a conditional-on-application-nonce guarantee.

**Why it works / fails:** The table in §8 ("Summary comparison") claims "Revocation without issuer leak: Yes (unified tree, stable credentialId, Poseidon preimage resistance)" unconditionally, while the body of §3 conditions this on a SHOULD-level nonce. The cryptographer reading the summary row would believe revocation issuer-hiding is a cryptographic property; the fine print reveals it is a conditional application-layer recommendation. For the claim to hold formally, either (a) the circuit must enforce a minimum-entropy random nonce in one attribute slot (circuit-level SHALL), or (b) the formal claim must be stated as conditional: *"revocation-tree issuer hiding holds under the assumption that credential attributes have min-entropy ≥ λ bits, e.g., enforced by a mandatory nonce."* As written, the "cryptographic" label in the summary table is not warranted.

**In-threat-model?** Yes — for the `A_REV` adversary class, this attack succeeds against non-nonce credentials, which are valid circuit inputs. The construction survives only with application-layer cooperation that no circuit constraint enforces.

1. **RFC 8693 Token Exchange** — each of the 4,600 NCUA credit unions exchanges a local credential for a token issued by a neutral, privacy-preserving federated NCUA identity broker (analogous to the NACHA/Mastercard Send federated identity models). The broker AS never stores which CU originated the exchange request in operational logs; it issues tokens with `iss=https://identity.ncua.gov`.

2. **Pairwise Subject Identifiers (OIDC PPID)** — the broker uses sector-specific `sub` values per RS. RS1 and RS2 each see different `sub` values for the same underlying CU agent. There is no global stable identifier in any party's view.

3. **RFC 7662 filtered introspection + signed JWT response** (draft-ietf-oauth-jwt-introspection-response) — the broker signs an introspection JWT asserting `chartered_by_NCUA=true` and omits `client_id`, `iss`-of-credential, and any CU-identifying claim. The broker caches and pre-signs these JWTs, going offline after issuance. The RS verifies the broker's signature without contacting the broker again.

4. **RFC 8707 Resource Indicators** — tokens are audience-bound to specific RSes, preventing reuse across RSes by a stolen token.

5. **RFC 9449 DPoP** — the agent holds a DPoP proof-of-possession key. Token theft is neutralized.

Under this deployment:
- The RS sees: `iss=https://identity.ncua.gov`, a pairwise `sub`, `chartered_by_NCUA=true`, no CU identity. This is RS-blind hiding by the construction's own definition.
- The broker AS sees: the Token Exchange request from CU X at issuance time. But if the broker is operated by a neutral third party (e.g., a CUSO like PSCU, or NACHA) with a published no-logging policy, audit-enforced by a supervisory board, the AS does not *retain* the issuer correlation. The AS-side "leak" is an organizational posture, not an algebraic certainty.
- Cross-session correlation by the broker: the broker pre-signs and caches introspection JWTs, then goes offline. Subsequent RS token validations hit the RS-local cache, not the broker. The broker never sees which RS is querying which token in production steady-state.

The construction's §8 Gap 6 response — "If the AS is compromised, colluding, subpoenaed, or simply misconfigured, RS-blind hiding fails silently" — applies equally to the on-chain `issuerRegistryRoot`. If the registry smart contract is compromised, the `issuerRegistryRoot` is corrupted and all proofs against it are invalid. The trust surfaces are analogous. The construction has moved trust from an AS operator to a smart contract operator and a Poseidon hash assumption, not eliminated trust.

**Why it works / why it fails against the construction:**
The construction's AS-blind hiding claim is strongest when:
- The issuer set is large and stable
- There is no federated broker that can aggregate issuer-specific entropy
- The deployment cannot support an organizational privacy policy

In the concrete NCUA scenario (§7), the NCUA already operates a federated supervisory identity infrastructure. The "NCUA simultaneously being the registry operator and supervisory authority" concern the construction raises is equally present on-chain: NCUA can observe which Merkle root is being used and correlate proof activity against known enrollment events (see Attack 3). The construction does not establish a cryptographic advantage over a carefully deployed federated broker — it establishes an advantage over a *naive, per-issuer* AS.

**In-threat-model?** Partially. The construction's comparison in §8 Gap 6 conflates "AS-blind hiding" (cryptographic) with "AS-blind hiding" (organizational). The baseline comparison should be against the federated broker architecture, not the naive per-issuer RFC 7662. The construction survives this attack only in deployments where a neutral broker is not feasible — but for the NCUA scenario specifically, federated CU identity infrastructure exists today and is expanding.

### Attack 3: Cross-Epoch Registry Correlation Reduces the Effective Anonymity Set — the IND-ISS Game Does Not Model This

**Attack:**
The IND-ISS game (§3) is defined over a **fixed issuer registry** with `n ≥ 2` enrolled issuers. The game challenger enrolls all issuers at setup time, and the adversary's adaptive queries occur against this static registry. The claimed anonymity bound is the full set of issuers in the registry tree.

The real NCUA deployment scenario (§7) states the registry is "updated quarterly." In a live deployment, the A_AS adversary observes:

- **Registry epoch transitions**: `issuerRegistryRoot_v1 → issuerRegistryRoot_v2` at a specific on-chain block. The transition is publicly observable — it is a smart contract call from the NCUA registry operator.
- **New enrollee disclosure**: the difference between `v1` and `v2` can be computed by any party who monitors the on-chain Merkle leaf insertions (standard Ethereum event logs). The set of *newly added issuers* at each epoch update is therefore public.
- **First-proof correlation**: if a prover generates a proof using `issuerRegistryRoot_v2` and this is the first proof observed under that root, and only one new issuer was added in the `v1 → v2` transition, the anonymity set for that first proof collapses from 4,600 to 1.

More generally: the effective anonymity set for any proof is not the full registry, but the *cohort of issuers who have generated at least one proof under the current root and whose enrollment epoch is consistent with the observed proof timing*. For a large, active registry with frequent proofs, this cohort approximates the full set. For a sparse or low-activity registry (e.g., the early rollout with only a handful of enrolled agents), the cohort may be small.

RFC 9728 Protected Resource Metadata (PRM) provides an analogous disclosure risk in OAuth deployments, but the mitigation there is well-understood: the RS doesn't reveal which AS validated a specific token. The construction has no analogous mitigation for registry epoch correlation. The `issuerRegistryRoot` is a public input to every proof and must match the current on-chain root — it cannot be blinded without defeating the registry membership check.

The IND-ISS security proof (§4) is unconditional on registry dynamics and does not reduce over cross-epoch adversaries. The proof sketch in §4 operates entirely within a single epoch. There is no composability argument covering sequential epochs.

**Why it works / why it fails against the construction:**
The construction partially addresses timing in §3 ("Timing is outside the algebraic model... it does not vary measurably with the issuer's position in the Merkle tree"). But this addresses microarchitectural timing side-channels within a single proof, not cross-epoch registry correlation, which is a protocol-layer information leak from publicly observable on-chain events. These are distinct attack vectors; the construction conflates them.

**In-threat-model?** Yes — but the construction does not fully address it. The IND-ISS game must be extended to model a dynamic registry with observable epoch transitions, or the construction must add a batching/delay mechanism that prevents on-chain observers from learning when a new issuer first generates a proof. Without this, the anonymity bound of "all issuers in the registry" only holds for issuers who were present at the inception of the registry — a condition that grows less favorable over time as the registry is updated.

### Attack 4: Verifier-Controlled Session Nonce Enables Colluding RS Cross-Credential Correlation Without Breaking the PRF

**Attack:**
The construction defines:

```
blindNullifier = Poseidon2(credentialId, sessionNonce)
```

and claims cross-session unlinkability under the Poseidon PRF assumption: "for any PPT adversary with oracle access to `f(·) = Poseidon2(credentialId, ·)`, distinguishing `f` from a random function has negligible advantage."

The standard PRF security definition assumes the PRF *key* (here, `credentialId`) is uniformly random and the *input* (`sessionNonce`) is adversarially chosen. The construction correctly notes the verifier controls `sessionNonce`. What the construction does **not** address is the multi-key PRF distinguishing attack available to a colluding RS consortium:

Consider a consortium of 10 partner credit unions, each acting as a verifier (RS). Each RS independently controls its own `sessionNonce`. They share a side channel (e.g., a consortium data-sharing agreement, which is standard in the CU industry — CUSOs exist specifically for this). The consortium observes:

| RS | sessionNonce | blindNullifier |
|----|------|------|
| RS1 | `n_1` | `Poseidon2(cid_A, n_1)` |
| RS2 | `n_2` | `Poseidon2(cid_A, n_2)` |
| RS3 | `n_1` | `Poseidon2(cid_B, n_1)` |

Each RS sees only its own row. The consortium sees all rows. Now: under the PRF assumption, even with shared observations at distinct nonces, no PPT adversary can correlate `cid_A` across RS1 and RS2 — the PRF outputs are indistinguishable from random. This is correct.

**But the attack is not a PRF attack.** The attack is a *cross-RS nonce coordination attack*:

All consortium RSes agree to issue the *same* `sessionNonce = n_fixed` to all agents in a specific time window. Now:
- Every agent that presents to RS1 with nonce `n_fixed` produces `blindNullifier = Poseidon2(cid_X, n_fixed)`.
- The same agent presenting to RS2 in the same window (having been redirected or solicited) also produces `Poseidon2(cid_X, n_fixed)`.
- RS1 and RS2 share their observed `(n_fixed, blindNullifier)` tuples. A matching tuple identifies the same credential across both RSes.

This is not a PRF attack — `n_fixed` is the same input at the same key, so the output is deterministically identical. The construction explicitly states "replay prevention within a session" is the purpose of `blindNullifier`, but it does not prevent cross-RS de-anonymization when colluding RSes coordinate nonces.

DPoP (RFC 9449) faces the analogous nonce coordination attack: two colluding RSes issuing the same DPoP nonce can correlate DPoP proof keys across RSes. RFC 9449 §8 explicitly acknowledges this: "A DPoP nonce can be used to correlate tokens across RS instances if the nonces are coordinated." The RFC's mitigation is that nonces should be RS-specific and should not be shared. The construction does not specify analogous guidance and does not flag this attack in the threat model.

The on-chain replay check (the registry rejects duplicate `blindNullifier` values) actively *enables* this attack: if RS1's on-chain registry and RS2's on-chain registry share state (e.g., a single on-chain smart contract for the NCUA registry), then coordinated-nonce submission by colluding RSes lets them match credential usage across the full RS set.

**Why it works / why it fails against the construction:**
The PRF security argument in §3 and §4 holds against a single adversary with oracle access. It does not hold against a coordinated multi-RS attack where all oracles share the same input. This is a standard shared-nonce attack that RFC 9449 documents and mitigates through RS-specific nonce generation requirements. The IssuerBlindPredicate construction grants nonce-generation authority entirely to the verifier without any constraint, creating this exposure.

**In-threat-model?** No. The construction must either (a) require that `sessionNonce` incorporates verifier-specific entropy that cannot be coordinated across RSes (e.g., include the RS's on-chain address in the nonce derivation), (b) have the prover contribute entropy to the nonce via a commit-reveal protocol, or (c) explicitly scope the cross-session unlinkability claim to non-colluding RSes and add this to the threat model as an accepted limitation. None of these options are currently specified.

### Attack 3: Registry Root Update Timing Channel Collapses the Anonymity Set

**Attack:** The IND-ISS game (§3) fixes the issuer registry at setup and does not model adaptive registry updates. In a real deployment (§7: NCUA registry of 4,600+ CUs, updated quarterly), the `issuerRegistryRoot` changes each quarter. Consider the following:

1. Quarter Q1: registry has 4,600 CUs, root = `R_1`. PenFed issues a credential bound to `R_1`.
2. Q2: 3 new CUs are admitted. Registry root becomes `R_2`. The on-chain verifier now accepts proofs against `R_2`.
3. In Q2, PenFed submits a proof with `issuerRegistryRoot = R_2` in the public inputs.

The adversary observes: (a) which CUs newly joined the registry between `R_1` and `R_2` (this is public on-chain); (b) that the proof uses `R_2`. If only the 3 newly admitted CUs have freshly rotated their credentials to `R_2` (because they just joined), and the proof uses `R_2`, the anonymity set is the 3 new CUs, not 4,603. More generally, whenever a subset of issuers are known to have migrated to a new root (because their enrollment or a known key rotation event is public), the effective anonymity set degrades from `|registry|` to `|migrated|`.

The construction's §3 bounds adversary advantage by `Adv^{ZK}_{PLONK} + 4 · Adv^{PRF}_{Poseidon}`, but this bound only holds when the registry root is fixed. With adaptive root updates, the adversary's advantage can approach 1 for newly admitted issuers who are the only parties with credentials bound to the fresh root.

SPIFFE handles this problem via bundle refresh: all workloads in a trust domain rotate SVIDs simultaneously on a TTL. The analogous defense here — requiring all issuers to rotate credentials simultaneously on each registry update — is an operational constraint the construction does not specify.

**Why it works / why it fails against the construction:** This attack is outside the algebraic model (it is a traffic-analysis attack on root commitment staleness) and is therefore not addressed by the IND-ISS reduction. The construction mentions the "A_AS adversary's extra observations" but dismisses timing by saying proof generation time doesn't vary with Merkle position — this misidentifies the timing channel. The relevant timing signal is not proof-generation latency but **which registry root the public input carries**, correlated with the known schedule of registry updates.

**In-threat-model?** No — the IND-ISS game does not model adaptive registry updates. The construction must either (a) add an anonymity-epoch assumption (all credentials are refreshed synchronously on each root update) or (b) bound the anonymity-set degradation as a function of update frequency and issuer churn rate.

### Attack 4: Schema Binding is Conventional, Not Cryptographic — A Registry Insider Can Exploit Semantic Divergence

**Attack:** Gadget 4a commits the structural predicate — `(clauseAttrIndex[i], clauseComparator[i], clauseThreshold[i])` for each clause — to the public `predicateHash`. The constraint is: the attribute at **index** `k` satisfies comparator `c` against threshold `t`. What Gadget 4a does NOT commit to is the **semantic meaning** of attribute index `k`.

In the §7 NCUA deployment, the predicate `chartered_by_NCUA == 1` is compiled to `clauseAttrIndex = [1, ...]` with the assumption that all NCUA-registry issuers place `chartered_by_NCUA` at attribute index 1. This is a convention established off-chain — the construction's predicate compiler and the issuers' credential schemas must agree on which index maps to which claim. The circuit enforces "attribute at index 1 equals 1"; it does not enforce "the attribute called `chartered_by_NCUA` equals 1."

A malicious issuer enrolled in the NCUA registry (a legitimate CU that turns adversarial) can issue credentials where:
- `attrValues[1] = 1` but this field in their schema is `active_member_count_nonzero` (trivially true for any active CU), not `chartered_by_NCUA`
- `attrValues[3] = 0` where index 3 is `chartered_by_NCUA` in their schema

This issuer generates a valid IssuerBlindPredicate proof with `predicateHash = Poseidon25(1, 0, 1, ...)` — the NCUA membership predicate. The proof verifies. The predicate evaluates to true (index 1 = 1). But the semantic claim being proven is *not* NCUA charter status — it is a structurally identical predicate over a semantically different attribute.

SPIFFE avoids this by using typed X.509 extension OIDs and a structured SVID format where the semantics of each claim are enforced by the SPIRE attestor's registration policy, not by convention. In the SPIFFE model, "chartered by NCUA" would be an OID-typed extension in the SVID, not an anonymous positional index.

**Why it works / why it fails against the construction:** The construction's defense is that only trusted issuers are in the registry. But the registry contains all NCUA-chartered CUs — and a CU can be legitimate at enrollment and malicious later. More importantly, even honest issuers can use divergent schemas if the schema agreement mechanism is off-chain and unspecified. The construction must either (a) include a schema commitment in the `predicateHash` (e.g., a hash of the schema definition that maps index 1 → `chartered_by_NCUA`), making the predicate semantically binding, or (b) define a schema registry and require in-circuit verification that the issuer's schema root matches a known commitment. Without this, predicate integrity (§3, Property 4) is incomplete: the Poseidon25 collision resistance argument defends structural substitution, not semantic substitution.

**In-threat-model?** No — §3 Property 4 ("Predicate integrity") and the Class 4 adversary (`A_PRED`) only model a prover who tries to change the clause *parameters* while holding the `predicateHash` fixed. The schema-binding attack is a prover who uses the *correct* clause parameters for a *different* attribute schema, producing a proof that is structurally valid but semantically fraudulent. This attack is not modeled and not defended against.
