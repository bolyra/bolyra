## Persona: auth0_pm

### Attack 1: Scope Blinder Is Unrecoverable State — Enterprises Cannot Accept This

- **Attack:** The entire unlinkability construction rests on `scopeBlinder` values stored in the "agent's local secure enclave" (§7, Step 2). The construction says nothing about what happens when that state is lost — device migration, agent restart, enclave wipe, or cloud redeployment. If a blinder is lost, the agent either (a) cannot generate valid proofs for existing scopes because `scopeNullifier` is deterministic per `(credential, scopeId, blinder)` — the RS's double-spend ledger has the old nullifier, the new blinder produces a different one, so the RS accepts a second token but now the AS has two distinct nullifier chains for the same scope, or (b) generates a fresh blinder, silently breaking the Sybil-detection invariant within a scope. The construction in §2 says the blinder is "stored locally" and then never mentions it again. There is no key recovery, no backup scheme, no rotation protocol. Enterprise customers run agents on auto-scaled fleets where ephemeral state is expected to be lost.

- **Why it works:** The construction is formally correct given persistent blinder storage, but the operational model is a silent assumption. Auth0's token store is recoverable (it's a database row). A lost scopeBlinder breaks the scope's double-spend ledger silently — the RS cannot detect that it has issued two authorization chains for the same logical agent at the same scope. No section of the construction addresses this.

- **In-threat-model?** No — the threat model in §3 assumes the adversary does not control agent local state, but says nothing about *unavailability* or *loss* of that state. The construction must define blinder persistence requirements, a secure backup/sync protocol across agent replicas, and what happens at rotation. Without this, no enterprise ops team can deploy it.

Under BSA §314(b), the CU may be required to share transaction information with law enforcement. If the AS never sees presentation proofs — which is the core security claim in §4 ("AS impotence") — the CU *cannot* meet these obligations without building a separate surveillance channel that re-introduces the linkability the construction was designed to eliminate. But it frames this as a *regulatory feature* for credit unions without acknowledging the BSA/AML counter-requirement. A CU's Chief Compliance Officer will immediately flag this: "You're telling me I can't see what my member's agent is doing, and that's a selling point?" Every credit union has a BSA officer. - **In-threat-model?** No — the construction's threat model treats AS impotence as a pure security win with no regulatory cost. The deployment scenario in §7 must be revised to either (a) acknowledge that CU deployments require a compliant audit overlay (re-introducing some linkability), or (b) propose a ZK-friendly AML approach (e.g., range-proof on transaction amount satisfying FINANCIAL_SMALL threshold without revealing merchant identity). Neither is present.

### Attack 3: 3-Second Proof Time Is a Non-Starter for Agentic Workloads

- **Attack:** §6 states the PLONK proving target is "<3s" for `ScopedPresentation`. The construction presents this as "latency-sensitive" and calls it acceptable. It is not. AI agent frameworks (LangChain, CrewAI, AutoGPT) make dozens to hundreds of tool calls per task. Each tool call that crosses a scope boundary requires a fresh `ScopedPresentation` proof. A single agent task touching three scopes (Amazon, pharmacy, insurance — §7) requires three sequential proofs: 3 × 3s = 9 seconds of cryptographic overhead before the agent can begin work. In comparison, WorkOS issues an OAuth token in <100ms via a standard `client_credentials` grant. The `presentationNonce` is fresh per-request (§2, Public Inputs), so proofs cannot be cached across requests. The construction targets <1.5s with Groth16 but requires a per-circuit Phase 2 ceremony — adding operational cost and ceremony management overhead that a solo-founder protocol cannot credibly sustain.

- **Why it fails partially:** The construction is correct that rapidsnark on commodity hardware achieves ~0.8s for 13K Groth16 constraints (§6). But "commodity hardware" for agent deployments is a Lambda function or a Kubernetes pod, not an M2 Mac. The benchmark reference (`circuits/scripts/bench_rapidsnark.js`) is local. The construction provides no evidence of proving time on AWS Lambda (arm64, 512MB) or a standard cloud container. On cold-start Lambda, native binary loading alone can add 2-5s.

- **In-threat-model?** No — the construction defines a timing side-channel extension (§3) that adds *additional* random jitter to proof submission, increasing the effective latency budget further. This widens the gap against <100ms incumbents. The construction must either demonstrate sub-1s cloud-native proving (e.g., via WASM snarkjs in V8 isolates with preloaded proving keys) or define a session token model where a single proof gates a time-bounded session, reducing per-call overhead.

### Attack 4: On-Chain Root Dependency Creates a Dual-Failure Liveness Problem

- **Attack:** Every RS verification in the construction requires checking `agentMerkleRoot` against the "on-chain root history" (§5, root history buffer, 30-entry circular buffer per tree). This means the RS must either maintain a live connection to a Base Sepolia / Base mainnet node or cache the root buffer locally. Both options create enterprise-blocking problems. A live node connection adds 100-500ms of RPC latency per verification and creates a hard dependency on EVM availability. A local cache creates a root-synchronization attack surface: if an attacker can delay the RS's cache update, a credential that was revoked on-chain (by removing it from the Merkle tree) will still verify successfully against the RS's stale root. The 30-entry buffer is a liveness window, not a security window — the construction in §2.1 and §5 describes it as a circular buffer but does not define: (a) how long a root stays valid before the RS must refresh, (b) what happens to in-flight proofs when the root rotates out of the buffer, or (c) whether an adversary can force rapid root turnover to evict a valid root before a proof is submitted (griefing the honest agent). Auth0 and WorkOS have globally distributed token validation CDN endpoints with <10ms latency. Bolyra's RS needs an Ethereum node. In enterprise procurement, a dependency on public blockchain RPC is an immediate IT security red flag ("we don't allow our production auth path to depend on a third-party blockchain node").

- **Why it works:** The construction is cryptographically sound on the root-check. But operationally, the 30-entry circular buffer and on-chain dependency introduce a failure mode that has no analog in OAuth: if Base Sepolia is congested or the RS's cache is stale, valid proofs fail. The construction must specify RS caching semantics, root TTL, revocation latency bounds, and either a fallback for chain unavailability or a clear statement that revocation is non-real-time (with quantified risk).

- **In-threat-model?** No — §3 defines the adversary as controlling up to N-1 RSes and the AS, but does not model a blockchain-level DoS or root rotation griefing attack. The deployment scenario in §7 (Step 3: "Amazon verifies proof against on-chain agentMerkleRoot") glosses over this entirely. A production RS operator will ask: "What is our SLA if Base is down?" The construction has no answer.

### Attack 1: Scope ID Collision by Adversarial RSes

- **Attack:** The adversary controls N-1 RSes. Those RSes coordinate to present the **same** `scopeId` value to the target agent. Since `scopeId` is a public input chosen by the verifier, nothing in the circuit or protocol prevents two colluding RSes from using `scopeId = Poseidon("shared-sentinel")`. The agent's `scopeBlinder` is keyed per `(credential, scopeId)` — not per `(credential, RS, scopeId)`. Identical `scopeId` → identical `innerHash` → identical `scopeNullifier`. Two colluding RSes see `scopeNullifier_α = scopeNullifier_β` and immediately link the agent.

- **Why it works:** §2's `scopeBlinder` design invariant ("same agent + different scope → different scopeNullifier") relies on `scopeId` being RS-unique. But this is an application-layer convention — the circuit enforces nothing. The SES-UNL game (§3) assumes distinct `scopeId_α` and `scopeId_β` are **independently chosen by the challenger**, but in deployment the adversary (controlling N-1 RSes) chooses the `scopeId` inputs. The game does not restrict the adversary's choice of `scopeId` at controlled RSes.

- **In-threat-model? NO.** The adversary is within the stated capability boundary (controls RSes, chooses all public inputs). The construction must either (a) enforce globally unique, unforgeable `scopeId` values via an on-chain registry the adversary cannot write to, or (b) key `scopeBlinder` on `(credential, RS_identifier, scopeId)` where `RS_identifier` is itself committed on-chain and non-collidable.

### Attack 3: Poseidon PRF Assumption Is Not Closed for Poseidon3 in Third-Argument Position

- **Attack:** This is a reduction gap, not a direct adversary strategy. The stated P-PRF assumption (§4) covers: *for random key k, the function x → Poseidon2(x, k) is computationally indistinguishable from a random function.* The key is the second argument. In the SES-UNL reduction, Hybrid H₂ → H₃ requires that `blindedScopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, scopeBlinder)` be computationally uniform given secret `scopeBlinder`. This requires the function `k → Poseidon3(a, b, k)` — with fixed `a = permissionBitmask` (drawn from an 8-bit sparse space: only ~256 values), `b = credentialCommitment` — to be a PRF in its **third argument**. This is a distinct assumption: call it **P-PRF3**. It is never stated.

- **Why it matters:** `permissionBitmask` is NOT uniformly distributed. In practice, agents cluster around a handful of permission tiers (e.g., `0x07`, `0x1F`, `0xFF`). An adversary with a quantum algebraic advantage over Poseidon (Gröbner basis attack over BN254's prime field) could exploit the low-entropy first input to `Poseidon3` to build a partial preimage distinguisher. Concretely: fixing `a` to one of 256 values reduces the algebraic system dimension and makes interpolation attacks cheaper. The existing benchmarks and MDS matrix design target uniform inputs; the low-entropy prefix case is unstudied in the Poseidon specification.

- **In-threat-model? NO** (as stated). The reduction in §4 does not invoke P-PRF3 as a named assumption, so the theorem's security guarantee has a hole. The construction must either (a) add P-PRF3 as an explicit named assumption with supporting evidence from the Poseidon design document, or (b) restructure `blindedScopeCommitment` so the key material is always in first/second position (matching the stated P-PRF assumption).

### Attack 4: Variable Merkle Depth Is a Timing Oracle — the Formal Proof Does Not Close It

- **Attack:** The circuit takes `merkleProofLength` as a private input (§2: "≤20, actual Merkle depth"). Circom witness generation and snarkjs/rapidsnark proof time scale with the number of active Poseidon2 calls, which depends on `merkleProofLength`. An agent enrolled early in a shallow tree (depth 3) generates a proof in ~0.4s; one enrolled later at depth 17 takes ~1.6s. Network-layer timing to the RS exposes this. The adversary (AS) knows the enrollment time and can infer approximate leaf index from the Merkle tree growth log (on-chain, public). Cross-correlating RS-observed proof timing with AS-held enrollment timestamps recovers a probabilistic binding between (RS-observed auth, AS-held agent identity) — exactly what both games are designed to prevent.

- **Why the "timing extension" in §3 doesn't fix it:** The timing extension says "the challenger samples proof generation time from a fixed distribution (uniform over [0, 500ms] jitter)" and calls this "an application-layer requirement, not a circuit constraint." But the reduction sketches in §4 do not include timing in the hybrid argument. Hybrid H₀ through H₃ operate on the proof's public outputs only; timing is treated as orthogonal. This means the formal security claim in §1 ("no PPT adversary controlling the AS and up to N-1 RSes wins the IND-UNL-AS game with advantage greater than negligible") is stated for a computational adversary, but a timing adversary is polynomial-time and the game does not include a timing oracle. The 500ms jitter is also insufficient: for a depth-1 vs depth-20 agent, the raw variance exceeds 1.2s on commodity hardware (per existing `bench_rapidsnark.js` data cited in §6).

- **In-threat-model? NO** (gap). The timing extension must be either (a) formally integrated into the IND-UNL-AS/SES-UNL game definitions as a timing oracle query that the adversary can make, with the reduction bounding the distinguishing advantage over the jitter distribution, or (b) the circuit must pad all Merkle paths to exactly depth 20 (constant-time witness generation), eliminating the side channel at the circuit level. The latter is the cleaner fix — `merkleProofLength` should be removed as a private input; the circuit always runs 20 Poseidon2 levels, with unused levels hashing fixed dummy siblings.

### Attack 1: `agentMerkleRoot` Is a Public Epoch Stamp — SES-UNL Proof Ignores Anonymity Set Collapse

The colluding RSes now have a 1-in-3 guessing advantage, far exceeding negligible.

- **In-threat-model?** **No — construction must address.** The construction needs either: (a) a minimum anonymity set guarantee (e.g., padding every batch to ≥ k enrollments), (b) a proof that the epoch window does not degrade the advantage bound, or (c) acknowledgment in §3 threat model that small-cohort deployments lose the privacy property.

### Attack 3: The SES-UNL Game's Identical-Attribute Assumption Hides Real-World Credential Heterogeneity

- **Attack:** Game 2 (§3) stipulates "Challenger enrolls three agents A₀, A₁, A₂ with *identical* `permissionBitmask` and `expiryTimestamp`." In production, different agents have different permission profiles. The circuit exposes `requiredScopeMask` as a public input — the RS supplies the minimum bitmask it demands. In Reduction 2 §4, the `blindedScopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, scopeBlinder)` is claimed to be computationally uniform because `scopeBlinder` is secret. However, if `permissionBitmask` takes only 4-8 distinct values across the deployment (e.g., READ\_DATA-only, FINANCIAL\_SMALL, FINANCIAL\_MEDIUM, FINANCIAL\_UNLIMITED), then `blindedScopeCommitment` is drawn from at most 4-8 Poseidon orbits, one per `permissionBitmask` value. A colluding RS-A and RS-B who each independently set `requiredScopeMask` tightly (to probe exact permission levels) can use the `requiredScopeMask` values they choose to narrow the candidate `permissionBitmask` to a small set — then apply a meet-in-the-middle across the two `blindedScopeCommitment` values. Specifically: for each candidate `permissionBitmask` in the small set, and for each candidate `credentialCommitment` in the enrollment set, sample a distribution of `blindedScopeCommitment` values and compare to the observed outputs. With a 4-value bitmask space and a 50-agent enrollment, this reduces the search space by a factor of ~200x versus a uniform PRF.

- **Why it works / why it fails:** This is not a PRF break — it is a *PRF with low-entropy input*. Poseidon3(m, c, b) is computationally indistinguishable from random when b is uniform random over F\_p. But the adversary doesn't need to break Poseidon; they exploit that the number of distinct (permissionBitmask, credentialCommitment) pairs is bounded by the deployment size. The SES-UNL proof in §4 steps through H₃ assuming that `blindedScopeCommitment` "distribution is identical in both worlds" — but this relies on `scopeBlinder` being the sole source of randomness. If the pre-image space is small enough, distinguishing b=0 (shared `credentialCommitment`) from b=1 (distinct `credentialCommitment`) may be feasible without breaking P-PRF in the standard sense, simply by exhaustive enumeration.

- **In-threat-model?** **Yes — construction survives with a fix, but must address.** The construction should require a *minimum anonymity set* for `(permissionBitmask, credentialCommitment)` pairs, or replace `permissionBitmask` in `blindedScopeCommitment` with a Poseidon hash of `(permissionBitmask ‖ scopeBlinder)` to ensure the PRF input entropy is always λ bits regardless of bitmask cardinality.

- **Attack:** The entire SES-UNL security argument (§4, Reduction 2, "AS impotence") rests on one sentence: "The adversary does NOT control: The agent's local state (secret key, scope blinders)." For human-operated devices this is reasonable. For AI agents — the construction's primary use case — this assumption is operationally inverted. AI agents run on *operator-controlled infrastructure*: the CU's cloud environment (credit union scenario, §7) or Kaiser's EHR cluster (healthcare scenario, §7). In both cases, the operator IS the issuer, and the issuer is the adversarial AS in the threat model. The "CU-as-AS must not see member merchant graph" scenario requires the CU to *both* operate the agent compute environment (and therefore have access to the file system where `scopeBlinder` values are stored) *and* be the party the construction claims is excluded from the presentation path. The RFC 7662 baseline does not require this trust split — PPID is enforced AS-side, not dependent on an agent secret that the AS's own infrastructure hosts.

- **Why it works / why it fails:** The threat model in §3 explicitly excludes agent local state from adversary control. But the deployment scenario in §7 places the agent on CU-operated infrastructure. This creates a gap: the formal security proof holds under the model, but the model's assumption does not hold in the advertised deployment. By contrast, in OAuth with per-RS PPID, the unlinkability property (at the RS level) is enforced by AS policy — and the AS in that role is *intended* to be trusted. Bolyra's construction inverts the trust relationship without addressing the hosting gap for cloud-hosted AI agents. The IND-UNL-AS game models an adversarial AS who observes all token-issuance metadata — but does not model an AS who also controls the agent runtime. That is the correct threat model for the stated deployment and the paper does not address it.

- **In-threat-model?** **No — construction must address.** The threat model needs a deployment-specific trust boundary: "the operator who hosts the agent is distinct from the AS." If they are the same party (the common case in enterprise AI deployments), the `scopeBlinder` isolation fails outside the circuit and the entire unlinkability claim reduces to operator policy, not cryptographic enforcement. The construction should either (a) require hardware-backed secure enclaves (TEE) for `scopeBlinder` storage on operator infrastructure, or (b) scope the security claim to deployments where the agent runtime is operated by a party distinct from the credential issuer.

### Attack 2: `agentMerkleRoot` is a trust-domain fingerprint — the formal games are blind to it

- **Attack:** Every `ScopedPresentation` proof outputs `agentMerkleRoot` as a **public** value. In a real deployment, different issuers (different CUs, different enterprises) maintain separate agent Merkle trees with distinct roots. The root is stable for the lifetime of the tree (updated only when new agents enroll, via the 30-entry root history buffer in §5). A coalition of RSes — even without the AS — can trivially cluster all presentations by root value. Within a root cluster, they know: (a) all these proofs came from agents enrolled in the same deployment, and (b) the deployment is identifiable because the root is public on-chain. In the CU scenario from §7, Amazon and the pharmacy don't need to link individual agents — they can determine "Alice's agent came from Navy Federal's tree" by matching the root against on-chain issuance events, and then narrow the window of candidates to the ~13M member enrollment cohort. - **Why it works:** The IND-UNL-AS and SES-UNL games (§3) fix both agents A₀, A₁ in the same tree (same root), so the root provides no signal within the game. But the games never model *cross-deployment* distinguishability. In SPIFFE terms: the `agentMerkleRoot` is equivalent to publishing the SPIFFE trust domain on every request. ### Attack 3: `scopeBlinder` is an unmanaged secret with no infrastructure — SPIRE has this; Bolyra doesn't

- **Attack:** The entire unlinkability property reduces to the secrecy of per-(credential, scopeId) `scopeBlinder` values. These are described in §2 as "agent-generated, stored locally" in the agent's "local secure enclave" (§7). The construction provides no mechanism for: (a) backup and recovery, (b) rotation when a blinder is suspected compromised, or (c) synchronisation across agent replicas (load-balanced agent processes). If the local store is lost, the agent must generate new blinders — producing new `scopeNullifier` values for existing scopes. This has two consequences. First, from the RS's perspective, Sybil detection breaks: the RS's double-spend set for a given `scopeNullifier` is now bypassed because the agent appears as a fresh identity. Second, a malicious agent can deliberately reset its blinders to evade per-scope rate limiting or revocation. SPIFFE addresses credential lifecycle through the Workload API: SPIRE agents hold SVIDs with defined TTLs, rotation is automated, and replicas share the same certificate via the Workload API socket. Bolyra delegates all of this to "local secure enclave" — a placeholder, not a protocol.

- **Why it works:** This is not a cryptographic break of the circuit; it is an operational break of the security assumption. The IND-UNL-AS reduction (§4) assumes the `scopeBlinder` is "agent-generated" and secret. If the blinder is recoverable (backed up), the backup store is a new attack surface. If it is not recoverable, losing it silently escalates to a Sybil bypass. Neither case is addressed. The WIMSE BCP (draft-ietf-wimse-workload-identity-bcp) §5 explicitly requires credential storage to be bound to the workload runtime — not to a locally managed file.

- **In-threat-model?** **No.** The construction must define a `scopeBlinder` management protocol: how blinders are generated (deterministically from a master secret, or randomly with backup?), where they are stored (HSM, TEE, SPIRE Workload API extension?), what happens on rotation, and how replica agents share blinders without leaking them. Deterministic derivation (`scopeBlinder = PRF(masterSecret, credentialCommitment || scopeId)`) would solve the recovery problem but adds a new assumption (masterSecret secrecy) and changes the threat model if the masterSecret is extractable.

### Attack 4: `scopeId` collision — no global namespace enforcement, adversarial AS can force linkage

- **Attack:** The `scopeId` is defined as `Poseidon("CU-A-merchant-read")` — an RS-controlled string hashed into a field element. The construction provides no enforcement mechanism for global uniqueness of scopeId strings. Two RSes that independently choose the same scope label — for example, both using `"read"` or `"payments-v1"` — will produce identical `scopeId` values. An agent presenting to both RSes will generate the same `scopeNullifier` at both, because `scopeNullifier = Poseidon2(scopeId, innerHash)` is deterministic. This collapses the unlinkability guarantee: the two RSes share the same nullifier and can directly link the agent's sessions. Worse, an adversarial AS can *influence* scopeId selection at RSes it controls (e.g., by dictating scope naming conventions to enrolled RSes) to force collisions and selectively de-anonymize specific agents. SPIFFE avoids this with the `spiffe://trust-domain/path` URI scheme — the trust domain is cryptographically bound to the issuer's certificate, making global uniqueness structurally enforced.

- **Why it works:** The SES-UNL game (§3) assumes `scopeId_α ≠ scopeId_β` — the two challenge scopes are distinct by construction. But it does not model the case where the adversary can influence scopeId assignment. If the AS controls RS onboarding (registering RSes in the CU deployment), it can register two RSes with the same scopeId and watch for nullifier collisions in the proofs sent to those RSes. This is a **chosen-scopeId attack** not modelled by either game.

- **In-threat-model?** **No.** The construction must either: (a) define a scopeId assignment authority with collision resistance guarantees (analogous to SPIFFE trust domain registration), (b) require RS-specific namespace prefixing enforced at the circuit level (e.g., `scopeId = Poseidon(rsPubkey, scopeLabel)` where `rsPubkey` is a public input), or (c) explicitly bound the unlinkability claim to "distinct scopeIds" and require that RS operators ensure uniqueness via out-of-band coordination. Option (b) is the cleanest: if the RS's public key (or on-chain identifier) is mixed into the `scopeId` derivation as a public circuit input, the adversarial AS cannot manufacture collisions without compromising the RS's key.
