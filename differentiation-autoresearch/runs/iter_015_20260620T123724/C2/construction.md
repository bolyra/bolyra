# Construction

## 1. Statement of claim

The same Bolyra agent accessing *k* distinct Resource Servers produces *k* authorization proofs that are computationally indistinguishable from proofs generated by *k* independent agents, even to an adversarial Authorization Server (AS) that controls the agent Merkle tree, observes all proof submissions, and colludes with up to *k−1* of the *k* RSes. Cross-scope unlinkability holds under the Poseidon PRF assumption on BN254 and the knowledge soundness of Groth16/PLONK.

## 2. Construction (gadgets, circuits, public/private inputs)

### New circuit: `ScopeBlindedAgent(MAX_DEPTH)`

This circuit extends `AgentPolicy` with a scope-blinding layer. The agent proves credential validity and scope satisfaction while producing a **scope-specific pseudonym** that is unlinkable across scopes. Critically, `scopeId` is a **private** input — it never appears on-chain. The RS confirms the proof targets its scope via a verifier-local consistency check on a blinded scope tag.

**Private inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `modelHash` | F_p | Hash of model identifier |
| `operatorPubkeyAx`, `operatorPubkeyAy` | F_p | Operator EdDSA public key |
| `permissionBitmask` | 64-bit | Permission bitfield |
| `expiryTimestamp` | 64-bit | Credential expiration |
| `sigR8x`, `sigR8y`, `sigS` | F_p | Operator EdDSA signature |
| `merkleProofLength`, `merkleProofIndex`, `merkleProofSiblings[MAX_DEPTH]` | — | Merkle inclusion proof |
| `blindingScalar` | F_p | Fresh random blinding value per proof |
| `scopeId` | F_p | RS-specific scope identifier — **private, never on-chain** |

**Public inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `requiredScopeMask` | 64-bit | Required permission bits |
| `currentTimestamp` | 64-bit | Current time |
| `sessionNonce` | F_p | Fresh per-session nonce |

**Public outputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `agentMerkleRoot` | F_p | Computed Merkle root |
| `scopePseudonym` | F_p | `Poseidon2(credentialCommitment, scopeId)` — unlinkable across scopes |
| `blindedScopeCommitment` | F_p | `Poseidon3(permissionBitmask, credentialCommitment, blindingScalar)` — rerandomized per proof |
| `sessionNullifier` | F_p | `Poseidon3(scopePseudonym, sessionNonce, blindingScalar)` — replay prevention per session×scope |
| `scopeTag` | F_p | `Poseidon2(scopeId, blindingScalar)` — verifier-local scope binding |

**Circuit constraints (11 steps):**

1. **Range checks:** `Num2Bits(64)` on `permissionBitmask`, `expiryTimestamp`, `currentTimestamp`.
2. **Credential commitment:** `credentialCommitment = Poseidon5(modelHash, operatorPubkeyAx, operatorPubkeyAy, permissionBitmask, expiryTimestamp)`.
3. **EdDSA signature:** `EdDSAPoseidonVerifier(operatorPubkey, credentialCommitment, sig)`.
4. **Merkle membership:** `BinaryMerkleRoot(MAX_DEPTH, credentialCommitment, proof) == agentMerkleRoot`.
5. **Scope satisfaction:** For each bit *i* in [0, 64): `requiredBits[i] * (1 - permBits[i]) === 0`.
6. **Cumulative bit encoding:** Standard Bolyra tier implication constraints on bits 2–4.
7. **Expiry check:** `currentTimestamp < expiryTimestamp` via `LessThan(64)`.
8. **Scope pseudonym:** `scopePseudonym = Poseidon2(credentialCommitment, scopeId)`.
9. **Blinded scope commitment:** `blindedScopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, blindingScalar)`.
10. **Session nullifier:** `sessionNullifier = Poseidon3(scopePseudonym, sessionNonce, blindingScalar)`.
11. **Scope tag:** `scopeTag = Poseidon2(scopeId, blindingScalar)`.

**Why `scopeTag` works — verifier-local consistency check:**

The RS already knows its own `scopeId` (computed as `Poseidon2(RS_domain, RS_public_key)`). The agent transmits `blindingScalar` to the RS over the direct TLS channel that already carries `scopeId` and `sessionNonce`. The RS recomputes `Poseidon2(scopeId, blindingScalar)` locally and checks that it equals the `scopeTag` public output from the proof. If they match, the proof was generated for this RS's scope. If they don't match, the proof targeted a different RS and the RS rejects.

**Why the AS cannot exploit `scopeTag`:**

The AS observes `scopeTag` on-chain but does not know `blindingScalar` (private input, transmitted only to the RS over a direct channel the AS does not control). Even if the AS has a dictionary of all possible `scopeId` values (e.g., all known RS domains), it cannot test candidates: each candidate `scopeId'` would require the correct `blindingScalar` to produce the observed `scopeTag`, and `blindingScalar` is drawn uniformly from F_p per proof. Under the Poseidon PRF assumption (A1), `Poseidon2(scopeId, blindingScalar)` is indistinguishable from random when `blindingScalar` is unknown, regardless of whether `scopeId` is drawn from a small dictionary.

**Why `blindingScalar` disclosure to the RS is safe:**

The RS learns `blindingScalar` but cannot use it to link proofs across scopes. To link a proof to a different RS's scope, the attacker would need the other proof's `blindingScalar` (which is independent — fresh per proof) and the other RS's `scopeId`. Even if RS_A colludes with the AS and learns `blindingScalar_A`, the proof for RS_B used an independent `blindingScalar_B` that neither RS_A nor the AS possesses.

**Constraint estimate:** ~11,514 constraints (prior estimate of ~10,714 + one additional Poseidon2 at ~800 for `scopeTag`). Fits within 2^14 (16,384); `pot16.ptau` covers.

### Protocol flow (scope-blinded handshake)

1. RS generates `(sessionNonce, scopeId)` where `scopeId = Poseidon2(RS_domain, RS_public_key)`. The RS sends both to the agent over a direct TLS channel. The AS never sees `scopeId`.

2. Agent generates `ScopeBlindedAgent` proof with fresh `blindingScalar ← random(F_p)`.

3. Agent submits proof to the on-chain registry. The registry sees `(agentMerkleRoot, scopePseudonym, blindedScopeCommitment, sessionNullifier, scopeTag, requiredScopeMask, currentTimestamp, sessionNonce)`. Note: `scopeId` does NOT appear in public inputs or outputs — the registry never sees it.

4. Registry verifies:
   - `sessionNonce` freshness (not reused).
   - `agentMerkleRoot` in root history buffer.
   - Groth16/PLONK proof validity.
   - `sessionNullifier` not previously recorded (replay prevention).

5. Agent transmits `blindingScalar` to the RS over the direct TLS channel (same channel used in step 1).

6. RS performs the **verifier-local consistency check**: compute `Poseidon2(scopeId, blindingScalar)` and assert equality with the `scopeTag` public output from the proof. If the check fails, the RS rejects the authorization. This check is performed locally by the RS — it does not require any on-chain interaction or AS involvement.

7. RS verifies that the proof's `requiredScopeMask` matches the scope policy the RS advertised. RS accepts the authorization.

### Batched submission (timing side-channel mitigation)

To prevent the AS from correlating proof submissions by timing, agents MAY use a **batch relay**:
- A relay collects proofs from multiple agents over a mixing window *Δt* (e.g., 30 seconds).
- The relay shuffles and submits proofs in a single multicall transaction.
- The AS sees *n* proofs arriving simultaneously with no per-agent timing signal.

This is an application-layer defense; the cryptographic unlinkability holds without it.

## 3. Threat model (adversary capabilities, game definition)

### Adversary capabilities

The adversary **A** controls:
- The Authorization Server (AS): full read access to the agent Merkle tree, all proof submissions, all public signals (including `scopeTag`).
- Up to *k−1* of *k* target RSes: A learns the `scopeId`, `blindingScalar`, `requiredScopeMask`, and proof for those RSes.
- Timing of proof submissions (unless batch relay is used).

The adversary **A** does NOT control:
- The agent's private key or `blindingScalar` for honest RSes.
- The ZK proving process (no malicious prover).
- The direct TLS channel between agent and at least one honest RS (the one whose traffic A tries to link).
- At least one RS that remains honest.

### IND-UNL-AS game

**Game IND-UNL-AS(λ):**

1. **Setup:** Challenger generates two agents `(cred₀, cred₁)` with identical permission bitmasks and enrolls both in the Merkle tree. Challenger generates *k* RS scope identifiers `{scopeId_1, …, scopeId_k}`.

2. **Challenge:** Challenger flips coin `b ← {0, 1}`. For each `scopeId_j` (j = 1…k), challenger generates a valid `ScopeBlindedAgent` proof using agent `cred_b` with fresh `blindingScalar_j` and `sessionNonce_j`.

3. **Oracle access:** A may:
   - Request proofs for any `scopeId` of A's choosing for either agent (adaptive queries, up to poly(λ) queries).
   - Observe all public signals from all proofs (including `scopeTag`).
   - Corrupt up to *k−1* RSes, learning their `scopeId` values **and the `blindingScalar` values** transmitted to those RSes.

4. **Restriction:** A may NOT query both agents on the same `scopeId` as the challenge scope (trivial distinguisher via Merkle root + known tree).

5. **A outputs** guess `b'`.

**Advantage:** `Adv_A = |Pr[b' = b] - 1/2|`.

**Claim:** For all PPT adversaries A, `Adv_A ≤ negl(λ)` under the Poseidon PRF assumption on BN254 and knowledge soundness of Groth16/PLONK.

### scopeTag under adversarial AS — why privacy holds

The AS sees `scopeTag = Poseidon2(scopeId, blindingScalar)` on-chain. Two attack vectors:

**Dictionary attack (AS knows candidate scopeIds):** The AS enumerates candidate `scopeId'` values. For each candidate, it would need to find `blindingScalar` such that `Poseidon2(scopeId', blindingScalar) = scopeTag`. Under Poseidon preimage resistance, this requires brute-forcing `blindingScalar` over F_p (~254 bits), which is computationally infeasible.

**Collusion attack (AS + corrupted RS):** When A corrupts RS_j, A learns both `scopeId_j` and `blindingScalar_j`. A can verify that `Poseidon2(scopeId_j, blindingScalar_j) = scopeTag_j` — this confirms which proof targeted RS_j, but A already knew this by corrupting RS_j. For the honest RS_h, A does not know `blindingScalar_h` (transmitted only over the direct TLS channel to RS_h), so `scopeTag_h` remains opaque. A cannot test whether `scopeTag_h` corresponds to `scopeId_h` (even if A guesses `scopeId_h`) without `blindingScalar_h`.

## 4. Security argument (named assumption + reduction sketch)

### Named assumptions

- **A1 — Poseidon PRF:** `Poseidon2(·, ·)` is a pseudorandom function when keyed on its first argument over the BN254 scalar field. Specifically, `f_k(x) = Poseidon2(k, x)` is computationally indistinguishable from a random function for any fixed key `k` unknown to the adversary.

- **A2 — Knowledge soundness of Groth16 (ROM):** In the random oracle model with the BN254 pairing, Groth16 satisfies knowledge soundness: a valid proof implies the prover knows a satisfying witness.

- **A3 — Discrete log hardness on Baby Jubjub:** Given `(Ax, Ay) = BabyPbk(s)`, recovering `s` is computationally intractable.

### Reduction sketch

**Theorem:** If there exists a PPT adversary A winning IND-UNL-AS with non-negligible advantage, then there exists a PPT distinguisher D breaking the Poseidon PRF assumption.

**Proof sketch:**

1. **Scope pseudonym unlinkability.** The scope pseudonym is `Poseidon2(credentialCommitment, scopeId)`. For two different scopes `scopeId_i ≠ scopeId_j`, the pseudonyms are `Poseidon2(cc, scopeId_i)` and `Poseidon2(cc, scopeId_j)`. Under A1, keying Poseidon on `cc` (which is hidden as a private input), the outputs are indistinguishable from random. D simulates IND-UNL-AS for A: when A distinguishes two pseudonyms as coming from the same `cc`, D uses this to distinguish Poseidon from random, contradicting A1.

2. **Scope tag unlinkability.** The scope tag is `Poseidon2(scopeId, blindingScalar)`. Since `blindingScalar` is drawn uniformly from F_p and is unknown to the AS, under A1 (keying Poseidon on `blindingScalar`), the output is indistinguishable from random regardless of the `scopeId` input. Two scope tags from the same agent with different `blindingScalar` values are independently pseudorandom. The AS gains no information about which RS the proof targets from the scope tag alone. If A could distinguish scope tags as targeting the same `scopeId` across two proofs (with independent blinding scalars), then A breaks the PRF assumption on Poseidon2 keyed by the blinding scalar — contradicting A1.

3. **Blinded scope commitment unlinkability.** The blinded scope commitment includes a fresh `blindingScalar` per proof. Even if `permissionBitmask` and `credentialCommitment` are identical across proofs, the output `Poseidon3(permissionBitmask, cc, blindingScalar)` is fresh each time. Under A1, this is indistinguishable from random for each new `blindingScalar`.

4. **Session nullifier unlinkability.** The session nullifier `Poseidon3(scopePseudonym, sessionNonce, blindingScalar)` depends on the already-unlinkable `scopePseudonym` and a fresh `blindingScalar`. Composing two PRF applications preserves pseudorandomness under A1.

5. **Witness hiding via ZK.** By the zero-knowledge property of Groth16/PLONK (A2), the proof transcript reveals nothing about private inputs (`credentialCommitment`, `blindingScalar`, `scopeId`, operator key, Merkle path). The adversary's view is limited to public signals, which are unlinkable per (1)–(4).

6. **Collusion resistance (k−1 RSes).** Corrupting RS_j reveals `scopeId_j` and `blindingScalar_j` to A. A can verify that `scopeTag_j = Poseidon2(scopeId_j, blindingScalar_j)` and can compute `Poseidon2(cc_guess, scopeId_j)` for candidate credential commitments against `scopePseudonym_j`. But `cc` is never revealed (private input), and the tree contains *N* agents. For the honest RS_h, A knows neither `scopeId_h` nor `blindingScalar_h`. Even if A guesses `scopeId_h`, it cannot verify the guess against `scopeTag_h` without `blindingScalar_h`. The advantage reduces to brute-forcing `blindingScalar_h` over F_p, which is negligible.

**QED (sketch).**

### Side-channel treatment

- **Timing:** Batch relay (§2) ensures proof submission times are shuffled within a mixing window. Without batch relay, timing correlation is possible but does not break cryptographic unlinkability — it is a traffic-analysis attack mitigated at the network layer.
- **Nonce freshness:** Each `sessionNonce` is unique per handshake. The `sessionNullifier` binds the proof to a specific `(scopePseudonym, sessionNonce, blindingScalar)` triple. Replaying a proof with a stale nonce fails the freshness check.
- **Merkle root correlation:** If the agent tree is small, the AS could narrow the anonymity set by observing which Merkle root is used. The 30-entry root history buffer and the shared tree across all agents mitigate this — every agent references the same tree.
- **`scopeTag` dictionary resistance:** The AS observing `scopeTag` on-chain cannot test candidate `scopeId` values without the corresponding `blindingScalar`. The blinding scalar is transmitted only over the direct agent-to-RS TLS channel, which the AS does not control. This holds even if the AS maintains a complete dictionary of all RS `scopeId` values.

## 5. Bolyra primitive mapping

| Construction element | Bolyra primitive | Source |
|---|---|---|
| Credential commitment | `Poseidon5(modelHash, Ax, Ay, permissionBitmask, expiryTimestamp)` | AgentPolicy circuit (spec §4.2) |
| Scope pseudonym | `Poseidon2(credentialCommitment, scopeId)` | NEW — analogous to human nullifier `Poseidon2(scope, secret)` |
| Blinded scope commitment | `Poseidon3(permissionBitmask, credentialCommitment, blindingScalar)` | NEW — rerandomized variant of `scopeCommitment` |
| Session nullifier | `Poseidon3(scopePseudonym, sessionNonce, blindingScalar)` | NEW — extends agent nullifier pattern |
| Scope tag | `Poseidon2(scopeId, blindingScalar)` | NEW — verifier-local scope binding; `scopeId` private, RS checks locally |
| Operator signature | EdDSA on Baby Jubjub over `credentialCommitment` | AgentPolicy circuit (spec §4.2) |
| Merkle membership | Lean Incremental Merkle Tree, depth 20, Poseidon2 node hash | Shared agent tree (spec §3.1) |
| Scope satisfaction | Bitwise AND check: `requiredBits[i] * (1 - permBits[i]) === 0` | AgentPolicy circuit (spec §4.2) |
| Cumulative encoding | Tier implication constraints on bits 2–4 | Permissions model (CLAUDE.md) |
| Proving system | PLONK (universal setup, no per-circuit ceremony) for agent circuits | Spec §3.3 (OPTIONAL for agent) |
| Delegation chain entry | `blindedScopeCommitment` replaces `scopeCommitment` as chain seed | Delegation circuit (spec §5) — requires adapter |

**Delegation compatibility:** The existing Delegation circuit links via `previousScopeCommitment = Poseidon2(delegatorScope, delegatorCredCommitment)`. To chain from a `ScopeBlindedAgent` proof, the delegator must reveal `credentialCommitment` to the delegation circuit as a private input and prove consistency with the `blindedScopeCommitment`. This requires a one-constraint adapter: `Poseidon3(delegatorScope, delegatorCredCommitment, blindingScalar) == blindedScopeCommitment` (blinding scalar also private to the delegator). The delegation circuit then proceeds with standard chain linking.

## 7. Concrete deployment scenario

### Credit union member agent — merchant graph privacy

**Stakeholder:** Desert Financial Credit Union (DFCU), a $9B credit union in Arizona with 400K members.

**Setup:** DFCU operates an Authorization Server for its members' AI agents. Members deploy agents to interact with merchants (Target, Costco, local pharmacies) for price comparison, coupon redemption, and loyalty point management. Each merchant operates a Resource Server.

**Problem under baseline:** DFCU-as-AS sees every token issuance: "Member #4821's agent requested `merchant-read` scope for Target at 14:02, then `pharmacy-read` scope for CVS at 14:07." DFCU reconstructs the member's complete merchant graph. Under NCUA Regulation §748 (member data protection), this is a compliance liability. Under the CFPB's proposed open-banking rules (§1033), the AS must not use authorization metadata to build behavioral profiles.

**Bolyra deployment:**

1. DFCU enrolls member agents in the Bolyra agent Merkle tree. Each agent has an operator-signed credential with `permissionBitmask = 0b00000101` (READ_DATA + FINANCIAL_SMALL).

2. Target's RS has `scopeId_Target = Poseidon2("target.com", Target_RS_pubkey)`. CVS has `scopeId_CVS = Poseidon2("cvs.com", CVS_RS_pubkey)`. These are computed locally by each RS and shared with the agent over a direct TLS channel. DFCU (AS) never receives these values.

3. Member's agent generates `ScopeBlindedAgent` proof for Target with `blindingScalar₁ ← random(F_p)`:
   - `scopeId_Target` is a **private input** — it does not appear on-chain.
   - `scopePseudonym_Target = Poseidon2(credCommitment, scopeId_Target)` — visible on-chain but unlinkable.
   - `scopeTag_Target = Poseidon2(scopeId_Target, blindingScalar₁)` — visible on-chain; DFCU cannot invert without `blindingScalar₁`.
   - `blindedScopeCommitment_Target = Poseidon3(0b00000101, credCommitment, blindingScalar₁)` — unique to this proof.

4. Agent transmits `blindingScalar₁` to Target over the direct TLS channel. Target recomputes `Poseidon2(scopeId_Target, blindingScalar₁)` and verifies it matches `scopeTag_Target` from the proof. Target accepts.

5. Same agent generates `ScopeBlindedAgent` proof for CVS with independent `blindingScalar₂ ← random(F_p)`:
   - `scopeId_CVS` is private. `scopePseudonym_CVS` differs from Target's. `scopeTag_CVS = Poseidon2(scopeId_CVS, blindingScalar₂)` — independent of `scopeTag_Target`.

6. Agent transmits `blindingScalar₂` to CVS. CVS performs its own local consistency check.

7. DFCU (AS) observes two proofs on-chain. Both reference the same `agentMerkleRoot` (shared tree with 400K agents). The `scopePseudonym` values differ. The `scopeTag` values differ. The `blindedScopeCommitment` values differ. `scopeId` never appears in any public signal. Even if DFCU knows that Target and CVS are possible RSes, it cannot test `Poseidon2(scopeId_Target, ?)` or `Poseidon2(scopeId_CVS, ?)` without the blinding scalars, which were transmitted only to the respective RSes.

**Quantified privacy:** DFCU's advantage in IND-UNL-AS is bounded by `negl(λ)` under Poseidon PRF. The anonymity set is the full agent tree (400K members). Even if DFCU colludes with Target (corrupting 1 of 2 RSes and learning `blindingScalar₁`), it cannot link the CVS proof to the Target proof because `scopeId_CVS` is private and `blindingScalar₂` is unknown to DFCU.

### Healthcare delegation — referral network privacy

**Stakeholder:** Kaiser Permanente (KP), a healthcare system with 12.6M members.

**Setup:** A primary care physician (PCP) delegates a scoped credential to a specialist referral agent. The agent interacts with Lab Corp (lab orders) and Walgreens (prescription fulfillment). KP operates the AS.

**Problem:** Under the baseline, KP sees the full referral chain: PCP → specialist → Lab Corp → Walgreens. This reveals the member's condition (specialist type), diagnostic path (lab orders), and treatment (prescriptions). HIPAA's Minimum Necessary Standard (45 CFR §164.502(b)) requires limiting data exposure to what is necessary for the specific purpose.

**Bolyra deployment:** The PCP's agent uses `ScopeBlindedAgent` with `permissionBitmask = 0b10000101` (READ_DATA + FINANCIAL_SMALL + ACCESS_PII). Both `scopeId_LabCorp` and `scopeId_Walgreens` are private inputs. The proof's `scopeTag` lets each RS verify locally that the proof targets its scope, while KP-as-AS sees only opaque tags it cannot invert. The PCP delegates to the specialist agent via the Delegation circuit, narrowing to `0b00000101`. The specialist's agent generates scope-blinded proofs for Lab Corp and Walgreens with independent `blindingScalar` and `scopeId` values — all private. KP-as-AS cannot reconstruct the referral topology.

## 8. Why the baseline cannot match

| Capability | Baseline (PPID + RFC 8707 + DPoP + BBS+) | ScopeBlindedAgent |
|---|---|---|
| **AS sees token issuance per RS** | Yes — every token is AS-issued, AS logs `(agent, RS, scope, time)` | No — agent generates ZK proof locally; AS sees only unlinkable pseudonyms and opaque scope tags |
| **AS sees which RS a proof targets** | Yes — RFC 8707 `resource` parameter visible at issuance | No — `scopeId` is a **private circuit input**; the `scopeTag` output is blinded and cannot be inverted without the per-proof `blindingScalar` |
| **RS can verify proof targets its scope** | Yes — RS reads the `aud` claim | Yes — RS performs verifier-local check: `Poseidon2(scopeId, blindingScalar) == scopeTag` using values received over the direct TLS channel |
| **AS can dictionary-attack RS identity** | N/A — AS already knows the RS from the token request | No — `scopeTag = Poseidon2(scopeId, blindingScalar)` requires the per-proof `blindingScalar` to test any candidate `scopeId`; blinding scalar is never revealed to the AS |
| **Collusion (AS + k−1 RSes)** | Trivial correlation — AS holds the PPID mapping table | Bounded by `negl(λ)` — corrupted RSes reveal their `scopeId` and `blindingScalar` but cannot link to honest RS's proof (independent blinding scalar, private `scopeId`) |
| **Issuer anonymity** | Absent — BBS+ exposes issuer public key | N/A — agent Merkle tree is shared; no issuer key in public signals |
| **Formal security definition** | None — no IND-UNL-AS game exists in OAuth/OIDC/BBS+ specs | IND-UNL-AS game defined (§3) with reduction to Poseidon PRF (§4) |
| **Timing side-channel** | No mitigation — DPoP `jti` and timestamps leak request timing to AS | Batch relay mixing window; cryptographic unlinkability holds even without it |
| **Delegation chain privacy from AS** | Absent — RFC 8693 every hop is AS-visible | Blinded scope commitment chains through Delegation circuit; AS sees only rerandomized outputs |
| **Rerandomization per proof** | Absent — same BBS+ credential, same issuer key | `blindingScalar` ensures each proof's public signals are fresh even for identical inputs |

**The structural impossibility:** The baseline requires the AS to issue every token. This is not a configuration flaw — it is the definition of OAuth 2.0. An AS that issues a token necessarily knows the requesting agent, the target RS, and the requested scope. No combination of PPIDs, DPoP, or BBS+ changes this because they operate *after* issuance. Bolyra's `ScopeBlindedAgent` eliminates the issuance step entirely: the agent proves its credential validity in zero knowledge without any AS interaction at proof time. The AS is reduced to a tree maintainer (enrollment) and cannot observe per-session authorization events — nor can it determine which RS any given proof targets, because `scopeId` is a private input and the `scopeTag` is cryptographically blinded. This is a categorical, not incremental, separation.
