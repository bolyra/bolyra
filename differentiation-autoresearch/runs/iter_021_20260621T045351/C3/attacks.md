### Attack 1: Blinding Factor Exfiltration Through Colluding Adjacent Hops

**Attack:** The adversary A controls N−1 of N chain participants. Consider a 4-hop chain where A controls hops 0, 1, and 3, leaving hop 2 honest. The delegation protocol requires the prover at each hop to communicate the blinding factor to the *next* hop's delegatee, so they can chain-link their scope commitment. Concretely:

- Hop 1 (A-controlled) generates `delegateeScopeBlinding[1]` and communicates it to hop 2 (honest) for chain linking. Hop 2 uses this as `delegatorScopeBlinding[2]` — A already has it.
- Hop 2 (honest) generates `delegateeScopeBlinding[2]` and communicates it to hop 3 (A-controlled) so hop 3 can verify `delegatorScopeBlinding[3] = delegateeScopeBlinding[2]`. A thus receives `delegateeScopeBlinding[2]` from hop 2.
- The delegation protocol also communicates `(delegateeScope[2], delegateeCredCommitment[2], delegateeExpiry[2])` in plaintext to hop 3.

A therefore holds all three preimage components of `Poseidon3(delegateeScope[2], delegateeCredCommitment[2], delegateeScopeBlinding[2])` and recovers the honest participant's terminal scope. Game 3 is broken against the threat model's stated adversary (≤ N−1 colluding participants).

**Why it works:** The blinding scheme is sound against a passive auditor who sees only public outputs — Theorem 3's reduction to Poseidon3 pseudorandomness holds in that model. But the threat model also allows the adversary to *control* chain participants, who receive blinding factors via the out-of-band delegation protocol channel. The reduction never accounts for this: an adversary that is simultaneously auditor and adjacent-hop participant is not distinguishable from the pure-auditor game.

**In threat model?** No. The game definition for Game 3 says A "plays as auditor" and queries from public outputs. It does not restrict A from combining auditor observations with private inputs received as a colluding participant. Any honest participant flanked by two A-controlled hops has their scope fully exposed through the protocol-level blinding-factor relay. The construction must either (a) scope the Game 3 adversary to passive observers only and disclaim privacy against colluding adjacent hops, or (b) redesign the blinding derivation so adjacent hops cannot infer each other's blinding factors (e.g., each hop derives its own blinding via a PRF over a session-scoped secret not shared with neighbors).

### Attack 3: Underconstrained `finalScopeCommitment` Multiplexer — Missing Circuit Constraint

**Attack:** The constraint logic section (§2) specifies per-hop constraints (0–8) but stops short of the circuit-level constraint that binds `finalScopeCommitment` to hop `lastActive = chainLength − 1`. The prose states "the prover selects the output of hop `chainLength − 1` via a multiplexer over the known-contiguous prefix." No multiplexer constraint is written. A dishonest prover could set `chainLength = 3` (with `hopActive = [1,1,1,0]`) and produce a valid-looking audit for three properly-narrowed hops — while supplying a `finalScopeCommitment` that is actually the blinded scope commitment of *hop 1*, not hop 2. If the multiplexer is implemented as:

```
for i in 0..MAX_HOPS:
    isLast[i] = (chainLength - 1 == i)   // selection signal
finalScopeCommitment = sum_i isLast[i] * hopBlindedOutput[i]
```

then `isLast[i]` must be constrained to be boolean and sum to exactly 1. Without those constraints explicitly in the R1CS, a prover can set multiple `isLast` signals to 1 or to non-boolean values, producing a `finalScopeCommitment` that is a linear combination of multiple hop outputs. This bypasses the monotonic narrowing guarantee for the terminal scope: the auditor's view of `finalScopeCommitment` no longer reliably reflects the scope at the last active hop.

**Why it works:** Knowledge soundness (A1) guarantees the prover knows a satisfying witness — but "satisfying" means satisfying every *written* constraint. If the multiplexer constraint is absent from the circuit, a satisfying witness can assign `finalScopeCommitment` to any value expressible as a constrained linear combination. The security argument in Theorem 1 asserts narrowing soundness but does not reference the multiplexer constraint because the multiplexer constraint is never stated. This is a spec gap, not necessarily an implementation bug — but the formal security argument is incomplete until the multiplexer constraint is written into the R1CS and its correctness is verified.

**In threat model?** Not fully addressed. Theorem 1's sketch argues that "if any delegatee bit is set where the delegator bit is not, this constraint evaluates to a nonzero value." This is correct for the per-hop bitwise subset constraint. But the theorem does not separately argue that `finalScopeCommitment` is correctly bound to the last hop's output, which is a distinct claim requiring a distinct constraint (and a distinct proof step). The distinction matters: a prover can satisfy all per-hop narrowing constraints while supplying a `finalScopeCommitment` that corresponds to a *different* hop than `chainLength − 1`, making the audit output misleading without violating any stated constraint.

**Attack:** The construction uses PLONK with `pot16.ptau` as the universal SRS for the Agent/Delegation circuit family (§5, Table: "PLONK with universal setup (pot16.ptau)"). Under PLONK, knowledge soundness holds only if the SRS was honestly generated (no party retained the toxic waste — the discrete log relation between SRS elements). If a single party generated `pot16.ptau` and retained the trapdoor, that party can forge PLONK proofs for arbitrary statements over any circuit that uses this SRS, including `DelegationAuditChain`. Specifically: forge a proof claiming `narrowingValid = 1` for a chain that expanded permissions (defeating Game 1), or forge `chainLength = 2` for a 4-hop chain with an expanded middle hop. **Why it works:** The threat model (§3) assumes "The CRS/SRS [is not controlled by the adversary]" — but this is a deployment assumption, not a cryptographic property. **In threat model?** Claimed out-of-model ("The CRS/SRS [is not controlled by the adversary]") but not justified. For the construction's stated use case — regulatory audit by a skeptical third party — this assumption is precisely what must not be taken for granted. The fix is a verifiable multi-party ceremony for `pot16.ptau` with a public transcript, or a shift to a transparent setup (e.g., STARK-based proofs or Plonky2 with no SRS) for the audit chain circuit specifically. The Semaphore v4 reuse for HumanUniqueness demonstrates Bolyra knows how to cite ceremony provenance — the same rigor must apply to `pot16.ptau`.

### Attack 4: The Blinding Factor Is a Key — Where Does It Live?

- **Attack:** The construction introduces `scopeBlinding ∈ F_p` as a fresh random field element per hop, sampled by the prover and "passed privately between delegator and delegatee alongside the delegation token" (§2, Blinded scope commitment). Theorem 3 (Scope Privacy) reduces to the adversary's inability to guess this ~254-bit value. This means the blinding factor is now **a cryptographic secret that must be generated, stored, transmitted, and eventually deleted** — a full key lifecycle. The construction is completely silent on how this happens. Under the **GLBA Safeguards Rule § 314.4(c)**, I must implement "access controls on customer information systems" and "encryption of customer information." If these blinding factors live in agent process memory, they are lost on restart. If they are persisted to disk or a database, they become a new category of sensitive cryptographic material requiring key management infrastructure (KMS integration, rotation policy, destruction schedule). If they are transmitted between hops over the wire, the "private" channel between delegator and delegatee is undefined — TLS? A separate secure channel? The same channel that carries member PII? My **Vendor Management Policy** and **SOC 2 Type II CC6** require me to document and control all cryptographic key material.

- **Why it works / why it fails:** The construction correctly identifies that blinding is necessary and that blinding factors must be passed privately between hops. But "passed privately" is not a key management specification. For the security argument in Theorem 3 to hold in practice, the blinding factors must never be recoverable by an adversary who compromises the system after the fact. This requires a key lifecycle that the construction does not specify. If blinding factors are retained (for reproof or audit), Theorem 3's guarantee weakens proportionally. If they are destroyed immediately after the proof, reproof is impossible — meaning a failed proof cannot be regenerated, and the member's transaction cannot be retried without re-initiating the entire delegation chain.

- **In-threat-model?** Partially. The construction's adversary (§3) cannot control the Poseidon hash function or the CRS, but it does not model an adversary who compromises key storage after the session ends. The construction must address: where blinding factors are generated (HSM? secure enclave? agent memory?), whether they are retained or destroyed after proof generation, what the reproof policy is, and whether blinding factor compromise after the fact retroactively breaks scope privacy for past sessions.

- **Attack:** C3's `gap_to_close` states the construction "must broaden" to general AI agent pipelines — not just privacy-sensitive niches. §8's differentiator column leads with hiding intermediate scopes and participants. Verification is O(N) for MAX_HOPS = 4 — four signature checks, not one pairing — a constant-factor difference, not a categorical one. The construction must define the boundary condition: *when* does the compliance use case require hidden intermediate scopes, and *when* does it not? ### Attack 4: `chainLength` + `sessionNonce` + On-Chain Timestamp Correlation Breaks Journalist/Source Anonymity at the Deployment Layer

- **Attack:** §7's journalist/source variant asserts *"intermediate node anonymity is cryptographically guaranteed by the ZK property."* But `chainLength`, `sessionNonce`, and `currentTimestamp` are public outputs and inputs, and `chainSeedScopeCommitment` is published on-chain at handshake time. In the described scenario, the editorial board knows its own pipeline: editorial tool at hop 1, secure drop tool at hop 2. Observing `chainLength = 3` confirms a third participant exists at a known chain position — i.e., a source is present. More critically, `chainSeedScopeCommitment` appears in the on-chain registry at the moment the human handshake occurs (§5: *"Chain seed: `lastScopeCommitment[sessionNonce]` from on-chain registry"*). A state adversary who monitors on-chain transaction timing can correlate the block timestamp of the `chainSeedScopeCommitment` registration with network-layer source contact events — recovering source identity without breaking any circuit constraint. `sessionNonce` is a unique per-session identifier; combined with `currentTimestamp`, it is a near-unique fingerprint. The §3 adversary model says A observes "all proof transcripts" but does not model on-chain metadata correlation or blockchain transaction deanonymization. RFC 7662 with PPIDs and per-RS introspection policy filtering generates no on-chain records — the AS retains the session mapping in a localized, warrant-resistant datastore with no blockchain footprint.

- **Why it works / fails against the construction:** §4 Theorem 2 (Participant Privacy) is correct that the proof reveals nothing about private inputs beyond what the public outputs imply. The theorem does not claim the *deployment* is anonymous. The circuit-level ZK guarantee is sound. The *system-level* journalist/source guarantee is not — the on-chain anchoring that enables "no trusted third party" and "self-verifying proof" directly creates the metadata leak that compromises the use case. This is a fundamental tension in the design: the properties that make the construction trustless (on-chain root history buffer, on-chain `auditDigest` registry, on-chain `chainSeedScopeCommitment`) are also the properties that expose session metadata to any blockchain observer.

- **In-threat-model?** No. The construction must address whether the journalist/source scenario can be deployed without on-chain transaction deanonymization — for example, via stealth addresses, off-chain proof submission through an anonymizing relay, or a separate private mempool. Without a deployment-layer anonymity mechanism, the "whistleblower-safe agent delegation" use case claimed in C3's `gap_to_close` is achieved at the circuit level and lost at the deployment level. The construction should either (a) specify the required deployment-layer anonymity measures, or (b) remove the journalist/source variant from the claimed use cases.

### Attack 1: The Enrollment Bootstrap Problem — Who Attests the Attester?

**Attack:** The construction's entire security chain depends on `agentMerkleRoot` representing a trustworthy set of enrolled agents. Enrollment gates on an operator EdDSA keypair signing `credentialCommitment = Poseidon5(modelHash, opPubAx, opPubAy, permBitmask, expiry)` (§3.2 / §5 primitive mapping). But the construction never specifies how the operator keypair itself is attested.

In SPIRE, every workload's identity is rooted in **node attestation** — the SPIRE agent on the node proves to the SPIRE server that the node is what it claims (via TPM-measured boot, AWS instance identity documents, GCP attestation tokens, k8s service account tokens, etc.). The workload API then issues SVIDs only to workloads that pass **workload attestation** (process UID, container image hash, pod annotations). This is a layered trust model: hardware → node → workload.

Bolyra's enrollment is `operator signs modelHash → leaf in Merkle tree`. There is no hardware root. A compromised or colluding operator key inserts arbitrary `modelHash` values into the tree. Since `modelHash` is just a field element (§3.2), an attacker with the operator private key enrolls a phantom agent with any model hash they choose. Once enrolled, this agent appears in every `agentMerkleRoot` and satisfies Constraint 7 for any hop. The "Phantom Enrollment" defense (§3, Game 5) only closes the circuit-level gap — it assumes `agentMerkleRoot` is clean to begin with.

**Why it works against the construction:** Section 5 (Threat model) explicitly states the adversary "cannot insert leaves without valid operator-signed credential commitments enrolled through the registry." This treats the operator key as a trust root without justifying it. If the operator is a single key or a small multisig, this is a weaker trust anchor than SPIRE's hardware attestation. The threat model labels this out-of-scope but the claim in §7 (NCUA scenario) says the auditor "confirms every delegatee in the chain is a legitimately enrolled agent." That claim does not hold if "legitimate" depends on an unattested operator key.

**In-threat-model?** No — construction must address. Either (a) specify what attests the operator keypair (HSM, SGX quote, SPIRE-issued SVID, multisig ceremony), or (b) weaken the enrollment soundness claim to "enrolled by the designated operator" and disclaim hardware-level attestation guarantees. The current language in §7 claims stronger guarantees than the cryptographic proof provides.

### Attack 2: The Global Merkle Root Is More Centralized Than SPIFFE Federation

**Attack:** §8 claims "Cross-org without shared trust anchor" and cites as evidence that the third-party fintech at hop 3 is "just another enrolled agent" in the same tree. But read §7 carefully: "The third-party fintech's market-data agent at hop 3 must also be enrolled in the same tree; if it is enrolled in a separate tree, the two organizations must use a shared agent registry (the Bolyra on-chain registry serves this role) **or the proof cannot be generated.**"

This is not trustless cross-org. This is a **global shared registry** that all organizations must write to. SPIFFE federation explicitly solves this without a global registry: org A's SPIRE server maintains `spiffe://orgA/` trust domain, org B maintains `spiffe://orgB/`, and they federate by exchanging trust bundles (X.509 CA certs). Each org keeps full autonomy over its own workload identities. Federation is bilateral and revocable.

The Bolyra construction collapses cross-org identity into a single Merkle tree. The `agentMerkleRoot` is a single public input — one tree root for all orgs. The on-chain registry is therefore a global coordination point: if the fintech doesn't enroll in Bolyra's registry, the NFCU cannot generate the audit proof at all. This is stronger centralization than the baseline, not weaker.

**Why it works against the construction:** §8 comparison row "Cross-org without shared trust anchor" is factually wrong. The construction requires NFCU and the third-party fintech to share a single Merkle tree, maintained by whoever controls the on-chain registry. SPIFFE federation requires only bilateral trust bundle exchange. The construction trades WIMSE's "shared AS" for a "shared Merkle tree" — same centralization, different cryptographic clothing. The auditor cannot produce the cross-org proof unless the global registry is available and the fintech has consented to enrollment in it.

**In-threat-model?** No — construction must address. The §8 comparison row must be retracted or qualified. Either acknowledge the shared registry requirement and justify why it's acceptable, or describe a tree-of-trees construction (each org maintains its own subtree, the root of roots is the audit anchor) and prove the cross-tree enrollment constraint still closes Game 5.

### Attack 3: `narrowingValid = 1` Does Not Satisfy NCUA Examination Requirements

**Attack:** The NCUA deployment scenario (§7) is the primary regulatory use case. The construction delivers `narrowingValid = 1`, `chainLength = 3`, `finalScopeCommitment` (blinded), and `auditDigest` to the examiner. It explicitly states the examiner does NOT receive "any intermediate scope bitmask." The scenario frames this as a feature: competitive intelligence about NFCU's internal authorization policy is protected.

NCUA examination under Part 748 (Information Security) and FFIEC IT Examination Handbook (Information Security Booklet) requires examiners to verify **specific controls** — not a boolean property over hidden values. An NCUA examiner attesting to "the AI pipeline had monotonically narrowing permissions" without seeing what those permissions were cannot satisfy: (a) the requirement to verify that PII access was appropriately controlled, (b) the requirement to verify that financial transaction limits were within policy, or (c) the requirement to produce a written finding that names the specific capabilities authorized at each stage. A `narrowingValid = 1` flag in a ZK proof is not a finding an NCUA examiner can cite in an examination report.

More concretely: the examiner must be able to answer "did the rate-lookup agent have financial transaction authority?" The construction's answer is "the proof guarantees it had no more than the credit-scoring agent, which had no more than the front-desk agent." That's not an answer — it's a proof about a relationship between hidden values. If the examiner suspects the front-desk agent was over-privileged, they have no way to check because `chainSeedScopeCommitment` is blinded.

**Why it works against the construction:** The §7 scenario conflates "mathematical proof of monotonic narrowing" with "regulatory compliance with NCUA authorization controls." These are not the same. The construction is genuinely useful for privacy-preserving audit in contexts where the auditor only needs to verify a process property (narrowing happened) without semantic understanding of what was narrowed. NCUA examination is not that context — it requires semantic verification. The scenario is selling the construction to a use case it cannot serve.

**In-threat-model?** Partially. The construction's mathematical claims are valid. The compliance framing is the attack surface. The construction must either (a) replace the NCUA scenario with a use case where a process property proof is actually sufficient (e.g., internal SLA audit where the auditor only needs "did narrowing occur" not "what were the scopes"), or (b) describe a selective-disclosure mode where the examiner can see specific scope values for hops they are entitled to inspect while intermediate hops remain hidden — which requires a fundamentally different approach (per-hop proof, not chain-level proof).

### Attack 4: This Should Be a SPIRE Attestor Plugin, Not a New Protocol

**Attack:** WIMSE (`draft-ietf-wimse-arch`) defines a workload-to-workload authentication model that explicitly includes token exchange, delegation, and audience binding. The draft's threat model covers workload impersonation, token replay, and scope escalation. The WIMSE working group is actively scoping selective disclosure for workload attributes.

The correct contribution of this construction to the standards ecosystem is a **ZK attestor for SPIRE** — a new attestor type that, instead of returning a SPIFFE SVID (X.509 or JWT), returns a ZK proof of workload enrollment. This would:

1. Slot into the existing SPIRE plugin architecture (attestor interface is well-defined).
2. Reuse SPIRE's node attestation for the enrollment bootstrap problem (Attack 1 resolved).
3. Compose with WIMSE token exchange for cross-org delegation (Attack 2 resolved — each org runs its own SPIRE, the ZK attestor produces enrollment proofs that are WIMSE-compatible).
4. Allow WIMSE's token exchange to carry the `finalScopeCommitment` and `auditDigest` as JWT claims, fitting into existing OAuth 2.0 authorization server infrastructure.

Instead, the construction defines a new DID method (`spec/did-method-bolyra.md`), a new IETF draft (`spec/draft-bolyra-mutual-zkp-auth-01.md`), a new on-chain registry, a new proving system pipeline, and a new audit proof format. Every one of these is a standards body engagement, an interop problem, and an adoption barrier.

The specific gap the ZK construction fills — proving bitwise subset relationships over hidden scopes — is a **gadget** that should live inside WIMSE's token exchange, not replace it. A WIMSE extension that says "the delegated token was derived via a ZK monotonic-narrowing proof; final scope commitment is [commitment]; audit digest is [digest]" gives existing WIMSE implementations the privacy property without requiring them to adopt a new identity layer.

**Why it works against the construction:** §8's entire "Why the baseline cannot match" table is written as "WIMSE cannot do X." The correct adversarial response is: "WIMSE cannot do X *today*; contribute X to WIMSE rather than building a parallel ecosystem." The construction's comparison implicitly assumes WIMSE is frozen, but WIMSE is an active WG with exactly this design space in scope. The construction's authors are not participating in WIMSE — they are racing to ship a parallel stack that will need to prove interoperability with WIMSE before any Fortune 500 deploys it. The adoption cost of a new protocol stack versus a WIMSE extension is not addressed anywhere in the construction.

**In-threat-model?** No — this is a positioning and adoption attack, not a cryptographic break. But it's the most dangerous attack for the claim's practical relevance. The construction must either (a) explain why a SPIRE attestor plugin + WIMSE extension is insufficient (identify a concrete gap that requires a full protocol stack), or (b) reframe the construction as a WIMSE extension proposal and demonstrate how it slots into `draft-ietf-wimse-arch` Section 4 (Token Exchange) with the ZK audit proof as a new token type. Claiming "no trusted third party" while operating an on-chain registry and a new DID method is not actually eliminating trusted parties — it is redistributing them.
