# Construction

## 1. Statement of claim

The same Bolyra agent accessing *k* distinct Resource Servers produces *k* authorization proofs that are computationally indistinguishable from proofs generated by *k* independent agents, even to an adversarial Authorization Server (AS) that controls the agent Merkle tree, observes all proof submissions, and colludes with up to *k‚àí1* of the *k* RSes. Cross-scope unlinkability holds under the Poseidon PRF assumption on BN254 and the knowledge soundness of Groth16/PLONK.

## 2. Construction (gadgets, circuits, public/private inputs)

### New circuit: `ScopeBlindedAgent(MAX_DEPTH)`

This circuit extends `AgentPolicy` with a scope-blinding layer and an **agent-held trapdoor**. The agent proves credential validity and scope satisfaction while producing a **scope-specific pseudonym** that is unlinkable across scopes. Critically, `scopeId` is a **private** input ‚Äî it never appears on-chain. The RS confirms the proof targets its scope via a verifier-local consistency check on a blinded scope tag.

**Enrollment change ‚Äî trapdoor pattern (mirrors Semaphore):**

The agent generates a random `agentSecret ‚Üê random(F_p)` at credential creation time. The tree leaf is no longer the bare `credentialCommitment` but an **enrollment commitment**:

```
enrollmentCommitment = Poseidon2(credentialCommitment, agentSecret)
```

The AS enrolls `enrollmentCommitment` into the agent Merkle tree. The AS **never** learns `credentialCommitment` or `agentSecret` individually ‚Äî it sees only the opaque leaf. The operator knows `credentialCommitment` (it created and signed the credential fields) but does not know `agentSecret` (generated solely by the agent). This mirrors Semaphore's trapdoor pattern where the identity commitment hides the secret from the tree maintainer.

**Private inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `modelHash` | F_p | Hash of model identifier |
| `operatorPubkeyAx`, `operatorPubkeyAy` | F_p | Operator EdDSA public key |
| `permissionBitmask` | 64-bit | Permission bitfield |
| `expiryTimestamp` | 64-bit | Credential expiration |
| `sigR8x`, `sigR8y`, `sigS` | F_p | Operator EdDSA signature |
| `merkleProofLength`, `merkleProofIndex`, `merkleProofSiblings[MAX_DEPTH]` | ‚Äî | Merkle inclusion proof |
| `agentSecret` | F_p | Agent-held trapdoor ‚Äî **never revealed to AS or operator** |
| `blindingScalar` | F_p | Fresh random blinding value per proof |
| `scopeId` | F_p | RS-specific scope identifier ‚Äî **private, never on-chain** |

**Public inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `requiredScopeMask` | 64-bit | Required permission bits |
| `currentTimestamp` | 64-bit | Current time |
| `sessionNonce` | F_p | Fresh per-session nonce |

**Public outputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `agentMerkleRoot` | F_p | Computed Merkle root |
| `scopePseudonym` | F_p | `Poseidon3(agentSecret, credentialCommitment, scopeId)` ‚Äî unlinkable across scopes, uninvertible by AS |
| `blindedScopeCommitment` | F_p | `Poseidon3(permissionBitmask, credentialCommitment, blindingScalar)` ‚Äî rerandomized per proof |
| `sessionNullifier` | F_p | `Poseidon3(scopePseudonym, sessionNonce, blindingScalar)` ‚Äî replay prevention per session√óscope |
| `scopeTag` | F_p | `Poseidon2(scopeId, blindingScalar)` ‚Äî verifier-local scope binding |

**Circuit constraints (12 steps):**

1. **Range checks:** `Num2Bits(64)` on `permissionBitmask`, `expiryTimestamp`, `currentTimestamp`.
2. **Credential commitment:** `credentialCommitment = Poseidon5(modelHash, operatorPubkeyAx, operatorPubkeyAy, permissionBitmask, expiryTimestamp)`.
3. **EdDSA signature:** `EdDSAPoseidonVerifier(operatorPubkey, credentialCommitment, sig)`.
4. **Enrollment commitment:** `enrollmentCommitment = Poseidon2(credentialCommitment, agentSecret)`.
5. **Merkle membership:** `BinaryMerkleRoot(MAX_DEPTH, enrollmentCommitment, proof) == agentMerkleRoot`. Note: the Merkle proof is over `enrollmentCommitment`, NOT bare `credentialCommitment`.
6. **Scope satisfaction:** For each bit *i* in [0, 64): `requiredBits[i] * (1 - permBits[i]) === 0`.
7. **Cumulative bit encoding:** Standard Bolyra tier implication constraints on bits 2‚Äì4.
8. **Expiry check:** `currentTimestamp < expiryTimestamp` via `LessThan(64)`.
9. **Scope pseudonym:** `scopePseudonym = Poseidon3(agentSecret, credentialCommitment, scopeId)`.
10. **Blinded scope commitment:** `blindedScopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, blindingScalar)`.
11. **Session nullifier:** `sessionNullifier = Poseidon3(scopePseudonym, sessionNonce, blindingScalar)`.
12. **Scope tag:** `scopeTag = Poseidon2(scopeId, blindingScalar)`.

**Why the prior `Poseidon2(cc, scopeId)` pseudonym was fatally broken and why `Poseidon3(agentSecret, cc, scopeId)` fixes it:**

In the prior construction, tree leaves were bare `credentialCommitment` values. The AS maintains the tree and therefore knows every leaf. An AS with a dictionary of RS `scopeId` values (computable from public RS data as `Poseidon2(RS_domain, RS_public_key)`) could enumerate all `(cc, scopeId)` pairs and compute `Poseidon2(cc, scopeId)` for each. Matching against the observed on-chain `scopePseudonym` would identify both the agent and the target RS in `O(N √ó R)` time where *N* is the number of enrolled agents and *R* is the number of known RSes. This is a trivial inversion ‚Äî no PRF assumption helps when the key is known.

The fix has two interlocking parts:

1. **Tree leaf is now `enrollmentCommitment = Poseidon2(cc, agentSecret)`:** The AS sees only the enrollment commitment. It cannot extract `cc` from the leaf without `agentSecret` (Poseidon preimage resistance). The operator knows `cc` but not `agentSecret`. Neither party alone can decompose the leaf.

2. **Scope pseudonym includes `agentSecret`:** `scopePseudonym = Poseidon3(agentSecret, cc, scopeId)`. Even if the AS could somehow guess `cc` and `scopeId`, it still cannot compute the pseudonym without `agentSecret`. Under the Poseidon PRF assumption keyed on `agentSecret` (unknown to A), the output is indistinguishable from random.

**Why `scopeTag` works ‚Äî verifier-local consistency check:**

The RS already knows its own `scopeId` (computed as `Poseidon2(RS_domain, RS_public_key)`). The agent transmits `blindingScalar` to the RS over the direct TLS channel that already carries `scopeId` and `sessionNonce`. The RS recomputes `Poseidon2(scopeId, blindingScalar)` locally and checks that it equals the `scopeTag` public output from the proof. If they match, the proof was generated for this RS's scope. If they don't match, the proof targeted a different RS and the RS rejects.

**Why the AS cannot exploit `scopeTag`:**

The AS observes `scopeTag` on-chain but does not know `blindingScalar` (private input, transmitted only to the RS over a direct channel the AS does not control). Even if the AS has a dictionary of all possible `scopeId` values (e.g., all known RS domains), it cannot test candidates: each candidate `scopeId'` would require the correct `blindingScalar` to produce the observed `scopeTag`, and `blindingScalar` is drawn uniformly from F_p per proof. Under the Poseidon PRF assumption (A1), `Poseidon2(scopeId, blindingScalar)` is indistinguishable from random when `blindingScalar` is unknown, regardless of whether `scopeId` is drawn from a small dictionary.

**Why `blindingScalar` disclosure to the RS is safe:**

The RS learns `blindingScalar` but cannot use it to link proofs across scopes. To link a proof to a different RS's scope, the attacker would need the other proof's `blindingScalar` (which is independent ‚Äî fresh per proof) and the other RS's `scopeId`. Even if RS_A colludes with the AS and learns `blindingScalar_A`, the proof for RS_B used an independent `blindingScalar_B` that neither RS_A nor the AS possesses.

**Constraint estimate:** ~12,314 constraints (prior 11,514 + one additional Poseidon2 at ~800 for `enrollmentCommitment`). Fits within 2^14 (16,384); `pot16.ptau` covers.

### Protocol flow (scope-blinded handshake)

1. RS generates `(sessionNonce, scopeId)` where `scopeId = Poseidon2(RS_domain, RS_public_key)`. The RS sends both to the agent over a direct TLS channel. The AS never sees `scopeId`.

2. Agent generates `ScopeBlindedAgent` proof with fresh `blindingScalar ‚Üê random(F_p)`, using its stored `agentSecret`.

3. Agent submits proof to the on-chain registry. The registry sees `(agentMerkleRoot, scopePseudonym, blindedScopeCommitment, sessionNullifier, scopeTag, requiredScopeMask, currentTimestamp, sessionNonce)`. Note: neither `scopeId` nor `agentSecret` nor `credentialCommitment` appears in public inputs or outputs ‚Äî the registry never sees them.

4. Registry verifies:
   - `sessionNonce` freshness (not reused).
   - `agentMerkleRoot` in root history buffer.
   - Groth16/PLONK proof validity.
   - `sessionNullifier` not previously recorded (replay prevention).

5. Agent transmits `blindingScalar` to the RS over the direct TLS channel (same channel used in step 1).

6. RS performs the **verifier-local consistency check**: compute `Poseidon2(scopeId, blindingScalar)` and assert equality with the `scopeTag` public output from the proof. If the check fails, the RS rejects the authorization. This check is performed locally by the RS ‚Äî it does not require any on-chain interaction or AS involvement.

7. RS verifies that the proof's `requiredScopeMask` matches the scope policy the RS advertised. RS accepts the authorization.

### Batched submission (timing side-channel mitigation)

To prevent the AS from correlating proof submissions by timing, agents MAY use a **batch relay**:
- A relay collects proofs from multiple agents over a mixing window *Œît* (e.g., 30 seconds).
- The relay shuffles and submits proofs in a single multicall transaction.
- The AS sees *n* proofs arriving simultaneously with no per-agent timing signal.

This is an application-layer defense; the cryptographic unlinkability holds without it.

## 3. Threat model (adversary capabilities, game definition)

### Adversary capabilities

The adversary **A** controls:
- The Authorization Server (AS): full read access to the agent Merkle tree (all `enrollmentCommitment` leaves), all proof submissions, all public signals (including `scopePseudonym` and `scopeTag`).
- Up to *k‚àí1* of *k* target RSes: A learns the `scopeId`, `blindingScalar`, `requiredScopeMask`, and proof for those RSes.
- A complete dictionary of RS `scopeId` values (A knows every RS's domain and public key and can compute `Poseidon2(RS_domain, RS_public_key)` for all RSes).
- Timing of proof submissions (unless batch relay is used).

The adversary **A** does NOT control:
- The agent's `agentSecret` or `blindingScalar` for honest RSes. The `agentSecret` is generated solely by the agent and is never transmitted to any external party ‚Äî not to the AS, not to the operator, not to any RS.
- The ZK proving process (no malicious prover).
- The direct TLS channel between agent and at least one honest RS (the one whose traffic A tries to link).
- At least one RS that remains honest.

### Why the prior construction's scopePseudonym was broken (and why this is now fixed)

**Prior vulnerability (now patched):** In the prior construction, tree leaves were bare `credentialCommitment` values known to the AS. The scope pseudonym was `Poseidon2(credentialCommitment, scopeId)`. The AS knows all `cc` values (it enrolled them) and can compute all `scopeId` values (public RS data). The AS computes `Poseidon2(cc_i, scopeId_j)` for all enrolled agents *i* and all known RSes *j*, then matches against the observed on-chain `scopePseudonym`. This is an `O(N √ó R)` dictionary attack ‚Äî no PRF assumption helps because both inputs are known to the adversary.

**Fix applied:** Two changes close this gap:

1. Tree leaves are now `enrollmentCommitment = Poseidon2(credentialCommitment, agentSecret)` ‚Äî the AS sees only the enrollment commitment and cannot extract `credentialCommitment` without `agentSecret` (Poseidon preimage resistance).

2. Scope pseudonym is now `Poseidon3(agentSecret, credentialCommitment, scopeId)` ‚Äî even if the AS could somehow guess `credentialCommitment` and `scopeId`, it cannot compute the pseudonym without `agentSecret`. Under the Poseidon PRF assumption keyed on `agentSecret`, the output is indistinguishable from random.

The AS's dictionary attack now requires guessing `agentSecret ‚àà F_p` (~254-bit space) for each candidate, which is computationally infeasible.

### IND-UNL-AS game

**Game IND-UNL-AS(Œª):**

1. **Setup:** Challenger generates two agents `(cred‚ÇÄ, agentSecret‚ÇÄ)` and `(cred‚ÇÅ, agentSecret‚ÇÅ)` with identical permission bitmasks. Computes `enrollmentCommitment‚ÇÄ = Poseidon2(cc‚ÇÄ, agentSecret‚ÇÄ)` and `enrollmentCommitment‚ÇÅ = Poseidon2(cc‚ÇÅ, agentSecret‚ÇÅ)`. Enrolls both in the Merkle tree. Challenger generates *k* RS scope identifiers `{scopeId_1, ‚Ä¶, scopeId_k}`.

2. **Challenge:** Challenger flips coin `b ‚Üê {0, 1}`. For each `scopeId_j` (j = 1‚Ä¶k), challenger generates a valid `ScopeBlindedAgent` proof using agent `(cred_b, agentSecret_b)` with fresh `blindingScalar_j` and `sessionNonce_j`.

3. **Oracle access:** A may:
   - Request proofs for any `scopeId` of A's choosing for either agent (adaptive queries, up to poly(Œª) queries).
   - Observe all public signals from all proofs (including `scopePseudonym` and `scopeTag`).
   - Corrupt up to *k‚àí1* RSes, learning their `scopeId` values **and the `blindingScalar` values** transmitted to those RSes.
   - Read all tree leaves (`enrollmentCommitment` values) ‚Äî but NOT the `credentialCommitment` or `agentSecret` preimages.
   - Compute `Poseidon2(RS_domain, RS_public_key)` for any RS (full `scopeId` dictionary).

4. **Restriction:** A may NOT query both agents on the same `scopeId` as the challenge scope (trivial distinguisher via Merkle root + known tree).

5. **A outputs** guess `b'`.

**Advantage:** `Adv_A = |Pr[b' = b] - 1/2|`.

**Claim:** For all PPT adversaries A, `Adv_A ‚â§ negl(Œª)` under the Poseidon PRF assumption on BN254 and knowledge soundness of Groth16/PLONK.

### scopeTag under adversarial AS ‚Äî why privacy holds

The AS sees `scopeTag = Poseidon2(scopeId, blindingScalar)` on-chain. Two attack vectors:

**Dictionary attack (AS knows candidate scopeIds):** The AS enumerates candidate `scopeId'` values. For each candidate, it would need to find `blindingScalar` such that `Poseidon2(scopeId', blindingScalar) = scopeTag`. Under Poseidon preimage resistance, this requires brute-forcing `blindingScalar` over F_p (~254 bits), which is computationally infeasible.

**Collusion attack (AS + corrupted RS):** When A corrupts RS_j, A learns both `scopeId_j` and `blindingScalar_j`. A can verify that `Poseidon2(scopeId_j, blindingScalar_j) = scopeTag_j` ‚Äî this confirms which proof targeted RS_j, but A already knew this by corrupting RS_j. For the honest RS_h, A does not know `blindingScalar_h` (transmitted only over the direct TLS channel to RS_h), so `scopeTag_h` remains opaque. A cannot test whether `scopeTag_h` corresponds to `scopeId_h` (even if A guesses `scopeId_h`) without `blindingScalar_h`.

### scopePseudonym under adversarial AS ‚Äî why privacy now holds

The AS sees `scopePseudonym = Poseidon3(agentSecret, credentialCommitment, scopeId)` on-chain. Three attack vectors considered:

**Dictionary attack (AS knows candidate cc and scopeId values):** In the prior construction, this was fatal ‚Äî the AS could enumerate `(cc_i, scopeId_j)` pairs and match. Now the AS does NOT know `credentialCommitment` values (tree leaves are `Poseidon2(cc, agentSecret)`, and the AS cannot extract `cc` without `agentSecret`). Even if the AS could somehow guess `cc`, it still lacks `agentSecret` ‚Äî a ~254-bit value never revealed to any external party. The attack requires brute-forcing `agentSecret`, which is infeasible.

**Collusion with operator:** The operator knows `credentialCommitment` (it created the credential) but NOT `agentSecret`. The operator could collude with the AS and supply `cc`. The AS then attempts `Poseidon3(agentSecret_guess, cc, scopeId_j)` for candidate `scopeId_j` values ‚Äî but must still guess `agentSecret ‚àà F_p`, which is infeasible.

**Collusion with corrupted RS:** A corrupted RS_j learns `scopeId_j` and `blindingScalar_j` but NOT `agentSecret` or `credentialCommitment`. Even with operator collusion providing `cc`, the AS cannot compute `Poseidon3(agentSecret, cc, scopeId_j)` without `agentSecret`.

## 4. Security argument (named assumption + reduction sketch)

### Named assumptions

- **A1 ‚Äî Poseidon PRF:** `Poseidon2(¬∑, ¬∑)` and `Poseidon3(¬∑, ¬∑, ¬∑)` are pseudorandom functions when keyed on their first argument over the BN254 scalar field. Specifically, `f_k(x) = Poseidon2(k, x)` and `g_k(x, y) = Poseidon3(k, x, y)` are computationally indistinguishable from random functions for any fixed key `k` unknown to the adversary.

- **A2 ‚Äî Knowledge soundness of Groth16 (ROM):** In the random oracle model with the BN254 pairing, Groth16 satisfies knowledge soundness: a valid proof implies the prover knows a satisfying witness.

- **A3 ‚Äî Discrete log hardness on Baby Jubjub:** Given `(Ax, Ay) = BabyPbk(s)`, recovering `s` is computationally intractable.

- **A4 ‚Äî Poseidon preimage resistance:** Given `y = Poseidon2(x‚ÇÅ, x‚ÇÇ)`, finding `(x‚ÇÅ, x‚ÇÇ)` is computationally intractable. (Used to argue that the AS cannot decompose `enrollmentCommitment` into `(credentialCommitment, agentSecret)`.)

### Reduction sketch

**Theorem:** If there exists a PPT adversary A winning IND-UNL-AS with non-negligible advantage, then there exists a PPT distinguisher D breaking the Poseidon PRF assumption.

**Proof sketch:**

1. **Scope pseudonym unlinkability (patched).** The scope pseudonym is `Poseidon3(agentSecret, credentialCommitment, scopeId)`. Under A1, keying Poseidon3 on `agentSecret` (which is generated solely by the agent and never revealed to any external party ‚Äî not to the AS, not to the operator, not to any RS), the output `g_{agentSecret}(cc, scopeId)` is indistinguishable from random for any `(cc, scopeId)` pair. The AS does not know `agentSecret`:
   - The tree leaf is `enrollmentCommitment = Poseidon2(cc, agentSecret)`. The AS sees the leaf but cannot extract `agentSecret` without inverting Poseidon2 (contradicting A4).
   - The operator knows `cc` but not `agentSecret`.
   - No RS learns `agentSecret` (only `blindingScalar` is transmitted to the RS).

   For two different scopes `scopeId_i ‚âÝ scopeId_j` from the same agent, the pseudonyms are `Poseidon3(agentSecret, cc, scopeId_i)` and `Poseidon3(agentSecret, cc, scopeId_j)`. Under A1 keyed on `agentSecret`, these are indistinguishable from independently random values. If A could distinguish them as originating from the same agent, D would use this to distinguish Poseidon3 from random, contradicting A1.

   **Why the prior Poseidon2(cc, scopeId) failed this step:** The PRF key was `cc`, which the AS knew directly (it was the tree leaf). No PRF assumption helps when the key is known to the adversary. The fix places the PRF key (`agentSecret`) in a position where no external party learns it ‚Äî mirroring Semaphore's trapdoor pattern.

2. **Scope tag unlinkability.** The scope tag is `Poseidon2(scopeId, blindingScalar)`. Since `blindingScalar` is drawn uniformly from F_p and is unknown to the AS, under A1 (keying Poseidon on `blindingScalar`), the output is indistinguishable from random regardless of the `scopeId` input. Two scope tags from the same agent with different `blindingScalar` values are independently pseudorandom. The AS gains no information about which RS the proof targets from the scope tag alone. If A could distinguish scope tags as targeting the same `scopeId` across two proofs (with independent blinding scalars), then A breaks the PRF assumption on Poseidon2 keyed by the blinding scalar ‚Äî contradicting A1.

3. **Blinded scope commitment unlinkability.** The blinded scope commitment includes a fresh `blindingScalar` per proof. Even if `permissionBitmask` and `credentialCommitment` are identical across proofs, the output `Poseidon3(permissionBitmask, cc, blindingScalar)` is fresh each time. Under A1, this is indistinguishable from random for each new `blindingScalar`. The AS does not know `cc` (hidden behind `enrollmentCommitment`), adding a second layer of protection.

4. **Session nullifier unlinkability.** The session nullifier `Poseidon3(scopePseudonym, sessionNonce, blindingScalar)` depends on the already-unlinkable `scopePseudonym` and a fresh `blindingScalar`. Composing two PRF applications preserves pseudorandomness under A1.

5. **Witness hiding via ZK.** By the zero-knowledge property of Groth16/PLONK (A2), the proof transcript reveals nothing about private inputs (`credentialCommitment`, `agentSecret`, `blindingScalar`, `scopeId`, operator key, Merkle path). The adversary's view is limited to public signals, which are unlinkable per (1)‚Äì(4).

6. **Collusion resistance (k‚àí1 RSes).** Corrupting RS_j reveals `scopeId_j` and `blindingScalar_j` to A. However, RS_j does NOT learn `agentSecret` or `credentialCommitment` ‚Äî only `blindingScalar_j` is transmitted to the RS. For the honest RS_h, A knows neither `agentSecret` nor `blindingScalar_h`. Even with operator collusion (providing `cc`), computing `Poseidon3(agentSecret, cc, scopeId_h)` requires `agentSecret`, which is infeasible to extract from `enrollmentCommitment` under A4. The advantage reduces to brute-forcing `agentSecret` over F_p, which is negligible.

7. **Enrollment commitment hides credentialCommitment from AS.** The AS observes `enrollmentCommitment = Poseidon2(cc, agentSecret)` as the tree leaf. Under A4 (Poseidon preimage resistance), the AS cannot recover `(cc, agentSecret)` from the leaf. This prevents the AS from mounting the dictionary attack that broke the prior construction: without `cc`, the AS cannot compute `Poseidon3(agentSecret_guess, cc, scopeId_j)` even for a single `(cc, scopeId_j)` pair.

**QED (sketch).**

### Side-channel treatment

- **Timing:** Batch relay (¬ß2) ensures proof submission times are shuffled within a mixing window. Without batch relay, timing correlation is possible but does not break cryptographic unlinkability ‚Äî it is a traffic-analysis attack mitigated at the network layer.
- **Nonce freshness:** Each `sessionNonce` is unique per handshake. The `sessionNullifier` binds the proof to a specific `(scopePseudonym, sessionNonce, blindingScalar)` triple. Replaying a proof with a stale nonce fails the freshness check.
- **Merkle root correlation:** If the agent tree is small, the AS could narrow the anonymity set by observing which Merkle root is used. The 30-entry root history buffer and the shared tree across all agents mitigate this ‚Äî every agent references the same tree.
- **`scopeTag` dictionary resistance:** The AS observing `scopeTag` on-chain cannot test candidate `scopeId` values without the corresponding `blindingScalar`. The blinding scalar is transmitted only over the direct agent-to-RS TLS channel, which the AS does not control. This holds even if the AS maintains a complete dictionary of all RS `scopeId` values.
- **`scopePseudonym` dictionary resistance (PATCHED):** The AS observing `scopePseudonym` on-chain cannot test candidate `(cc, scopeId)` pairs because (a) `cc` is hidden behind `enrollmentCommitment` (Poseidon preimage resistance, A4), and (b) even with operator collusion providing `cc`, the computation requires `agentSecret` which is a ~254-bit value never revealed to any external party (PRF assumption, A1).

**Delegation compatibility:** The existing Delegation circuit links via `previousScopeCommitment = Poseidon2(delegatorScope, delegatorCredCommitment)`. To chain from a `ScopeBlindedAgent` proof, the delegator must reveal `credentialCommitment` to the delegation circuit as a private input and prove consistency with the `blindedScopeCommitment`. This requires a one-constraint adapter: `Poseidon3(delegatorScope, delegatorCredCommitment, blindingScalar) == blindedScopeCommitment` (blinding scalar also private to the delegator). The delegation circuit then proceeds with standard chain linking.

## 7. Concrete deployment scenario

### Credit union member agent ‚Äî merchant graph privacy

**Stakeholder:** Desert Financial Credit Union (DFCU), a $9B credit union in Arizona with 400K members.

**Setup:** DFCU operates an Authorization Server for its members' AI agents. Members deploy agents to interact with merchants (Target, Costco, local pharmacies) for price comparison, coupon redemption, and loyalty point management. Each merchant operates a Resource Server.

**Problem under baseline:** DFCU-as-AS sees every token issuance: "Member #4821's agent requested `merchant-read` scope for Target at 14:02, then `pharmacy-read` scope for CVS at 14:07." DFCU reconstructs the member's complete merchant graph. Under NCUA Regulation ¬ß748 (member data protection), this is a compliance liability. Under the CFPB's proposed open-banking rules (¬ß1033), the AS must not use authorization metadata to build behavioral profiles.

**Bolyra deployment:**

1. DFCU enrolls member agents in the Bolyra agent Merkle tree. Each agent generates `agentSecret ‚Üê random(F_p)` locally. The agent computes `credentialCommitment = Poseidon5(modelHash, Ax, Ay, 0b00000101, expiry)` and `enrollmentCommitment = Poseidon2(credentialCommitment, agentSecret)`. The agent submits `enrollmentCommitment` to DFCU for tree insertion. **DFCU never learns `credentialCommitment` or `agentSecret` individually.**

2. Target's RS has `scopeId_Target = Poseidon2("target.com", Target_RS_pubkey)`. CVS has `scopeId_CVS = Poseidon2("cvs.com", CVS_RS_pubkey)`. These are computed locally by each RS and shared with the agent over a direct TLS channel. DFCU (AS) never receives these values.

3. Member's agent generates `ScopeBlindedAgent` proof for Target with `blindingScalar‚ÇÅ ‚Üê random(F_p)`:
   - `scopeId_Target` is a **private input** ‚Äî it does not appear on-chain.
   - `scopePseudonym_Target = Poseidon3(agentSecret, credCommitment, scopeId_Target)` ‚Äî visible on-chain but uninvertible by DFCU (requires `agentSecret`).
   - `scopeTag_Target = Poseidon2(scopeId_Target, blindingScalar‚ÇÅ)` ‚Äî visible on-chain; DFCU cannot invert without `blindingScalar‚ÇÅ`.
   - `blindedScopeCommitment_Target = Poseidon3(0b00000101, credCommitment, blindingScalar‚ÇÅ)` ‚Äî unique to this proof.

4. Agent transmits `blindingScalar‚ÇÅ` to Target over the direct TLS channel. Target recomputes `Poseidon2(scopeId_Target, blindingScalar‚ÇÅ)` and verifies it matches `scopeTag_Target` from the proof. Target accepts.

5. Same agent generates `ScopeBlindedAgent` proof for CVS with independent `blindingScalar‚ÇÇ ‚Üê random(F_p)`:
   - `scopeId_CVS` is private. `scopePseudonym_CVS = Poseidon3(agentSecret, credCommitment, scopeId_CVS)` differs from Target's pseudonym (different `scopeId`). `scopeTag_CVS = Poseidon2(scopeId_CVS, blindingScalar‚ÇÇ)` ‚Äî independent of `scopeTag_Target`.

6. Agent transmits `blindingScalar‚ÇÇ` to CVS. CVS performs its own local consistency check.

7. DFCU (AS) observes two proofs on-chain. Both reference the same `agentMerkleRoot` (shared tree with 400K agents). The `scopePseudonym` values differ ‚Äî and DFCU cannot test whether they come from the same agent because it doesn't know any agent's `agentSecret` (never revealed) or `credentialCommitment` (hidden behind `enrollmentCommitment`). The `scopeTag` values differ and are independently blinded. Even if DFCU knows that Target and CVS are possible RSes and can compute their `scopeId` values, it cannot test `Poseidon3(agentSecret_guess, cc_guess, scopeId_Target)` against the observed `scopePseudonym` without both `agentSecret` and `cc` ‚Äî neither of which it possesses.

**Quantified privacy:** DFCU's advantage in IND-UNL-AS is bounded by `negl(Œª)` under Poseidon PRF (A1) and Poseidon preimage resistance (A4). The anonymity set is the full agent tree (400K members). Even if DFCU colludes with Target (corrupting 1 of 2 RSes and learning `blindingScalar‚ÇÅ`), it cannot link the CVS proof to the Target proof because (a) `scopeId_CVS` is private, (b) `blindingScalar‚ÇÇ` is unknown to DFCU, and (c) `agentSecret` is unknown to DFCU.

### Healthcare delegation ‚Äî referral network privacy

**Stakeholder:** Kaiser Permanente (KP), a healthcare system with 12.6M members.

**Setup:** A primary care physician (PCP) delegates a scoped credential to a specialist referral agent. The agent interacts with Lab Corp (lab orders) and Walgreens (prescription fulfillment). KP operates the AS.

**Problem:** Under the baseline, KP sees the full referral chain: PCP ‚Üí specialist ‚Üí Lab Corp ‚Üí Walgreens. This reveals the member's condition (specialist type), diagnostic path (lab orders), and treatment (prescriptions). HIPAA's Minimum Necessary Standard (45 CFR ¬ß164.502(b)) requires limiting data exposure to what is necessary for the specific purpose.

**Bolyra deployment:** The PCP's agent uses `ScopeBlindedAgent` with `permissionBitmask = 0b10000101` (READ_DATA + FINANCIAL_SMALL + ACCESS_PII). The agent's `agentSecret` was generated at credential creation and never shared with KP. Both `scopeId_LabCorp` and `scopeId_Walgreens` are private inputs. The proof's `scopeTag` lets each RS verify locally that the proof targets its scope, while KP-as-AS sees only opaque tags and pseudonyms it cannot invert ‚Äî because it knows neither `agentSecret` (held only by the agent) nor `credentialCommitment` (hidden behind `enrollmentCommitment`). The PCP delegates to the specialist agent via the Delegation circuit, narrowing to `0b00000101`. The specialist's agent generates scope-blinded proofs for Lab Corp and Walgreens with independent `blindingScalar` and `scopeId` values ‚Äî all private. KP-as-AS cannot reconstruct the referral topology.

## 8. Why the baseline cannot match

| Capability | Baseline (PPID + RFC 8707 + DPoP + BBS+) | ScopeBlindedAgent |
|---|---|---|
| **AS sees token issuance per RS** | Yes ‚Äî every token is AS-issued, AS logs `(agent, RS, scope, time)` | No ‚Äî agent generates ZK proof locally; AS sees only unlinkable pseudonyms (keyed on `agentSecret` unknown to AS) and opaque scope tags |
| **AS sees which RS a proof targets** | Yes ‚Äî RFC 8707 `resource` parameter visible at issuance | No ‚Äî `scopeId` is a **private circuit input**; the `scopeTag` output is blinded and cannot be inverted without the per-proof `blindingScalar` |
| **AS can invert scope pseudonym via tree enumeration** | N/A ‚Äî AS already knows agent identity | **No (PATCHED)** ‚Äî tree leaves are `enrollmentCommitment = Poseidon2(cc, agentSecret)`, AS cannot extract `cc`; pseudonym `Poseidon3(agentSecret, cc, scopeId)` requires `agentSecret` (~254-bit, never revealed) |
| **RS can verify proof targets its scope** | Yes ‚Äî RS reads the `aud` claim | Yes ‚Äî RS performs verifier-local check: `Poseidon2(scopeId, blindingScalar) == scopeTag` using values received over the direct TLS channel |
| **AS can dictionary-attack RS identity** | N/A ‚Äî AS already knows the RS from the token request | No ‚Äî `scopeTag = Poseidon2(scopeId, blindingScalar)` requires the per-proof `blindingScalar` to test any candidate `scopeId`; `scopePseudonym = Poseidon3(agentSecret, cc, scopeId)` requires `agentSecret` to test any candidate ‚Äî both values are never revealed to the AS |
| **Collusion (AS + k‚àí1 RSes)** | Trivial correlation ‚Äî AS holds the PPID mapping table | Bounded by `negl(Œª)` ‚Äî corrupted RSes reveal their `scopeId` and `blindingScalar` but NOT `agentSecret`; cannot link to honest RS's proof |
| **Collusion (AS + operator)** | N/A ‚Äî AS is typically the operator | Bounded by `negl(Œª)` ‚Äî operator knows `cc` but NOT `agentSecret`; cannot compute `Poseidon3(agentSecret, cc, scopeId)` |
| **Issuer anonymity** | Absent ‚Äî BBS+ exposes issuer public key | N/A ‚Äî agent Merkle tree is shared; no issuer key in public signals |
| **Formal security definition** | None ‚Äî no IND-UNL-AS game exists in OAuth/OIDC/BBS+ specs | IND-UNL-AS game defined (¬ß3) with reduction to Poseidon PRF (¬ß4) |
| **Timing side-channel** | No mitigation ‚Äî DPoP `jti` and timestamps leak request timing to AS | Batch relay mixing window; cryptographic unlinkability holds even without it |
| **Delegation chain privacy from AS** | Absent ‚Äî RFC 8693 every hop is AS-visible | Blinded scope commitment chains through Delegation circuit; AS sees only rerandomized outputs |
| **Rerandomization per proof** | Absent ‚Äî same BBS+ credential, same issuer key | `blindingScalar` ensures each proof's public signals are fresh even for identical inputs |

**The structural impossibility:** The baseline requires the AS to issue every token. This is not a configuration flaw ‚Äî it is the definition of OAuth 2.0. An AS that issues a token necessarily knows the requesting agent, the target RS, and the requested scope. No combination of PPIDs, DPoP, or BBS+ changes this because they operate *after* issuance. Bolyra's `ScopeBlindedAgent` eliminates the issuance step entirely: the agent proves its credential validity in zero knowledge without any AS interaction at proof time. The AS is reduced to a tree maintainer (enrollment) and cannot observe per-session authorization events ‚Äî nor can it determine which RS any given proof targets, because `scopeId` is a private input and the `scopeTag` is cryptographically blinded. Critically, even the tree enumeration attack is closed: the AS cannot extract `credentialCommitment` from `enrollmentCommitment` leaves (Poseidon preimage resistance) and cannot compute scope pseudonyms without `agentSecret` (Poseidon PRF). This is a categorical, not incremental, separation.
