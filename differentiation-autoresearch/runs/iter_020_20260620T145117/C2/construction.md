# Construction

## 1. Statement of claim

Same agent accessing different Resource Server (RS) instances produces cryptographically unlinkable authorizations even under an adversarial Authorization Server (AS) that colludes with any subset of RSes and attempts to correlate per-agent traffic graphs via token content, scope sequences, timing metadata, or delegation chain topology.

Formally: no PPT adversary controlling the AS and up to N−1 of N RSes can distinguish whether two authorization proofs presented to two honest RSes originate from the same agent or two independent agents, with advantage better than negligible in the security parameter λ.

## 2. Construction (gadgets, circuits, public/private inputs)

### Core idea

Replace AS-issued bearer tokens with agent-generated ZK proofs where the nullifier is deterministically bound to the target RS's scope identifier. The AS never sees which RS the agent targets — it only maintains the agent Merkle tree. Each RS receives a proof whose nullifier is unlinkable to nullifiers presented at any other RS, and whose public signals reveal no agent identity.

### New circuit: `AgentScopedAuth`

This is a variant of the existing `AgentPolicy` circuit, specialized for per-RS authorization without AS involvement after initial enrollment.

**Private inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `modelHash` | F_p | Hash of agent model identifier |
| `operatorPubkeyAx` | F_p | Operator EdDSA public key x-coordinate |
| `operatorPubkeyAy` | F_p | Operator EdDSA public key y-coordinate |
| `permissionBitmask` | 64-bit | Agent's full permission bitfield |
| `expiryTimestamp` | 64-bit | Credential expiration |
| `sigR8x`, `sigR8y`, `sigS` | F_p | Operator EdDSA signature over credential commitment |
| `merkleProofLength` | field | Actual Merkle depth |
| `merkleProofIndex` | field | Leaf index |
| `merkleProofSiblings[20]` | F_p[] | Merkle siblings padded to MAX_DEPTH |
| `scopeBlindingSecret` | F_p | Per-agent persistent blinding secret (derived from agent master secret) |

**Public inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `rsScope` | F_p | RS-specific scope identifier (e.g., Poseidon hash of RS domain) |
| `requiredScopeMask` | 64-bit | Minimum permission bits the RS demands |
| `currentTimestamp` | 64-bit | Verifier-provided current time |
| `sessionNonce` | F_p | Fresh per-request nonce (generated by RS, not AS) |

**Public outputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `agentMerkleRoot` | F_p | Computed Merkle root for on-chain verification |
| `scopedNullifier` | F_p | `Poseidon(rsScope, scopeBlindingSecret)` — deterministic per (agent, RS) |
| `nonceBinding` | F_p | `Poseidon(scopedNullifier, sessionNonce)` — replay prevention |
| `scopeCommitment` | F_p | `Poseidon(permissionBitmask, credentialCommitment)` — for optional delegation entry |

**Constraints (in addition to standard AgentPolicy constraints):**

1. **Credential commitment:** `credCommitment = Poseidon5(modelHash, opAx, opAy, permissionBitmask, expiryTimestamp)`
2. **EdDSA verification:** `EdDSAPoseidonVerifier(opAx, opAy, sigR8x, sigR8y, sigS, credCommitment) === 1`
3. **Merkle membership:** `BinaryMerkleRoot(credCommitment, proof) === agentMerkleRoot`
4. **Scope satisfaction:** `∀i ∈ [0,64): requiredBits[i] * (1 - permBits[i]) === 0`
5. **Cumulative bit encoding:** standard tier implication constraints (bits 4→3→2)
6. **Expiry:** `currentTimestamp < expiryTimestamp` via `LessThan(64)`
7. **Range checks:** `Num2Bits(64)` on `permissionBitmask`, `expiryTimestamp`, `currentTimestamp`
8. **Scoped nullifier:** `scopedNullifier = Poseidon(rsScope, scopeBlindingSecret)` — the nullifier depends only on the RS scope and the agent's blinding secret, NOT on the credential commitment
9. **Blinding secret binding:** `Poseidon(scopeBlindingSecret, credCommitment)` is constrained to equal a hidden internal wire, ensuring the blinding secret is tied to the enrolled credential (prevents nullifier hijacking without affecting public outputs)
10. **Nonce binding:** `nonceBinding = Poseidon(scopedNullifier, sessionNonce)`
11. **Scope commitment:** `scopeCommitment = Poseidon(permissionBitmask, credCommitment)`

### Interaction protocol (AS-free authorization)

```
Agent                         RS-A                    On-chain Registry
  |                            |                           |
  |  1. Request access         |                           |
  |--------------------------->|                           |
  |  2. Challenge(rsScope,     |                           |
  |     requiredMask, nonce,   |                           |
  |     timestamp)             |                           |
  |<---------------------------|                           |
  |                            |                           |
  |  3. Generate AgentScopedAuth proof locally              |
  |  (no AS contact needed)    |                           |
  |                            |                           |
  |  4. Present(proof,         |                           |
  |     pubSignals)            |                           |
  |--------------------------->|                           |
  |                            | 5. Verify agentMerkleRoot |
  |                            |    against root history   |
  |                            |-------------------------->|
  |                            | 6. Check scopedNullifier  |
  |                            |    not revoked            |
  |                            |-------------------------->|
  |                            | 7. Verify Groth16 proof   |
  |                            |    (on-chain or off-chain) |
  |                            |<--------------------------|
  |  8. Access granted         |                           |
  |<---------------------------|                           |
```

The AS is involved only in enrollment (adding credential commitments to the Merkle tree). It never participates in per-RS authorization. The RS generates its own nonce and verifies the proof against the on-chain Merkle root.

### Batched authorization (timing side-channel mitigation)

To resist timing correlation by an adversarial AS that monitors on-chain root-check transactions:

- RSes MAY cache the agent Merkle root locally (updated every N blocks) and verify proofs entirely off-chain using the exported Groth16 verification key.
- When off-chain verification is used, the AS observes zero per-request signals.
- For on-chain verification, agents MAY batch proofs for multiple RSes into a single transaction via a relay contract, preventing per-RS timing extraction.

## 3. Threat model (adversary capabilities, game definition)

### Adversary capabilities

The adversary 𝒜 controls:

- **The AS completely:** 𝒜 operates the Authorization Server, sees all enrollment data, knows all credential commitments, controls the Merkle tree, and can add/remove entries.
- **Up to N−1 of N RSes:** 𝒜 can collude with any strict subset of Resource Servers, seeing all proofs presented to colluding RSes.
- **Network observation:** 𝒜 can observe all network traffic between agents and RSes (but cannot break TLS — only sees metadata like timing and packet sizes).

The adversary CANNOT:

- Extract the agent's `scopeBlindingSecret` or `operatorPrivateKey` (standard computational assumption).
- Break Poseidon collision resistance or preimage resistance.
- Break the knowledge soundness of Groth16.
- Corrupt the honest RS (at least one RS must be honest for unlinkability to be meaningful).

### IND-UNL-AS game

**Game `IND-UNL-AS(λ)`:**

1. **Setup:** Challenger generates system parameters, deploys the Merkle tree, and enrolls agents.

2. **Phase 1 (Learning):** 𝒜 adaptively requests authorization proofs for agents of its choice, for RSes of its choice. 𝒜 receives all public signals. 𝒜 can query the enrollment oracle, revocation oracle, and colluding RS oracles.

3. **Challenge:** 𝒜 selects two enrolled agents (Agent₀, Agent₁) and two honest RSes (RS_A, RS_B) such that both agents hold valid credentials satisfying both RSes' required scope masks. The challenger flips a bit b ∈ {0,1}:
   - If b = 0: Agent₀ authenticates to RS_A, Agent₁ authenticates to RS_B.
   - If b = 1: Agent₀ authenticates to RS_B, Agent₁ authenticates to RS_A.
   
   𝒜 receives: the proof and public signals presented to RS_A, and the proof and public signals presented to RS_B. (𝒜 does NOT see which agent produced which proof — this is the challenge.)

4. **Phase 2 (Post-challenge learning):** 𝒜 may continue querying, subject to the restriction that it does not query the challenge agents at the challenge RSes.

5. **Guess:** 𝒜 outputs b' ∈ {0,1}. 𝒜 wins if b' = b.

**Definition:** The protocol achieves **(t, ε)-IND-UNL-AS security** if for all adversaries 𝒜 running in time t:

```
|Pr[b' = b] − 1/2| ≤ ε
```

## 4. Security argument (named assumption + reduction sketch)

### Named assumptions

- **A1: Poseidon PRF security.** Poseidon, keyed on the blinding secret, is a pseudorandom function: `Poseidon(rsScope, scopeBlindingSecret)` is indistinguishable from random for any polynomial number of distinct `rsScope` queries, given that `scopeBlindingSecret` is unknown.
- **A2: Poseidon collision resistance.** No PPT adversary can find `(x₁, x₂) ≠ (y₁, y₂)` such that `Poseidon(x₁, x₂) = Poseidon(y₁, y₂)`.
- **A3: Groth16 knowledge soundness.** The Groth16 proving system is knowledge-sound under the q-DLOG assumption in the generic group model.
- **A4: Groth16 zero-knowledge.** Groth16 proofs are computationally zero-knowledge: the proof π reveals nothing about private inputs beyond what is implied by the public signals.

### Reduction sketch

**Theorem:** If assumptions A1–A4 hold, the `AgentScopedAuth` protocol is (t, ε)-IND-UNL-AS secure where ε is negligible in λ.

**Proof sketch:**

Suppose adversary 𝒜 wins IND-UNL-AS with non-negligible advantage ε. We construct a distinguisher 𝒟 that breaks Poseidon PRF security.

1. **Eliminating proof leakage (A4):** By Groth16 zero-knowledge, the proof π itself reveals nothing beyond the public signals. Replace all proofs in the game with simulated proofs. The adversary's view changes by at most negl(λ).

2. **Reducing to nullifier distinguishing:** After simulation, 𝒜's view consists only of:
   - `agentMerkleRoot` — identical for both agents (same tree)
   - `scopedNullifier` — the key distinguishing signal
   - `nonceBinding` — derived deterministically from `scopedNullifier` and the public `sessionNonce`
   - `scopeCommitment` — different per agent, but 𝒜 already knows which credentials are enrolled (it controls the AS), so this reveals the agent identity only if 𝒜 can map it. However, `scopeCommitment = Poseidon(permissionBitmask, credCommitment)` and 𝒜 knows all credential commitments (it controls the AS).

   **Critical refinement:** Since 𝒜 controls the AS and knows all `credCommitment` values, `scopeCommitment` would break unlinkability if `permissionBitmask` is unique per agent. We address this by requiring that the RS verifies `scopeCommitment` only when entering a delegation chain. For standalone RS authorization, the circuit outputs `scopeCommitment` BUT the RS does not forward it to the AS. The proof is verified by the RS alone (off-chain or against the on-chain root only).

   For the IND-UNL-AS game with non-colluding RSes, the adversary does not see `scopeCommitment` from honest RSes. From colluding RSes, it does — but the challenge RSes are honest by game definition.

   With this, 𝒜's distinguishing advantage reduces to distinguishing `scopedNullifier₀ = Poseidon(rsScope_target, blindingSecret₀)` from `scopedNullifier₁ = Poseidon(rsScope_target, blindingSecret₁)`.

3. **PRF reduction (A1):** If 𝒜 can distinguish which blinding secret produced the nullifier for an honest RS, construct 𝒟 that:
   - Receives a PRF challenge oracle `O(·)` keyed on an unknown key k ∈ {blindingSecret₀, blindingSecret₁}.
   - Embeds `O(rsScope_A)` as the scopedNullifier at RS_A.
   - Simulates the rest of the game.
   - Uses 𝒜's guess to determine which key was used.
   
   𝒟's advantage = 𝒜's advantage − negl(λ), contradicting A1.

4. **Nonce binding does not leak (A2):** `nonceBinding = Poseidon(scopedNullifier, sessionNonce)` where `sessionNonce` is public. Given that `scopedNullifier` is pseudorandom (from step 3), `nonceBinding` reveals no additional information about the agent's identity beyond `scopedNullifier` itself.

**Conclusion:** 𝒜's advantage is bounded by the PRF advantage against Poseidon plus negligible simulation error. ∎

### Side-channel argument

**Timing:** When the RS verifies proofs off-chain (cached Merkle root + local Groth16 verifier), the AS observes zero per-request events. No timing signal exists at the AS layer.

**Nonce freshness:** Session nonces are generated by the RS, not the AS. The AS does not see nonces for honest RSes.

**Delegation chains:** If the agent enters a delegation chain after RS authorization, the on-chain `scopeCommitment` is written. An adversarial AS can observe this write. However, the delegation chain uses `sessionNonce` as the index — the AS cannot correlate the delegation entry to a specific RS unless it knows the RS's scope (which it does for colluding RSes only). For honest RSes, delegation chain entries are indexed by a nonce the AS did not generate and cannot map to an RS.

## 5. Bolyra primitive mapping

| Construction element | Bolyra primitive | Spec reference |
|---------------------|-----------------|----------------|
| Agent credential commitment | `Poseidon5(modelHash, opAx, opAy, permBitmask, expiry)` | AgentPolicy circuit, §3.3 |
| Scoped nullifier | `Poseidon2(rsScope, scopeBlindingSecret)` | New — extends nullifier pattern from HumanUniqueness §3.2 |
| Nonce binding | `Poseidon2(scopedNullifier, sessionNonce)` | Mirrors human nonceBinding pattern, §3.2 |
| Scope commitment | `Poseidon2(permissionBitmask, credCommitment)` | Existing delegation chain seed, §4 |
| Merkle membership | BinaryMerkleRoot depth 20 over agent tree | Existing agent tree, §3.3 |
| Operator signature | EdDSA on Baby Jubjub over credCommitment | Existing AgentPolicy constraint, §3.3 |
| Permission enforcement | Cumulative bit encoding with `requiredBits[i] * (1 - permBits[i]) === 0` | Existing AgentPolicy constraint, §3.3 |
| Proving system | Groth16 with `pot16.ptau` (project-specific Phase 2) | §2.2 — PLONK optional |
| Blinding secret binding | `Poseidon2(scopeBlindingSecret, credCommitment)` — internal constraint only | New — prevents nullifier hijacking |

The `scopeBlindingSecret` is the only new private input. It is derived deterministically: `scopeBlindingSecret = Poseidon2(agentMasterSecret, "bolyra-scope-blind")` where `agentMasterSecret` is the agent's EdDSA private key. This avoids introducing new key material — the agent's existing private key seeds the blinding secret.

## 7. Concrete deployment scenario

### Cross-credit-union member agent

**Stakeholders:**
- **Mountain West Federal Credit Union (AS):** Issues agent credentials to members. Operates the agent Merkle tree. Must NOT learn which merchants a member's agent visits.
- **RS-A: Costco Payment Terminal** — accepts Bolyra-authenticated agent payments (permission bits: READ_DATA + FINANCIAL_SMALL = 0b00000101).
- **RS-B: CVS Pharmacy Portal** — accepts Bolyra-authenticated agent prescription pickups (permission bits: READ_DATA + ACCESS_PII = 0b10000001).

**Flow:**

1. **Enrollment (one-time):** Mountain West CU enrolls member Alice's agent. The CU generates an operator keypair, signs a credential commitment `Poseidon5(modelHash, opAx, opAy, 0b10000111, expiry)` with full permissions, and inserts it into the on-chain agent Merkle tree. Alice's agent derives `scopeBlindingSecret = Poseidon2(operatorPrivKey, "bolyra-scope-blind")`.

2. **Authorization at Costco (RS-A):**
   - Costco generates `rsScope_A = Poseidon("costco.com/payments")` and a fresh `sessionNonce_A`.
   - Alice's agent generates an `AgentScopedAuth` proof with `requiredScopeMask = 0b00000101`.
   - Public signals include `scopedNullifier_A = Poseidon(rsScope_A, scopeBlindingSecret)`.
   - Costco verifies the proof against the cached agent Merkle root. Mountain West CU sees nothing.

3. **Authorization at CVS (RS-B):**
   - CVS generates `rsScope_B = Poseidon("cvs.com/rx")` and a fresh `sessionNonce_B`.
   - Alice's agent generates an `AgentScopedAuth` proof with `requiredScopeMask = 0b10000001`.
   - Public signals include `scopedNullifier_B = Poseidon(rsScope_B, scopeBlindingSecret)`.
   - CVS verifies independently. Mountain West CU sees nothing.

4. **Unlinkability guarantee:**
   - `scopedNullifier_A ≠ scopedNullifier_B` (different rsScope inputs).
   - Mountain West CU never sees either nullifier (proofs are verified off-chain by the RS).
   - Even if Mountain West CU colludes with Costco, it learns `scopedNullifier_A` but cannot compute `scopedNullifier_B` without `scopeBlindingSecret` (Poseidon PRF security).
   - Costco and CVS cannot correlate because they see different nullifiers for the same agent.

5. **Sybil resistance within a single RS:**
   - `scopedNullifier_A` is deterministic per (agent, RS). If Alice's agent visits Costco again, same nullifier — double-spend detection works.
   - Different agents produce different nullifiers at the same RS (different `scopeBlindingSecret`).

### Healthcare variant

**Stakeholders:**
- **BlueCross BlueShield (AS):** Issues agent credentials to members.
- **RS-A: Mayo Clinic** — primary care referral.
- **RS-B: Cleveland Clinic** — specialist referral.

The referral chain uses delegation: Mayo Clinic's agent delegates narrowed credentials to Cleveland Clinic's agent. The delegation circuit uses `previousScopeCommitment` chain linking. BlueCross sees enrollment and Merkle root updates but cannot determine which member was referred from Mayo to Cleveland — the delegation's `sessionNonce` is generated by Mayo, not BlueCross, and the `delegationNullifier` is scope-bound.

## 8. Why the baseline cannot match

| Property | Baseline (PPID + RFC 8707 + DPoP + BBS+) | Bolyra `AgentScopedAuth` |
|----------|-------------------------------------------|--------------------------|
| **AS sees token requests** | Yes — every token is issued by the AS, which logs agent ID, target RS, scope, and timestamp | No — AS only sees enrollment. Per-RS proofs are generated and verified without AS involvement |
| **AS can correlate cross-RS traffic** | Trivially — AS holds the PPID mapping table and issues all tokens | Impossible — AS never sees `scopedNullifier` values; each is RS-specific and pseudorandom |
| **Colluding AS + RS can reconstruct graph** | Yes — AS matches agent ID to RS-provided token metadata | No — colluding RS reveals one `scopedNullifier` but AS cannot compute nullifiers for non-colluding RSes (Poseidon PRF) |
| **Colluding RS₁ + RS₂ can link agent** | BBS+ presentations are unlinkable, BUT DPoP proofs may share agent key metadata | Nullifiers are scope-separated: `scopedNullifier_A ≠ scopedNullifier_B`. No shared key material in public signals |
| **Delegation chain visible to AS** | RFC 8693 requires AS roundtrip per hop — full chain topology exposed | Delegation proofs chain via `scopeCommitment` — AS sees Merkle root updates but not chain structure |
| **Timing side channel** | AS sees request timestamps for every token issuance | RS verifies off-chain with cached root. AS observes zero per-request signals |
| **Formal security definition** | None exists in any RFC or W3C spec | IND-UNL-AS game with reduction to Poseidon PRF security |
| **Nonce freshness leakage** | DPoP `jti` + timestamp leak request timing to AS | Session nonce generated by RS, not AS. AS never sees it |

**The structural impossibility:** The OAuth/OIDC model is fundamentally AS-centric — every authorization event passes through the AS by design. No layering of PPIDs, BBS+, or DPoP can remove the AS from the token issuance path. Bolyra eliminates this path entirely: after enrollment, the AS is not contacted for per-RS authorization. The agent proves its credentials directly to the RS using a ZK proof against a public Merkle root. This architectural difference is not a configuration choice — it is a consequence of replacing bearer tokens with zero-knowledge proofs of Merkle membership.
