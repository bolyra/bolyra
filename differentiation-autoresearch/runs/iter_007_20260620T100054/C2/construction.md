# Construction

## 1. Statement of claim

Same agent accessing different RS instances produces cryptographically unlinkable authorizations even under an adversarial AS that (a) controls enrollment, (b) logs every issuance event, and (c) colludes with any strict subset of RS instances. Formally: no PPT adversary wins the IND-UNL-AS game (defined in §3) with advantage greater than negl(λ).

The baseline (OIDC PPID + RFC 8707 + DPoP + BBS+) fails because the AS observes every token issuance, including the target RS audience claim, enabling trivial cross-scope correlation from the AS issuance log alone.

## 2. Construction (gadgets, circuits, public/private inputs)

### Design principle

Decouple credential enrollment from scope-specific authorization. The AS enrolls an agent's credential commitment into the agent Merkle tree exactly once. All subsequent per-scope authorizations are computed locally by the agent using a new circuit — **ScopeIsolatedAuth** — that produces scope-bound nullifiers unlinkable across scopes. The AS is never contacted at authorization time and therefore learns nothing about which scopes the agent exercises.

### New circuit: ScopeIsolatedAuth (PLONK)

This circuit sits between the existing AgentPolicy (enrollment) and per-RS authorization. It re-proves Merkle membership but derives a scope-specific nullifier and a randomized authorization tag.

**Private inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `blindingSalt` | F_p (254 bits) | Agent-local secret, generated by CSPRNG, never shared with the AS |
| `modelHash` | F_p | Hash of model identifier |
| `operatorPubkeyAx`, `operatorPubkeyAy` | F_p | Operator EdDSA public key |
| `permissionBitmask` | uint64 | Agent's enrolled permission bitfield |
| `expiryTimestamp` | uint64 | Credential expiry |
| `merkleProofLength` | uint | Actual depth |
| `merkleProofIndex` | uint | Leaf index |
| `merkleProofSiblings[20]` | F_p[20] | Sibling hashes |
| `blindingNonce` | F_p | Fresh random value per proof |

**Public inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `scopeId` | F_p | RS-specific scope identifier (e.g., hash of RS URI) |
| `requiredScopeMask` | uint64 | Required permission bits for this RS |
| `currentTimestamp` | uint64 | Verifier-supplied current time |

**Public outputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `agentMerkleRoot` | F_p | Computed Merkle root (checked against on-chain root history buffer) |
| `scopeNullifier` | F_p | `Poseidon2(scopeId, agentSecret)` where `agentSecret = Poseidon2(credCommitment, blindingSalt)` — deterministic per (agent, scope), unlinkable across scopes |
| `authTag` | F_p | `Poseidon2(scopeNullifier, blindingNonce)` — randomized per-presentation, prevents RS-to-RS correlation on the nullifier itself |

**Constraints (11 groups):**

1. **BlindingSalt range**: `Num2Bits(254)` on `blindingSalt`, ensuring `blindingSalt ∈ [0, 2^254)`.

2. **Credential commitment reconstruction**: `credCommitment = Poseidon5(modelHash, operatorPubkeyAx, operatorPubkeyAy, permissionBitmask, expiryTimestamp)`.

3. **Agent secret derivation**: `agentSecret = Poseidon2(credCommitment, blindingSalt)`. The `credCommitment` is a public on-chain value (the AS knows it from enrollment), but `blindingSalt` is the PRF key — knowledge of `credCommitment` alone cannot recover `agentSecret`. This replaces the prior `Poseidon2(operatorSecret, modelHash)` derivation which was fatally broken when the operator IS the AS.

4. **Secret range**: `Num2Bits(251)` on `agentSecret`, ensuring `agentSecret ∈ [0, 2^251)`.

5. **Merkle membership**: `BinaryMerkleRoot(20)` with `credCommitment` as leaf must produce `agentMerkleRoot`.

6. **Scope satisfaction**: For each bit i in [0, 64): `requiredBits[i] * (1 - permBits[i]) === 0`.

7. **Cumulative bit encoding**:
   - `permBits[4] * (1 - permBits[3]) === 0`
   - `permBits[4] * (1 - permBits[2]) === 0`
   - `permBits[3] * (1 - permBits[2]) === 0`

8. **Expiry**: `currentTimestamp < expiryTimestamp` via `LessThan(64)`.

9. **Range checks**: `Num2Bits(64)` on `permissionBitmask`, `expiryTimestamp`, `currentTimestamp`.

10. **Scope nullifier derivation**: `scopeNullifier = Poseidon2(scopeId, agentSecret)`. This is the critical unlinkability gadget — identical in structure to the human nullifier from HumanUniqueness, but keyed on the agent's derived secret.

11. **Auth tag derivation**: `authTag = Poseidon2(scopeNullifier, blindingNonce)`. The blinding nonce randomizes each presentation so that even within the same scope, two presentations to the same RS are unlinkable unless the RS checks the scope nullifier.

### Public signal layout (PLONK verifier, 6 signals)

| Index | Signal | Description |
|-------|--------|-------------|
| 0 | `agentMerkleRoot` | Merkle root for enrollment verification |
| 1 | `scopeNullifier` | Per-scope replay detection |
| 2 | `authTag` | Randomized per-presentation tag |
| 3 | `requiredScopeMask` | Required permission bits |
| 4 | `currentTimestamp` | Freshness bound |
| 5 | `scopeId` | Scope identifier |

### Protocol flow (authorization, post-enrollment)

1. Agent has been previously enrolled via the standard AgentPolicy circuit (one-time, AS-visible).
2. Agent wants to access RS-A with `scopeId = Poseidon("https://rs-a.example/")`.
3. Agent locally computes `agentSecret = Poseidon2(credCommitment, blindingSalt)`, then `scopeNullifier_A = Poseidon2(scopeId_A, agentSecret)` and generates a ScopeIsolatedAuth PLONK proof with a fresh `blindingNonce`.
4. Agent sends `(proof, publicSignals)` directly to RS-A. **No AS interaction occurs.**
5. RS-A verifies: (a) PLONK proof validity, (b) `agentMerkleRoot` is in the on-chain root history buffer, (c) `scopeNullifier` is not revoked/reused, (d) `currentTimestamp` is fresh.
6. RS-A accepts. It sees `scopeNullifier_A` and `authTag_A` — neither of which is linkable to `scopeNullifier_B` or `authTag_B` that the same agent presents to RS-B.

### Integration with existing Bolyra handshake

The ScopeIsolatedAuth circuit is used **after** the mutual handshake for per-RS authorization. The handshake establishes the session and delegation chain seed; ScopeIsolatedAuth proves per-RS authorization without further AS involvement. The `agentSecret` is derived as `Poseidon2(credCommitment, blindingSalt)` where `blindingSalt` is generated locally by the agent and never shared with the AS or operator.

### BlindingSalt lifecycle

**Generation**: The agent generates `blindingSalt` via a 256-bit CSPRNG at credential creation time. The AS MUST NOT be involved in salt generation. The salt MUST NOT be derived from any value the AS or operator can reconstruct.

**Storage** (in descending preference): (1) HSM-backed keystore, (2) OS keychain (macOS Keychain, Android Keystore), (3) encrypted file with a passphrase the AS cannot access.

**Rotation**: The agent MAY rotate `blindingSalt` at any time by generating a new salt. Rotation does not require any on-chain transaction — it changes `agentSecret` and therefore all future `scopeNullifier` values, effectively creating a new unlinkable identity within the same enrolled credential.

**Recovery**: There is no AS-assisted recovery path. If `blindingSalt` is lost, the agent's scope-specific identity is irrecoverable. Optional: Shamir secret sharing across trusted parties (not including the AS).

**MUST NOT rules:**
1. `blindingSalt` MUST NOT be transmitted to the AS at any point.
2. `blindingSalt` MUST NOT be derived from `operatorSecret` or any operator-controlled seed.
3. `blindingSalt` MUST NOT be stored in any system the AS administers.
4. `blindingSalt` MUST NOT appear in enrollment transactions or their metadata.
5. `blindingSalt` MUST NOT be logged by any component in the proof generation pipeline.

## 3. Threat model (adversary capabilities, game definition)

### Adversary power levels

The adversary A operates at one of three power levels. Each level strictly subsumes the previous.

**Level 1 — Passive AS (A_passive):**
- Observes the enrollment log: every `credentialCommitment` inserted into the agent Merkle tree, including timing and transaction metadata.
- Observes the `operatorSecret` (the operator may be the AS itself).
- Observes encrypted network traffic metadata (timing, packet sizes) between agents and all RS instances.
- Does NOT inject, modify, or replay enrollment entries.
- Does NOT receive any proof or public signal presented to any RS.

**Level 2 — Active AS (A_active):**
- Everything in Level 1.
- Chooses which agents to enroll, with what credential fields (adaptive enrollment).
- May generate its own agents and enroll them to probe the system.
- May query an authorization oracle: given a `scopeId`, receive a valid `(proof, publicSignals)` tuple for any non-challenge agent (modeling the AS controlling some agents directly).
- Does NOT collude with any RS.

**Level 3 — Colluding AS+RS (A_collude):**
- Everything in Level 2.
- Colludes with up to `k-1` of `k` RS instances. Receives every `(proof, publicSignals)` tuple — including `scopeNullifier` and `authTag` — presented to the colluding RS instances.
- May correlate colluding RS observations with the enrollment log and timing metadata.

**What A does NOT control at any level:**
- The agent's local computation environment (specifically, `blindingSalt` is not leaked).
- At least one RS instance (otherwise A trivially sees all scopes).

### IND-UNL-AS game definition

The game is parameterized by adversary level L ∈ {1, 2, 3} and timing model T ∈ {T-none, T-batch, T-real}.

```
Game IND-UNL-AS[L, T](λ):

  Setup(1^λ):
    1. Challenger C generates Bolyra system parameters (BN128 curve, Poseidon
       parameters, PLONK SRS, Merkle trees, verifier contracts).
    2. C samples a set of n agents {a_1, ..., a_n}. For each agent a_i:
       - C generates blindingSalt_i ←$ {0,1}^256
       - C computes credCommitment_i = Poseidon5(modelHash_i, opAx_i, opAy_i, perm_i, exp_i)
       - C computes agentSecret_i = Poseidon2(credCommitment_i, blindingSalt_i)
       - C enrolls credCommitment_i into the agent Merkle tree.
    3. A receives: the full enrollment log (all credCommitment_i values, timing,
       operator public keys, permission bitmasks, expiry timestamps).
       If L ≥ 2: A may adaptively choose some agents' credential fields before
       enrollment (A still does not choose blindingSalt).

  Query phase (adaptive, polynomially many queries):
    - Auth(i, scopeId): A requests a ScopeIsolatedAuth proof for agent a_i
      on scope scopeId. C returns (proof, publicSignals) including
      scopeNullifier and authTag.
      Restriction: not available for the challenge agent a* on challenge scopes
      (scopeId_0, scopeId_1).
    - If L = 3: Collude(j, scopeId): A designates RS_j as colluding.
      A then receives all future Auth outputs directed to RS_j.
      Restriction: at most k-1 of k RS instances may be designated.
    - Timing oracle T:
      - T-none: no timing information returned with any query.
      - T-batch: A receives a batch timestamp window [t - δ, t + δ] covering
        all proofs generated in the same batch, but not individual proof times.
      - T-real: A receives the exact wall-clock time of each proof generation.

  Challenge(scopeId_0, scopeId_1):
    1. A selects a challenge agent a* ∈ {a_1, ..., a_n} and two distinct
       target scopes scopeId_0, scopeId_1.
       Restriction: if L = 3, neither scopeId_0 nor scopeId_1 may be a scope
       associated with a colluding RS (otherwise A trivially wins by observing
       the colluding RS's nullifier).
    2. C samples b ←$ {0,1}.
    3. C generates two ScopeIsolatedAuth proofs for a*:
       - π_0: proof for a* on scopeId_b     (with fresh blindingNonce_0)
       - π_1: proof for a* on scopeId_{1-b} (with fresh blindingNonce_1)
    4. C sends to A:
       - (π_0, scopeId_0, scopeNullifier_0*, authTag_0*)
       - (π_1, scopeId_1, scopeNullifier_1*, authTag_1*)
       where scopeNullifier_0* is the nullifier from π_0 and scopeId_0 is
       the label (A must determine which nullifier belongs to which scope).

  Post-challenge queries:
    A may continue issuing Auth and Collude queries subject to the same
    restrictions (no Auth for a* on scopeId_0 or scopeId_1; no Collude
    designating an RS serving scopeId_0 or scopeId_1).

  Guess:
    A outputs b' ∈ {0,1}.

  Winning condition:
    A wins if b' = b. The advantage is:
      Adv^{IND-UNL-AS}_{L,T}(A) = |Pr[b' = b] - 1/2|

  Security claim:
    The construction is (L, T)-cross-scope-unlinkable if for all PPT A:
      Adv^{IND-UNL-AS}_{L,T}(A) ≤ negl(λ)
```

**Winning condition in plain language:** A wins if it can determine which of the two challenge proofs was generated for which scope, even given the full enrollment log, the operator's secret, adaptive enrollment control (Level 2+), and scope nullifiers from colluding RSes on non-challenge scopes (Level 3).

### Side-channel parameterization

The timing model T is orthogonal to the adversary level L:

- **T-none**: Pure cryptographic game. No timing information. This is the primary security target.
- **T-batch**: Agent batches all scope authorizations into a single timing window (±δ). A sees only the batch, not individual proof generation times. Security reduces to T-none plus the indistinguishability of proof generation order within the batch.
- **T-real**: A sees per-proof generation timestamps. Security degrades — if each scope is accessed at a distinct, characteristic time, timing alone may leak the association. This is mitigated operationally (random delay ∈ [0, 2s], jitter) but is NOT covered by the cryptographic reduction.

### Scope of the formal claim

The construction achieves IND-UNL-AS security at **all three levels** in the **T-none** model. At **T-batch**, security holds under the additional assumption that proof generation times within a batch are indistinguishable. **T-real** is explicitly out of scope for the formal claim; it is addressed operationally in §7.

## 4. Security argument (named assumption + reduction sketch)

### Named assumptions

1. **Poseidon-PRF**: Poseidon2, keyed on its second input, is a pseudorandom function. Formally: for any PPT distinguisher D, `|Pr[D(Poseidon2(x, k)) = 1] - Pr[D(R(x)) = 1]| < negl(λ)` where k is uniform over F_p and R is a truly random function.

2. **Knowledge soundness of PLONK**: The PLONK proving system (with universal SRS) satisfies knowledge soundness — from any valid proof, an extractor can recover a valid witness.

3. **Collision resistance of Poseidon**: Finding `(x, x')` with `x ≠ x'` and `Poseidon(x) = Poseidon(x')` requires `Ω(2^{128})` work over F_p.

4. **DLP on Baby Jubjub**: Given `(Ax, Ay) = s * G`, recovering `s` is infeasible.

### Reduction: Level 1 (Passive AS)

**Theorem 1**: If Poseidon-PRF holds and PLONK is zero-knowledge, no PPT A_passive wins IND-UNL-AS[1, T-none] with non-negligible advantage.

**Proof sketch**:

A_passive sees only the enrollment log. It never receives any `scopeNullifier` or `authTag` except the two challenge tuples.

- **Hybrid 0**: Real game. A receives `(π_0, π_1)` for a* on scopes `(scopeId_b, scopeId_{1-b})`.

- **Hybrid 1**: Replace `agentSecret* = Poseidon2(credCommitment*, blindingSalt*)` with a uniform random `s* ∈ F_p`. By Poseidon-PRF keyed on `blindingSalt*` (which A_passive does not know — even if A knows `operatorSecret`, it does not know `blindingSalt`), Hybrids 0 and 1 are computationally indistinguishable. Note: `credCommitment*` is known to A from the enrollment log, but it is the PRF *input*, not the key. The key is `blindingSalt*`, which is uniform and A-independent by the MUST NOT rules (§2).

- **Hybrid 2**: Replace `scopeNullifier_0 = Poseidon2(scopeId_0, s*)` with uniform `r_0 ∈ F_p`. Since `s*` is now uniform (Hybrid 1), this is a second application of Poseidon-PRF keyed on `s*`. Indistinguishable.

- **Hybrid 3**: Replace `scopeNullifier_1 = Poseidon2(scopeId_1, s*)` with uniform `r_1 ∈ F_p`. Same argument. Now both nullifiers are independent uniform values.

- **Hybrid 4**: Replace both `authTag` values with uniform random (Poseidon-PRF on the now-random nullifier inputs as keys, with `blindingNonce` as input).

- **Hybrid 5**: In Hybrid 4, A's view consists of two PLONK proofs with uniformly random public outputs `(scopeNullifier, authTag)`. By PLONK zero-knowledge, the proofs reveal nothing about private inputs. The public inputs `(scopeId_0, scopeId_1, requiredScopeMask, currentTimestamp)` are symmetric — they appear in both orderings. A's advantage in Hybrid 5 is exactly 0.

Advantage bound: `Adv(A_passive) ≤ 3 · Adv_PRF(λ) + negl_ZK(λ)`.

**Why the old derivation was fatal:** If `agentSecret = Poseidon2(operatorSecret, modelHash)`, a passive AS that is also the operator knows both inputs and can compute `agentSecret` directly. Hybrid 1 collapses — the PRF key (`modelHash` or `operatorSecret`) is known to A, so the replacement with a uniform value is not justified. The `blindingSalt` derivation fixes this: the AS knows `credCommitment` (the PRF input) but not `blindingSalt` (the PRF key), so the PRF assumption applies.

### Reduction: Level 2 (Active AS)

**Theorem 2**: If Poseidon-PRF holds and PLONK is zero-knowledge, no PPT A_active wins IND-UNL-AS[2, T-none] with non-negligible advantage.

**Proof sketch**: A_active can adaptively choose credential fields for some agents before enrollment, but cannot choose `blindingSalt*` for the challenge agent (the MUST NOT rules enforce this). The Auth oracle for non-challenge agents on arbitrary scopes reveals `scopeNullifier_i = Poseidon2(scopeId, agentSecret_i)` for `i ≠ *`. These are PRF evaluations under keys `agentSecret_i` independent of `agentSecret*` (different `blindingSalt` values). Adaptive enrollment control lets A choose `credCommitment*`'s ingredients (modelHash, operator keys, permissions, expiry), but `credCommitment*` is the PRF input to the `agentSecret` derivation — the key `blindingSalt*` remains uniform and A-independent.

The hybrid argument from Level 1 applies unchanged. The Auth oracle answers are simulable because they involve different PRF keys. Adaptive credential field choice does not help because the PRF key (`blindingSalt*`) is independent of all credential fields.

Advantage bound: `Adv(A_active) ≤ 3 · Adv_PRF(λ) + q · negl_ZK(λ)` where q is the number of Auth queries.

### Reduction: Level 3 (Colluding AS+RS)

**Theorem 3**: If Poseidon-PRF holds and PLONK is zero-knowledge, no PPT A_collude wins IND-UNL-AS[3, T-none] with non-negligible advantage.

**Proof sketch**: A_collude additionally receives `scopeNullifier` values for the challenge agent a* on non-challenge scopes via colluding RSes. Let the colluding scopes be `{scopeId_c1, ..., scopeId_cm}` where each `scopeId_cj ∉ {scopeId_0, scopeId_1}`.

From these, A learns the values `Poseidon2(scopeId_cj, agentSecret*)` for `j = 1, ..., m`. These are m evaluations of the PRF keyed on `agentSecret*` at known, distinct points — none of which equals `scopeId_0` or `scopeId_1` (by the game restriction).

- **Hybrid 0→1**: Replace `agentSecret*` with uniform `s*`. The PRF assumption must now hold even given m evaluations at non-challenge points. This is precisely the standard multi-query PRF security definition: given `{(scopeId_cj, Poseidon2(scopeId_cj, k))}_{j=1}^{m}`, distinguish `Poseidon2(scopeId_0, k)` from random. By PRF security under polynomial queries, this step introduces advantage at most `Adv_PRF(λ)`.

- **Hybrids 1→4**: Identical to Level 1 (replace challenge nullifiers and auth tags with uniform values).

- **Hybrid 5**: A sees m colluding-scope nullifiers (now PRF evaluations under the replaced uniform key s*, still consistent with the PRF oracle) plus two uniform challenge outputs. The colluding-scope nullifiers are correctly simulated by evaluating the PRF on `s*` at the colluding scope IDs. A's view is independent of b.

Advantage bound: `Adv(A_collude) ≤ 3 · Adv_PRF(λ) + q · negl_ZK(λ)`.

The critical observation: colluding-scope nullifiers are PRF evaluations at points *distinct* from the challenge points. Standard PRF security guarantees that knowing polynomially many evaluations at non-challenge inputs does not help predict or distinguish evaluations at challenge inputs.

### Lifecycle-reduction binding

The five MUST NOT rules in §2 (blindingSalt lifecycle) are not operational niceties — they are necessary conditions for the uniformity assumption on `blindingSalt*` that underpins Hybrid 0→1 at every level. If any MUST NOT rule is violated:

- Rule 1 violated (salt transmitted to AS): A knows the PRF key; Hybrid 1 collapses.
- Rule 2 violated (salt derived from operatorSecret): A can reconstruct the PRF key; same collapse.
- Rule 3 violated (salt stored on AS-administered system): A can read the PRF key; same collapse.
- Rule 4 violated (salt in enrollment metadata): A extracts the PRF key from on-chain data.
- Rule 5 violated (salt logged): A reads the PRF key from logs.

Each violation reduces to: A learns `blindingSalt*`, the PRF key is no longer uniform from A's perspective, and the IND-UNL-AS advantage becomes 1.

### Collusion resistance summary

| Adversary level | What A sees for challenge agent | Why it doesn't help |
|---|---|---|
| L1 (Passive) | Enrollment log only | No scope nullifiers at all; challenge outputs are PRF values under unknown key |
| L2 (Active) | Enrollment log + Auth oracle for non-challenge agents | Non-challenge agents have independent PRF keys (`blindingSalt` values) |
| L3 (Collude) | Enrollment log + non-challenge scope nullifiers for a* + Auth oracle | Colluding-scope nullifiers are PRF evaluations at non-challenge points; standard multi-query PRF security separates them from challenge-point evaluations |

## 7. Concrete deployment scenario

### Stakeholder: State Employees' Credit Union (SECU), North Carolina

**Context**: SECU members use AI agents (financial advisors, payment bots) to interact with multiple resource servers — merchant payment processors, loan originators, and insurance providers. SECU operates the Authorization Server. Regulatory requirement: SECU-as-AS must not learn which merchants a member's agent transacts with (GLBA financial privacy, Reg P).

**Critical detail: operator = AS.** In this deployment, SECU is both the operator (it signs the agent credential with its EdDSA key) and the AS. SECU therefore knows `operatorSecret`. Under the old `agentSecret = Poseidon2(operatorSecret, modelHash)` derivation, SECU could compute every agent's `agentSecret` and therefore every `scopeNullifier` — cross-scope unlinkability would be zero. Under the `blindingSalt` derivation, SECU knows `credCommitment` (from enrollment) but not `blindingSalt` (generated on the member's device). SECU cannot compute `agentSecret` and therefore cannot derive any `scopeNullifier`. This is an L2 (Active AS) deployment: SECU controls enrollment adaptively but does not collude with merchants.

**Deployment**:

1. **Enrollment (one-time, AS-visible)**: SECU enrolls each member agent's credential commitment into the agent Merkle tree via the standard AgentPolicy circuit. SECU sees the enrollment but learns only that the agent has permission bits `[payments, loans, insurance]` with an expiry. The member's device generates `blindingSalt` via CSPRNG and stores it in the iOS Keychain / Android Keystore. This is the last time SECU observes the agent's authorization activity.

2. **Per-merchant authorization (AS-invisible)**: When the agent accesses Merchant RS-A (scope: `Poseidon("https://merchant-a.secu.org/")`), it computes `agentSecret = Poseidon2(credCommitment, blindingSalt)` locally, then generates a ScopeIsolatedAuth proof. The proof is sent directly to RS-A. SECU is not contacted.

3. **Cross-scope unlinkability (IND-UNL-AS[2, T-batch])**: If the same agent accesses Insurance RS-B (scope: `Poseidon("https://insurance-b.secu.org/")`), the `scopeNullifier_B` is cryptographically independent of `scopeNullifier_A`. By Theorem 2, even with adaptive enrollment control, SECU's advantage in linking the two is bounded by `3 · Adv_PRF(λ) + negl_ZK(λ)`.

4. **Replay detection within scope**: Merchant RS-A stores seen `scopeNullifier` values. If the same agent presents twice to RS-A, the nullifier matches and RS-A can enforce rate-limiting or replay rejection. This is scope-local and does not leak to other RS instances.

5. **Timing side-channel mitigation**: The agent batches proof generation for all scopes within a configurable window (default: 500ms jitter + random delay ∈ [0, 2s]). Under the T-batch model, SECU observing network metadata sees a single burst, not individual per-RS proof generation events.

### Healthcare variant

**Stakeholder**: Kaiser Permanente (credential issuer for healthcare agents)

An agent delegated by a patient accesses Provider-A (primary care) and Provider-B (specialist referral). Kaiser-as-AS must not learn the referral network. This is an L3 (Colluding) deployment if Kaiser colludes with Provider-A: Kaiser receives `scopeNullifier` for Provider-A's scope but cannot derive Provider-B's nullifier (Theorem 3 — the colluding-scope nullifier is a PRF evaluation at a non-challenge point). The patient's device holds `blindingSalt`; Kaiser never sees it.

## 8. Why the baseline cannot match

The baseline's fundamental architectural constraint is that **every authorization requires AS participation at issuance time**. This is not a limitation of specific specs — it is structural to the OAuth 2.0 grant model:

| Property | Baseline (OIDC PPID + RFC 8707 + DPoP + BBS+) | ScopeIsolatedAuth |
|----------|------------------------------------------------|-------------------|
| AS sees which RS is accessed | **Yes** — AS signs audience-bound tokens per RS | **No** — agent generates proofs locally post-enrollment |
| AS can correlate cross-RS access | **Yes** — issuance log contains (agent_id, RS, timestamp) tuples | **No** — AS sees only the one-time enrollment event |
| Colluding AS+RS can de-anonymize | **Yes** — AS can reverse-map PPID; DPoP proofs are AS-visible | **No** — colluding AS+RS sees enrollment + one scope's nullifier; cannot derive other scopes' nullifiers (Poseidon-PRF, Theorem 3) |
| Formal IND-UNL-AS security | **No** — no spec defines this game; PPID is prose-level | **Yes** — three-level game with explicit reductions (§3–§4) |
| Passive/Active/Colluding distinction | **Not modeled** — baseline assumes AS trustworthiness | **Formally separated** — Theorems 1/2/3 cover each level |
| Nullifier separation per scope | **Impossible** — no OAuth primitive supports scope-bound unlinkable identifiers | **Native** — `Poseidon2(scopeId, agentSecret)` per scope |
| Operator = AS resilience | **N/A** — operator IS the AS by design | **Yes** — `blindingSalt` ensures `agentSecret` is AS-independent even when operator = AS |
| Side-channel treatment | **Silent** — no spec addresses timing correlation | **Parameterized** — T-none/T-batch/T-real with explicit scope of formal claim |
| Proof of scope separation to RS | **None** — RS trusts AS policy claim | **Cryptographic** — PLONK proof that required bits are satisfied without revealing full bitmask |

The core impossibility: in OAuth/OIDC, the AS is the token factory. Every token carries an audience claim that the AS computed and signed. Removing the AS from the per-scope authorization path is not possible without abandoning the OAuth grant model entirely. PPID hides the subject from RS-to-RS correlation, but the AS itself is the PPID issuer — it can trivially reverse any PPID it generated.

BBS+ addresses holder-to-verifier unlinkability but does not remove the issuer from the authorization path. The issuer (AS) still signs the credential containing the scope. BBS+ selective disclosure hides attributes from verifiers, not from issuers.

ScopeIsolatedAuth breaks this structural dependency: the AS enrolls the credential once (broad permission bitmask in Merkle tree), and all subsequent per-scope authorizations are computed locally by the agent using a ZK proof of Merkle membership and scope satisfaction. The AS is architecturally excluded from the per-scope authorization path, making cross-scope correlation cryptographically infeasible rather than policy-dependent — now formalized across three adversary power levels with explicit reductions to Poseidon-PRF security.
