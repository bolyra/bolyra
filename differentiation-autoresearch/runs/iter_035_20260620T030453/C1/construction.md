# Construction

## 1. Statement of claim

An AI agent proves to a resource server (RS) that its 64-bit permission bitmask satisfies an RS-specified required-scope mask predicate — i.e., `permissionBitmask & requiredMask == requiredMask` — without revealing any bits of the permission bitmask beyond what the predicate logically implies. The proof is:

- **AS-blind**: generated by the agent alone from a previously-issued credential commitment anchored in the on-chain Merkle tree; no Authorization Server roundtrip or cooperation is needed at presentation time.
- **Constant-size**: exactly 8 group elements (Groth16) or ~1 KB regardless of bitmask width or number of permission tiers.
- **Adversarial-AS-resilient**: the RS verifies the proof against the on-chain agent Merkle root, not against any AS attestation. A compromised AS that lies about the agent's scope cannot forge a valid proof because it lacks the agent's EdDSA private key and Merkle witness.
- **Runtime-adaptive**: the RS supplies `requiredMask` as a public input at verification time; the agent evaluates the bitwise predicate inside the circuit at proof-generation time. No token re-issuance or AS policy-table lookup is needed when the RS changes its required mask.

No composition of RFC 7662, jwt-introspection-response, RFC 8693, RFC 8707, DPoP, and BBS+ can simultaneously achieve all four properties. BBS+ achieves holder-driven selective disclosure over discrete claims but cannot evaluate bitwise AND with cumulative-bit implication closure over a bitmask. RFC 7662 variants require AS cooperation and trust.

## 2. Construction (gadgets, circuits, public/private inputs)

### Circuit: `SelectiveScopeProof`

This is a variant of the existing `AgentPolicy` circuit, stripped of the session-nonce nullifier (which is handshake-specific) and re-targeted for standalone RS presentation. It can also be understood as a "presentation mode" of the AgentPolicy circuit.

**Private inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `modelHash` | field | Poseidon hash of model identifier |
| `operatorPubkeyAx` | field | Operator EdDSA public key x-coordinate |
| `operatorPubkeyAy` | field | Operator EdDSA public key y-coordinate |
| `permissionBitmask` | uint64 | Full 64-bit permission bitfield |
| `expiryTimestamp` | uint64 | Credential expiration (Unix epoch) |
| `sigR8x`, `sigR8y`, `sigS` | field×3 | Operator EdDSA signature over credential commitment |
| `merkleProofLength` | uint8 | Actual depth of enrollment Merkle proof |
| `merkleProofIndex` | uint20 | Leaf index in enrollment tree |
| `merkleProofSiblings[20]` | field[20] | Enrollment sibling hashes padded to MAX_DEPTH=20 |
| `blindingNonce` | field | Fresh random value for presentation unlinkability |
| `revocationPathBits[20]` | bit[20] | Path in the sparse Merkle revocation tree derived from the credential commitment |
| `revocationSiblings[20]` | field[20] | Sibling hashes in the sparse Merkle revocation tree |

**Public inputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `requiredScopeMask` | uint64 | RS-specified required permission bits |
| `currentTimestamp` | uint64 | Verifier-supplied current time |
| `resourceIdentifier` | field | Poseidon hash of RS URI (audience binding) |
| `revocationTreeRoot` | field | Current root of the on-chain sparse Merkle revocation tree |

**Public outputs:**

| Signal | Type | Description |
|--------|------|-------------|
| `agentMerkleRoot` | field | Computed enrollment Merkle root (verified against on-chain root history buffer) |
| `presentationTag` | field | `Poseidon3(credentialCommitment, resourceIdentifier, blindingNonce)` — RS-specific, unlinkable across RSes |

### Gadgets and constraints

1. **Range checks** (standard):
   - `Num2Bits(64)` on `permissionBitmask`, `expiryTimestamp`, `currentTimestamp`

2. **Credential commitment** (Poseidon5):
   ```
   credentialCommitment = Poseidon5(modelHash, operatorPubkeyAx, operatorPubkeyAy, permissionBitmask, expiryTimestamp)
   ```

3. **EdDSA signature verification** (EdDSAPoseidonVerifier):
   ```
   EdDSAPoseidonVerifier(credentialCommitment, operatorPubkeyAx, operatorPubkeyAy, sigR8x, sigR8y, sigS)
   ```
   Proves the credential was signed by the operator's private key. The RS never learns the operator key — only that a valid signature exists.

4. **Merkle membership** (BinaryMerkleRoot):
   ```
   agentMerkleRoot = BinaryMerkleRoot(20, credentialCommitment, merkleProofIndex, merkleProofSiblings, merkleProofLength)
   ```
   Proves the credential commitment is enrolled on-chain. The RS checks `agentMerkleRoot` against the on-chain root history buffer.

5. **Selective scope predicate** (64 R1CS constraints):
   ```
   permBits = Num2Bits(64)(permissionBitmask)
   reqBits  = Num2Bits(64)(requiredScopeMask)
   for i in 0..63:
       reqBits[i] * (1 - permBits[i]) === 0
   ```
   This enforces `requiredScopeMask & permissionBitmask == requiredScopeMask` without revealing any bit of `permissionBitmask` that is not logically implied by the predicate.

6. **Cumulative bit encoding** (3 R1CS constraints):
   ```
   permBits[4] * (1 - permBits[3]) === 0
   permBits[4] * (1 - permBits[2]) === 0
   permBits[3] * (1 - permBits[2]) === 0
   ```
   Enforces the implication hierarchy (FINANCIAL_UNLIMITED → FINANCIAL_MEDIUM → FINANCIAL_SMALL).

7. **Expiry check** (LessThan(64)):
   ```
   currentTimestamp < expiryTimestamp
   ```

8. **Revocation non-membership** (SparseMerkleNonMembership(20)):
   ```
   revocationKey = Poseidon2(credentialCommitment, 0)   // domain-separated revocation key
   computedRevRoot = SparseMerkleRoot(20, revocationKey, revocationPathBits, revocationSiblings)
   computedRevRoot === revocationTreeRoot
   leafAtPath = SparseMerkleLeaf(20, revocationPathBits, revocationSiblings)
   leafAtPath === 0   // non-membership: the leaf slot is empty
   ```
   The sparse Merkle tree (SMT) has depth 20, matching the enrollment tree. Each leaf is either `0` (empty — credential not revoked) or `1` (credential revoked). The path through the SMT is derived by truncating `Poseidon2(credentialCommitment, 0)` to 20 bits. The circuit proves that the leaf at the credential's path is `0` and that the computed SMT root matches the public input `revocationTreeRoot`.

   **SMT structure:** The tree is keyed by `Poseidon2(credentialCommitment, 0)` truncated to 20 bits. A depth-20 SMT supports up to 2^20 ≈ 1M distinct revocation slots. Collision probability under Poseidon is negligible for realistic credential populations (< 100K agents). If two credential commitments hash to the same 20-bit path, the on-chain contract detects the collision at insertion time and rejects the second revocation — the operator must re-enroll the agent with a fresh credential commitment. This is operationally equivalent to the enrollment tree's depth-20 capacity bound.

   **Non-membership gadget detail:** The gadget computes the SMT root from the leaf value (constrained to be `0`) up through 20 levels of Poseidon2 hashing, selecting left/right sibling at each level via `revocationPathBits[i]`. This is structurally identical to the enrollment `BinaryMerkleRoot` gadget but with the leaf value constrained to zero rather than to a credential commitment.

9. **Presentation tag** (Poseidon3):
   ```
   presentationTag = Poseidon3(credentialCommitment, resourceIdentifier, blindingNonce)
   ```
   The `blindingNonce` is freshly sampled per presentation, making the tag unlinkable across RSes. The `resourceIdentifier` audience-binds the proof. The `credentialCommitment` ensures the tag is identity-bound (the same agent presenting to the same RS with a different blinding nonce produces a different tag, preventing correlation).

**Access control:** Only the credential's operator (verified via EdDSA signature over the revocation message) or the registry admin can call `revokeCredential`. This prevents a compromised AS from revoking credentials it didn't issue — the operator key is the trust anchor, consistent with the adversarial-AS model.

**Revocation latency SLA:** Revocation takes effect within **1 block** of the `revokeCredential` transaction being mined (~2 seconds on Base, ~12 seconds on Ethereum L1). The 30-entry root history buffer defines the maximum staleness window: an agent whose credential was revoked can continue presenting proofs against older roots in the buffer. On Base with 2-second blocks, the buffer spans ~60 seconds. On Ethereum L1, ~6 minutes. The RS MAY enforce a tighter staleness bound by rejecting roots older than a configurable threshold (e.g., only the 5 most recent roots, bounding staleness to ~10 seconds on Base).

**Revocation finality guarantee:** Once a credential commitment is inserted into the SMT (leaf set to `1`), the non-membership proof for that credential becomes infeasible — no valid witness exists that makes the leaf equal to `0` at that path while producing the updated root. This is unconditional on Poseidon collision resistance: forging a non-membership proof for a revoked credential requires finding a second preimage for the SMT path hash.

### Verification (on-chain or off-chain)

The RS calls the Groth16 verifier with public signals `[agentMerkleRoot, presentationTag, requiredScopeMask, currentTimestamp, resourceIdentifier, revocationTreeRoot]` and checks:

1. `agentMerkleRoot` is in the on-chain enrollment root history buffer (30-entry circular buffer).
2. `revocationTreeRoot` is in the on-chain revocation root history buffer (30-entry circular buffer).
3. The Groth16 proof verifies.
4. `requiredScopeMask` matches the RS's own policy.
5. `currentTimestamp` is within an acceptable window of the RS's clock.

No AS is contacted. No token introspection occurs. Revocation status is cryptographically proven inside the ZKP, not queried from an external service.

## 3. Threat model (adversary capabilities, game definition)

### Adversary model

The adversary **A** controls:

- The Authorization Server (AS) — can issue, revoke, or lie about credentials
- Network communication between agent and RS
- Up to `n-1` colluding RSes (trying to link presentations)
- The `requiredScopeMask` public input (chosen adaptively)

The adversary **does not** control:

- The agent's EdDSA private key (operator key)
- The on-chain Merkle tree state (assumed to reflect ground truth via smart contract access control)
- The on-chain revocation SMT state (same trust assumption as enrollment tree — smart contract access control)
- The BN128 pairing or Baby Jubjub discrete log problems

### Revocation-specific adversary capabilities

The adversarial-AS model introduces a subtlety for revocation: if the AS is adversarial, who can revoke? The construction resolves this cleanly:

- **Operator-initiated revocation:** The operator signs a revocation message with the same EdDSA key that signed the credential. The smart contract verifies this signature before updating the SMT. An adversarial AS cannot forge operator signatures (Baby Jubjub DLP hardness).
- **Self-revocation:** The agent itself can revoke its own credential by proving knowledge of the credential's private inputs (modelHash, operator key) in a separate on-chain transaction. This is a safety valve: if the operator is unreachable, the agent can still revoke.
- **AS-initiated revocation is explicitly excluded:** A compromised AS cannot unilaterally revoke honest agents' credentials. This is a deliberate design choice — revocation authority follows the same trust anchor (operator EdDSA key) as credential issuance.

### Game: Selective Scope Soundness (SSS)

```
Game SSS(λ):
  1. Challenger runs Setup(1^λ) → (pk, vk) for SelectiveScopeProof circuit.
  2. Challenger enrolls agents {A_1, ..., A_n} with credential commitments {cc_1, ..., cc_n}
     into the on-chain Merkle tree, producing root R.
     Challenger initializes the revocation SMT with revoked set V ⊆ {cc_1, ..., cc_n},
     producing revocation root R_rev.
  3. A is given vk, R, R_rev, and the enrolled credential commitments (but NOT the EdDSA
     private keys or Merkle witnesses of honest agents).
  4. A chooses requiredScopeMask*, currentTimestamp*, resourceIdentifier* adaptively.
  5. A outputs a proof π* and public signals (R*, tag*, requiredScopeMask*, currentTimestamp*,
     resourceIdentifier*, R_rev*).
  6. A wins if:
     a. The Groth16 verifier accepts π* under vk, AND
     b. R* is a valid root in the enrollment root history buffer, AND
     c. R_rev* is a valid root in the revocation root history buffer, AND
     d. There is NO enrolled, unrevoked agent whose permissionBitmask satisfies
        (permissionBitmask & requiredScopeMask* == requiredScopeMask*) AND
        whose credential is unexpired at currentTimestamp*.
        ("Unrevoked" means cc ∉ V at the time R_rev* was the revocation root.)
```

**Claim**: Under the knowledge soundness of Groth16, collision resistance of Poseidon, and hardness of the discrete logarithm problem on Baby Jubjub, Pr[A wins Game SSS] ≤ negl(λ).

### Game: Revocation Completeness (RC)

```
Game RC(λ):
  1. Setup as in SSS.
  2. Challenger revokes agent A_j by inserting cc_j into the revocation SMT,
     producing new root R_rev'.
  3. A is given the revoked agent A_j's full private state (EdDSA key, Merkle witness,
     permissionBitmask — everything).
  4. A outputs a proof π* with R_rev* = R_rev' (the post-revocation root).
  5. A wins if the Groth16 verifier accepts π*.
```

**Claim**: Under Poseidon collision resistance, Pr[A wins Game RC] ≤ negl(λ). The revoked credential's SMT leaf is `1`; no witness can make it `0` while producing root R_rev'.

### Game: Presentation Unlinkability (PU)

```
Game PU(λ):
  1. Setup as above.
  2. A chooses two RSes (resourceIdentifier_0, resourceIdentifier_1) and two requiredScopeMasks.
  3. Challenger generates presentations (proof_0, tag_0) and (proof_1, tag_1)
     for the SAME agent to both RSes, using fresh blindingNonces.
  4. A receives (proof_0, tag_0, proof_1, tag_1) and must output b ∈ {0, 1}
     indicating whether proof_0 and proof_1 came from the same agent.
  5. A wins if it correctly identifies the linkage with probability > 1/2 + negl(λ).
```

**Claim**: Under the zero-knowledge property of Groth16 and Poseidon PRF assumption, Pr[A wins Game PU] ≤ 1/2 + negl(λ).

**Note on revocation and unlinkability:** The revocation non-membership proof does not leak the credential commitment to the verifier. The SMT path is derived from `Poseidon2(credentialCommitment, 0)` which is a private input; the verifier sees only the `revocationTreeRoot` (public input shared across all agents) and the `presentationTag` (blinded). Two presentations from the same agent against the same revocation root do not reveal a common SMT path — the path bits and siblings are private witnesses, hidden by Groth16 zero-knowledge.

## 4. Security argument (named assumption + reduction sketch)

### Assumptions

1. **Knowledge soundness of Groth16** (in the algebraic group model + random oracle model): Given a valid proof, there exists an extractor that recovers the witness with overwhelming probability.
2. **Collision resistance of Poseidon** over BN254 scalar field: Finding `(x, x')` with `x ≠ x'` and `Poseidon(x) = Poseidon(x')` is computationally infeasible.
3. **Discrete logarithm hardness on Baby Jubjub**: Given `(Ax, Ay) = s * G`, recovering `s` is infeasible.
4. **Poseidon PRF assumption**: `Poseidon(key, ·)` is indistinguishable from a random function when the key is secret.

### Reduction sketch for SSS (with revocation)

Suppose adversary A wins Game SSS with non-negligible probability ε. We construct an extractor E:

1. By Groth16 knowledge soundness, E extracts a witness `w = (modelHash, opPkAx, opPkAy, permissionBitmask, expiryTimestamp, sig, merkleProof, blindingNonce, revocationPathBits, revocationSiblings)` from A's proof π*.
2. From the extracted witness, E computes `credentialCommitment = Poseidon5(modelHash, opPkAx, opPkAy, permissionBitmask, expiryTimestamp)`.
3. The Merkle membership constraint forces `credentialCommitment` to be a leaf under root R*. By Poseidon collision resistance, this leaf must be one of the enrolled `{cc_1, ..., cc_n}`.
4. The EdDSA verification constraint forces the signature to be valid under `(opPkAx, opPkAy)`. By discrete log hardness on Baby Jubjub, A could only produce this signature if it held the corresponding EdDSA private key — contradicting the assumption that A does not hold honest agents' keys.
5. The scope predicate constraint forces `permissionBitmask & requiredScopeMask* == requiredScopeMask*` inside the circuit. By soundness of the R1CS, the extracted `permissionBitmask` satisfies this predicate.
6. The expiry constraint forces `currentTimestamp* < expiryTimestamp` inside the circuit.
7. **The revocation non-membership constraint forces the SMT leaf at the credential's path to be `0` under root R_rev\*.** By Poseidon collision resistance, the extracted `revocationPathBits` and `revocationSiblings` must correspond to an authentic empty leaf in the SMT at root R_rev*. Since R_rev* is in the revocation root history buffer, this means the credential was not revoked at that SMT snapshot. If the credential was revoked (leaf = `1`), no valid witness exists — the circuit constrains the leaf to `0`, and Poseidon collision resistance prevents finding an alternative path that maps to an empty leaf while producing the same root.

Therefore, the extracted credential is enrolled, operator-signed, scope-satisfying, unexpired, **and unrevoked** — contradicting A's win condition (6d). So ε ≤ negl(λ).

### Reduction sketch for RC (revocation completeness)

Suppose adversary A wins Game RC: it produces a valid proof for a revoked credential against the post-revocation root R_rev'. By Groth16 knowledge soundness, extract the witness. The non-membership gadget constrains the SMT leaf to `0`. But the revocation inserted leaf = `1` at the credential's path in the SMT producing R_rev'. For the extracted witness to satisfy both `computedRevRoot = R_rev'` and `leafAtPath = 0`, the adversary must find sibling values that hash to R_rev' with a `0` leaf at the credential's path — but the honest SMT has a `1` leaf at that path. This requires a second preimage for at least one Poseidon2 hash in the SMT path, contradicting Poseidon collision resistance.

### Reduction sketch for PU

The presentation tag is `Poseidon3(credentialCommitment, resourceIdentifier, blindingNonce)` where `blindingNonce` is freshly random per presentation. By the Poseidon PRF assumption (keyed by `blindingNonce`), each tag is pseudorandom. The Groth16 proof itself reveals nothing about private inputs by the zero-knowledge property — this includes the revocation path bits and siblings, which are private witnesses. Therefore, two presentations from the same agent are computationally indistinguishable from two presentations from different agents.

The `SelectiveScopeProof` circuit reuses the same gadgets as `AgentPolicy` (EdDSAPoseidonVerifier, BinaryMerkleRoot, Num2Bits, LessThan) and adds the presentation tag (one Poseidon3 call), the revocation key derivation (one Poseidon2 call), and the SMT non-membership verification (structurally identical to BinaryMerkleRoot with a zero-constrained leaf). No new cryptographic primitives are introduced — all components use Poseidon2 hashing over BN254, the same primitive used throughout the Bolyra circuit stack.

- Groth16 (snarkjs, browser/Node): ~12–18 seconds
- Groth16 (rapidsnark, native): ~1.2–2.0 seconds

## 7. Concrete deployment scenario

**Stakeholder:** Navy Federal Credit Union (NFCU), the largest credit union in the United States with 13M+ members.

**Scenario:** NFCU deploys an AI agent platform where member-authorized agents perform financial operations — balance inquiries, bill payments, loan applications, investment trades. Each agent holds a Bolyra credential with a 64-bit permission bitmask encoding NFCU's internal permission taxonomy (extending Bolyra's 8-bit base to include domain-specific bits for loan origination, wire transfers, account closure, PII access, etc.).

**Problem without Bolyra:** NFCU's current OAuth2 infrastructure uses RFC 7662 introspection. When a member's agent contacts the bill-payment RS, the RS calls the AS to check scope. The AS returns the agent's full scope string: `"read write financial_small financial_medium bill_pay"`. The RS now knows the agent also holds `financial_medium` even though it only needed `bill_pay`. A compromised RS can use this information to craft targeted phishing or social engineering attacks against the member. Worse: NFCU's CISO cannot guarantee that the AS (operated by a third-party identity vendor) won't be compromised — the AS is a single point of trust.

**Revocation scenario:** An NFCU member reports their agent credentials compromised. The member's operator (NFCU's agent platform) signs a revocation transaction with the same EdDSA key that signed the credential. Within one Base block (~2 seconds), the credential commitment is inserted into the revocation SMT. Any subsequent `SelectiveScopeProof` generated by the compromised agent will fail: the non-membership gadget cannot produce a valid witness because the SMT leaf at the credential's path is now `1`. The RS does not need to contact any revocation endpoint — it simply verifies the proof against the current `revocationTreeRoot` from the on-chain history buffer.

**Revocation SLA:** NFCU's security policy requires credential revocation to take effect within 60 seconds. The construction delivers:
- **Revocation insertion latency:** 1 block on Base (~2 seconds) from the `revokeCredential` transaction being submitted to the SMT root updating.
- **Staleness window:** The 30-entry revocation root history buffer on Base spans ~60 seconds (30 × 2s blocks). An agent presenting a proof against a pre-revocation root in the buffer could theoretically use the credential for up to 60 seconds after revocation. NFCU configures the RS to accept only the 5 most recent revocation roots, narrowing the window to ~10 seconds — well within the 60-second SLA.
- **Comparison to RFC 7662:** OAuth token revocation via RFC 7009 requires the RS to query the AS for revocation status (or wait for a short-lived token to expire). CRL/OCSP-style revocation lists introduce similar latency. The Bolyra construction makes revocation status a cryptographic proof input, not a network query — eliminating the AS from the revocation hot path.

**With Bolyra SelectiveScopeProof (full flow including revocation):**

1. At onboarding, NFCU enrolls the agent's credential commitment into the on-chain Merkle tree. The credential includes `permissionBitmask = 0b00001111` (READ_DATA + WRITE_DATA + FINANCIAL_SMALL + FINANCIAL_MEDIUM) and the operator's EdDSA signature.

2. When the agent contacts the bill-payment RS, the RS publishes `requiredScopeMask = 0b00000100` (FINANCIAL_SMALL), `resourceIdentifier = Poseidon("billpay.nfcu.org")`, and the current `revocationTreeRoot` from the on-chain registry.

3. The agent generates a `SelectiveScopeProof` locally (no AS contact) proving:
   - It holds a credential enrolled in NFCU's Merkle tree
   - Its permission bitmask satisfies `bitmask & 0b00000100 == 0b00000100`
   - The credential is operator-signed and unexpired
   - The credential is not revoked (SMT non-membership against `revocationTreeRoot`)
   - The presentation tag is bound to the bill-payment RS

4. The RS verifies the Groth16 proof against the on-chain enrollment root and revocation root. It learns only that the agent has at least FINANCIAL_SMALL permission and that the credential is not revoked. It does not learn whether the agent also has FINANCIAL_MEDIUM, SIGN_ON_BEHALF, ACCESS_PII, or any other capability.

5. A compromised AS cannot forge proofs because it lacks the operator's EdDSA private key. A compromised AS cannot revoke honest agents' credentials because revocation requires an operator EdDSA signature. A colluding set of RSes cannot link the agent's presentations because each uses a fresh `blindingNonce`.

**Regulatory context:** NCUA examiners reviewing NFCU's agent authorization infrastructure can verify that the on-chain Merkle tree is append-only, that the revocation SMT is operator-controlled (not AS-controlled), and that proof verification is deterministic — no AS policy-table interpretation or revocation endpoint availability is required.

## 8. Why the baseline cannot match

The baseline (RFC 7662 + jwt-introspection-response + RFC 8707 + DPoP + BBS+) fails on four properties that the `SelectiveScopeProof` achieves simultaneously:

| Property | SelectiveScopeProof | Best baseline attempt | Why baseline fails |
|----------|--------------------|-----------------------|-------------------|
| **AS-blind presentation** | Agent generates proof locally from on-chain credential. No AS roundtrip. | BBS+ allows holder-driven disclosure, but the AS issued and signed the BBS+ credential. The AS was present at issuance and can refuse to issue, revoke, or issue a credential with wrong claims. The agent cannot present anything the AS didn't explicitly sign. | BBS+ selective disclosure is *structurally* AS-dependent: the credential's claim set is fixed at AS issuance. The agent selects *which claims to show*, but cannot evaluate a predicate the AS didn't pre-compute. Bolyra's agent holds the raw bitmask and operator signature; the circuit evaluates any predicate at runtime without AS involvement. |
| **Runtime-adaptive bitwise predicate** | RS supplies `requiredScopeMask` as a public input. The circuit evaluates `bitmask & mask == mask` with cumulative-bit implication closure. The same credential works for any mask without re-issuance. | BBS+ supports equality and range predicates over hidden attributes. Bitwise AND with implication closure requires circuit-level evaluation. No BBS+ extension in draft-irtf-cfrg-bbs-signatures covers this. The AS would need to pre-issue a separate credential for every possible mask value — combinatorial explosion for 64 bits. | The predicate `bitmask & mask == mask` with cumulative implication rules is a *non-linear Boolean function* over a multi-bit field. BBS+ predicates are limited to linear relations and equality over individual messages. There is no BBS+ gadget for bitwise AND, and no way to express "bit 4 implies bits 3 and 2" without circuit constraints. |
| **Adversarial-AS resilience** | Soundness depends on the on-chain Merkle root (smart contract state) and operator EdDSA signature (agent-held key). A compromised AS cannot forge proofs or unilaterally revoke honest credentials — revocation requires operator EdDSA signature. | RFC 7662: RS trusts the AS's introspection response. BBS+: RS trusts the AS's signature on the credential. In both cases, a malicious AS can lie about the agent's permissions or revoke access unilaterally. | The fundamental issue is *who signs and who revokes*. In the baseline, the AS controls both issuance and revocation. In Bolyra, the *operator* signs, revocation requires operator authorization, and the Merkle trees are maintained by a smart contract with its own access control. The AS is not in the trust path for any operation. |
| **Constant-size proof** | 256 bytes (Groth16: 3 group elements) regardless of bitmask width or revocation set size. The revocation non-membership proof adds zero bytes to the proof — it is absorbed into the same 3-element Groth16 output. | BBS+ derived proof: O(|undisclosed|) group elements. JWT introspection response: O(|scopes|) string length. CRL/OCSP revocation checks add a separate network round-trip and response payload. | Groth16 proof size is determined by the pairing-based verification equation (constant: 2 G1 + 1 G2 elements), not by the witness size. The revocation status, scope predicate, Merkle membership, and EdDSA signature are all compressed into the same 256-byte proof. No baseline construction achieves this — revocation checking always adds either a network call or a separate proof artifact. |

**The irreducible gap (updated for revocation):** The baseline's revocation mechanisms (RFC 7009, CRL, OCSP, short-lived tokens) all require either AS cooperation or a separate revocation-status channel. Bolyra's `SelectiveScopeProof` folds revocation non-membership into the same constant-size ZKP that proves scope satisfaction — the RS verifies a single 256-byte proof and learns simultaneously that the credential is enrolled, unrevoked, scope-satisfying, and unexpired. The baseline cannot compose these properties into a single artifact because each property is handled by a different specification with a different trust model, verification endpoint, and proof format.
