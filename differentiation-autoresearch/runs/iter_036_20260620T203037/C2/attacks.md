### Attack 1: IND-UNL-AS Game §3 Contradicts the On-Chain Deployment Model

**Attack:**
The IND-UNL-AS game states in step 3 that the adversary "sees only that *some* proof was submitted for `scopeId_B`, but not the public signals, since the honest RS does not collude." The game then uses this limited view to argue that the adversary cannot observe `scopePseudonym_B = Poseidon2(scopeId_B, sbs_b)` or `scopeBoundDelegationAnchor_B = Poseidon4(scopeId_B, sbs_b, permBitmask_b, cc_b)` at the honest RS.

But the construction's own §7 (Concrete Deployment Scenario) describes proofs being submitted to an on-chain Solidity verifier (§5 primitive mapping cites on-chain `lastScopeBoundAnchor` mapping, on-chain Merkle tree, ~220k–300k gas per verification). On a public blockchain, **all public inputs and outputs of every submitted proof are globally observable.** The adversary—who controls the AS and the network—trivially reads all on-chain calldata. There is no mechanism in the construction preventing the adversary from observing `scopePseudonym_B` and `scopeBoundDelegationAnchor_B`.

**Why it matters:**
If the adversary sees both `(scopePseudonym_A, scopeBoundDelegationAnchor_A)` at the colluding RS **and** `(scopePseudonym_B, scopeBoundDelegationAnchor_B)` at the honest RS (via on-chain observation), the reduction in §4 is attempting a strictly harder problem than modeled. The adversary now holds:

```
scopePseudonym_A    = Poseidon2(scopeId_A, sbs_b)
scopePseudonym_B    = Poseidon2(scopeId_B, sbs_b)
scopeAnchor_A       = Poseidon4(scopeId_A, sbs_b, permBitmask_b, cc_b)
scopeAnchor_B       = Poseidon4(scopeId_B, sbs_b, permBitmask_b, cc_b)
```

for known `(scopeId_A, scopeId_B, permBitmask_b, cc_b)`. The PRF argument still holds for each individual value (each looks pseudorandom under unknown key `sbs_b`), but the game as written explicitly assumed the adversary does NOT see honest-RS signals. The reduction sketch in §4 was constructed under this assumption. Giving the adversary all four values does not obviously break POS-PRF (it reduces to distinguishing two correlated PRF evaluations from independent random values under the same key at two inputs), but **the construction never proves this stronger statement.** The game needs to be re-stated with the on-chain adversary, and the reduction must be redone against the stronger view.

**Concretely:** the construction silently assumes an off-chain verification model for honest RSes while describing an on-chain deployment. These two assumptions are mutually exclusive without an explicit argument for why on-chain public signals don't help the adversary.

**In-threat-model?** No — the construction must either (a) restrict the deployment to off-chain verification at honest RSes and explain the trust model, or (b) re-prove the IND-UNL-AS game with the full on-chain adversary view, explicitly handling the case where the adversary sees all four PRF outputs simultaneously.

### Attack 3: Correlated PRF Key Across Poseidon2 and Poseidon4 — The ROM Independence Claim Is Unproven

**Attack:**
The joint leakage hybrid argument in §4.3 requires replacing `scopePseudonym_A = Poseidon2(scopeId_A, sbs_b)` and `scopeBoundDelegationAnchor_A = Poseidon4(scopeId_A, sbs_b, permBitmask_b, cc_b)` with independent random values in two successive hybrids. The construction acknowledges (in the "Note on correlated key usage") that this requires modeling Poseidon2 and Poseidon4 as **independent** random oracles, and defers to "(a) modeling Poseidon in the random oracle model, where evaluations of different arities are independent random oracles."

This is not a theorem. It is an additional unproven modeling assumption. In Poseidon's specification, Poseidon-2 (t=3: 1 capacity + 2 rate elements) and Poseidon-4 (t=5: 1 capacity + 4 rate elements) are **the same sponge permutation** applied to state vectors of different widths, or different permutation instantiations using the same round constant generation process. The construction does not specify which Poseidon variant is being used across arities. If both use a width-t permutation with `t = max_inputs + 1`, and the first two inputs are identical (`scopeId_A, sbs_b`), then Poseidon4's first two rounds of its wider state encode the same values as Poseidon2's entire input. The diffusion properties of MDS matrices do not guarantee independence.

**Concrete distinguisher sketch:**
Let `H_2(x, k) = Poseidon2(x, k)` and `H_4(x, k, a, b) = Poseidon4(x, k, a, b)`.

Suppose there exists a non-negligible algebraic relationship `f(H_2(x, k), H_4(x, k, a, b)) = 0` holding for all `x, k, a, b` over the BN254 field (or even with non-negligible probability over the round constants). Since Poseidon uses sparse-but-fixed MDS matrices and alternating full/partial round structures, such algebraic leakage is not excluded by the construction's security argument — it is simply assumed away. Finding such an `f` is an open problem, but the reduction does not prove its non-existence; it assumes independence.

**Why this matters for the construction's claim:**
In Hybrid 0→1, the argument replaces `Poseidon2(scopeId_A, sbs_b)` with a random value `r_1`. But the adversary still sees `Poseidon4(scopeId_A, sbs_b, permBitmask_b, cc_b)` in Hybrid 1. If Poseidon2 and Poseidon4 are NOT independent random oracles over the same key `sbs_b`, then replacing `Poseidon2` with `r_1` while leaving `Poseidon4` unchanged is not a valid hybrid step — the adversary can potentially detect the inconsistency. The construction acknowledges this as needing "(a) ROM or (b) related-key PRF assumption" but provides no citation for related-key PRF security results for Poseidon, and "independent random oracles for different arities" is a non-standard extension of the ROM that has not been justified.

**In-threat-model?** Yes for standalone auth (single PRF evaluation per scope), but **No for the joint leakage argument** across `scopePseudonym` and `scopeBoundDelegationAnchor` at the same scope. The construction must either (a) use separate secrets for the two PRF evaluations (e.g., derive them from a KDF applied to `sbs_b`), or (b) provide a formal Poseidon-specific argument justifying independence across arities, or (c) explicitly state the related-key PRF assumption and justify it.

### Attack 4: `scopeBlindingSecret` Is Persistent — No Forward Secrecy or Post-Compromise Security

**Attack:**
Section §2 and §7 describe `scopeBlindingSecret` as "a per-agent persistent blinding scalar, generated once at agent enrollment and stored alongside the agent's credential material." It is the sole secret protecting cross-scope unlinkability. The construction provides no key rotation mechanism, no forward secrecy definition, and no post-compromise security (PCS) guarantee.

**Forward secrecy failure:** Suppose at time `T`, an adversary compromises the agent's local storage and recovers `sbs`. All `scopePseudonym` values ever published on-chain are of the form `Poseidon2(scopeId_X, sbs)` for publicly known `scopeId_X` values (domain hashes of RS endpoints). Since `scopeId_X` is public and `sbs` is now known, the adversary can compute `Poseidon2(scopeId_X, sbs)` for every RS that ever appeared on-chain and retroactively identify which RSes the agent visited, in what order, across its entire lifetime. In the credit-union scenario (§7), a compromised `sbs` reveals Alice's complete merchant transaction graph going back to enrollment.

For comparison: TLS 1.3 forward secrecy guarantees that compromising a long-term key does not compromise past session keys. The construction has no analogous guarantee — it uses a single long-term secret as the PRF key for all scopes, forever.

**PCS failure:** Even if the agent rotates its credential (re-enrolls with a new `modelHash` or new operator key), the `scopeBlindingSecret` is described as independent of the credential. The construction provides no protocol for rotating `sbs` without breaking existing `scopePseudonym` continuity at every RS where the agent has an established account.

**IND-UNL-AS game gap:** The game is defined at a single point in time. It does not address multi-epoch security where the adversary adaptively corrupts agents between sessions. A forward-secure variant of the game would require that compromise of `sbs` at time `T` gives no advantage over activity before `T`. The current reduction is a static reduction that says nothing about this.

**Key reuse across delegation hops (interaction with Attack 3):** In `ScopeBlindDelegation`, the delegator supplies their `delegatorBlindingSecret` as a private witness to satisfy the chain-linking constraint. This means the delegation protocol requires the delegator to use `sbs` as a private circuit input. Every party who receives a delegated credential from the delegator sees a proof that was computed using `sbs` (as a private witness). If the prover's local proving environment is compromised during a delegation, the adversary recovers `sbs` directly from the witness trace. There is no blinding or re-randomization step for the `sbs` input itself.

**Minimum bar the construction must clear:** Provide an explicit security definition for persistent-secret systems — either (a) a forward-secrecy game that shows the construction achieves some weaker variant, or (b) a claim that FS is explicitly out of scope with a clear statement of the attack surface, or (c) a key-rotation protocol for `sbs` that handles pseudonym migration at existing RSes. Asserting POS-PRF without bounding the adversary's ability to corrupt long-term secrets is not sufficient for a claim that targets "adversarial AS that actively attempts to correlate per-agent traffic graphs" — a motivated AS will attempt physical or software compromise of the agent's storage precisely because the cryptographic path is closed.

**In-threat-model?** No — the construction explicitly does not address `sbs` compromise in the threat model. The adversary capabilities in §3 state "The adversary does NOT control: the agent's local proving environment." This assumption is stated without justification and without a forward-secrecy theorem. For a deployed financial privacy system, it needs a formal scope statement: "FS is not claimed, and compromise of `sbs` retroactively de-anonymizes all past activity."

If the AS operates any delegatee, it recovers the blinding secret. Furthermore, the transmission channel itself is unspecified — the construction says "encrypted via the delegatee's public key" without specifying which public key, which authenticated channel, or which key infrastructure. RFC 8693 Token Exchange requires no transmission of any long-term secret between principals; the AS is the intermediary and signing keys never cross party boundaries.

- **Why it fails against the construction / why it works:** The construction explicitly excluded delegatee compromise from the threat model. This is a gap, not a feature. The privacy guarantee collapses to the weakest delegatee's operational security. The IND-UNL-AS game never models an adversary who coerces or operates delegatees.

- **In-threat-model?** **No — construction must address this.** Either (a) expand the threat model to explicitly bound the number of compromised delegatees and prove the game holds under that bound, or (b) redesign blinding secret delegation using a ratcheting scheme where each delegatee gets a *derived* secret (e.g., HKDF of `sbs || scopeId || delegateeCC`) that does not expose the root `sbs`, so a compromised delegatee reveals nothing about proofs at other RSes.

### Attack 2: `requiredScopeMask` is a Public Quasi-Identifier That Collapses Anonymity Set Size

- **Attack:** Every proof publishes `requiredScopeMask` as a public input. The AS knows the full permission bitmask of every enrolled agent — it issued those credentials. The AS can therefore compute, for any proof, the set `S = { agent_i : permBitmask_i ⊇ requiredScopeMask }`. This is the *effective anonymity set* of the proof. The IND-UNL-AS game in §3 requires that the adversary select `agent_0` and `agent_1` — implicitly both must satisfy the RS's required scope. In production deployments the permission space is not uniform: §5 (permissions model) defines 8 distinct bits including `FINANCIAL_UNLIMITED` (bit 4), `SIGN_ON_BEHALF` (bit 5), `SUB_DELEGATE` (bit 6), and `ACCESS_PII` (bit 7), all of which are rare grants. If only 3 agents out of 10,000 enrolled agents hold `ACCESS_PII | SIGN_ON_BEHALF`, any proof submitted to an RS requiring those bits reduces the anonymity set to 3. At `|S| = 3`, the adversary wins the linkability game with probability `≥ 1/3` by guessing — far above negligible. The §8 "Structural Impossibility" argument never addresses anonymity set size; it only addresses the AS's inability to evaluate the PRF. PRF security is irrelevant when the population of plausible provers is small enough to enumerate. By contrast, an RFC 7662 baseline AS can enforce k-anonymity at issuance time: refuse to issue a token to agent `i` for RS `j` unless at least `k` other agents have issued tokens for an RS in the same scope class. The ZK construction has no equivalent policy hook.

- **Why it works:** The unlinkability argument assumes a large anonymity set hiding the agent among all enrolled agents. But `requiredScopeMask` (public) combined with the AS's knowledge of `permBitmask_i` for every agent (issuer knowledge) performs set intersection that is entirely outside the ZK proof and independent of the PRF assumption.

- **In-threat-model?** **No — construction must address this.** The formal game in §3 must add a liveness/population condition: both `agent_0` and `agent_1` must be drawn from a set of size ≥ k for some security parameter k, AND the construction must specify the minimum anonymity set size below which the privacy claim degrades. Alternatively, show that the batch relayer withholds `requiredScopeMask` from the public transcript — but that requires the on-chain verifier not to log public inputs, which contradicts standard EVM verification contract behavior.

### Attack 3: The IND-UNL-AS Game Gives the Adversary Less Information Than a Public Blockchain Actually Provides — But Then the Reduction Is Incomplete

- **Attack:** §3 ("Challenge phase") states: *"Proof `π_B` for `agent_b` at `scopeId_B` (adversary sees only that some proof was submitted for `scopeId_B`, but not the public signals, since the honest RS does not collude)."* In a public EVM deployment (Base Sepolia, §7), every call to the on-chain verifier contract is a public transaction. The adversary can read the chain and see ALL public signals — `scopePseudonym_B`, `nonceBinding_B`, `scopeBoundDelegationAnchor_B`, `agentMerkleRoot`, `requiredScopeMask_B`, `currentTimestamp_B` — for EVERY proof submitted to EVERY RS, regardless of whether that RS is compromised. The game as stated gives the adversary strictly less information than a real deployment. This means the IND-UNL-AS game as written is **necessary** for security but **not sufficient** — passing the weakened game does not prove security in the full-information setting where the adversary holds both `scopePseudonym_A` (from RS-A, which it controls) and `scopePseudonym_B` (from the public chain at RS-B). The correct game should give the adversary ALL public signals from ALL RSes. The §4 hybrid argument is informally extended to this case in §4 item 3 ("Joint leakage"), but only for signals observed at the compromised RS. There is no hybrid argument for joint leakage across the compromised and honest RS when the adversary also sees honest RS signals. The construction claims `scopePseudonym_A = Poseidon2(scopeId_A, sbs)` and `scopePseudonym_B = Poseidon2(scopeId_B, sbs)` are unlinkable by POS-PRF — this is plausible but the formal reduction in §4 does not explicitly reduce the adversary's advantage in the full-chain-observable setting to the PRF assumption. The joint hybrid in §4.3 covers Poseidon2 and Poseidon4 at the *same* RS (`scopeId_A`). Cross-RS joint leakage — the adversary simultaneously holding `P2(scopeId_A, sbs)`, `P4(scopeId_A, sbs, pm, cc)`, `P2(scopeId_B, sbs)`, and `P4(scopeId_B, sbs, pm, cc)` — requires a two-dimensional hybrid over the key `sbs`, and the random oracle argument must cover correlated queries with distinct first-position inputs (`scopeId_A ≠ scopeId_B`) but shared key position. The §4.3 footnote acknowledges the correlated-key issue but defers to the ROM without writing the explicit hybrid.

- **Why it works / why it partially fails:** The PRF intuition is sound and the attack almost certainly does not yield a practical break — but the formal security argument as written has a gap between the game definition and the realistic threat model. This is a definitional flaw, not a construction flaw.

- **In-threat-model?** **Partially — the construction must close the proof gap.** Fix the IND-UNL-AS game to give the adversary all public chain signals. Then write the full hybrid: (1) replace `P2(scopeId_A, sbs)` with random; (2) replace `P4(scopeId_A, sbs, pm, cc)` with random; (3) replace `P2(scopeId_B, sbs)` with random; (4) replace `P4(scopeId_B, sbs, pm, cc)` with random. In the ROM each Poseidon evaluation with distinct inputs is independent, so each replacement step loses only a PRF advantage. Total adversary advantage is then bounded by `4ε_PRF`, negligible. This proof exists but is not written.

### Attack 4: JWT Introspection Response + Pre-Issuance Narrows (But Does Not Eliminate) the AS Visibility Gap

- **Attack:** Using `draft-ietf-oauth-jwt-introspection-response` plus RFC 8707 Resource Indicators, an AS can issue a signed JWT introspection bundle at enrollment time that covers all RSes the agent is pre-authorized for. The RS caches and validates locally — it never calls the AS per-request. The agent then presents tokens to RSes; the AS sees no per-request traffic. Combined with OIDC PPID, the `sub` the RS sees is RS-specific and the AS holds only the mapping, not the RS. In this mode: (a) the AS sees only which RSes were pre-authorized at enrollment, not which are actually visited; (b) each RS sees a distinct PPID; (c) there is no per-request AS call. The construction's §8 "Structural Impossibility 1" claim — "every token is issued by the AS [which] necessarily sees the (agent, RS, scope, timestamp) tuple at issuance time" — assumes per-request issuance and overstates the baseline's per-request visibility. The §8 argument does not engage with pre-issuance bulk authorization (one token batch issued at enrollment covering all allowed RS scopes), which removes the per-request visibility it cites as the structural impossibility. The residual gap is: the AS still sees the enrollment-time set of authorized RSes (even if not per-visit), and the PPID mapping is AS-held and therefore AS-recoverable. A determined AS can still reconstruct the authorization graph from enrollment data. But the claim "AS sees the full (agent, RS, scope, timestamp) tuple at issuance time" is not true in the pre-issuance model — the AS sees only (agent, {RS set}, {scope set}) at enrollment, not individual visits.

- **Why it partially works:** The construction's §8 analysis correctly identifies that PPID is AS-computed (AS holds the inverse map) and that timing is unmitigated in OAuth baselines. But the "Structural Impossibility 1" argument, as written, implicitly assumes per-request token issuance and does not address the pre-issuance + JWT introspection response model. The real comparative advantage of Bolyra is: the AS cannot reconstruct ANY subset of the visit graph even given all enrollment data, because `scopeBlindingSecret` is never transmitted to the AS. In the JWT pre-issuance baseline, the AS holds the full authorized RS list and the PPID mapping — it cannot see individual visit events, but it knows the domain of possible visits and holds the identity mapping. Bolyra is strictly stronger. But §8's "structural impossibility" framing overstates the case — it is a quantitative advantage, not a categorical impossibility, and a sophisticated RFC baseline can close most (not all) of the gap.

- **In-threat-model?** **Yes — construction survives, but §8 framing must be corrected.** Restate Structural Impossibility 1 as: "An RFC 7662 + JWT introspection response + pre-issuance AS can eliminate per-request AS visibility but CANNOT eliminate enrollment-time graph knowledge or the AS-held PPID inverse mapping. Bolyra eliminates both: the AS never learns which RSes are visited and holds no pseudonym mapping because pseudonyms are derived client-side from `scopeBlindingSecret`." This is accurate and still a meaningful differentiator — but "structural impossibility" is too strong. The honest comparison is: Bolyra provides visit-graph secrecy even against the credential issuer; no RFC baseline does.

### Attack 1: The Batch Relayer Is a Correlation Oracle — `scopeId` Is Public

- **Attack:** The batch relayer collects `ScopeBlindAuth` proofs before on-chain submission. `scopeId = Poseidon(RS domain)` is a **public input** to the circuit (§2, Public Inputs table). The RS domain space is small and enumerable — the construction itself quantifies this: "~10,000 merchants in the CU*Answers network" (§7). The relayer can trivially precompute `Poseidon(domain)` for every known RS and match incoming proofs to RSes. Within a 30-second epoch the relayer sees which proofs target which RSes. If multiple proofs from the same agent appear in the same epoch — plausible for an agent making simultaneous calls to Merchant-A and Merchant-B — the relayer can reconstruct the exact RS visit pattern the construction claims to protect. The §3 timing sub-game assumes the adversary "sees epoch boundaries but not per-proof timing within an epoch," but it says nothing about the relayer learning `scopeId` per proof. The relayer is granted the same RS correlation capability the construction was designed to deny the AS.

- **Why it works:** The construction explicitly states "the relayer sees proofs but cannot link them — each proof's `scopePseudonym` is scope-specific and the `credentialCommitment` is hidden." This is true for agent identity, but the claim slides over `scopeId`. The relayer does not need to link proofs to agents; it only needs to link proofs to RSes — which `scopeId` provides directly. The batch relayer is a new trusted party introduced without a trust model section.

The batch relayer is not modeled as an adversary at all. A formal treatment requires either (a) showing the relayer cannot correlate `scopeId`-to-agent (it can), or (b) routing proofs through an oblivious relay (e.g., PIR or onion routing) that hides `scopeId` from the relay itself, or (c) restricting the relayer to a fixed-permutation shuffle of proofs with blinded `scopeId` — none of which are specified.

### Attack 2: SPIFFE Already Removes the AS from the Per-Request Path — §8.1 Is Wrong

- **Attack:** §8.1 ("Structural impossibility 1") states: "In OAuth/OIDC, every token is issued by the AS. The AS necessarily sees the `(agent, RS, scope, timestamp)` tuple at issuance time." This is presented as a categorical impossibility of the OAuth/OIDC baseline. It is false for SPIFFE/SPIRE. Once SPIRE issues an X.509 SVID or JWT-SVID to a workload, that credential is used directly for mTLS or bearer auth per resource access — the SPIRE server is **not contacted per request**. The SVID TTL is typically 1 hour; the Workload API rotates it automatically in the background. For the duration of one SVID, the workload makes arbitrarily many resource calls with zero AS visibility per call. The AS (SPIRE server) sees only SVID issuance events, not individual resource accesses. The frequency of SVID rotations gives the AS only a coarse-grained signal, not a per-request merchant graph.

- **Why it works:** The construction's §8 case rests on claiming the OAuth/baseline is *structurally* incapable of removing the AS from the per-request path. SPIFFE disproves this claim by construction — and has done so in production at Fortune 500 scale for years. If the baseline already achieves "AS not on the critical path per request," then the comparative advantage in §8.1 collapses. The remaining claim must be scoped narrowly to: *ZK provides cross-scope unlinkability that SPIFFE does not, even when the AS is not on the per-request path.* That narrower claim may be true, but the construction does not make it — it argues against a strawman AS.

- **In-threat-model?** **No — construction must address.** Section 8 is the core competitive differentiation argument. Claiming structural impossibility against a baseline that demonstrably does not have the limitation described is a falsifiable misrepresentation that a WIMSE reviewer or standards body would reject on the first pass. The construction needs to specifically engage with the SPIFFE workload-attestation + SVID architecture and explain what unlinkability gap persists *given* that AS is already removed from per-request flow.

### Attack 4: `scopeBlindingSecret` Has No Rotation — Historical Merchant Graph Is Recoverable

- **Attack:** The `scopeBlindingSecret` is described as "generated once at agent enrollment and stored alongside the agent's credential material" (§5, Bolyra primitive mapping). It is a single 251-bit scalar with **no rotation mechanism** anywhere in the construction. If this secret is exfiltrated — through keystore compromise, a side-channel against the agent's "trusted local proving environment," or a malicious delegatee who received the secret for chain linking (§2, delegation extension: "the delegator must communicate their `scopeBlindingSecret` to the delegatee through a private channel") — the adversary immediately recovers the full historical RS visit graph for the agent. For every on-chain `scopeBoundDelegationAnchor` ever published, the adversary computes `Poseidon4(scopeId_candidate, recoveredSecret, permBitmask, cc)` over the ~10,000 candidate merchant `scopeId` values and reconstructs every RS the agent has ever visited, retroactively. The SPIFFE Workload API rotates SVIDs every hour by default; compromise of one SVID leaks only the accesses within that TTL window. Bolyra's blinding secret compromise leaks the complete lifetime history. This is a forward-secrecy failure.

- **Why it works:** The construction's security argument (§4) correctly proves that an adversary *without* `scopeBlindingSecret` cannot correlate across scopes. But it provides no argument for what happens when the secret is compromised. The delegation chain explicitly requires transmitting the delegator's `scopeBlindingSecret` out-of-band to each delegatee — this is a new exfiltration surface per delegation hop. In the healthcare scenario (§7), Kaiser's agent transmits its `scopeBlindingSecret` to Bob's agent "through a private channel (e.g., encrypted via the delegatee's public key)." Bob's agent now holds a secret whose compromise retroactively deanonymizes Kaiser's agent's complete RS visit history. The construction inherits the exact same "single long-term secret" weakness it attributes to OAuth `sub` values, but with higher stakes — the `sub` only leaks correlation across RSes that already see it; the blinding secret leaks the full historical graph to any downstream delegatee.

- **In-threat-model?** **No — construction must address.** The IND-UNL-AS game (§3) does not model an adversary who recovers `scopeBlindingSecret` post-compromise. A complete treatment requires either (a) a forward-secrecy argument with per-epoch or per-session blinding secret derivation (e.g., `sbs_t = PRF(masterSecret, epoch_t)`) so that rotation limits retroactive graph recovery, or (b) an explicit acknowledgment that the security guarantee is computational-unlinkability-given-secret-remains-secret, with a lifecycle model specifying SVID-analogous rotation. The delegation transmission pattern for `scopeBlindingSecret` should be replaced with a delegatee-specific blinding contribution that does not expose the delegator's secret.
