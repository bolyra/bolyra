# Construction

## 1. Statement of claim

An AI agent proves that its 64-bit permission bitmask satisfies an RS-specified mask predicate (`permissionBitmask & requiredScopeMask == requiredScopeMask`) without revealing any permission bits beyond the predicate result. The proof is:

- **AS-blind**: generated by the agent at the moment of use with zero AS roundtrips.
- **Runtime-adaptive**: the RS specifies `requiredScopeMask` at verification time, not at credential issuance.
- **Constant-size**: a single Groth16 or PLONK proof regardless of bitmask width.
- **Adversarial-AS resilient**: the RS's assurance derives from the on-chain Merkle root and proof verification, not from any AS attestation.
- **Identity-bound**: the proof cryptographically binds the predicate result to a specific model hash, operator key, and credential commitment enrolled in the agent Merkle tree.

No configuration of RFC 7662 + jwt-introspection-response + RFC 8693 + RFC 8707 + DPoP + BBS+ can simultaneously achieve all five properties.

## 2. Construction (gadgets, circuits, public/private inputs)

The construction uses the existing **AgentPolicy** circuit from the Bolyra spec (§4.2) with two targeted modifications: (1) the `scopeCommitment` output is randomized per session to eliminate cross-RS fingerprinting, and (2) the agent contributes a private blinding factor to the nonce derivation, defeating nonce-coordination attacks by colluding RSes.

### Modification 1: Session-randomized scope commitment (from prior iteration)

```
scopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, blindedNonce)
```

### Modification 2: Agent-side nonce blinding (closes nonce coordination gap)

**Problem**: If colluding RSes select identical `sessionNonce` values, a construction without agent blinding produces identical `scopeCommitment` and `nullifierHash` outputs, reducing CROSS-RS-UNLINK advantage to 1.

**Fix**: The agent samples a fresh private blinding factor `agentBlind` per proof. The circuit derives an internal `blindedNonce` that mixes RS-provided and agent-provided entropy:

```
blindedNonce = Poseidon2(sessionNonce, agentBlind)
```

All session-dependent outputs use `blindedNonce` instead of raw `sessionNonce`:

```
scopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, blindedNonce)
nullifierHash   = Poseidon2(credentialCommitment, blindedNonce)
```

The RS-provided `sessionNonce` remains a public input for replay detection at the RS layer. The agent's `agentBlind` is a private input — never revealed. Even if two RSes submit identical `sessionNonce` values, different `agentBlind` values produce computationally independent `blindedNonce` values, and therefore independent outputs.

**Agent obligation**: The agent MUST sample `agentBlind` uniformly at random from F_p for each proof. This is a local operation requiring no coordination or AS involvement.

### Commit-then-prove sub-protocol (bounds adaptive leakage)

Each selective-scope proof inherently reveals one bit of information: whether the agent's bitmask satisfies the RS's required mask. An adaptive adversary that freely chooses masks based on prior results can reconstruct the full bitmask in 64 queries (one per bit position). The construction bounds adaptive leakage via a **commit-then-prove** interaction that prevents the RS from conditioning its mask choice on prior proof results within the same session and gives the agent informed consent over what it discloses.

**Protocol (replaces the verification protocol's step 1–2)**:

1. **RS commits**: RS computes `maskCommitment = Poseidon2(requiredScopeMask, maskNonce)` where `maskNonce` is RS-sampled randomness, and sends `(maskCommitment, sessionNonce)` to the agent.
2. **Agent acknowledges**: The agent receives the commitment and signals readiness to prove (or aborts if the session context is suspicious — see agent-side policy below).
3. **RS opens**: RS sends `(requiredScopeMask, maskNonce)` in the clear.
4. **Agent verifies and proves**: The agent checks `Poseidon2(requiredScopeMask, maskNonce) == maskCommitment`. If the check passes **and** the agent's consent policy permits proving against this mask, the agent generates the proof. If the agent's bitmask does not satisfy the mask, the agent aborts silently (revealing no information beyond the abort itself — see §3 Game 2 for the leakage accounting).

The commitment step is **off-chain** — it adds no on-chain cost and no circuit modifications. The `maskCommitment` is not a public signal of the proof; it is a protocol-layer message. The circuit and its public/private signal layout are unchanged from the prior iteration.

**Agent-side consent policy** (protocol-layer, not circuit-layer):

The agent MAY enforce a per-RS or per-epoch query budget:

- **Per-RS mask budget**: The agent tracks the number of distinct `requiredScopeMask` values it has proven against for a given RS identity within a sliding window (e.g., 1 hour). If the count exceeds a threshold `Q_max`, the agent refuses further proofs to that RS. A recommended default is `Q_max = 3` (sufficient for typical access patterns: one mask per resource tier).
- **Minimum Hamming weight**: The agent MAY refuse to prove against masks with Hamming weight 1 (single-bit probes), which are the most efficient for bitmask reconstruction. This is a policy decision, not a cryptographic constraint.

These policies are local to the agent and require no protocol changes. They bound the *practical* leakage to `Q_max` bits per RS per window, rather than the theoretical 64-bit maximum.

**Why commitment matters**: Without the commit-then-prove step, the RS could observe the agent's latency, side-channel behavior, or willingness to engage before deciding which mask to send — enabling adaptive probing within a single interaction. The commitment forces the RS to fix its mask before any agent-dependent signal is observable.

### Gadgets (all from Bolyra spec, two modified)

1. **Num2Bits(64)** on `permissionBitmask` → `permBits[0..63]`. Prevents field-overflow attacks.
2. **Num2Bits(64)** on `requiredScopeMask` → `requiredBits[0..63]`.
3. **Scope satisfaction** (the selective-scope core):
   ```
   for i in [0, 64):
       requiredBits[i] * (1 - permBits[i]) === 0
   ```
   This is a bitwise AND check: every bit set in `requiredScopeMask` must be set in `permissionBitmask`. The RS learns only that the predicate holds — not which additional bits are set.
4. **Cumulative-bit implication closure**:
   ```
   permBits[4] * (1 - permBits[3]) === 0
   permBits[4] * (1 - permBits[2]) === 0
   permBits[3] * (1 - permBits[2]) === 0
   ```
   Enforces that financial tiers are hierarchically consistent even when hidden.
5. **Poseidon5** for credential commitment.
6. **EdDSAPoseidonVerifier** for operator signature over credential commitment.
7. **BinaryMerkleRoot(20)** for enrollment proof.
8. **LessThan(64)** for expiry: `currentTimestamp < expiryTimestamp`.
9. **Poseidon2** for `blindedNonce = Poseidon2(sessionNonce, agentBlind)`.
10. **Poseidon3** for `scopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, blindedNonce)`.
11. **Poseidon2** for `nullifierHash = Poseidon2(credentialCommitment, blindedNonce)`.

### Delegation chain linking (updated for agent-blinded nonces)

Within a delegation chain, the delegator must communicate their `blindedNonce` to the delegation prover so the chain-linking constraint can be verified. The `blindedNonce` is passed as a **private input** to the Delegation circuit — it is never public, preserving unlinkability.

**Chain-linking constraint** (Delegation circuit):
```
Poseidon3(delegatorScope, delegatorCredCommitment, delegatorBlindedNonce) == previousScopeCommitment
```

Where `delegatorBlindedNonce` is a private input provided by the delegator. The circuit does NOT recompute it from `sessionNonce` + `agentBlind` — the delegator simply passes the derived value. This is safe because:
- The constraint is satisfied only if `delegatorBlindedNonce` is the value actually used to produce `previousScopeCommitment` (Poseidon collision resistance, A2).
- The delegatee cannot substitute a different `blindedNonce` without finding a Poseidon3 collision.

**Delegatee output**: The delegatee samples their own `delegateeBlind`, computes `delegateeBlindedNonce = Poseidon2(sessionNonce, delegateeBlind)` inside the circuit, and outputs:
```
newScopeCommitment = Poseidon3(delegateeScope, delegateeCredCommitment, delegateeBlindedNonce)
```

This ensures each hop in the delegation chain has its own blinding factor, preventing cross-hop correlation by colluding RSes even within the same session.

**Delegation circuit private inputs** (changes from prior iteration):

| Signal | Change | Purpose |
|--------|--------|---------|
| `delegatorBlindedNonce` | new (replaces internal recomputation) | Chain-linking verification |
| `delegateeBlind` | new | Delegatee's fresh blinding factor |

**Delegation circuit internal signal** (new):
```
delegateeBlindedNonce = Poseidon2(sessionNonce, delegateeBlind)
```

### Verification protocol (RS perspective)

1. RS computes `maskCommitment = Poseidon2(requiredScopeMask, maskNonce)` and sends `(maskCommitment, sessionNonce)` to the agent.
2. Agent acknowledges readiness (or aborts based on consent policy).
3. RS opens `(requiredScopeMask, maskNonce)`. Agent verifies the commitment.
4. Agent checks consent policy (per-RS query budget, minimum Hamming weight). If refused, agent aborts.
5. Agent samples a fresh `agentBlind`, generates a Groth16 (or PLONK) proof with the RS's public inputs and its own private credential + blinding factor. No AS is contacted. If the agent's bitmask does not satisfy the mask, the agent aborts silently.
6. RS verifies:
   - Proof validity (pairing check or PLONK verification).
   - `agentMerkleRoot` ∈ on-chain root history buffer (last 30 roots).
   - `sessionNonce` matches the one it generated.
   - `currentTimestamp` is within acceptable skew.
7. RS records `nullifierHash` for replay detection within the session scope.

The RS learns: the agent holds a valid, non-expired, operator-signed credential enrolled in the on-chain tree, and its permission bitmask satisfies the required mask. The RS learns nothing else about the permission bitmask, model hash, operator identity, Merkle position, or blinding factor. Even if two RSes coordinate and use the same `sessionNonce`, the agent's independent `agentBlind` values produce computationally independent outputs.

## 3. Threat model (adversary capabilities, game definition)

### Adversary model

The adversary A controls:
- The Authorization Server (AS) — can forge, alter, or withhold attestations.
- The network between agent and RS — can observe, replay, and modify messages.
- Up to `N - 1` of `N` enrolled agents' secrets (static corruption).
- **One or more colluding RSes** that share all received public signals (proofs, scopeCommitments, nullifierHashes) and attempt to correlate them to the same agent.
- **RSes may coordinate their `sessionNonce` values** — including selecting identical nonces, sequential nonces, or nonces derived from agent-observable state.

The adversary sees:
- All public signals (outputs + public inputs) of every proof.
- The on-chain Merkle roots and root history buffer.
- All AS-issued tokens, introspection responses, and credentials.
- **The SRS/CRS (structured reference string / common reference string)** for both Groth16 and PLONK. These are public parameters by design.
- **Agent abort decisions**: whether the agent produced a proof or aborted for each query. This is modeled explicitly in Game 2 below.

The adversary does NOT control:
- The on-chain verifier contract (trusted smart contract model).
- The honest agent's private inputs, **including `agentBlind`**.
- The honest agent's local randomness source.
- **The SRS toxic waste** (see assumption A1a below). For Groth16, the adversary did not participate in the ceremony as its sole contributor. For PLONK, the SRS is universal and publicly verifiable.

### Trust boundary: SRS ceremony integrity

The Groth16 proving system requires a circuit-specific structured reference string (SRS) generated via a two-phase ceremony:

- **Phase 1 (Powers of Tau)**: Produces universal parameters. Bolyra reuses `pot16.ptau` (2^16 constraints) from the Hermez/PSE community ceremony for AgentPolicy and Delegation circuits, and the Semaphore v4 ceremony for HumanUniqueness. These ceremonies had hundreds of independent contributors; security requires that **at least one contributor destroyed their toxic waste**.
- **Phase 2 (circuit-specific)**: Produces per-circuit proving and verification keys. Bolyra runs project-specific Phase 2 ceremonies for AgentPolicy and Delegation. The HumanUniqueness circuit reuses Semaphore v4's Phase 2 (depth 20).

**The adversary model explicitly scopes SRS trust**: the adversary is assumed NOT to have recovered the toxic waste from every contributor to both Phase 1 and Phase 2 ceremonies. This is formalized as assumption A1a below. An adversary who recovers the complete toxic waste can forge proofs for any statement — this would break knowledge soundness (A1b) and is a catastrophic failure mode shared by all Groth16 deployments, not specific to this construction.

**Mitigation for deployments that cannot tolerate ceremony trust**: The AgentPolicy and Delegation circuits support PLONK as an OPTIONAL proving system (§3.3 of the Bolyra spec). PLONK uses a universal SRS (not circuit-specific) with publicly verifiable update ceremonies and no per-circuit Phase 2. Deployments operating under regulatory constraints that prohibit reliance on multi-party ceremony integrity (e.g., certain NCUA examination standards) SHOULD use PLONK for agent and delegation proofs while retaining Groth16 for HumanUniqueness (which inherits the battle-tested Semaphore v4 ceremony with 500+ contributors).

### Game 1: Selective-scope soundness (permission forgery)

```
Game SCOPE-FORGE(A, λ):
  1. Challenger enrolls agent with permissionBitmask = B in Merkle tree.
  2. A receives: agentMerkleRoot, all public signals from prior proofs.
  3. Challenger samples requiredScopeMask = M where B & M ≠ M
     (i.e., the agent does NOT satisfy the predicate).
  4. A wins if it produces (π, pubSignals) such that:
     a. Verify(vk, π, pubSignals) = 1
     b. pubSignals.requiredScopeMask = M
     c. pubSignals.agentMerkleRoot ∈ root history buffer

  Advantage: Adv[SCOPE-FORGE](A) = Pr[A wins]
```

**Claim**: Adv[SCOPE-FORGE](A) ≤ ε_srs (SRS ceremony integrity, A1a) + ε_ks (knowledge soundness of Groth16 under honest SRS, A1b) + ε_col (Poseidon collision resistance, A2) under the stated assumptions. See §4 for the reduction.

### Game 2: Selective-scope privacy (adaptive bitmask reconstruction)

The prior formulation of SCOPE-HIDE was a single-query indistinguishability game. This is insufficient: an adaptive adversary controlling the RS can issue Q sequential queries with different `requiredScopeMask` values, each chosen based on prior proof results (or aborts), and extract up to Q bits of information about the hidden bitmask. With Q = 64 single-bit masks, the full 64-bit bitmask is recoverable.

The reformulated game captures this adaptive threat and proves that per-query leakage is bounded to exactly 1 bit (the predicate result or the abort decision), and that the commit-then-prove protocol plus agent-side consent policy bounds total practical leakage.

```
Game SCOPE-HIDE-ADAPTIVE(A, λ, Q):
  1. Challenger samples permissionBitmask B uniformly from {0,1}^64
     (subject to cumulative-bit constraints).
  2. Challenger enrolls agent with bitmask B in the Merkle tree.
  3. For q = 1, ..., Q (adaptive queries):
     a. A chooses requiredScopeMask M_q.
     b. A sends maskCommitment_q = Poseidon2(M_q, maskNonce_q) to Challenger.
     c. A opens (M_q, maskNonce_q).
     d. If the agent's consent policy rejects (per-RS budget exceeded
        or Hamming weight below threshold), Challenger returns ⊥_policy.
     e. If B & M_q ≠ M_q, Challenger returns ⊥_unsatisfied.
     f. Otherwise, Challenger generates proof π_q with fresh agentBlind_q
        and returns (π_q, pubSignals_q).
     g. A sees the response and adaptively chooses M_{q+1}.
  4. A outputs a guess B' for the full bitmask B.
  5. A wins if B' = B.

  Advantage: Adv[SCOPE-HIDE-ADAPTIVE](A, Q) = Pr[B' = B] - 2^{-|B_eff|}
  where |B_eff| is the number of bits of B not determined by A's queries
  and cumulative-bit constraints.
```

**Leakage accounting per query**:

Each query q yields exactly one of three outcomes, each leaking at most 1 bit:

| Outcome | What A learns | Bits leaked |
|---------|---------------|-------------|
| ⊥_policy | Nothing about B (policy is independent of B) | 0 |
| ⊥_unsatisfied | B & M_q ≠ M_q (at least one required bit is unset) | ≤ 1 |
| (π_q, pubSignals_q) | B & M_q = M_q (all required bits are set) | ≤ 1 |

When a proof is returned, the public signals `(agentMerkleRoot, nullifierHash, scopeCommitment)` leak **zero additional bits** beyond the predicate result. This is because:

- `nullifierHash = Poseidon2(credentialCommitment, blindedNonce)` — randomized by fresh `agentBlind_q`, computationally independent across queries (A5).
- `scopeCommitment = Poseidon3(permissionBitmask, credentialCommitment, blindedNonce)` — similarly randomized. Two bitmasks B₀, B₁ that both satisfy M_q produce computationally indistinguishable `scopeCommitment` values under A4 (zero-knowledge) and A5 (PRF).
- `agentMerkleRoot` — shared across all agents, no signal.
- The Groth16/PLONK proof elements — simulatable under A4.

**Claim**: For each query q (conditioned on not being rejected by consent policy), the adversary's posterior entropy about B decreases by at most 1 bit. Over Q queries, the adversary learns at most Q bits. The proof elements, scope commitment, and nullifier hash contribute zero additional leakage. See §4 for the formal argument.

**Optimality**: This 1-bit-per-query leakage is **inherent and optimal**. Any protocol that allows an RS to make an access-control decision (grant/deny) based on a permission predicate necessarily reveals the predicate's Boolean result. The construction achieves this information-theoretic lower bound: the RS learns the minimum possible — exactly the 1-bit answer to its yes/no question — and nothing more.

**Practical bound via consent policy**: With agent-side consent policy enforcing `Q_max` queries per RS per window (recommended `Q_max = 3`), the practical leakage is bounded to 3 bits per RS per window. An RS attempting to reconstruct a 64-bit bitmask would need at least 22 windows (with optimal single-bit probing) or be rate-limited to infeasibility over the agent's configured window duration.

**Commit-then-prove role in adaptive security**: The commitment step ensures that within each query, the RS cannot adapt its mask based on timing side-channels or partial agent behavior. The RS must fix M_q before any agent-dependent signal (including proof-generation latency or willingness to engage) is observable. Without this, an RS could send a tentative mask, observe whether the agent begins computation (indicating satisfaction), and abort/change the mask — effectively getting partial information without consuming a query against the agent's budget.

### Game 3: AS-blindness (AS cannot influence selective disclosure)

```
Game AS-BLIND(A_AS, λ):
  1. Adversary A_AS controls the AS entirely.
  2. Honest agent holds credential enrolled in on-chain Merkle tree.
  3. RS specifies requiredScopeMask = M and sessionNonce.
  4. Agent generates proof without contacting AS.
  5. A_AS wins if it can cause RS to accept a proof where
     the agent's actual bitmask does NOT satisfy M,
     OR cause RS to reject a proof where it DOES satisfy M.

  Advantage: Adv[AS-BLIND](A_AS) ≤ ε_srs + ε_ks + ε_col
```

The AS has no role in proof generation or verification. The proof's validity depends only on the on-chain Merkle root and the circuit constraints. A compromised AS cannot alter the agent's enrolled credential commitment (it would need to forge a Poseidon preimage or corrupt the Merkle tree contract).

### Game 4: Cross-RS unlinkability (colluding RSes cannot correlate) — HARDENED

```
Game CROSS-RS-UNLINK(A, λ):
  1. Challenger enrolls two agents with credentials C₀, C₁ in the Merkle tree.
  2. A (controlling both RS₁ and RS₂) chooses sessionNonce₁ and sessionNonce₂.
     These MAY be equal (nonce coordination attack).
  3. Challenger flips coin b ∈ {0, 1}:
     - Generates proof π₁ for RS₁ using credential C_b with fresh agentBlind₁.
     - Generates proof π₂ for RS₂ using credential C_b with fresh agentBlind₂.
     (agentBlind₁ ≠ agentBlind₂ with overwhelming probability over uniform sampling.)
  4. A receives (π₁, pubSignals₁, π₂, pubSignals₂) and all on-chain state.
     A outputs guess b'.
  5. A wins if b' = b.

  Advantage: Adv[CROSS-RS-UNLINK](A) = |Pr[b' = b] - 1/2|
```

Even when `sessionNonce₁ = sessionNonce₂ = n`:
- `blindedNonce₁ = Poseidon2(n, agentBlind₁)`
- `blindedNonce₂ = Poseidon2(n, agentBlind₂)`
- Since `agentBlind₁ ≠ agentBlind₂` (with probability `1 - 1/|F_p|` ≈ `1 - 2^{-254}`), the Poseidon PRF assumption (A5) guarantees `blindedNonce₁` and `blindedNonce₂` are computationally independent.

**Claim**: Adv[CROSS-RS-UNLINK](A) ≤ ε_zk + ε_prf + 1/|F_p|, where the `1/|F_p|` term accounts for the negligible probability of `agentBlind` collision.

**Why the adversary cannot force `agentBlind` reuse**: `agentBlind` is a private input sampled by the honest agent from its local randomness source. The adversary does not control or observe this value.

### Named assumptions

The simulation trapdoor (used for ZK) is the element `δ` in the SRS; the extraction trapdoor (used for soundness) is the element `α`. ### Relationship between A1a, A1b, and A1c

The dependency structure is:

```
A1a (ceremony integrity) ──► A1b (Groth16 knowledge soundness)
                                    │
                                    ▼
                          SCOPE-FORGE, AS-BLIND games

A1c (PLONK knowledge soundness) ──► SCOPE-FORGE, AS-BLIND games
     [no ceremony trust needed]      [when PLONK is selected]

A4 (Groth16 ZK) + A5 (Poseidon PRF) ──► SCOPE-HIDE-ADAPTIVE, CROSS-RS-UNLINK games
   [unconditional — does NOT depend on A1a]
```

**Key observation**: A compromised SRS ceremony (A1a violated) breaks **soundness** (an adversary could forge proofs for false statements) but does NOT break **zero-knowledge** (the adversary still cannot learn the agent's private permission bits from a proof). This means:

- **SCOPE-FORGE and AS-BLIND** degrade if the ceremony is compromised: an adversary with toxic waste could prove scope satisfaction for permissions it does not hold.
- **SCOPE-HIDE-ADAPTIVE and CROSS-RS-UNLINK** are unaffected: even with toxic waste, the adversary learns nothing about the hidden bitmask beyond the predicate result.

Deployments that need soundness without ceremony trust SHOULD select PLONK (A1c) for AgentPolicy and Delegation circuits. The HumanUniqueness circuit is Groth16-only (reusing Semaphore v4's battle-tested ceremony), but its soundness property (Sybil detection) is orthogonal to the selective-scope claim of this construction.

### Reduction sketch for SCOPE-FORGE

Suppose adversary A wins SCOPE-FORGE with non-negligible advantage. We construct a reduction B that breaks either Groth16 knowledge soundness (A1b, which presupposes A1a) or Poseidon collision resistance (A2):

1. B runs A to obtain a valid proof π for `requiredScopeMask = M` where the enrolled bitmask B does not satisfy `B & M = M`.
2. By Groth16 knowledge soundness (A1b — which holds given an honestly generated SRS per A1a), B extracts a witness (modelHash', operatorPubkey', permissionBitmask', expiry', sig', merkleProof', agentBlind') from π.
3. The circuit enforces `requiredBits[i] * (1 - permBits'[i]) === 0` for all i, so `permissionBitmask' & M = M`.
4. The circuit enforces `credentialCommitment' = Poseidon5(modelHash', ...)` and Merkle membership of `credentialCommitment'` against `agentMerkleRoot`.
5. Case (a): `credentialCommitment' = credentialCommitment` (the enrolled credential). Then `permissionBitmask'` satisfies M but the enrolled `permissionBitmask` does not. Since `credentialCommitment = Poseidon5(..., permissionBitmask, ...)`, having two distinct bitmasks map to the same commitment yields a Poseidon collision. Contradiction with A2.
6. Case (b): `credentialCommitment' ≠ credentialCommitment` but is still a valid Merkle leaf. Then either a different credential was enrolled (not a break — the adversary controls that credential legitimately) or the Merkle proof is forged (Poseidon collision in the tree). Contradiction with A2.

Therefore: Adv[SCOPE-FORGE](A) ≤ ε_srs + ε_ks + ε_col, where ε_srs captures the probability that A1a is violated (all ceremony contributors' toxic waste was recovered) and ε_ks is Groth16's knowledge soundness conditioned on an honest SRS.

**When PLONK is used**: The reduction replaces A1a + A1b with A1c. PLONK's knowledge soundness holds in the algebraic group model + ROM without any ceremony trust assumption beyond the universal Powers of Tau (which is updatable — any single honest updater suffices). The bound becomes Adv[SCOPE-FORGE](A) ≤ ε_ks(PLONK) + ε_col with no ε_srs term.

Note: the addition of `agentBlind` does not affect this reduction. The `agentBlind` participates only in `blindedNonce` derivation, which feeds `scopeCommitment` and `nullifierHash` — neither of which is involved in the soundness argument. The scope satisfaction constraint operates directly on `permBits`, which are derived from `permissionBitmask` via Num2Bits(64), independent of `agentBlind`.

### Reduction sketch for SCOPE-HIDE-ADAPTIVE (1-bit-per-query bound)

We prove that no PPT adversary can extract more than 1 bit of information about the hidden bitmask B per query, beyond what is deducible from the predicate result and cumulative-bit constraints.

**Theorem**: For any PPT adversary A making Q adaptive queries in SCOPE-HIDE-ADAPTIVE, the mutual information between A's view and the hidden bitmask B satisfies:

```
I(B ; View_A) ≤ Q    (bits)
```

where `View_A` is the adversary's complete transcript (proof elements, public signals, and abort/success indicators for all Q queries).

**Proof sketch** (per-query decomposition):

Fix query q. A has chosen M_q and received either ⊥ or (π_q, pubSignals_q). We show the proof and public signals contribute zero bits beyond the predicate result.

**Step 1: Predicate result leaks exactly 1 bit.** The outcome (proof returned vs. ⊥_unsatisfied) reveals whether B & M_q = M_q. This is a single Boolean — 1 bit. This is inherent: any access-control protocol must reveal the access decision.

**Step 2: Public signals leak zero additional bits.** Conditioned on the predicate result being "satisfied" (proof returned), we show the public signals are computationally indistinguishable across all bitmasks B that satisfy M_q.

Consider two bitmasks B₀, B₁ with B₀ & M_q = M_q and B₁ & M_q = M_q and B₀ ≠ B₁. We construct a hybrid argument:

- **Hybrid H₀**: Real proof with witness using B₀.
- **Hybrid H₁**: Simulated proof (using Groth16 simulator from A4) with `scopeCommitment` and `nullifierHash` computed from B₀.
- **Hybrid H₂**: Simulated proof with `scopeCommitment` and `nullifierHash` replaced by random field elements.
- **Hybrid H₃**: Simulated proof with `scopeCommitment` and `nullifierHash` computed from B₁.
- **Hybrid H₄**: Real proof with witness using B₁.

Transitions:
- H₀ → H₁: Indistinguishable by A4 (perfect zero-knowledge of Groth16).
- H₁ → H₂: Indistinguishable by A5 (Poseidon PRF). Since `agentBlind_q` is fresh and uniform, `blindedNonce_q = Poseidon2(sessionNonce_q, agentBlind_q)` is pseudorandom. The outputs `Poseidon3(B₀, cred, blindedNonce_q)` and `Poseidon2(cred, blindedNonce_q)` are computationally indistinguishable from random under A5 (keyed by the pseudorandom `blindedNonce_q`).
- H₂ → H₃: Indistinguishable by A5 (symmetric argument, B₁ direction).
- H₃ → H₄: Indistinguishable by A4 (symmetric to H₀ → H₁).

Total: |Pr[A distinguishes B₀ from B₁ in query q]| ≤ 2·ε_zk + 2·ε_prf.

**Step 3: Adaptive composition.** By a standard hybrid argument over Q queries, the total advantage in distinguishing two bitmasks B₀, B₁ that agree on all Q predicate results is:

```
|Pr[A outputs B₀] - Pr[A outputs B₁]| ≤ Q · (2·ε_zk + 2·ε_prf)
```

which is negligible for polynomial Q. Therefore, the only information A extracts is the Q predicate results — exactly Q bits.

**Step 4: Commit-then-prove prevents intra-query adaptation.** Without the commitment step, the adversary could observe side-channels (e.g., proof generation latency, which might differ based on specific bit patterns due to implementation artifacts) before finalizing M_q. The commitment forces M_q to be fixed before any agent-dependent signal is observable, ensuring the clean decomposition in Steps 1–3 holds. Formally, the commitment binds M_q at a point where the adversary's view is independent of B (it has seen only prior query results), so the adaptive choice of M_q depends only on prior predicate results — exactly the Q-bit leakage we account for.

**Practical leakage bound**: With agent consent policy enforcing Q_max queries per RS per window, total practical leakage per RS per window ≤ Q_max bits. For the recommended Q_max = 3, an RS learns at most 3 bits per window — insufficient to reconstruct a 64-bit bitmask. Over K windows, the agent's consent policy enforces a total leakage of K · Q_max bits, but the RS cannot probe the same bits repeatedly (it already knows those predicate results), so reconstruction requires at least ⌈64 / Q_max⌉ = 22 windows with optimal single-bit masks. At a 1-hour window, this is 22 hours of sustained probing — a detectable anomaly for any monitoring system.

### Reduction sketch for CROSS-RS-UNLINK (hardened)

Suppose adversary A wins CROSS-RS-UNLINK with non-negligible advantage. A controls both RSes and chooses both `sessionNonce₁` and `sessionNonce₂` (possibly equal).

The public signals that vary between the two agents are:
- `scopeCommitment_j = Poseidon3(B_b, cred_b, Poseidon2(nonce_j, agentBlind_j))` for j ∈ {1, 2}
- `nullifierHash_j = Poseidon2(cred_b, Poseidon2(nonce_j, agentBlind_j))` for j ∈ {1, 2}
- The Groth16 proof elements themselves (simulatable under A4 — unconditionally).

**Case 1: `sessionNonce₁ ≠ sessionNonce₂`**. Both the nonce difference and the `agentBlind` difference contribute independence. The prior reduction applies with strictly more randomness.

**Case 2: `sessionNonce₁ = sessionNonce₂ = n`** (the nonce coordination attack). We reduce to the Poseidon PRF game:

1. Construct reduction B that receives a PRF challenge oracle O(·) for Poseidon2(n, ·).
2. B samples `agentBlind₁, agentBlind₂` uniformly, queries `y₁ = O(agentBlind₁)` and `y₂ = O(agentBlind₂)`.
3. B uses `y₁` as `blindedNonce₁` and `y₂` as `blindedNonce₂` to compute `scopeCommitment` and `nullifierHash` values (or simulates the proofs under A4).
4. If A distinguishes which credential was used, B distinguishes Poseidon2(n, ·) from a random function — contradicting A5.

Therefore: Adv[CROSS-RS-UNLINK](A) ≤ ε_zk + ε_prf + 1/|F_p|. This bound holds even if A1a is violated (ZK is unconditional, PRF security is independent of SRS trust).

### Impact on delegation chain security

The chain-linking constraint `Poseidon3(delegatorScope, delegatorCredCommitment, delegatorBlindedNonce) == previousScopeCommitment` remains sound under A2. The `delegatorBlindedNonce` is a private input to the Delegation circuit; the constraint verifies that it matches the value used to produce the prior hop's output. An adversary cannot substitute a different delegator's scope/credential because Poseidon3 collision resistance prevents finding `(scope', cred', nonce')` that collides with `(scope, cred, blindedNonce)` without breaking A2.

The delegation circuit computes `delegateeBlindedNonce = Poseidon2(sessionNonce, delegateeBlind)` internally, ensuring the delegatee's blinding factor is fresh and independent. The `sessionNonce` is a public input verified by both prover and on-chain contract, so it cannot be manipulated. The delegator's `blindedNonce` is passed as a private input — the delegatee learns this value but cannot recover `agentBlind` from it (Poseidon preimage resistance under A2).

### SRS trust summary by game

| Game | Depends on A1a (ceremony)? | Depends on A1b (Groth16 KS)? | Alternative via A1c (PLONK)? |
|------|---------------------------|------------------------------|------------------------------|
| SCOPE-FORGE | Yes | Yes | Yes — eliminates A1a dependency |
| SCOPE-HIDE-ADAPTIVE | **No** (A4 + A5 only) | No | N/A — already ceremony-free |
| AS-BLIND | Yes | Yes | Yes — eliminates A1a dependency |
| CROSS-RS-UNLINK | **No** (A4 + A5 only) | No | N/A — already ceremony-free |

This decomposition means that even a catastrophic ceremony compromise (all contributors collude or are compromised) affects only soundness games (can I forge a proof for permissions I don't hold?) but never privacy games (can the RS learn my hidden permissions?). For the privacy properties that are central to the selective-scope claim — SCOPE-HIDE-ADAPTIVE and CROSS-RS-UNLINK — the construction's security is unconditional with respect to ceremony trust.

## 7. Concrete deployment scenario

### Scenario: Federated Credit Union AI Agent Access Control

**Stakeholder**: A mid-size credit union (e.g., Navy Federal Credit Union, 13M members) deploying AI agents for member services under NCUA examination guidelines.

**Setup**: The credit union operates three resource servers:
- **RS₁**: Member account inquiry (requires `READ_DATA`, bit 0)
- **RS₂**: Loan pre-qualification (requires `READ_DATA` + `FINANCIAL_SMALL`, bits 0+2)
- **RS₃**: Wire transfer initiation (requires `READ_DATA` + `WRITE_DATA` + `FINANCIAL_MEDIUM`, bits 0+1+3, which implies bit 2 via cumulative encoding)

The credit union's compliance officer enrolls an AI agent with `permissionBitmask = 0b00001111` (bits 0-3: READ, WRITE, FINANCIAL_SMALL, FINANCIAL_MEDIUM) into the on-chain agent Merkle tree, signed by the operator's EdDSA key.

**SRS ceremony trust in this deployment**: The Groth16 verifier for the AgentPolicy circuit uses a project-specific Phase 2 ceremony atop the Hermez/PSE `pot16.ptau` Phase 1 (100+ contributors). The credit union's IT department can independently verify the ceremony transcript (published as a chain of contributor commitments) to confirm that the SRS was honestly generated. For deployments where the NCUA examiner requires eliminating ceremony trust entirely, the credit union deploys the PLONK verifier contract instead — the universal SRS is updatable, and the credit union can contribute its own update to guarantee integrity regardless of all other participants.

**Runtime flow** (agent accessing RS₂ — with commit-then-prove):

1. RS₂ samples `maskNonce₂`, computes `maskCommitment₂ = Poseidon2(0b00000101, maskNonce₂)`, generates `sessionNonce₂`, and sends `(maskCommitment₂, sessionNonce₂)` to the agent.
2. The agent acknowledges. Its consent policy checks: RS₂ has made 0 queries this window (< Q_max = 3). Proceeds.
3. RS₂ opens `(requiredScopeMask = 0b00000101, maskNonce₂)`. The agent verifies `Poseidon2(0b00000101, maskNonce₂) == maskCommitment₂`. Check passes.
4. The agent inspects the mask: Hamming weight = 2 (≥ minimum threshold). Consent policy permits.
5. The agent checks: `0b00001111 & 0b00000101 == 0b00000101`. Predicate satisfied. Proceeds to prove.
6. The agent samples a fresh `agentBlind₂` from its local CSPRNG, generates a Groth16 proof using its private credential and blinding factor. No AS is contacted.
7. RS₂ receives the proof and verifies:
   - Proof is valid (pairing check).
   - `agentMerkleRoot` matches an on-chain root (last 30).
   - The circuit enforced `permissionBitmask & 0b00000101 == 0b00000101`.
8. RS₂ learns: the agent has READ_DATA and FINANCIAL_SMALL. RS₂ does **not** learn that the agent also has WRITE_DATA and FINANCIAL_MEDIUM.

**Adaptive probing resistance**: Suppose RS₂ attempts to reconstruct the agent's full bitmask by issuing sequential single-bit probes:

| Query | requiredScopeMask | Result | Bits learned | Budget remaining |
|-------|-------------------|--------|-------------|-----------------|
| 1 | 0b00000101 | Proof returned | bit 0 + bit 2 set | 2 |
| 2 | 0b00000010 | Proof returned | bit 1 set | 1 |
| 3 | 0b00010000 | ⊥_unsatisfied | bit 4 unset | 0 |
| 4 | 0b00001000 | ⊥_policy | (budget exhausted) | — |

After 3 queries, RS₂ has learned 3 bits. The remaining 61 bits are unknown. The agent's consent policy blocks further probing until the next window.

**Nonce coordination attack defeated**: Suppose RS₂ and RS₃ are operated by the same core banking vendor (e.g., Jack Henry) and deliberately set `sessionNonce₂ = sessionNonce₃` to correlate agent traffic:

| Signal | RS₂ sees | RS₃ sees | Correlatable? |
|--------|----------|----------|---------------|
| `sessionNonce` | n | n | Identical (adversary's choice) |
| `blindedNonce` | Poseidon2(n, blind₂) | Poseidon2(n, blind₃) | **No** — distinct `agentBlind` values, PRF independence |
| `scopeCommitment` | Poseidon3(0xF, cred, blindedNonce₂) | Poseidon3(0xF, cred, blindedNonce₃) | **No** — derived from independent blindedNonces |
| `nullifierHash` | Poseidon2(cred, blindedNonce₂) | Poseidon2(cred, blindedNonce₃) | **No** — same reasoning |
| `agentMerkleRoot` | root | root | Shared across all agents — no signal |

The agent-blinded construction reduces the adversary's advantage to `ε_prf + 1/|F_p|` ≈ negligible.

**Delegation within a session**: If the agent delegates to a sub-agent within the same session (same `sessionNonce₂`), it passes its `blindedNonce₂` as a private input to the delegation circuit. The sub-agent samples its own `delegateeBlind`, computes its own `blindedNonce` inside the circuit, and outputs a fresh `newScopeCommitment`. The chain-linking constraint verifies against the delegator's `blindedNonce₂` without revealing it publicly.

**Why this matters for the credit union**:
- **Examiner audit**: The NCUA examiner can verify on-chain that all enrolled agents have operator-signed credentials without accessing any agent's full permission set. The examiner can also independently verify the SRS ceremony transcripts, or the credit union can deploy PLONK to eliminate ceremony trust entirely.
- **Least-privilege compliance**: RS₂ cryptographically cannot learn about permissions it didn't ask for — satisfying least-privilege requirements without relying on AS-side policy configuration. The commit-then-prove protocol and consent policy provide auditable evidence that the agent was not subjected to excessive probing.
- **Adversarial AS resilience**: If the credit union's OAuth AS is compromised (e.g., by a supply-chain attack on the AS vendor), the RS still has cryptographic assurance of the agent's permissions via the on-chain Merkle root. The AS cannot retroactively alter what the agent can prove.
- **Vendor collusion resistance**: Even when RS₁, RS₂, and RS₃ share a vendor that coordinates nonces, the agent's private blinding factor prevents cross-RS correlation.
- **Reconstruction resistance**: With Q_max = 3 per RS per hour, a vendor operating all three RSes learns at most 9 bits per hour across all endpoints — requiring over 7 hours of coordinated sustained probing to reconstruct the full 64-bit bitmask, a pattern easily flagged by the credit union's security operations.

## 8. Why the baseline cannot match

The baseline (RFC 7662 + jwt-introspection-response + RFC 8707 + DPoP + BBS+) fails on five independent axes. Each axis alone is sufficient; together they form an unbridgeable gap.

### Axis 1: AS-blind presentation is structurally impossible

In every RFC 7662 variant, the AS is the sole authority that determines the content of the introspection response. The agent cannot unilaterally decide, at the moment it contacts a new RS, which subset of its permissions to prove — the AS must have been involved at issuance or introspection time. BBS+ allows holder-selective disclosure of discrete claims, but the AS still issued and signed the original credential. If the AS is offline, compromised, or refuses to issue a scoped token, the agent has no recourse.

In the Bolyra construction, the agent generates the proof entirely from its local private inputs (credential, Merkle proof, operator signature, blinding factor). The AS is not in the protocol at all. The only trust anchor is the on-chain Merkle root, which is a public, immutable, verifier-independent commitment.

### Axis 2: Runtime-adaptive bitwise predicate is inexpressible

The RS specifies `requiredScopeMask` at verification time — after credential issuance, after any BBS+ derivation, after any token exchange. The circuit evaluates `permissionBitmask & requiredScopeMask == requiredScopeMask` over private bits with cumulative implication closure. This is a runtime-evaluated Boolean predicate over a hidden bitmask.

RFC 7662 scopes are fixed string sets. BBS+ supports equality and range predicates over individual hidden attributes, but bitwise AND over a 64-bit field with cascading implication constraints (bit 4 → bit 3 → bit 2) has no BBS+ extension. The RS would need to pre-enumerate all valid permission combinations and request BBS+ equality proofs for each — exponential in bitmask width.

### Axis 3: Adversarial-AS model breaks the trust foundation

The entire RFC 7662 stack's integrity rests on AS honesty. A compromised AS can:
- Claim an agent has permissions it does not (`scope: "financial_unlimited"` in introspection response for an agent with only `read`).
- Claim an agent lacks permissions it holds (denial of service by omission).
- The RS has no recourse: the signed introspection JWT proves only that the AS said what it said.

In the Bolyra construction, the AS could be entirely adversarial — the proof's validity depends on:
1. The credential commitment being a leaf in the on-chain Merkle tree (public, immutable).
2. The operator's EdDSA signature over the credential commitment (verifiable without AS).
3. The Groth16/PLONK proof satisfying the circuit constraints (verified by the RS or on-chain contract).

No AS attestation is in the verification path. The trust assumptions are explicit and decomposed: SRS ceremony integrity (A1a) for Groth16 soundness, which can be eliminated entirely by choosing PLONK (A1c). This is a strictly more transparent trust model than the baseline's implicit reliance on AS honesty.

### Axis 4: Constant-size proof is unachievable at scale

A jwt-introspection-response grows linearly with disclosed scope count. A BBS+ derived proof grows with the number of disclosed messages (O(|disclosed|)). For a 64-bit permission space — let alone the 2^64 theoretical space mentioned in the candidate scenarios — string enumeration is computationally infeasible and BBS+ proof size scales with disclosed claims.

The Bolyra Groth16 proof is exactly 192 bytes (3 BN254 G1/G2 elements) regardless of whether the bitmask is 8 bits or 64 bits or hypothetically 256 bits (at proportional constraint cost). The PLONK proof is ~768 bytes. Both are constant in the number of permissions being checked.

### Axis 5: No cryptographic binding to runtime agent identity

`client_id` in OAuth is a static registration string. Neither RFC 7662 nor DPoP nor BBS+ bind the token to a specific `modelHash` (which model is executing), `operatorPubkey` (under whose authority), and `permissionBitmask` (with what capability set) — all committed together in a single `credentialCommitment = Poseidon5(modelHash, opAx, opAy, permissionBitmask, expiry)` that is the Merkle leaf.

The Bolyra proof cryptographically binds the scope-satisfaction result to a specific agent runtime identity. The RS knows that the entity satisfying the permission predicate is the same entity whose model hash, operator key, permission set, and expiry were committed at enrollment. No component of the baseline can express this binding.

### Axis 6: Adaptive reconstruction resistance with bounded per-query leakage

BBS+ selective disclosure reveals each disclosed claim in full — there is no mechanism to reveal only a predicate result over hidden claims while bounding the number of predicate evaluations. An RS receiving BBS+ presentations can issue unlimited selective-disclosure requests, each revealing a different claim subset, with no holder-enforced rate limit in the protocol.

In the Bolyra construction, each proof reveals exactly 1 bit (the predicate result), and the commit-then-prove protocol combined with agent-side consent policy bounds the total number of queries per RS per window. The SCOPE-HIDE-ADAPTIVE game (§3 Game 2) formally proves that no additional information leaks from proof elements, scope commitments, or nullifier hashes beyond the predicate result. This 1-bit-per-query bound is information-theoretically optimal — it is the minimum leakage for any protocol that enables access-control decisions — and the agent-enforced query budget makes full bitmask reconstruction practically infeasible.

The baseline has no analog: BBS+ presentations leak full claim values (not predicate results), RFC 7662 introspection responses leak full scope strings, and neither protocol gives the credential holder a mechanism to rate-limit verifier queries at the cryptographic or protocol layer.

### Axis 7: Cross-RS unlinkability without AS-level correlation — robust to nonce coordination

BBS+ presentations are unlinkable at the RS layer, but the AS that issued the credential knows the holder's full permission set and can correlate issuance events. PPID (OIDC Core §8.1) prevents RS-vs-RS `sub` correlation but explicitly does not prevent AS-level correlation.

In the Bolyra construction with agent-blinded, session-randomized scope commitments, there is no AS in the loop to correlate anything. The agent generates proofs locally, and the only public outputs (`scopeCommitment`, `nullifierHash`) are randomized via the agent's private `agentBlind`. This unlinkability holds even when colluding RSes coordinate their `sessionNonce` values. The security of this property (A4 + A5) does not depend on SRS ceremony integrity — it is unconditional with respect to ceremony trust.

### Summary: structural vs. configurational gap

The baseline's limitations are not deployment gaps fixable by better AS policy or additional extensions. They are structural: the information-flow architecture of RFC 7662 requires an honest AS in the verification path, operates over string-enumerated scopes rather than algebraic predicates, and has no mechanism for holder-computed proofs over committed bitmasks with bounded per-query leakage. The Bolyra construction's trust assumptions are explicit, decomposed (SRS ceremony integrity for soundness, unconditional for privacy), and eliminable (PLONK removes ceremony trust entirely). The baseline's trust in AS honesty is implicit, monolithic, and inescapable. The construction achieves the information-theoretic minimum of 1 bit leaked per access-control decision, with protocol-layer mechanisms to bound the total number of decisions — a property that has no counterpart in any configuration of the baseline stack. These are properties of the protocol architecture, not of any particular implementation.
