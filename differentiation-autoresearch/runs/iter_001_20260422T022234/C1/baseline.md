# Baseline: Selective Scope Proof (C1)

## Best Alternative: RFC 7662 + JWT Introspection Response + BBS+ Selective Disclosure (Composite)

The strongest non-ZK baseline for C1 is a composite of two independent stacks: the full RFC 7662 OAuth 2.0 introspection suite (with JWT introspection response, RFC 8693 token exchange, RFC 8707 audience binding, and RFC 9449 DPoP), layered with W3C Verifiable Credentials using BBS+ signatures for the credential presentation leg. No single specification closes all the gaps independently; the composite represents the best achievable without zero-knowledge circuits.

---

## What This Baseline CAN Do Against C1

### 1. AS-Side Scope Filtering (RFC 7662 + JWT Introspection Response)

The Authorization Server may enforce per-RS policy at introspection time, returning a filtered `scope` field containing only the scopes the requesting RS is authorized to learn about. The signed JWT introspection response ([draft-ietf-oauth-jwt-introspection-response](https://datatracker.ietf.org/doc/draft-ietf-oauth-jwt-introspection-response/)) then allows the RS to verify this filtered response offline without a second AS roundtrip. For a 2^64 permission space, AS-side policy tables are the standard approach: the AS evaluates the predicate centrally and returns a compact result.

### 2. Audience-Bound, Sender-Constrained Tokens

Combining [RFC 8707](https://datatracker.ietf.org/doc/html/rfc8707) (resource indicators) with [RFC 9449](https://datatracker.ietf.org/doc/html/rfc9449) (DPoP), the baseline issues tokens bound to a specific RS audience and cryptographically constrained to the agent's ephemeral keypair. This prevents cross-RS replay and bearer-token theft. Scope leakage across resource servers is substantially reduced — a token for RS-A does not disclose its contents to RS-B.

### 3. Per-Hop Scope Narrowing via Token Exchange

[RFC 8693](https://datatracker.ietf.org/doc/html/rfc8693) token exchange allows a delegation chain to progressively narrow scope at each hop. An orchestrator agent can exchange its token for a narrower actor-token before passing it to a subagent. The final RS sees only the narrowed scope.

### 4. BBS+ Selective Disclosure of Credential Claims

Using [draft-irtf-cfrg-bbs-signatures](https://datatracker.ietf.org/doc/draft-irtf-cfrg-bbs-signatures/) over a W3C VC ([VC Data Model 2.0](https://www.w3.org/TR/vc-data-model-2.0/)), the holder can generate a derived presentation that discloses only a subset of credential claims. Multiple presentations of the same credential are unlinkable. If permissions are encoded as individual VC claims, the agent can prove possession of a specific permission claim without disclosing the full claim set. This is the closest the baseline comes to a selective disclosure primitive that does not require an AS roundtrip.

### 5. Predicate-Adjacent Capabilities

BBS+ supports equality checks and range proofs over hidden messages via NIZK extensions. For ordinal or bounded-integer permissions, the baseline can prove "permission level ≥ threshold" without disclosing the exact level. OIDC pairwise subject identifiers ([OIDC Core §8.1](https://openid.net/specs/openid-connect-core-1_0.html#PairwiseAlg)) prevent RS-vs-RS correlation on the subject identifier.

## What This Baseline Fundamentally CANNOT Do

### 1. Boolean Bitmask Predicate Without Disclosure

The C1 claim requires proving `(permissionBitmask & requiredMask) == requiredMask` without revealing `permissionBitmask`. BBS+ handles equality and range over individual hidden messages. It does not natively support bitwise AND predicates over a packed bitmask. Encoding 2^64 permissions as 2^64 individual VC claims is computationally infeasible. Encoding as a packed integer collapses BBS+'s per-message selective disclosure. **This gap is not closeable within BBS+ without external circuit compilation.** RFC 7662 cannot express this at all — it returns a scope string; the RS sees what the AS sends, nothing less.

### 2. AS-Blind Presentation (Agent Chooses Disclosure at Runtime, No AS Involvement)

RFC 7662 requires the AS to be present at introspection time, or the AS must have pre-signed the JWT introspection response at issuance. In the latter case, the scope content is fixed at issuance — the agent cannot adaptively select which subset of permissions to reveal at the moment of use without a new AS roundtrip. BBS+ partially addresses this: a derived VP is generated by the holder without contacting the issuer. However, BBS+ VCs are not natively issued by OAuth ASes. A hybrid architecture would require a separate VC issuance layer, which is not part of any standardized OAuth flow. **No single RFC in the introspection suite supports agent-side, runtime-adaptive predicate selection with zero AS involvement after initial credential issuance.**

### 3. Cryptographic Assurance Against a Lying AS

The C1 scenarios include an adversarial AS that cannot lie about scope membership. Under RFC 7662 and the JWT introspection response, the RS must cryptographically trust the AS's signature over the introspection response. If the AS is semi-trusted and willing to forge filtered responses, the RS has no recourse — the AS's signature is the only attestation available. BBS+ separates the issuer (who signed the credential) from the presentation (holder-generated), but it does not prove that the credential accurately reflects the true permission set as recorded in some independent registry. **There is no mechanism in this baseline for an RS to verify scope membership against a commitment the AS cannot retroactively alter.** RFC 8707 audience binding and RFC 9449 DPoP do not address AS honesty — they address token binding and replay, not issuer integrity.

### 4. Constant-Size Proof Regardless of Permission Space Width

The JWT introspection response size scales with the number of scopes returned. BBS+ derived proof size is O(|disclosed claims|). For a 2^64 permission bitmask where the RS must be convinced that a predicate holds over the full space, neither approach produces a proof whose size is independent of the bitmask width. **Constant-size proofs over arbitrary-width predicates are not expressible in any combination of RFC 7662 variants or BBS+ without a succinct argument system.**

### 5. Unlinkable Cross-RS Scope Proofs Without AS Correlation

The AS sees every token issuance and every introspection call. Even with OIDC PPIDs preventing RS-vs-RS correlation on `sub`, the AS can correlate the agent's full cross-RS activity. BBS+ presentations are mutually unlinkable at the RS level, but the VC issuer (the AS, in a hybrid model) still sees issuance events. **Full cross-RS unlinkability including issuer-blindness is not achievable in this baseline.** [SPIFFE/WIMSE](https://datatracker.ietf.org/wg/wimse/about/) addresses workload identity transport but explicitly does not provide selective attribute disclosure or cross-scope unlinkability.

## Capability Matrix

| Property | RFC 7662 Suite | BBS+ VC | Composite |
|---|---|---|---|
| Scope filtering per RS | Yes (AS-enforced) | N/A | Yes |
| Offline RS verification | Yes (JWT sig) | Yes (derived proof) | Yes |
| Runtime-adaptive predicate, no AS roundtrip | No | Partial (holder-derived) | Partial |
| Bitmask AND predicate without disclosure | No | No | No |
| Adversarial-AS protection | No | No | No |
| Constant-size proof over wide permission space | No | No | No |
| Cross-RS unlinkability incl. issuer-blind | No | No | No |

**Bar to beat:** Prove `(permissionBitmask & requiredMask) == requiredMask` in constant size, with no AS roundtrip, with cryptographic soundness against a semi-honest AS that controls both issuance and the introspection endpoint — a property no combination of RFC 7662 variants, JWT introspection responses, token exchange, DPoP, or BBS+ selective disclosure can provide.
