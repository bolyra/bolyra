### Attack 1: The Adversarial-AS Claim Only Fires in Multi-Party Consortia — Which Is Not the Market

**Attack:**
The construction's decisive differentiator — "security holds even when the AS is fully adversarial" — requires a very specific deployment topology: the AS holds zero governor keys, enrollment governors are drawn from distinct institutions with conflicting interests, and there is a meaningful threat that one institution's AS would lie to another institution's RS. The construction's own deployment scenario (Section 7) is explicit about this: it requires a 12-CU consortium where no single CU controls more than 1 of 12 governor keys.

But the overwhelming majority of MCP deployments in 2026 are single-org: Salesforce AI agents against Salesforce data, a fintech's agents against its own risk API, a bank's automation against its own loan platform. In these deployments, the same organization that runs the AS *also* appoints the enrollment governors. The CISO who controls the AS also controls the HSMs holding the governor keys. A compromised AS-side attacker in a single-org deployment would compromise the governor keys via the same blast radius — same org, same personnel, same key management infrastructure.

In the single-org case, the adversarial-AS advantage collapses completely. The k-of-n enrollment reduces to "1-of-1 where the 1 is the operator." The construction explicitly warns in Section 3: "A deployment MAY designate the same organization as both AS and enrollment governor, but this collapses the trust separation." That sentence is a concession that the construction's primary differentiator does not apply to most deployments.

Auth0 MCP auth + Client Attestation (draft-ietf-oauth-attestation-based-client-auth) covers the single-org case with hardware-backed identity, SOC 2, <100ms token verification, and no blockchain dependency. For multi-party consortia, Cloudflare Access with custom claims handles federation.

**Why it works:** The construction never scopes Properties 1–5 to multi-party deployments. It presents them as universal. Procurement at a single-org enterprise will ask "is your AS adversarial?" and the answer is "no, it's Auth0." The ZK proof adds 500ms latency, requires blockchain infrastructure, and introduces circuit-level complexity — all with zero benefit over the incumbent.

**In-threat-model?** No. The construction must add an explicit scope statement: the adversarial-AS benefit and Properties 1–5 are only distinguishing in multi-party, cross-institutional deployments where participants do not share a single governance authority. Single-org deployments must be documented as a degenerate case where the construction provides no security improvement over OAuth + Client Attestation.

### Attack 2: You Swapped AS Availability Risk for L2 Availability Risk — and Lost the SLA Comparison

**Attack:**
The construction argues in Section 7 that the adversarial AS causes a "14-hour outage during a CrowdStrike-style event" and that AS-blind presentation eliminates this dependency. But the RS now reads `onChainAgentRoot` from the Bolyra on-chain registry (Base Sepolia / Base mainnet). During a Base sequencer outage (which occurred in June 2024 for ~45 minutes), RSes cannot fetch new roots. A credential enrolled 31 Merkle-root rotations ago — after the 30-entry buffer expires — would fail verification with no fallback. The construction has replaced "trust the AS to be online" with "trust Base L2 to be online and serving fresh roots."

Cloudflare Access has 300+ PoPs with global anycast and publishes a formal 100% uptime SLA for enterprise contracts. Auth0 publishes 99.99% SLA. Base Sepolia has no enterprise SLA. **Why it works:** The enterprise procurement question is availability. "Our agent auth uses a public L2 blockchain with no SLA" fails the same procurement check as "our auth provider has a CrowdStrike dependency." **In-threat-model?** Partially. ### Attack 3: The Latency Math Breaks at Microservice Scale — and Failing Proofs Make It Worse

**Attack:**
Section 6 claims "< 500 ms (rapidsnark native)" and frames the T_prove constant-time budget overhead as "negligible." This is true in isolation. It breaks at production microservice scale.

A typical enterprise AI agent workflow against a regulated API involves 5–20 RS authorization checks per user-visible request: read member profile (RS-1), read credit summary (RS-2), pull transaction history (RS-3), call risk-scoring endpoint (RS-4), write pre-qualification result (RS-5). With T_prove = 500ms per proof, that's 2.5–10 seconds of pure authorization overhead per user request, even before any business logic. Auth0/WorkOS issue a token once; downstream RS calls verify an RS256 JWT signature in < 1ms with local key caching. The per-call overhead is zero after the initial token fetch.

The chosen-predicate side-channel attack also creates a real-world DoS vector the construction underweights. Section 3's constant-time discipline requires the agent to sleep the full T_prove on a failed proof. A malicious RS (or a compromised consortium partner RS) that sends unsatisfied predicates forces the agent to burn T_prove per probe. With 64 single-bit probes sent serially to the agent, the agent is locked for 64 × 500ms = 32 seconds — not just blocked from the malicious RS, but blocked from ALL RSes if the constant-time budget is agent-global (which it must be to prevent timing side-channels via parallelism). Section 3 does not address this agent-level DoS, only the information leakage angle of the timing attack.

WorkOS MCP auth uses a standard token with short TTL + refresh; a malicious RS can at worst cause a 401 and force a token refresh. There is no 32-second blast radius.

**Why it works:** Enterprise SLAs for user-facing APIs are typically p99 < 200ms. A construction that introduces 500ms baseline per-RS overhead — regardless of predicate satisfaction — is dead on arrival for any interactive user flow. The construction presents latency as a solved property; the multiplication effect and the DoS-via-constant-time gap show it is not.

**In-threat-model?** No on the DoS angle (not addressed), partially on latency (addressed in isolation, not at scale). The construction must specify: (a) whether T_prove is per-agent or per-RS-connection; (b) rate-limit policy for unsatisfied-predicate probes from a single RS; (c) latency budget for multi-hop RS workflows.

### Attack 1: Chosen-Predicate Full Bitmask Recovery — Privacy Game Is Vacuously Satisfied

**Attack:** The adversary queries the agent 64 times with single-bit masks: `requiredScopeMask ∈ {0x01, 0x02, 0x04, ..., 0x8000000000000000}`. After 64 binary outcomes (proof returned vs. `scope_insufficient`), the adversary has recovered the full 64-bit `permissionBitmask` exactly.

The constant-time discipline (Section 2, "Agent-side response discipline") closes the **timing** channel — both branches take `T_prove`. But the construction does nothing about the **semantic** channel: each query is a direct 1-bit oracle on the hidden bitmask.

**Why the mqSE-IND game doesn't save it:** The game (Section 3, Privacy game) requires the adversary to choose challenge bitmasks B0, B1 such that "for every predicate R queried in Phase 1, B0 satisfies R iff B1 satisfies R." For 64 single-bit queries, this constraint forces B0 = B1. The game is trivially satisfied — but it's trivially satisfied because no two distinct bitmasks survive the Phase 1 constraint. The reduction to PLONK SE-ZK is sound, but it proves a statement with no content: the adversary cannot distinguish between two agents that are *required by the game to be identical*. This is not zero-knowledge; it's a privacy claim that the game structure makes circular.

Section 3 states: "A 64-query sweep where each query sets a single bit yields at most 64 binary outcomes — but these match exactly the outcomes the adversary would see if the agent had ANY subset of permissions." This is **false**. A fully-privileged agent (all 64 bits set) returns 64 successes. A fully-unprivileged agent returns 64 failures. An agent with `permissionBitmask = 0b...10010111` returns a mixed pattern that uniquely identifies it. The 64 outcomes are a bijection to the 64-bit bitmask, not a set that "matches any subset."

**Why it works:** The ZK property of PLONK guarantees that no individual proof reveals more than what is implied by the public inputs and outputs. But the **predicate satisfaction boolean is itself a bit of the bitmask**. Chosen-predicate probing is a semantic attack, not a structural one. The simulator cannot hide that `b & 0x01 == 1` when the RS asks `requiredScopeMask = 0x01` and receives a valid proof.

**In-threat-model?** Yes — the adversary controls `requiredScopeMask` (explicitly modeled in Section 3). The construction survives the mqSE-IND game as written, but **the game definition is too weak to preclude full bitmask recovery**. The construction should replace or supplement the privacy game with a k-query bit-extraction game that establishes a lower bound on the number of queries required to recover the bitmask. Under the current game, the privacy guarantee is: "you cannot distinguish two agents unless you can distinguish two agents." That's not a guarantee.

### Attack 2: SRS Provenance — The PLONK Soundness Argument Cites an Unnamed Ceremony

**Attack:** The construction's proof system is PLONK over BN254 with a universal SRS generated from `pot16.ptau` (Section 5 and CLAUDE.md). The entire soundness argument (Section 4, all three reduction sketches) reduces to **Assumption 1: knowledge soundness of PLONK with universal SRS in the AGM + ROM**. This assumption holds only if the SRS was generated with a destroyed toxic waste. If any party knows the toxic waste `τ` from the Powers of Tau ceremony, they can construct an adversarial SRS and forge accepting PLONK proofs for **any** circuit — including ones where the `permissionBitmask` does not satisfy the predicate.

**Concrete form of the attack:** Given toxic waste `τ`, construct a forged SRS `{τ^i G1, τ^i G2}`. The PLONK polynomial commitment scheme (KZG) is computationally binding only under the SRS's hiding of `τ`. With `τ` known, the adversary evaluates the PLONK round polynomials directly, constructs a valid transcript for G5-violating witness, and produces an accepting proof. The RS verifies with the legitimate verification key (computed from the same SRS) and accepts.

**Why it matters here:** The construction cites `pot16.ptau` without naming its provenance. Three candidates:
1. **snarkjs development artifact** (`pot16_final.ptau` shipped with the snarkjs npm package or tutorials): generated by snarkjs maintainers; toxic waste status unknown; NOT suitable for production.
2. **Hermez Perpetual Powers of Tau** (ptau file up to 2^28): multi-party ceremony with verifiable transcript; toxic waste is assumed destroyed; widely accepted as production-grade.
3. **Project-generated local ceremony**: generated by running `snarkjs powersoftau` without a ceremony; toxic waste is the entropy passed at generation; if retained, an insider can forge proofs.

The construction claims "PLONK avoids per-circuit ceremony" — correct for the per-circuit phase. But the **universal SRS phase is not a ceremony-free step**. PLONK trades per-circuit trusted setup for a single universal trusted setup. The reduction sketch in Section 4 asserts PLONK knowledge soundness as an assumption, but **the assumption only holds if the SRS is trustworthy**. Under a subverted `pot16.ptau`, the Rogue Enrollment Forgery sub-game, the Selective Scope Forgery game, and the Implication Closure Forgery game all collapse simultaneously — the adversary can win all three without breaking ECDSA, Poseidon, or EdDSA.

**In-threat-model?** The threat model (Section 3) grants the adversary control of the AS, up to k-1 governors, and network position — but does NOT mention the SRS as an adversary asset. The adversary model is silent on trusted setup compromise. Under the full adversarial-AS model the construction advertises, a party who is simultaneously the AS *and* ran the `pot16.ptau` ceremony (an entirely plausible deployment scenario for a consortium member) can compromise the entire construction without attacking any of the named assumptions. This is not in the threat model — **the construction must state the SRS provenance and include SRS integrity as a named assumption, or commit to a specific Hermez ptau artifact with a verifiable transcript hash**.

### Attack 3: Nullifier Rate-Limiting Is Incoherent — Fresh BlindingNonce Makes Per-Agent Rate Limiting Impossible

**Attack:** The `agentNullifier = Poseidon3(credentialCommitment, requiredScopeMask, blindingNonce)` is regenerated with a fresh `blindingNonce` per presentation. Two presentations of the same credential to the same RS with the same `requiredScopeMask` produce **distinct, unlinkable nullifiers**. The RS cannot associate the two presentations with the same agent.

The construction (Section 2, G9) claims this "enables rate-limiting without cross-session linkage." These goals are in direct contradiction:

- **Rate-limiting** requires the RS to associate multiple presentations with the same agent across a time window — i.e., linkability within a window.
- **Cross-session unlinkability** requires that presentations are indistinguishable regardless of which agent produced them.

A nullifier that is (a) fresh per presentation and (b) randomized by a secret field element cannot satisfy both. The RS sees a stream of distinct nullifiers with no structure. It can count total nullifiers in a window (coarse request-rate limit), but it cannot determine that credential `C*` has been used 1,000 times in the past second vs. 1,000 distinct credentials each used once.

The construction uses this incoherence to claim an advantage over RFC 7662 ("enables rate-limiting") while simultaneously claiming unlinkability. This is not a tradeoff the construction successfully navigates — it claims both properties while technically providing neither in any meaningful form. The rate-limiting function of the nullifier as written is equivalent to a monotonic request counter on the RS, which requires no nullifier scheme at all.

**Why it matters:** If the intended rate-limiting mechanism is "prevent replay of a specific (proof, nullifier) tuple," then the `blindingNonce` achieves anti-replay (different nonce → different nullifier → cannot reuse the same proof), but anti-replay is already handled by the EdDSA signature and Merkle membership — the nullifier adds nothing new for anti-replay. If the intended mechanism is "limit how many times credential `C*` can be used at RS `r`," then the scheme fails because the RS cannot compute `C*` from the nullifier without breaking Poseidon preimage resistance.

**In-threat-model?** The sub-game ImplicationClosureForgery and SelectiveScopeForgery do not address denial-of-service via credential flooding. A malicious agent (or an adversary-controlled agent in the threat model, per Section 3: "up to N-1 of N enrolled agents") can generate proofs at ~2 proofs/second (snarkjs) or ~2 proofs/millisecond (rapidsnark) and flood the RS with distinct nullifiers, exhausting the RS's nullifier storage. The construction does not bound the RS's nullifier table size or specify a staleness policy. The rate-limiting claim should either be removed or replaced with a concrete per-agent accountability mechanism (e.g., a deterministic nullifier family derived from a per-session nonce that the RS supplies, removing the agent's freedom to randomize).

### Attack 4: Variable-Depth Merkle Proof — `merkleProofLength` Is a Private Input Without a Circuit Range Check

**Attack:** The circuit takes `merkleProofLength` as a **private input** of type `uint8`, but the gadget list (Section 2, G1–G9) does not include a range check or a Num2Bits decomposition for `merkleProofLength`. G1 applies `Num2Bits(64)` to `permissionBitmask`, `expiryTimestamp`, `currentTimestamp`, and `requiredScopeMask` — but NOT to `merkleProofLength`.

The `BinaryMerkleRoot(credentialCommitment, merkleProofSiblings, merkleProofIndex, merkleProofLength)` gadget (G4) must use `merkleProofLength` to determine how many sibling hashes to include in the path computation. In Circom implementations of variable-depth Merkle trees, this is typically done with a multiplexer: for each level `i`, include sibling `i` if `i < merkleProofLength`, else use a dummy value (usually `0` or the current node). If `merkleProofLength = 0`, the gadget should return the leaf itself as the root — i.e., `computedRoot = credentialCommitment`.

**Exploit scenario:** Adversary sets `merkleProofLength = 0`, `merkleProofSiblings = [0, ..., 0]`, `merkleProofIndex = 0`. The circuit computes `computedRoot = credentialCommitment`. The constraint G4 requires `computedRoot === onChainAgentRoot`. The adversary needs `credentialCommitment === onChainAgentRoot` — which seems to require a Poseidon preimage attack. But consider:

1. The adversary is also choosing `credentialCommitment` via G2: `credentialCommitment = Poseidon5(modelHash, opAx, opAy, permissionBitmask, expiryTimestamp)`. The adversary has freedom over ALL private inputs.
2. The adversary needs `Poseidon5(modelHash, opAx, opAy, permissionBitmask, expiryTimestamp) === onChainAgentRoot`.
3. `onChainAgentRoot` is a public input — a specific field element the RS reads from the chain.
4. The adversary must find private inputs such that `Poseidon5(...)` equals this specific value — a **Poseidon preimage attack**. Under Assumption 2 (Poseidon collision resistance / preimage resistance), this fails.

**Why this still matters:** The construction's Table 1 (public inputs) shows `onChainAgentRoot` is RS-supplied. The RS reads a root from the chain — but the RS might read a **historical root from the 30-entry root history buffer**, not the current root. If the Merkle tree has evolved and the old root corresponds to a tree where `credentialCommitment` WAS the only leaf (or the root of a subtree), the `merkleProofLength = 0` attack becomes a **length-extension attack on a subtree root**. The circuit must enforce that `merkleProofLength` is consistent with the actual tree depth, not just that the computed root matches any valid historical root. The 30-entry buffer creates 30 distinct public inputs against which the adversary can attempt the attack — expanding the attack surface by a factor of 30.

More broadly, the omission of `merkleProofLength` from G1 is an **under-specification**: the circuit description does not demonstrate that the Merkle depth is pinned to a canonical value. A complete construction must either (a) fix `merkleProofLength = 20` (constant depth, padding all shorter paths to 20 levels with zero siblings), removing the private depth signal entirely, or (b) add an explicit range check `Num2Bits(5)` on `merkleProofLength` AND prove that the `BinaryMerkleRoot` gadget correctly handles all values in `[1, 20]` without degenerate edge cases at 0.

**In-threat-model?** The SelectiveScopeForgery game (Section 3) covers "adversary produces a proof for a predicate the honest agent doesn't satisfy." The Merkle depth manipulation is an attempt to forge Merkle membership, which the game does cover. The attack is blocked by Poseidon preimage resistance — **construction survives** for the historical-root case IF the preimage attack fails. But the **under-specification of G1** (missing `merkleProofLength` range check) means the circuit as described is incomplete, and any concrete Circom implementation must handle the degenerate `length = 0` case explicitly. The reduction in Section 4 assumes `BinaryMerkleRoot` is a correct Merkle implementation — but this correctness depends on implementation details the construction does not pin down. A formal reduction requires a formal gadget specification, not a table entry.

### Attack 4: Constant-Time Budget Degrades Every Member-Facing Transaction to Worst-Case Latency

- **Attack:** The construction's Section 2 mandates a constant-time budget `T_prove` (default 3 seconds for snarkjs, 500 ms for rapidsnark) for ALL proof attempts — successful or not — as a timing-channel defense. In the loan pre-qualification scenario (Section 7), an AI agent contacts a partner CU's RS to pull credit summaries. The partner RS specifies `requiredScopeMask = 0b00000101`. The agent runs the prover, generates a valid proof in 400 ms (rapidsnark), and then waits until `T_prove = 500 ms` before responding. Every legitimate member-facing transaction has a mandatory floor latency of `T_prove`, even when the prover succeeds quickly. If the consortium deploys snarkjs (e.g., in a Node environment without native rapidsnark), this floor is 3 seconds. My Tier 1 ops team will receive member support calls about slow loan response times. My SLAs with the consortium require sub-2-second API responses for member-facing operations. The construction enforces a mandatory minimum latency that may breach these SLAs — and the latency cannot be reduced without breaking the constant-time discipline (reintroducing the timing oracle the adversary exploits per Section 2's chosen-predicate attack).

- **Why it works / why it fails:** This is a genuine operational tension. ### Attack 4: The Chosen-Predicate Bit-Extraction Attack Survives Constant-Time Discipline via Adaptive Predicate Composition

- **Attack:** Section 2's "Agent-side response discipline" and Section 3's mqSE-IND game assume the adversary probes with *atomic single-bit predicates* (`requiredScopeMask = 0x01, 0x02, 0x04, ...`). The constant-time discipline neutralizes timing on these because the boolean outcome is the only signal. But the formal claim in Section 3 ("A 64-query sweep where each query sets a single bit yields at most 64 binary outcomes — but these match exactly the outcomes the adversary would see if the agent had ANY subset of permissions") is wrong. Consider an adversary using *adaptive predicate composition*: query `mask = 0b00000001` → success (bit 0 set). Query `mask = 0b00000011` → failure (bit 1 not set). Query `mask = 0b00000101` → success (bit 2 set). Each response is a binary outcome, but an adaptive adversary can binary-search the bitmask in `O(64)` queries by composing masks based on prior responses, and — critically — mask queries of the form `required = known_bits | candidate_bit` that definitively distinguish whether `candidate_bit` is set. The construction's Section 3 mqSE-IND game constrains the adversary to choose `B0, B1` that are indistinguishable on all queried predicates — but this game definition *already assumes* the adversary cannot distinguish them, which is exactly what constant-time is trying to prove. The game conflates the *definition* of indistinguishability with a *proof* of it. The actual claim should be: after `q` adaptive chosen-predicate queries, the adversary has recovered at most `q` bits of the bitmask. For a 64-bit space, 64 adaptive queries deterministically recover the full bitmask. This is not "at most 64 binary outcomes indistinguishable from any subset" — it is "64 queries that fully determine the bitmask." The constant-time discipline prevents timing attacks but does not prevent information-theoretic extraction via chosen predicates.

- **Why it fails against the construction / why it partially succeeds:** The construction partially survives: this is not an attack on the *cryptographic* soundness (the adversary cannot forge a proof) but on the *privacy* claim. The construction must concede that an adversary with unlimited chosen-predicate query access to the honest agent's `proveSelectiveScope()` oracle recovers the full bitmask in exactly 64 queries, regardless of constant-time discipline. The blinded `agentNullifier` (G9) prevents cross-session correlation but does not prevent within-session bitmask extraction. The construction's privacy argument holds only in the adversarial model where the *RS* is the adversary (the RS sees only the proofs it requests), but an adversary who controls the RS and can send arbitrary `requiredScopeMask` values fully recovers the bitmask in 64 oracle calls. Section 3's mqSE-IND game definition restricts "Phase 1" queries but does not bound the query count or require the adversary's mask choices to be non-adaptive.

- **In-threat-model?** Yes — this is a genuine gap the construction must address. **Section 3's mqSE-IND game must either (a) add a query-count bound `q << 64` that prevents full bitmask extraction and argue why the RS is rate-limited to fewer queries than the bitmask width, or (b) acknowledge that a malicious RS with unlimited query access recovers the full bitmask and reframe the privacy claim as "honest-RS privacy" not "malicious-RS privacy."** The agentNullifier rate-limit table (Section 2, verification step 5) is the natural mechanism to enforce this — but it must be made a *protocol requirement* with a bound of `< log2(permission_space)` queries per session, not an optional optimization.
